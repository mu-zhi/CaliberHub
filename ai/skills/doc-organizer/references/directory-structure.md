# 项目文档目录结构设计指南

## 一、设计原则

| 原则 | 说明 | 实施要点 |
|------|------|---------|
| **扁平化** | 目录深度 ≤ 3 层 | 避免不必要嵌套，优先分类而非深层级 |
| **按业务领域** | 同模块的需求/后端/前端聚合 | 不按技术栈拆散同一模块 |
| **唯一性** | 每个内容只在一处存在 | 使用交叉引用而非重复内容 |
| **渐进式** | 初期不必规划所有类型 | 优先保障：产品→架构→模块→测试 |

## 二、推荐目录结构

### 2.1 标准项目结构

```
[项目名称]/
├── README.md                              # 文档导航索引
├── assets/                                # 统一资源目录（图片、附件）
│
├── 01-产品设计/
│   ├── 01-产品概述.md                     # 项目背景、目标、范围
│   └── 02-UI-UX设计规范.md                # 统一的界面设计规范（SSOT）
│
├── 02-系统架构/
│   ├── 01-总体架构设计.md                 # 系统整体架构
│   ├── 02-环境与兼容性要求.md
│   ├── 03-数据库设计.md                   # ER 图/表结构（SSOT）
│   ├── 04-API规范.md                      # 全局 API 规范（SSOT）
│   ├── 05-测试架构.md                     # 测试策略与工具
│   └── 06-部署架构.md                     # 部署与运维
│
├── 03-模块设计/
│   ├── 模块总览.md                        # 模块列表、依赖关系图
│   ├── 核心模块/                          # ⭐ 系统核心价值功能
│   │   ├── [模块A]/
│   │   │   ├── 需求规格.md                # 功能需求、交互规则（SSOT）
│   │   │   ├── API接口.md                 # 模块接口契约（SSOT）
│   │   │   ├── 后端实现.md                # 业务逻辑、缓存、事务
│   │   │   ├── 前端实现.md                # 组件设计、状态管理
│   │   │   └── 测试用例.md                # 测试点、步骤、断言
│   │   └── [模块B]/
│   │       └── ...
│   └── 基础模块/                          # 🔧 系统基础支撑功能
│       ├── [模块C]/
│       │   └── ...
│       └── [模块D]/
│           └── ...
```

### 2.2 模块内部文件标准

每个模块目录包含以下标准文件：

| 文件 | 必需 | SSOT 内容 | 说明 |
|------|------|----------|------|
| `需求规格.md` | ✅ | 功能需求、交互规则、验收标准 | 模块功能的唯一权威 |
| `API接口.md` | ✅ | 接口路径、参数、响应 | 接口契约的唯一权威 |
| `后端实现.md` | 可选 | 业务逻辑、缓存、事务 | 引用需求和 API |
| `前端实现.md` | 可选 | 组件设计、状态管理 | 引用需求和 API |
| `客户端实现.md` | 可选 | Flutter/Android/iOS 实现 | 引用需求和 API |
| `测试用例.md` | ✅ | 测试点、步骤、断言 | 引用需求和 API |

## 三、SSOT 位置速查

| 内容类型 | SSOT 位置 | 说明 |
|---------|----------|------|
| **UI-UX 通用规范** | `01-产品设计/02-UI-UX设计规范.md` | 断点、色彩、通用交互 |
| **数据库设计** | `02-系统架构/03-数据库设计.md` | ER 图、表结构、索引 |
| **全局 API 规范** | `02-系统架构/04-API规范.md` | 通用错误码、认证方式 |
| **模块功能需求** | `03-模块设计/.../[模块]/需求规格.md` | 功能点、交互规则 |
| **模块接口契约** | `03-模块设计/.../[模块]/API接口.md` | 路径、参数、响应 |

## 四、目录深度参考

### 4.1 层级深度标准

| 层级 | 示例路径 | 说明 |
|------|---------|------|
| 第 1 层 | `01-产品设计/` | 大类目录 |
| 第 2 层 | `03-模块设计/核心模块/` | 分组目录 |
| 第 3 层 | `03-模块设计/核心模块/用户管理/` | 模块目录 |
| 第 4 层 | `03-模块设计/核心模块/用户管理/需求规格.md` | 文档文件 |

> ⚠️ 建议目录深度 ≤ 3 层，最多不超过 4 层

### 4.2 文件命名规则

| 规则 | 示例 | 说明 |
|------|------|------|
| 数字前缀 + 中文 | `01-产品概述.md` | 一级目录文件 |
| 纯中文 | `需求规格.md` | 模块内部文件 |
| kebab-case | `api-interface.md` | 英文项目 |

## 五、迁移检查清单

新建或重构目录结构时，检查以下项目：

- [ ] 目录深度 ≤ 3 层
- [ ] 同模块文档聚合在一个目录
- [ ] SSOT 内容只在一处存在
- [ ] 有 README.md 或导航索引
- [ ] 资源文件统一存放在 assets/
- [ ] 文件命名统一风格
