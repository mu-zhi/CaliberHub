# 特性文档校验与补写（feature-doc-authoring）

## Modes

本 skill 固定支持两种模式：

1. `gate mode`
   - 只做门禁检查
   - 适用于“检查特性文档能否进入实施计划”
2. `authoring mode`
   - 允许直接补写现有特性文档或新建目标特性文档
   - 适用于用户明确要求“补齐特性文档”，或自动化链路从 `feature-doc-coverage-mapping` 传入了需补写目标

## Workflow

1. 读取 `docs/engineering/standards/scenario-feature-doc-standard.md`。
2. 识别当前模式：
   - 如果用户要求“检查 / 门禁 / review”，进入 `gate mode`
   - 如果用户要求“补齐 / 完善 / 新建”，或输入来自自动补文档链路，进入 `authoring mode`
3. 如果输入来自 `feature-doc-coverage-mapping`，先提取其中标记为 `部分覆盖`、`缺失` 或“建议补写 / 新建”的目标清单。
4. 读取对应特性文档；批量模式下按“风险最高、最接近进入 `writing-plans` 的文档优先”排序，不做无差别全量扫库。
5. 在 `authoring mode` 下：
   - 对已有文档：补齐缺失段落、指标、边界与不负责范围
   - 对缺失文档：在建议路径下新建文档，并补齐最低完备性所需结构
   - 如主文档已有正式约束，以主文档为准落文，不自造并行口径
6. 对每份文档运行最低完备性检查，完整清单以 `scenario-feature-doc-standard.md` 第五节为准，至少覆盖：
   - 创建时必填头注（迭代归属、来源主文档）
   - 对应实施计划是否已同步产出；若无，是否有经批准的豁免与回补时机
   - 场景背景
   - 目标
   - 指标（含门禁型指标、效果型指标、分子、分母、统计窗口）
   - 功能说明
   - 涉及治理对象
   - 接口或数据边界
   - 前端承载页面或工作台
   - 本场景不负责什么
   - 引用材料可追溯性
7. 对每份文档明确输出：
   - 变更文件
   - 缺失项
   - 通过 / 不通过
   - 是否允许进入 `writing-plans`
   - 若不通过，建议先补哪一节
8. 批量模式下追加汇总：
   - 通过文档列表
   - 不通过文档列表
   - 共性缺失模式
   - 是否存在需要退回主方案或新增专题文档的情况

## Model routing

自动补文档链路固定采用以下模型约定：

1. 产出主模型：`gpt-5.4 + xhigh`
2. 产出备选：`claude-sonnet-4-6-20260218`
3. 校验主模型：`claude-sonnet-4-6-20260218`
4. 校验备选：`gpt-5.4 + xhigh`

若使用本地 `Claude Code` 作为备选产出链路，统一走：

```bash
bash scripts/claude_mux.sh wenwen claude-sonnet-4-6-20260218 --print "<prompt>"
```

## Guardrails

1. 一句话占位文档不通过。
1a. 新增或重写特性文档时，如缺少同步实施计划且无批准豁免，不通过。
2. 特性文档与主文档冲突时，以主文档为准，并要求当次同步修正。
3. 本 skill 负责检查，不负责自动补正文；除非用户明确要求改文档，否则只输出门禁结论和缺失项。
4. 若输入其实是“主方案还没拆到特性文档”的问题，先切换到 `feature-doc-coverage-mapping`，不要把缺失映射误当成单文档质量问题。
5. 自动补写只允许修改目标特性文档，不替代主文档回写动作。
