# 主方案能力到特性文档映射（feature-doc-coverage-mapping）

## Overview

这个 skill 用于把主方案里的能力拆成“原子能力项”，再映射到现有特性文档或建议新增的专题文档，并同步检查这些特性文档是否已经配对产出实施计划。它只做映射、归属和缺口判断，不直接补特性文档正文，也不改迭代顺序。

## Inputs

默认先读这些文件：

1. `docs/architecture/system-design.md`
2. `docs/architecture/frontend-workbench-design.md`
3. `docs/architecture/features/README.md`
4. `docs/engineering/current-delivery-status.md`
5. 必要时读取已存在的目标特性文档

## Workflow

1. 从主方案中提取原子能力项，优先拆这些类型：
   - 治理对象
   - 接口契约
   - 事件契约
   - 发布门禁
   - 运行降级
   - 审批与审计要求
   - 图谱投影与读源规则
   - `NFR（非功能需求，Non-Functional Requirements）`
   - 测试、迁移、回退与运维门禁
2. 每个能力项只指定一个主归属特性文档；其他文档只记为依赖或消费方，不并行维护。
3. 先尝试映射到现有 `docs/architecture/features/` 文档；只有满足“明显横切、现有文档接不住”的条件，才建议新增专题文档。
4. 对每个能力项同时检查“特性文档状态”和“实施计划状态”：
   - `已覆盖`：特性文档已经把对象、边界、门禁、接口或前端承载明确展开，且存在对应实施计划
   - `部分覆盖`：已有特性文档，但只提到能力名或局部规则，没有形成可排期的场景化定义
   - `缺失`：主方案有正式约束，但特性文档没有接住
   - `计划缺口`：特性文档已存在且场景定义基本成立，但缺少同步实施计划；该情况必须视为显式缺口，不得按 `已覆盖` 处理
5. 输出映射清单时，固定包含：
   - `能力项`
   - `主方案来源`
   - `能力说明`
   - `建议归属特性文档`
   - `对应实施计划`
   - `当前状态`
   - `判断依据`
   - `备注`
6. 若当前链路目标是“自动补充特性文档”，则在输出末尾追加：
   - 建议交给 `feature-doc-authoring` 的目标清单
   - 每个目标是“更新现有文档”还是“新建文档”
   - 建议优先级

## Suggested Grouping

默认按下面 8 组组织清单，避免章节式机械复制：

1. 知识生产输入与候选隔离层
2. 正式治理对象与建模约束
3. 推理资产与推理门禁
4. 来源契约、元数据与物理落位
5. 复核、缺口与发布前门禁协同
6. 运行面、发布面与图谱投影能力
7. 前端工作台壳层与跨台上下文
8. 样板场景、运维门禁与上线保障

## When To Suggest A New Feature Doc

只有满足任一条件时才建议新增专题文档：

1. 同一能力同时约束 3 个及以上特性文档
2. 该能力有独立对象、独立门禁或独立验收口径
3. 变更会同时影响接口契约、测试门禁或运维流程
4. 主方案已把它定义为统一基线，而不是单场景特例
5. 强行塞入现有文档会让单文档同时承担两个不相干职责

## Guardrails

1. 默认不直接补特性文档正文；补写动作交由 `feature-doc-authoring` 执行。
2. 不调整现有迭代顺序，除非用户明确要求重排。
3. 不把“当前排期没有进入状态表”误判成“文档缺失”；排期缺口、文档缺口和“实施计划缺口”要分开写。
4. 如果能力已在主文档和特性文档发生冲突，以主文档为准，并把该项至少标为 `部分覆盖`。
5. 输出必须可直接供 `feature-doc-authoring` 或后续写计划使用，不能只给章节摘要。
6. 只要发现“已有特性文档但缺实施计划”，必须在结果中直接写成 `计划缺口`，并给出需同步补齐的计划路径建议；不得用“后续补计划”轻描淡写带过。
