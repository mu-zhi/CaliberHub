## 项目启动方式（本地联调）

### 1. 启动后端（Spring Boot）

```bash
cd /Users/rlc/Code/数据直通车/backend
mvn -q spring-boot:run
```

默认地址：`http://127.0.0.1:8080`

可用性检查（建议）：

```bash
curl -sS http://127.0.0.1:8080/api/system/llm-preprocess-config
```

说明：
1. 当前阶段默认使用 H2 内存库，适合本地开发与联调。
2. `/actuator/health` 可能因可选依赖（如 Redis）显示 `DOWN`，请以业务接口可访问性为准。
3. 大模型调用优先使用系统管理中“用户实际配置”的 API Key。

### 2. 启动前端（Vite）

```bash
cd /Users/rlc/Code/数据直通车/frontend
npm install
npm run dev
```

默认地址：`http://127.0.0.1:5173`

### 3. 运行规范（强制）

#### 代码修改后的服务启动原则

> **默认触发原则：每次代码修改完成后，必须自动启动前后端服务**
>
> 这是项目的核心原则之一。任何代码变更（前端或后端）完成后，必须启动/重启前后端服务，确保系统处于可访问状态。

1. **修改后必重启**：任何代码变更（前端或后端）完成后，必须启动/重启前后端服务，确保系统处于可访问状态。
2. **自动可用性校验**：服务启动后，自动执行可用性校验：
```bash
curl -sS http://127.0.0.1:8080/api/system/llm-preprocess-config
curl -sS -I http://127.0.0.1:5173
```
3. **前后端联调要求**：本地联调默认要求前后端同时在线（`8080` + `5173`），除非当前任务明确声明仅后端或仅前端。
4. **服务中断处理**：若服务中断（端口丢失、进程退出），应立即重启并在任务反馈中说明恢复时间与状态。

#### 手动启动命令

```bash
# 启动后端（Spring Boot）
cd /Users/rlc/Code/数据直通车/backend
mvn -q spring-boot:run

# 启动前端（Vite）
cd /Users/rlc/Code/数据直通车/frontend
npm run dev
```

### 4. 常见问题

1. 端口被占用（`8080` 或 `5173`）：
```bash
lsof -nP -iTCP:8080 -sTCP:LISTEN
lsof -nP -iTCP:5173 -sTCP:LISTEN
```
2. 前端已启动但页面空白：确认后端接口可访问，再刷新前端页面。
3. 导入接口耗时较长：大文档会分块处理；若 LLM 不可用会重试后回退规则抽取，耗时会增加。

### 5. 系统化测试流程（推荐）

```bash
bash /Users/rlc/Code/数据直通车/scripts/run_system_test_flow.sh
```

该脚本会顺序执行：
1. 前端 `lint` / `test` / `build`
2. 前端静态产物同步到后端
3. 后端 `mvn -q test`
4. 前后端轻量可达性检查（如服务已在线）

## 项目文档同步更新规范（12条，生效版）

1. 文档分层固定为三类：`01-设计文档`、`02-开发文档`、`03-其他文档`。  
2. 文件夹与文件名均使用两位编号前缀，按阅读与维护顺序排列（如 `01-...`、`02-...`）。  
3. 一个主题只保留一份主文档，禁止并行维护多个主版本。  
4. 禁止使用“最终版/修改版/xxx版”等过程性后缀。  
5. 每份主文档（`README`、JSON、SQL 与纯索引文件除外）开头必须写清：目标、范围、读者、当前阶段定位。  
6. 英文术语首次出现必须带中文解释，例如：`NL2SQL（自然语言转SQL）`。  
7. 术语以 `/Users/rlc/Code/数据直通车/01-设计文档/08-数据直通车-统一术语表.md` 为唯一标准。  
8. 面向业务统一使用"数据地图"；技术常量（如 `CALIBER_IMPORT_V2`）可保留，但必须补充中文注释。  
9. 战略口径固定：`数据直通车2.0` 是远期蓝图，`口径治理` 是基础，`知识梳理服务` 是口径治理的大模型场景。  
10. 改名、迁移、新增、下线文档时，必须同步修复链接并更新 `/Users/rlc/Code/数据直通车/README.md` 导航。  
11. 每次“方案更新”必须先分类再落文：设计类写入 `01-设计文档`，开发类写入 `02-开发文档`，外部参考与调研写入 `03-其他文档`，并对全量文档执行一致性扫描。  
12. 文档中不保留“旧文档/旧目录迁移过程”等历史性描述，只保留当前有效信息。

## 接口文档维护规范（强制）

1. 后端新增接口时，必须同步更新 `/Users/rlc/Code/数据直通车/02-开发文档/04-数据直通车-接口文档.md`。  
2. 后端接口变更时（路径、方法、参数、返回、错误码），必须同步更新接口文档。  
3. 后端接口下线时，必须在接口文档标注下线日期与替代方案。  
4. 控制器代码变更与接口文档变更不一致时，不允许提测。  
5. 接口文档是前后端联调与测试验收的唯一协作基线。  

### 固定触发语

#### 1. 更新项目文档

当用户输入”更新项目文档”时，执行以下流程：

1. 收集本次最新方案变更点（新增、调整、下线）。
2. 按第 11 条规则将变更分门别类写入目标文档。
3. 若涉及接口新增/变更/下线，必须同步更新接口文档并补充变更记录。
4. 扫描全部项目文档，统一术语、口径与链接。
5. 同步更新 `README.md` 的快速阅读路径与目录结构（如有变化）。
6. 输出一次”文档同步结果”，包含：已更新文件、核心变更、已检查未修改文件。

#### 2. 修改代码后

当用户完成代码修改或功能调整时，默认触发以下流程：

1. **自动启动后端**：执行 `mvn -q spring-boot:run`（如已运行则重启）。
2. **自动启动前端**：执行 `npm run dev`（如已运行则重启）。
3. **自动可用性校验**：检查 `8080` 和 `5173` 端口服务状态。
4. **输出服务状态**：报告前后端地址与可用性验证结果。

> 注意：若当前任务明确声明仅后端或仅前端工作，可只启动对应服务。
