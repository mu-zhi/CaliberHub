# 业界参考-文档转知识图谱开源项目调研报告

> **目标**：梳理 GitHub 上可直接用于“文档转知识图谱”的开源项目，识别主流实现路径、开箱即用能力和适用边界，为数据直通车后续“知识梳理服务（Knowledge Curation Service）”与口径治理（Caliber Governance）能力预研提供参考。
> **范围**：聚焦“将 PDF、Office、Markdown、HTML 等文档解析后，抽取实体与关系，并落入知识图谱或图检索增强生成链路”的开源项目，不覆盖纯论文、纯算法仓库和纯向量检索项目。
> **读者**：产品经理、架构师、后端开发、前端开发、算法工程师、数据治理相关角色。
> **当前阶段定位**：外部方案调研文档，结论用于方案预研和技术选型，不作为当前开发基线。
> **调研时间**：2026-03-08。

---

## 1. 结论摘要

本轮调研的核心结论如下：

1. 当前 GitHub 上“文档直接转知识图谱”的主流路线，已经从传统规则抽取转向了“文档解析 + 大语言模型（LLM，Large Language Model）抽取 + 图数据库/向量数据库混合检索”的 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 方案。
2. 真正强调“把文档严谨地转成结构化知识图谱”的项目并不多，更多项目本质上是“带知识图谱能力的检索增强生成平台”，重点在问答与检索，而不是知识工程本身。
3. 若关注“最快看到效果”，优先看 `LightRAG`；若关注“文档转图谱本身”，优先看 `docling-graph`；若关注“方法论与行业影响力”，优先看 `microsoft/graphrag`；若关注“专业领域逻辑推理”，优先看 `OpenSPG/KAG`。
4. 业界成熟实现几乎都不是“只建图不建索引”，而是同时维护图结构、文本块、向量索引和溯源信息，查询时做混合检索与图推理。
5. 对数据直通车而言，最值得借鉴的不是单个仓库，而是三层组合思路：
   文档解析层、知识抽取层、图谱/检索服务层分离，避免把“文档清洗”“关系抽取”“用户问答”耦合在一个黑盒里。

---

## 2. 调研口径与筛选标准

本次筛选主要看 6 个维度：

1. 是否支持文档输入，而不是只接收纯文本。
2. 是否能生成显式的实体、关系或图结构，而不是只做向量化。
3. 是否提供较完整的开箱即用能力，例如 `CLI（命令行工具，Command Line Interface）`、`API（应用程序接口，Application Programming Interface）`、`Web UI（网页界面，Web User Interface）`、`Docker`。
4. 是否提供图检索增强生成（GraphRAG）或图数据库落库能力。
5. 是否有较明确的工程化文档，便于本地验证。
6. 是否在 GitHub 上有一定活跃度或社区关注度。

说明：

- 星标数量来自 GitHub 仓库页面或公开 `API（应用程序接口，Application Programming Interface）`，统计时间为 2026-03-08。
- “开箱即用程度”分为高、中、低，是基于仓库文档、启动方式、界面能力和默认组件完整度的综合判断，不等于最终落地难度。

## 3. 项目全景分型

从工程形态看，当前项目大致分为三类：

### 3.1 产品型平台

特点是具备 `Web UI（网页界面，Web User Interface）`、上传文档、索引构建、图谱浏览、问答和运维操作，适合快速演示。

- `HKUDS/LightRAG`
- `stevereiner/flexible-graphrag`

### 3.2 方法型框架

特点是强调 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 或领域推理能力，往往需要自行装配数据源、模型与数据库。

- `microsoft/graphrag`
- `OpenSPG/KAG`
- `neo4j/neo4j-graphrag-python`

### 3.3 抽取型工具链

特点是强调“文档转图谱”的抽取过程本身，适合做前置知识抽取和知识清洗。

- `docling-project/docling-graph`
- `stair-lab/kg-gen`
- `rahulnyk/knowledge_graph_maker`

### 4.2 项目逐项分析

#### A. LightRAG

**定位判断**

`LightRAG` 是当前最接近“上传文档即可体验知识图谱问答”的开源项目之一。它的重点不是严谨本体建模，而是以较低门槛把文档接入、知识图谱构建、图谱可视化和问答整合在同一套服务里。

**值得关注的能力**

1. 官方明确提供 `Web UI（网页界面，Web User Interface）` 与 `API（应用程序接口，Application Programming Interface）`，支持文档索引、知识图谱浏览与简单问答。
2. 支持 `local/global/hybrid/mix` 多种检索模式，体现出典型的“图 + 向量 + 全文”混合检索思路。
3. 支持实体删除、关系删除、实体合并、文档删除后自动重建图谱，说明其工程重点已经延伸到图谱运维，而不仅是抽取。
4. 2025 年后集成了多模态文档处理，支持 PDF、图片、Office、表格等格式。
5. 支持引用溯源，便于回到原文档定位证据。

**适合借鉴的点**

- 适合用来观察“图谱能力如何嵌入用户交互产品”。
- 适合参考其索引状态管理、删除重建、图谱浏览和混合检索设计。

**不适合直接照搬的点**

- 如果目标是“形成高质量、可长期维护的领域知识图谱”，LightRAG 的知识抽取机制仍然偏向问答服务配套，而不是严谨知识建模。

#### B. Microsoft GraphRAG

**定位判断**

`microsoft/graphrag` 是当前行业影响力最高的 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 方法仓库之一。它更像“方法学与参考实现”，而不是“一键式产品”。

**值得关注的能力**

1. 官方明确说明，该项目是一个从非结构化文本中抽取结构化数据的数据处理与转换套件。
2. 其核心思路是通过图结构增强 `LLM（大语言模型，Large Language Model）` 对私有数据的推理能力。
3. 官方明确提示索引成本较高，而且建议做 `Prompt Tuning（提示词调优）`，说明它更适合认真做实验和指标调优，而不是直接拿来即用。

**适合借鉴的点**

- 适合借鉴“社区摘要 + 实体关系图 + 全局/局部问答”的总体思路。
- 适合在数据量较大、文档主题复杂时作为中长期技术路线参考。

**不适合直接照搬的点**

- 如果当前目标是尽快搭一个演示系统，它的工程复杂度偏高。

#### C. OpenSPG/KAG

**定位判断**

`KAG` 不是传统意义上的“文档上传问答平台”，而是面向专业领域知识库的逻辑推理与检索框架。它想解决的是传统向量检索增强生成（RAG，Retrieval-Augmented Generation）在专业领域推理不足的问题。

**值得关注的能力**

1. 官方定位非常明确：面向专业领域知识库，强调逻辑形式引导的推理与检索。
2. 仓库强调 4 个关键能力：知识与文本块双向索引、概念语义对齐、模式约束知识构建、逻辑形式引导的混合推理与多跳问答。
3. 支持把非结构化数据、结构化数据和专家经验统一进业务知识图谱，更贴近企业知识工程场景。

**适合借鉴的点**

- 对“口径治理 + 规则推理 + 多跳问答”这类场景，KAG 的设计理念比一般 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 更贴切。
- 对数据直通车这种有专业术语、业务规则和统计限定条件的领域，KAG 的“逻辑形式”思路值得重点研究。

**不适合直接照搬的点**

- 体系偏重，适合中后期做专业知识库升级，不一定适合第一阶段快速验证。

#### D. Neo4j GraphRAG Python

**定位判断**

这是 `Neo4j` 官方维护的 Python 开发包，适合“已经决定用图数据库（Graph Database）承载知识图谱”的团队。

**值得关注的能力**

1. 提供知识图谱构建管线，官方说明 `SimpleKGPipeline` 可以更容易地构建知识图谱。
2. 支持直接从文本与 PDF 构建知识图谱。
3. 提供向量索引、混合检索、图遍历检索、`Text2Cypher` 等典型能力。

**适合借鉴的点**

- 若数据直通车未来确定采用 `Neo4j`，这是最直接的工程底座参考。
- 它把“图构建”和“图检索”放在一个稳定生态里，方便后续扩展。

**不适合直接照搬的点**

- 需要自建数据库与周边组件，不是上传文档就能直接用的完整平台。

#### E. Docling Graph

**定位判断**

这是当前最接近“文档转知识图谱工具链”本身的项目。它不是优先做聊天，而是优先保证抽取过程的结构化、可验证和可导出。

**值得关注的能力**

1. 官方明确说明：先把文档转换为 `Pydantic` 校验对象，再生成具有显式语义关系的有向知识图谱。
2. 支持 PDF、图片、Markdown、Office、HTML 等多种输入格式。
3. 支持 `LLM（大语言模型，Large Language Model）` 与 `VLM（视觉语言模型，Vision Language Model）` 两条抽取路径。
4. 支持导出 `CSV（逗号分隔值文件，Comma-Separated Values）`、`Cypher` 等图谱友好格式，并支持交互式 HTML 可视化。

**适合借鉴的点**

- 适合做“文档入库前的结构化抽取前置层”。
- 非常适合需要模板、模式和字段约束的严肃场景。

**不适合直接照搬的点**

- 需要较强的模式设计意识，适合技术团队主导，不是业务人员自助式工具。

#### F. Flexible GraphRAG

**定位判断**

这是一个偏“全家桶”的工程化平台，目标是把多数据源接入、文档处理、图谱构建、检索、前端界面和 `MCP（模型上下文协议，Model Context Protocol）` 工具链整合起来。

**值得关注的能力**

1. 支持 13 种数据源、10 种向量数据库、8 种图数据库。
2. 提供 `Docling` 或 `LlamaParse` 文档处理、自动知识图谱构建、混合检索和多套前端界面。
3. 提供 `FastAPI` 后端、`REST API（表述性状态转移接口，Representational State Transfer Application Programming Interface）`、`MCP（模型上下文协议，Model Context Protocol）` 服务和监控能力。

**适合借鉴的点**

- 适合看完整产品工程怎么拆层。
- 适合参考其“多源接入 + 后台处理 + 搜索问答 + 前端”一体化结构。

**不适合直接照搬的点**

- 仓库较新，更适合作为工程参考，而不是生产基线。

#### G. KG-Gen

**定位判断**

`kg-gen` 更像一个“从任意文本抽图谱”的研究型实用工具，适合做快速实验和抽取效果验证。

**值得关注的能力**

1. 明确支持从任意纯文本抽取知识图谱。
2. 支持 OpenAI、Ollama、Anthropic、Gemini、DeepSeek 等多种模型提供方。
3. 适合做图谱辅助检索、图合成数据生成和概念关系分析。

**适合借鉴的点**

- 适合做轻量 `POC（概念验证，Proof of Concept）`。
- 适合验证提示词、模型和抽取契约对图谱质量的影响。

**不适合直接照搬的点**

- 它处理“文本”很强，但不是完整的复杂文档处理平台。

#### H. Knowledge Graph Maker

**定位判断**

这是一个本体驱动的轻量知识图谱生成工具，特点是结构简单、容易读懂，适合做最小样板。

**值得关注的能力**

1. 基于给定本体从文本生成知识图谱。
2. 直接接受文档列表，按文档生成子图，再汇总成全图。
3. 可选落库到 `Neo4j`。

**适合借鉴的点**

- 适合作为“最小抽取管线”的参考实现。
- 对理解“本体约束如何影响抽取结果”很有帮助。

**不适合直接照搬的点**

- 产品化和运维能力较弱，适合实验，不适合直接承担平台职责。

## 5. 业界主流实现路线

综合上述仓库，当前行业做法基本收敛到以下五步：

### 5.1 文档解析

先把 PDF、图片、Office、HTML 等文档解析成相对稳定的结构片段，包括：

- 标题层级
- 正文块
- 表格
- 图片说明
- 页码与位置信息

这一步常见能力来自 `Docling`、`LlamaParse` 或项目自带解析层。

结论是：如果文档解析质量不稳定，后续图谱质量通常也不会稳定。

### 5.2 知识抽取

再用 `LLM（大语言模型，Large Language Model）` 或 `VLM（视觉语言模型，Vision Language Model）` 抽实体、关系、事件、属性和上下文约束。

当前两种路线最常见：

1. **开放抽取路线**：先让模型抽，再做归一化与合并。
   优点是速度快、门槛低；缺点是噪声多。
2. **模式约束路线**：先定义模板、本体、实体类型、关系类型，再让模型按结构输出。
   优点是结果更稳；缺点是前期设计成本更高。

### 5.3 双存储或多存储

成熟方案一般不会只存一份图谱，而是至少维护以下几类存储：

- 文本块存储
- 图结构存储
- 向量索引
- 索引状态或任务状态

这样做的原因很直接：

图适合多跳关系推理，向量适合召回语义近邻，原文本块适合证据回溯。

### 5.4 混合检索

查询阶段通常不是纯向量检索，而是混合以下能力：

- 向量检索
- 关键词检索
- 图遍历
- 规则过滤
- 多跳推理

`LightRAG` 的 `local/global/hybrid/mix`，`KAG` 的逻辑形式引导混合推理，`Neo4j GraphRAG` 的图遍历增强检索，本质上都是这一思路的不同实现。

### 5.5 图谱运维

真正工程化的项目，都会补这几类运维能力：

- 实体合并
- 关系修正
- 增量索引
- 文档删除后的图谱重建
- 溯源定位
- 图谱可视化

这部分工作的重要性不低于抽取本身。

很多系统之所以“看起来能跑、实际上难落地”，原因就在于缺少图谱运维能力。

## 6. 业界共识与分歧

### 6.1 已形成共识的部分

1. 单纯依赖向量检索增强生成（RAG，Retrieval-Augmented Generation）很难覆盖复杂专业领域问答。
2. 文档转知识图谱必须保留来源片段与页码上下文，否则难以审计和纠错。
3. 复杂领域需要模式约束、术语归一、实体合并和关系治理，不能完全依赖一次性抽取。
4. 真正可用的系统一定包含“抽取、存储、检索、运维、展示”闭环。

### 6.2 尚未统一的部分

1. 图谱是否必须有严格本体。
   有的项目偏轻量抽取，有的项目强调强模式建模。
2. 是否必须上图数据库（Graph Database）。
   有的项目先用内存图或 `NetworkX`，有的直接用 `Neo4j` 等图数据库。
3. 问答优先还是知识工程优先。
   产品型项目更关心“用户能问”，抽取型项目更关心“图谱是否干净可维护”。

## 7. 对数据直通车的启示

结合数据直通车当前“口径治理（Caliber Governance） + 知识梳理服务（Knowledge Curation Service）”的方向，建议按三层能力理解这类项目，而不是直接替换现有系统：

### 7.1 可优先借鉴的能力

1. **文档解析前置层**
   参考 `docling-graph` 或 `Docling` 生态，先把口径文档、制度文档、接口文档稳定切块并保留版式与页码。
2. **知识抽取与归一层**
   参考 `KAG`、`Knowledge Graph Maker` 的模式约束思路，把“术语、指标、口径限定、时间条件、数据来源、依赖关系”抽成可治理的对象。
3. **图检索与问答层**
   参考 `LightRAG` 或 `Neo4j GraphRAG` 的混合检索链路，把图谱用于问答增强，而不是直接让问答结果替代图谱本身。

### 7.2 更适合当前阶段的组合建议

如果目标是尽快预研并看到效果，建议优先考虑两条路线：

**路线 A：快速演示路线**

- 文档解析：`LightRAG`
- 图谱浏览与问答：`LightRAG`
- 目标：快速验证“文档能否转成图谱并参与问答”

这条路线优点是快，缺点是知识模型控制力一般。

**路线 B：稳妥抽取路线**

- 文档解析与结构化抽取：`docling-graph`
- 图数据库承载：`Neo4j GraphRAG` 或自建 `Neo4j`
- 逻辑推理增强：后续再研究 `KAG`
- 目标：先把“文档转知识”做好，再考虑面向用户的图谱问答

这条路线优点是更适合口径治理，缺点是初期搭建成本高于路线 A。

### 7.3 对当前项目最重要的现实判断

对数据直通车而言，难点不在于“有没有仓库能把文档变成图”，而在于以下三点：

1. 口径文档中的实体、指标、限定条件、统计范围是否有稳定本体。
2. 同一个业务术语在不同文档中的别名、缩写、历史口径是否能归并。
3. 抽出的知识是否能回溯到原文、被人工校验、支持持续修正。

因此，本项目不建议直接把“图检索增强生成（GraphRAG，Graph Retrieval-Augmented Generation）平台”当作最终答案，而应把它看成后续交互层的一部分。

真正的基础仍然是：术语治理、模式定义、证据溯源和知识维护流程。

## 10. 最终判断

如果只用一句话概括这次调研结论：

> 业界并没有一个“万能的一键文档转图谱标准答案”，主流做法是把文档解析、知识抽取、图谱存储、混合检索和用户问答拆成多层能力，再用 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 方式把它们串起来。

对数据直通车来说，最优先的不是挑一个最火的仓库，而是先明确：

- 要抽哪些实体与关系；
- 哪些口径必须可追溯；
- 哪些问答需要多跳推理；
- 哪些知识允许人工校正后持续沉淀。

只有这几个问题先明确，开源项目选型才会稳定。
