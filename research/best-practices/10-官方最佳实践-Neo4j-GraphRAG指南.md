# 官方最佳实践-Neo4j-GraphRAG指南

本文整理 `Neo4j（图数据库产品）` 官方 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 指南里，与数据直通车知识图谱、数据地图和运行时检索最相关的部分。

## 1. 官方来源

1. 来源：[Neo4j（图数据库产品）`GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 官方指南](https://neo4j.com/labs/genai-ecosystem/graphrag/)
2. 来源方：`Neo4j（图数据库产品）` 官方指南
3. 访问日期：2026-03-12

## 2. 这份指南最核心的观点

这份指南反复强调同一件事：

`GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 不是只靠向量检索找到文本块，而是先把文档里的实体、关系和主题结构提出来，再让图检索和向量检索协同工作。

官方页面里比较关键的几条原意可以压成下面 4 点：

1. 图检索和向量检索要组合工作。
2. 实体和关系要从非结构化文本里抽出来。
3. 相互关联的信息要在图里显式连起来，而不是靠外部代码维护一堆额外引用。
4. 图结构能提供更强的可解释性、透明度和审计性。

## 3. 官方指南里最值得我们关注的部分

### 3.1 图和文本不是替代关系

这份指南明确说，基础 `RAG（检索增强生成，Retrieval-Augmented Generation）` 更偏向量检索文本块，而 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 是在检索阶段把图结构一起用起来。

对数据直通车来说，这意味着：

1. 口径文档原文不能丢。
2. 业务术语、规则、表间关系也不能只写在文档里，必须被组织进图。

### 3.2 关系要显式保存，而不是靠代码临时拼

官方页面强调，如果不是图系统，就要存大量额外标识并写很多额外代码来追这些关系；而在图系统里，只要把关系显式建出来，就可以在检索中直接沿关系扩展。

这和我们当前把表间关联关系、派生规则、证据、版本都作为主图对象的方向一致。

### 3.3 图比纯向量更适合解释和审计

官方页面直接指出：在关键应用里，质量之外还需要解释性和透明度。图查询能把“这次为什么查到这些结果”这件事留下一条更细的执行轨迹。

这对数据直通车尤其重要，因为我们的目标从来不只是“查到数据”，还包括：

1. 解释路径
2. 回到证据
3. 说明版本
4. 支撑人工复核

### 3.4 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 不只是一种检索

官方页面把检索模式进一步拆成：

1. 文本驱动的向量 + 图组合检索
2. 实体驱动的局部检索
3. 聚类驱动的全局检索
4. 基于图查询语言的检索

这说明 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 真正值钱的不是“有图”，而是“图能按不同问题参与不同检索”。

## 4. 对数据直通车的直接启发

### 4.1 数据地图和 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 可以共用一套主图

这份指南支持我们现在的一个关键判断：主图不应该只服务下游生成，也要服务浏览、解释和人工复核。

### 4.2 运行时不能只返回表字段

如果按这份官方指南来理解，运行时最有价值的返回物，不是单纯的表清单，而是能继续沿关系扩展的一块问题相关业务子图。

### 4.3 文档入图要和关系治理一起做

这份指南强调实体抽取、关系抽取、主题聚类和原文关联。对数据直通车来说，等价动作就是：

1. 从口径文档中抽术语、规则、路径和证据。
2. 把这些对象连回来源表、字段、`SQL（结构化查询语言，Structured Query Language）` 片段和原文证据。

## 5. 我们不直接照搬的地方

这份官方指南更偏通用知识系统和图问答。落到数据直通车时，还要额外收紧两层：

1. 场景边界必须先收住，不能只按相似关系自由扩展。
2. 版本、时间和发布确认必须进入治理，不然“解释链条”会很快失真。

## 6. 当前最值得吸收的结论

这份指南最值得带回来的结论是：

图谱真正的价值，不只是比向量多找几条信息，而是把对象之间的关系、来源和解释路径显式保存下来，让检索和回答都可追溯。
