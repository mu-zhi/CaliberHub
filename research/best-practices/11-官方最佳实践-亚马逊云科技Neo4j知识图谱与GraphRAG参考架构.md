# 官方最佳实践-亚马逊云科技Neo4j知识图谱与GraphRAG参考架构

本文整理亚马逊云科技与 `Neo4j（图数据库产品）` 的官方参考架构页面，重点看多源数据如何进入知识图谱，以及知识图谱如何进入 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）`。

## 1. 官方来源

1. 来源：[亚马逊云科技与 `Neo4j（图数据库产品）` 知识图谱和 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 参考架构](https://docs.aws.amazon.com/architecture-diagrams/latest/knowledge-graphs-and-graphrag-with-neo4j/knowledge-graphs-and-graphrag-with-neo4j.html)
2. 来源方：亚马逊云科技官方架构图文档
3. 页面发布时间：2024-11-26
4. 访问日期：2026-03-12

## 2. 这份参考架构在讲什么

这份架构文档的重点不是查询模式，而是“图从哪里来”。

官方页面一开始就把输入源定成了三类：

1. 结构化数据
2. 非结构化数据
3. 半结构化数据

然后再把这些数据送入知识图谱建设链路，最后让应用基于图做 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）`。

## 3. 对数据直通车最有价值的部分

### 3.1 官方默认多源建图

这份参考架构很清楚地说明，知识图谱不应该只从文档来，也不应该只从数据库元数据来，而应该同时接多类来源。

这和数据直通车当前的真实输入是一致的：

1. 口径文档
2. 历史 `SQL（结构化查询语言，Structured Query Language）`
3. 工单样例
4. 数据库元数据
5. 人工确认结果

### 3.2 图数据库不是唯一重点，建图链路才是重点

页面把“建图”放在整条架构链路中间，而不是只强调图数据库本身。这对我们是个提醒：

真正难的不是把点边存进去，而是：

1. 数据怎么接进来
2. 关系怎么抽出来
3. 实体怎么统一
4. 结果怎么供下游消费

### 3.3 图建完以后要服务应用

页面给出的目标并不是“完成建图”，而是让应用能利用这个知识图谱进入 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）`。

对数据直通车来说，对应的应用就是：

1. `NL2SQL（自然语言转 SQL，Natural Language to SQL）` 上游缩域
2. 数据地图浏览
3. 路径解释
4. 影响分析

## 4. 这份架构与我们方案的对应关系

| 官方架构中的环节 | 数据直通车中的对应物 |
| --- | --- |
| 多类源数据接入 | 口径文档、历史 `SQL（结构化查询语言，Structured Query Language）`、工单样例、数据库元数据、人工确认 |
| 知识图谱建设链路 | 知识加工层与知识组织层 |
| 图数据库 / 图应用层 | 主图、数据地图、问题相关业务子图服务 |
| `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 应用 | 下游 `NL2SQL（自然语言转 SQL，Natural Language to SQL）`、解释与复核 |

## 5. 我们不直接照搬的地方

这份官方架构更偏云上参考图，讲的是大体组合方式。落到数据直通车时，还要进一步明确：

1. 哪些对象是当前主图合法对象
2. 哪些关系要显式治理
3. 哪些证据和版本是发布门槛
4. 上游到底输出什么给下游

所以它更适合作为“多源建图的官方依据”，不适合作为我们最终的对象模型定义。

## 6. 当前最值得吸收的结论

这份参考架构最值得带回来的结论是：

知识图谱建设从第一天就应该按多源数据接入来设计，而不是先假设所有知识都只来自一种文档格式。
