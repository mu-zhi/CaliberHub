# 数据直通车 · 前端设计留痕与执行约束

> 目标：把前端设计决策沉淀为可追溯的项目基线，确保后续页面迭代不偏航。  
> 范围：覆盖导航结构、页面布局、文案语境、未实现能力标注、数据地图图谱展示、系统管理入口边界。  
> 读者：前端开发、产品经理、交互设计、测试、评审人员。  
> 当前阶段定位：M1 已交付后的持续优化阶段（兼容 M2/M3 前端演进），按统一信息架构持续落地。

---

## 1. 生效目标

1. 前端改造遵循统一的信息架构，不再按页面零散演进。
2. 每次前端变更都可追溯到明确的决策条目，避免口径漂移。
3. 界面语言保持中文语境，降低业务用户理解成本。
4. 已实现与未实现能力在界面与文档中明确区分，避免误导。
5. 本文“已确认的前端基线决策”优先级高于“变更记录”；历史记录仅用于追溯，不作为当前口径。

## 2. 已确认的前端基线决策

### 2.1 导航与命名

1. 顶部左侧品牌固定为“数据直通车”。
2. 顶部主导航固定为：`首页`、`口径治理`、`数据地图`、`系统管理`、`个人中心`。
3. 路由命名去掉“中心”后缀，路径语义化。
4. “数据地图”是“语义资产”的业务化表达，业务场景、语义视图、字典、派生规则、主题节点、服务说明均归属数据地图。
5. 系统管理为治理支撑域，入口位于`数据地图`与`个人中心`之间，不放在业务工作台主流程内。
6. 顶部导航采用无底框文字导航样式，激活态使用底部高亮线表达当前模块。
7. 系统管理模块固定包含：`系统介绍`、`大模型配置`、`预处理提示词`；采用按角色显隐，系统管理员可见全部，其他角色默认不显示系统管理入口。

### 2.2 页面布局与交互

1. 主流程页面采用单列步骤流，不采用并排双流程。
2. 导入口径文档后在系统内生成草稿，用户编辑草稿内容，不要求直接编辑 JSON。
3. 导入流程按顺序展示：导入预处理、抽取质量判断、抽取结果、场景编辑与发布。
4. 说明性文字弱化呈现，不干扰主任务操作。
5. 已实现页面不展示“当前页面已实现”提示；仅对未实现能力给出说明与替代路径。
6. 桌面端采用“顶部导航与左侧导航悬浮固定、右侧内容区独立滚动”的交互方式；首页场景默认隐藏左侧导航，聚焦搜索主任务。
7. 左侧导航支持“收缩/展开”状态切换，并记住用户上次选择；收缩态以精简标签展示菜单，移动端默认关闭该交互。
8. 口径治理页新增“流程导航条 + 发布前检查清单”，用任务状态引导用户从导入、判断到发布处理闭环完成。
9. 首页新增“快捷任务入口”，支持一跳进入导入、手动创建、数据地图、我的待办。
10. 首页采用 Search-first 布局：搜索栏置于首屏视觉中心，搜索后展示结果，默认展示最近更新列表与热门业务领域入口。
11. 系统管理中的“系统手册”需支持在线阅读正文（目录可切换、章节可连续阅读），不展示本地磁盘路径作为主信息。
12. 导入口径页的多场景切换采用 Master-Detail：左侧可滚动场景队列，右侧显示“原文对照 + 编辑表单”，不再使用下拉切换同级复杂场景。
13. 导入口径页必须展示导入执行明细：总耗时、阶段耗时拆解、当前阶段文案、实时进度条；不得仅展示“处理中…”黑盒状态。
14. 导入成功后默认预创建全部识别场景草稿，编辑行为只允许更新当前 `sceneId`，禁止通过“保存”再次创建新场景。
15. 导入口径页按钮语义固定为：`保存当前场景`、`发布当前场景`、`弃用当前场景`；`手动创建草稿`仅保留在手动创建路由。
16. 导入口径页移除“05 发布检索”区块；当全部场景处理完成后，以完成弹层引导用户跳转“数据地图 > 业务场景”继续消费。

### 2.3 中文语境与文案

1. 面向业务文案优先中文，技术缩写仅在必要处保留（如 LLM、SQL）。
2. 来源类型、字段名、状态描述不显示英文词汇。
3. 文档与页面不使用“P0/P1”分级表述，改用直接中文描述（如“首批上线”“后续建设”）。
4. 前端与文档中不出现特定机构名称。

### 2.4 能力边界解释

1. 服务说明：发布态资产，强调可对外消费的输入输出契约。
2. 取数指南：面向业务用户的操作说明，强调如何使用服务说明完成取数。
3. 业务场景：治理资产本体，承载结构化口径与取数方案。
4. 场景集市：面向消费侧的浏览与发现入口，强调检索、推荐、复用，不替代业务场景管理。

### 2.5 数据地图图谱要求

1. 采用“一页三块”：顶部检索、左侧主题树、中间图谱画布，右侧详情抽屉按需展开。
2. 图谱支持至少三类交互：浏览模式、路径模式、影响模式。
3. 节点与关系使用可区分的视觉编码，保证一眼识别资产类型与关系方向。
4. 业务分类节点初始化以 `/Users/rlc/LingChao_Ren/1.2、数据直通车/03-其他文档/01-当前业务侧的业务场景分类` 为准。
5. 布局采用新版信息架构，视觉风格保持当前已定调性（纸质感与专业稳重风格）。

## 3. 未实现能力标注规范

1. 菜单项保留但显示 `[后续建设]` 标签。
2. 未实现页面必须给出三段信息：功能目标、预计阶段、当前替代方式。
3. 已实现页面不得出现“已实现”冗余提示。
4. 选项中若包含未接入能力（如图片 OCR），在选项后明确“暂未接入”。

## 4. 前端改动留痕规则

1. 任何影响导航、命名、布局、文案、交互模式的变更，必须同步更新本文件。
2. 任何新增接口导致前端交互变化，必须同步更新 `/Users/rlc/Code/数据直通车/02-开发文档/04-数据直通车-接口文档.md`。
3. 前端评审意见落地后，必须在本文件追加“变更记录”。
4. 前端评审类请求优先使用 `web-design-guidelines`，再按项目口径执行 `frontend-design-governance`。
5. 外部技能接入或替换（新增、下线、升级）属于前端治理决策，必须在本文件留痕。

### 4.1 外部前端技能接入基线

1. 项目级技能目录：`/Users/rlc/Code/数据直通车/.agents/skills`。
2. 已接入技能（2026-02-14）：
   - `web-design-guidelines`：用于 UI 规范、可访问性、UX 审查。
   - `vercel-react-best-practices`：用于 React 性能与工程实践审查。
   - `vercel-composition-patterns`：用于组件组合模式与可维护性审查。
3. 评审协同顺序：先做通用 UI 规范审查，再做项目口径一致性校验，再给出改造建议。
