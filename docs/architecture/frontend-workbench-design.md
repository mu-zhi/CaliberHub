# 数据直通车 · 前端界面与工作台设计

本文承接“受控知识底座”在前端的产品化表达，是当前唯一前端主文档，统一定义整站信息架构、页面骨架、中文界面口径、实现硬约束、评审规则和前端决策留痕。主方案继续由 [知识图谱与数据地图方案](system-design.md) 统一定义对象、边界、规则、接口和运行约束；本文负责把这些对象和规则落到可设计、可实现、可验收的前端工作台。

项目级文档路由原则固定如下：凡属前端页面结构、导航、交互、状态表达、中文界面口径和前端评审留痕，统一写入本文；凡属方案对象、运行主线、接口契约和治理边界，统一写入主方案。同一需求若同时影响两类内容，必须拆分更新两份主文档，不把前端页面结构塞回主方案，也不把方案对象与接口细节挤入本文。

## 1. 界面定位与边界

数据直通车前端不是“查数工具前端”，也不是“图数据库控制台”。它服务三件事：

1. 把知识沉淀进去。
2. 把知识安全发布出去。
3. 把知识有边界地消费出来。

前端统一承接两类工作台和一条追踪主线：

1. 知识生产工作台：承接材料接入、解析抽取、资产建模、元数据对齐、复核发布。
2. 运行消费工作台：承接数据地图浏览、运行决策解释、审批导出、监控审计。
3. 追踪主线：所有关键动作都能回到版本、证据、原因、影响和审计。

本文不替代主方案，不单独定义后端对象模型、接口载荷和运行变量，也不下沉到像素级视觉稿。视觉稿、前端实现、评审和留痕均以本文为前端基线。

## 2. 全局信息架构

### 2.1 一级导航

整站一级导航固定为 7 个业务工作台：

| 一级导航  | 主要用户          | 主任务                       | 默认进入内容                    |
| ----- | ------------- | ------------------------- | ------------------------- |
| 首页总览  | 全角色           | 读取当前系统状态、版本、风险与待办         | 默认进入状态总览，首屏展示一级状态卡、任务分发区与重点风险 |
| 数据地图  | 治理、分析、审批、审计   | 从业务视角浏览知识结构、覆盖边界、影响范围与证据  | 业务视图                      |
| 知识生产台 | 治理、数据产品、研发    | 把材料转成正式治理资产并推进复核          | 材料接入                      |
| 发布中心  | 治理、研发、审批负责人   | 检查候选版本、评估影响、执行发布与回滚       | 当前候选发布包                   |
| 运行决策台 | 运行支持、分析、治理、审计 | 解释一次请求为什么命中、为什么拒绝、为什么需要审批 | 请求解释工作台                   |
| 审批与导出 | 合规、安全、审批人、申请人 | 处理审批任务、查看导出结果和导出审计信息      | 待审批列表                     |
| 监控与审计 | 运维、治理、审计、研发   | 监控运行健康、元数据漂移、告警、审计链路      | 告警与追踪总览                   |

### 2.1.1 桌面端全局壳层

桌面端站点壳层固定采用“顶部横向轨道 + 左侧模块内导航 + 浅色工作面”，不在一级导航之外再发明并行站点结构，也不再使用独立的全局状态条压住正文。

1. 顶部横向轨道承载品牌、7 个一级工作台和全局工具区；一级导航始终完整可见，不因角色不同而隐藏导航项。
2. 一级导航采用“轨道式激活”表达：当前模块以底部亮线、字重提升和轻量高亮区分，不使用大面积门户标签块。
3. 搜索、消息、帮助、角色、个人入口固定放在顶部轨道右侧的全局工具区，不下沉到业务页正文。
4. 进入某个一级工作台后，左侧只展示该模块的二级导航；当模块无有效二级导航时，左侧可折叠，让内容区回到全宽。
5. 页级标题、副说明、关键状态卡和主操作位于顶部轨道之下的内容层，不再单独占一条站点级状态栏。
6. 无权限工作台采用“顶栏可见但置灰、不可点击”的表达方式，不进入页内后再拦截。
7. 内容工作面统一为浅色画布，导航轨略深；深色只服务导航和系统定位，不侵入主阅读区。

### 2.2 全局工具区

全局工具区固定承载：

1. 当前角色与角色切换。
2. 我的待办与个人最近操作。
3. 系统设置入口。
4. 用户手册与帮助入口。
5. 通知与消息入口。

系统设置属于支撑域，不进入一级业务工作台主链路。

### 2.3 角色可用性与焦点差异

一级导航不再按角色裁剪；角色差异通过“是否可操作、首页重点卡片、默认推荐入口”表达。

| 角色 | 主要可操作工作台 | 默认重点卡片 / 操作 |
| --- | --- | --- |
| 数据治理 / 数据产品 | 数据地图、知识生产台、发布中心、运行决策台、监控与审计 | 待复核、待补缺、待发布、最近风险 |
| 数据研发 / 平台研发 | 知识生产台、发布中心、运行决策台、监控与审计 | 元数据对齐异常、发布阻断、运行告警 |
| 合规 / 安全 / 审计 | 数据地图、审批与导出、监控与审计、运行决策台 | 待审批、高敏访问、审计追踪、阻断发布 |
| 运行支持 / 分析人员 | 数据地图、运行决策台、监控与审计 | 运行健康、命中率、最近风险、快捷查询 |
| 下游消费方 | 数据地图、运行决策台 | 知识包查询、覆盖说明、场景浏览 |

默认规则如下：

1. 所有角色都可见 7 个一级工作台，但仅对有权限工作台开放点击与进入。
2. 无权限工作台在顶栏置灰，不使用“点进去再报错”的拦截方式。
3. 首页卡片、快捷入口和主操作允许按角色裁剪，避免把无关任务堆给当前用户。
4. 治理对象允许被查看时，首屏先展示业务语义，细节区再展开治理对象、逻辑编码和技术辅助信息。

### 2.4 对象曝光层级

界面遵循“业务语义优先、治理对象下钻、技术编码折叠”的三层曝光：

1. 首屏先讲业务领域、业务场景、时间口径、覆盖边界、风险等级、当前状态。
2. 二级区块和详情抽屉再展开方案资产、契约视图、覆盖声明、策略对象、来源契约、证据片段、版本快照。
3. 逻辑编码、追踪编号、原因编码、版本编号和接口辅助信息只出现在详情、抽屉、复制区或审计区，不进入首屏主标题。

## 3. 全局界面规范

### 3.1 中文命名词典

整站主文案统一使用中文，不使用混合中英标题。需要保留英文编码时，放到二级信息区。

| 中文名称   | 用途                     |
| ------ | ---------------------- |
| 业务场景   | 表达稳定业务问题与消费入口          |
| 方案资产   | 表达可运行的受控解法             |
| 输出契约   | 表达系统正式承诺的输出范围          |
| 契约视图   | 表达按角色、用途和审批上下文裁剪后的生效视图 |
| 覆盖声明   | 表达覆盖边界、排除项和回退动作        |
| 策略对象   | 表达允许、审批、拒绝和审计要求        |
| 来源接入契约 | 表达材料接入阶段的结构化登记要求       |
| 来源契约   | 表达运行时可信来源约束            |
| 证据片段   | 表达可回溯的原始依据             |
| 版本快照   | 表达发布、回滚和运行时生效边界        |
| 缺口任务   | 表达材料和治理缺失项             |
| 复核任务   | 表达业务、技术、合规复核闭环         |
| 审计事件   | 表达发布、审批、导出和查询的审计记录     |

### 3.2 状态与决策文案

状态机展示统一采用“双轨约束”：代码、接口字段、内部状态仓库和枚举值可以保留英文状态编码，但所有面向业务用户的状态名称、筛选项、徽标、步骤条、时间轴和提示文案必须统一落成中文，不直接暴露英文状态值。

#### 资产状态

1. 草稿
2. 已复核
3. 已发布
4. 已退役
5. 已驳回

#### 发布任务状态

1. 待检查
2. 检查中
3. 已通过
4. 已阻断
5. 已切换
6. 已归档

#### 审批任务状态

1. 已提交
2. 预检查中
3. 已通过
4. 已拒绝
5. 已执行
6. 已归档

#### 覆盖等级

1. 完整覆盖
2. 部分覆盖
3. 存在缺口

#### 决策结果

1. 允许返回
2. 需要审批
3. 拒绝返回

### 3.3 按钮与提示语

按钮统一使用中文动词开头，不混用英文：

1. 新建接入
2. 开始解析
3. 发起复核
4. 执行发布
5. 执行回滚
6. 查看影响
7. 查看证据
8. 提交审批
9. 发起导出
10. 复制追踪编号

提示语统一遵循“说明原因、说明边界、说明下一步”，不使用空泛报错。推荐写法如下：

1. 当前角色不满足该契约视图的访问要求。
2. 当前请求命中高敏规则，需进入审批流程。
3. 本次发布被阻断：存在必需输出字段无法落位。
4. 当前请求仅部分覆盖，建议缩小时间范围或发起审批。
5. 图查询超时，系统未返回过期结果。

### 3.3.1 错误码与前端提示语映射

主方案 §9.3 定义的错误码必须在前端有统一中文提示语，不允许前端开发自行编写或直接展示英文错误码。

| 错误码 | 前端提示语 | 建议动作提示 |
| --- | --- | --- |
| `SCENE_NOT_FOUND` | 未找到匹配的业务场景。 | 建议调整查询条件或浏览数据地图选择场景。 |
| `PLAN_NOT_ELIGIBLE` | 当前场景下没有可运行的方案资产。 | 请确认输入参数是否完整，或联系治理人员检查方案发布状态。 |
| `COVERAGE_PARTIAL` | 当前请求仅部分覆盖，部分时间段或来源存在缺口。 | 建议缩小时间范围，或查看覆盖声明了解缺口详情。 |
| `POLICY_APPROVAL_REQUIRED` | 当前请求命中高敏规则，需进入审批流程。 | 请提交审批申请，填写用途和字段需求。 |
| `POLICY_DENIED` | 当前角色不满足该场景的访问要求，请求被拒绝。 | 请确认角色权限，或联系合规人员了解受限原因。 |
| `EVIDENCE_INSUFFICIENT` | 当前资产缺少必要的证据支撑，无法返回完整结果。 | 请联系治理人员补充证据片段。 |
| `METADATA_DRIFT_BLOCKED` | 来源表或字段已失效，当前结果不可信。 | 请联系数据研发检查元数据变更情况。 |
| `GRAPH_TIMEOUT` | 图查询超时，系统未返回过期结果。 | 请稍后重试，或缩小查询范围。 |
| `PUBLISH_BLOCKED` | 本次发布被阻断，存在未通过的检查项。 | 请查看发布检查矩阵，逐项修复阻断问题。 |

提示语展示规则如下：

1. 错误提示必须同时包含"发生了什么"和"建议怎么做"两部分。
2. 错误码本身只在底部控制台和审计追踪中展示，不进入主提示文案。
3. 当一次请求命中多个错误码时，按严重程度排序展示，`POLICY_DENIED` > `METADATA_DRIFT_BLOCKED` > `PUBLISH_BLOCKED` > `COVERAGE_PARTIAL` > 其他。

### 3.4 颜色、徽标与组件

视觉方向统一为“浅色画布上的克制技术产品台”，不走宣传式品牌页，也不走整站深色控制台。

1. 内容工作面使用低饱和浅灰蓝作为基础画布；主内容卡片保持白色或极浅冷灰，优先保证表单、图谱、审批和列表的可读性。
2. 顶部横向导航轨和全局工具区使用低饱和深灰蓝，但深色只停留在导航系统，不侵入正文主阅读区。
3. 强调色固定收敛为一套偏冷的青绿色，用于当前激活态、系统健康、关键确认和少量主强调；不同时混用米黄、琥珀、青蓝多套强调语言。
4. 绿色表示允许、已发布、健康；橙色表示需要审批、部分覆盖、风险提醒；红色表示拒绝、阻断、漂移和告警；灰色表示归档、退役、历史版本和辅助信息。
5. 主界面标题统一采用无衬线中文字体；数字、版本号、时间戳、原因码和局部英文标签允许局部使用等宽字体，但只用于系统感信息，不做大面积铺陈。

优先使用的基础组件如下：

1. 状态徽标
2. 一级状态卡
3. 二级业务卡
4. 表格
5. 抽屉 / 检查面板
6. 时间轴
7. 步骤条
8. 影响分析面板
9. 证据片段预览区

组件层级固定遵循以下规则：

1. 一级状态卡承担页面第一视觉，允许更强对比、更大数字和更硬朗的标签表达。
2. 二级业务卡回到白底、细边框和轻阴影，不把所有卡片做成同等重量。
3. 主按钮使用深色实底，次按钮使用浅底描边，幽灵按钮只保留文字和轻边界，不把所有操作都做成实心按钮。
4. 顶部一级导航的激活态采用横向轨道亮线；左侧二级导航的激活态采用浅底高亮卡片 + 更深字重，不使用大面积纯色按钮。
5. 图谱只在依赖、影响、证据回溯三个场景承担主角色，其余页面优先使用卡片、列表、表格和结构化详情。

### 3.5 参考项目界面取舍

本项目前端界面参考 `MiroFish（开源项目）` 的不是品牌皮肤本身，而是材料进入系统后的阶段表达方式、工作台组织方式和图谱联动方式。参考范围只限于页面布局、信息组织、阶段反馈和图谱联动，不直接继承其业务对象模型、英文命名、品牌插画、全站深色控制台外壳和报告主题。

具体取舍如下：

1. 参考其“单任务聚焦 + 阶段反馈前置”的写法，把当前阶段、任务状态和是否可继续操作放在页面上沿，而不是埋进正文区。
2. 参考其“主工作区 + 右侧详情区 + 可折叠辅助区”的工作台结构，在知识生产台、数据地图、运行决策台和发布中心承接主任务、对象详情和校验输出。
3. 参考其图谱画布与详情联动方式，节点、关系或资产点击后在当前页右侧展开详情，不跳离当前上下文。
4. 参考其“先上传 / 配置，再处理，再看结果”的任务顺序组织知识生产台，不把系统设置、帮助和调试信息放在主任务之前。
5. 参考其页首“大标题 + 副说明 + 轻量统计块”的组织方式，但首页总览仍保持“指挥页”定位，不做宣传式品牌首屏。
6. 参考其底部系统输出区思路，但统一改造为“可折叠运行日志 / 校验输出区”，只在解析、发布检查、导出执行等场景出现，默认折叠。
7. 不参考其全站深色控制台皮肤和单任务沉浸式站点壳层；本项目固定采用“浅色工作面 + 略深导航轨”的企业级工作台外壳。
8. 不参考其品牌插画、英文主标题、预测报告主题和开放式聊天替代正式治理操作的表达方式。
9. 不参考“实体抽取结果即全系统主对象”的建模方式；本系统继续以 `Scene（业务场景）`、`Plan（方案资产）`、`Output Contract（输出契约）`、`Contract View（契约视图）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）` 等治理资产为正式对象。
10. 图谱只在数据地图、影响分析、证据回溯、运行解释等适合的页面承担主角色，不作为全站默认表达方式。

### 3.5.1 MiroFish 视觉风格吸收细则

以下规则基于 `MiroFish（开源项目）` 6 张运行截图提炼，定义本项目在图谱可视化、布局密度和交互模式上的具体吸收口径。

#### 图谱画布风格

1. 吸收其力导向布局下的节点散布方式：节点按语义聚类自然收敛，不强制网格对齐；聚类之间保留呼吸空间，避免全图均匀铺满。
2. 吸收其节点着色体系：按节点类型分配固定色相，同类节点同色。本项目的节点色相映射如下：
   - 业务场景节点：靛蓝（主色）。
   - 方案资产节点：深青。
   - 覆盖声明 / 策略对象节点：深靛（与业务场景同色系但降一阶饱和度，表达控制资产语义）。
   - 推理结论 / 推理链节点：青绿（与方案资产同色系但偏绿，表达推理资产语义）。
   - 来源表 / 字段节点：暖灰。
   - 术语 / 口径 / 规则 / 字典节点：琥珀。
   - 表间关联关系节点：浅琥珀（与术语同色系但降一阶饱和度，表达物理关联语义）。
   - 证据片段节点：淡紫。
   - 风险 / 告警关联节点：红色。
3. 吸收其边的视觉分层方式：普通关系边使用低饱和灰色细线；当前选中路径或高亮关系使用高饱和主色粗线；风险关联边使用红色。不使用全彩色边的"彩虹图"风格。
4. 吸收其画布默认留白比例：图谱首次加载时可见节点占画布面积不超过 60%，四周保留留白，避免首屏即视觉压迫。
5. 不吸收其超大规模全图一次性渲染方式（截图 6 展示的数百节点全铺开）。本项目图谱默认按领域或场景缩域后展示局部子图，全域总览仅在数据地图的业务视图中作为可选视角。

#### 节点与详情联动

1. 吸收截图 2、3、6 的右侧浮层详情面板方式：点击节点后在画布右侧展开详情卡片，卡片内展示名称、类型、创建时间、摘要、标签和操作按钮，不跳离画布上下文。
2. 详情面板使用浮层卡片而非全高抽屉，保证画布仍可见且可继续交互。面板宽度固定为 320px–400px，不超过画布宽度的 35%。
3. 吸收其底部图例条方式：画布左下角固定展示当前可见节点类型的颜色图例，使用小圆点 + 类型名称，不使用大色块。

#### 边与关系交互

本项目中边（关系）与节点同等重要，不能让关系成为画布上的隐形对象。知识图谱底层的 13+ 种正式关系类型（如 `Domain 1:N Scene`、`Scene 1:N Plan`、`Plan 1:N Coverage Segment`、`Plan 1:N Contract View`、`Derivation Rule 1:N Inference Rule`、`Inference Rule 1:N Inference Assertion`、`Scene / Plan N:N Evidence Fragment` 等）在数据地图前端必须具备以下交互能力：

1. 点击边后，右侧浮层详情面板切换到 `关系详情` 页签，展示关系类型（中文，如"包含方案"、"覆盖分段"、"证据支撑"、"推理产出"）、关系方向、关系版本、关联的治理对象双端摘要和证据引用。
2. 顶部工具栏的治理过滤器增加 `关系类型` 维度，允许用户只显示特定类型的关系（如只看控制资产关系、只看推理关系、只看来源关系、只看语义关系），以降低全图复杂度。
3. 边的视觉分层增加关系类型区分：控制资产关系（场景到方案、方案到覆盖、方案到策略等）使用主色系实线；推理关系（推理规则到推理结论、推理结论到推理链）使用青绿色虚线；语义关系（术语到口径、口径到规则）使用琥珀色细线；元数据关系（方案到来源表、来源表到字段）使用暖灰色细线。
4. 关系标签默认隐藏，但在右下角工具栏提供"关系标签显隐"开关，开启后在边的中点显示中文关系名称（如"包含"、"覆盖"、"支撑"、"推理"）。
5. 当用户聚焦某一节点时，与该节点直接相关的边自动高亮，不需要用户逐边点击。

#### 页面布局密度

1. 吸收截图 1 的"大标题 + 副说明 + 工作流步骤"首屏密度：页面进入后首屏只展示标题、当前阶段和主操作入口，不堆叠多个平行信息区。
2. 吸收截图 2、3 的三区密度比例：图谱画布占 55%–60%，右侧详情面板占 30%–35%，底部控制台默认折叠仅占一行标签栏。
3. 吸收截图 4、5 的左正文右面板的双栏密度：左侧结构化内容区占 60%–65%，右侧辅助面板占 35%–40%。本项目中运行决策台采用此密度；对于资产建模与元数据对齐，推荐弱化双栏，改用“主视觉表单 + 辅助参考抽屉”以减少视觉拥挤。

#### 不吸收项（视觉层面补充）

1. 不吸收其品牌鱼形插画和宣传式首屏。
2. 不吸收截图 4、5 中的开放式 Agent Chat 对话面板替代正式治理操作；本项目右侧面板始终是结构化检查面板 + 操作面板，不做聊天式交互。
3. 不吸收截图 6 中节点文字直接铺在画布上的密集标注方式；本项目节点默认只显示缩略名称，完整信息在详情面板展开。

### 3.6 统一工作台壳层与页面模式

当前前端仍以 7 个一级工作台作为全站产品结构。统一的是站点壳层、页面模式和辅助面板纪律，不是把所有页面都改造成同一种图谱工作台。`MiroFish（开源项目）` 提供的是工作台表达方式，不是站点信息架构模板。落地原则如下：

1. 全站共享“顶部横向轨道 + 左侧模块内导航 + 浅色工作面”的基础壳层。
2. 首页总览采用“总览页模式”，知识生产台、发布中心、运行决策台采用“工作台页模式”，审批与导出、监控与审计采用“浏览页模式”。
3. 数据地图、影响分析和证据回溯保持画布优先；其他页面不得为了统一风格而强行图谱化或卡片化。
4. 首页总览继续作为指挥页，不改成“上传材料即首页默认动作”的生产入口页。

共享壳层固定包含四层：

1. 顶部横向轨道：承载品牌、一级导航和全局工具区。
2. 左侧模块内导航：承载当前一级工作台下的二级导航，不承担跨模块切换职责。
3. 页头与关键状态区：显示当前页面标题、副说明、关键状态卡和主操作。
4. 主工作区：根据页面类型承载总览区、画布区、结构化清单、检查面板和可折叠运行输出区。

在共享壳层内部，允许使用三种页面模式：

1. `总览页模式`：页头区 + 一级状态卡 + 两列业务区块。适用于首页总览、监控与审计总览等“先看状态、再分发任务”的页面。
2. `工作台页模式`：页头区 + 关键状态条 + 主工作区；必要时带右侧 `Inspector（检查面板）` 和底部 `Console（控制台）` / `Trace（追踪）` 折叠区。适用于知识生产台、发布中心、运行决策台。
3. `浏览页模式`：页头区 + 筛选区 + 主内容区。适用于审批与导出、审计检索以及以查找、筛选、比对为主的页面。

三种页面模式的边界如下：

1. 图谱不是全开放查询图，不向用户传达“点哪都能查”的数据库浏览器预期。
2. 数据地图和影响分析等图谱重页面必须先缩域，再看图，再解释路径，不跳过 `Scene（业务场景）`、`Plan（方案资产）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）` 等硬门禁对象。
3. 运行态最终返回的是 `Knowledge Package（知识包）`，不是裸字段结果、裸 `SQL（结构化查询语言，Structured Query Language）` 结果或报告正文。
4. 字段级安全必须在前台可感知，检查面板要明确显示当前角色可见字段、已脱敏字段、需审批字段和禁止返回字段。

## 4. 核心页面设计

### 4.1 首页总览

首页总览是“状态优先、任务分发次之”的总览页，不是宣传页，也不是上传材料后的生产默认页。用户进入系统后，需要在 10 秒内回答五个问题：

1. 当前是否处在稳定版本。
2. 今天最重要的风险在哪里。
3. 我上次处理中对象是什么。
4. 我今天要处理什么。
5. 哪些工作台值得立即进入。

页面结构固定为“页头区 + 一级状态卡 + 两列主区 + 辅助入口带”：

1. 页头区：默认只展示轻量页面标识，不在页头重复铺陈整站状态条；当一级状态卡已经承担首页第一认知时，允许省略额外大标题和副说明，避免挤占首屏高度。
2. 一级状态卡：固定 4 张，分别表达当前稳定版本、阻断发布、待审批、运行健康；承担首页第一视觉，不继续膨胀为通用大盘。
3. 两列主区：左列承载最近变更、最近处理中对象和发布动态；右列承载重点风险、我的待办和角色相关提醒。
4. 辅助入口带：固定 4 个一级动作，进入数据地图、新建材料接入、查看发布中心、查看审批队列；入口带降级为辅助区，不抢首屏主焦点。
5. 搜索与快捷定位保留为辅助动作，默认放在主区下沿或侧区，不回到首页最上方成为第一操作。

关键交互如下：

1. 点击当前稳定版本进入发布中心版本详情。
2. 点击阻断发布进入发布中心并筛选“已阻断”。
3. 点击待审批进入审批与导出。
4. 点击运行健康中的异常指标进入监控与审计。
5. 点击最近处理中对象中的“继续处理”，直接进入对应对象所属工作台，不经过中间门户页。
6. 点击最近查看的场景、方案资产或审批任务，优先回到原工作台上下文，不只打开孤立详情。

首页卡片规则固定如下：

1. `一级状态卡` 是首页第一层级，不用大段说明文替代；卡面只放核心数字和一句状态说明。
2. 当页头区已经提供轻量页面标识时，不再额外叠加解释性长标题；首页的解释责任由一级状态卡和主区摘要承担。
3. `最近处理中对象` 和 `重点风险` 必须前置到首屏可见区域，不允许被快捷入口或普通统计卡压到下方。
4. `动态区` 固定拆成“最近变更 / 发布动态”和“重点风险 / 我的待办”两列，不做单列流水。
5. `辅助入口带` 只保留一级动作，不在首页承接复杂筛选和闭环执行。
6. 首页不承接编辑、发布、审批等重操作；首页只负责摘要、续办和准确分发。
7. 首页默认态要求展示当前生效版本和角色相关事项；空态只允许出现在局部区块内；无权限态不显示受限操作卡；风险态区块必须前置到首屏可见区域。

### 4.2 数据地图

数据地图是整套产品最核心的解释页，回答“知识如何组织、覆盖到哪里、为什么能发布、为什么能返回”。

数据地图必须保持“画布优先、卡片从属”的原则，不把首页式状态卡和入口卡平移到图谱页。站点级左侧二级导航只负责数据地图模块内子页切换；页面内部的领域树、过滤器、证据列和详情面板都属于主工作区，不与站点级导航混用。

页面骨架固定为六区，采用“内容主切换、布局次级切换”的桌面端工作台结构：

1. 顶部工具栏：搜索框、业务领域筛选、场景类型筛选、敏感等级筛选、状态筛选、是否仅看已发布、当前版本选择、导出当前视图。
2. 主视图切换条：`业务视图`、`路径视图`、`覆盖视图` 三选一，默认进入 `业务视图`。这一层先回答“当前在看什么”。
3. 页面内左侧导航树：只保留 `业务领域` 一级，不在树内继续展开到业务场景或治理对象；节点上显示该领域下的已发布场景数、风险和覆盖微标识，作为所有主视图共享的缩域入口。
4. 中央主区：承载当前主视图的核心内容；其标题栏内再提供次级布局切换：`图谱（graph）`、`双栏（split）`、`工作台（workbench）`。这一层只回答“当前内容怎么摆”。
5. 右侧检查面板：固定常驻当前工作台上下文，承载定义、覆盖、策略、证据、版本、影响等详情与动作；在 `业务视图` 中默认页签顺序固定为 `对象概况` -> `覆盖检查` -> `风险与影响`，不使用打断式模态弹窗。
6. 底部联动区：证据片段、版本差异、影响范围、关联路径四个标签区，默认折叠，仅在追查时展开。

这一定义意味着：数据地图先让用户选择 `业务视图 / 路径视图 / 覆盖视图`，再在当前主视图内切换 `图谱 / 双栏 / 工作台`。布局切换不再承担“分析模式”的主入口角色。

结合 `MiroFish（开源项目）` 的实地取舍，数据地图明确吸收的是“图谱工作台组织方式”，不是其业务对象模型，也不是其默认全图表达。落地结论如下：

1. 吸收其“主工作区 + 右侧详情区 + 底部可折叠输出区”的工作台结构，让图谱、详情和追踪在同一上下文内联动。
2. 吸收其刷新、最大化、关系标签开关、画布图例和同页详情的微交互，让图谱可观察、可追踪、可回到上下文。
3. 吸收其长任务的阶段反馈思路，让图谱加载、影响分析、证据追踪等异步动作在页内可感知。
4. 不吸收其“默认全图铺开”的表达方式；数据地图必须先缩域到业务场景或焦点对象，再展示局部子图。
5. 不吸收其“实体抽取结果直接等于正式对象”的建模方式；数据地图中正式对象仍然是 `Scene（业务场景）`、`Plan（方案资产）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Output Contract（输出契约）` 等治理资产。
6. 不吸收其单项目长流程式站点结构；`MiroFish（开源项目）` 更适合“单任务沉浸式流程”，本项目仍坚持 7 个一级工作台的企业级导航体系。

当前一期实现补充约束如下：

1. 已落地 `graph / split / workbench` 三态布局切换，但其定位调整为次级布局层；默认进入 `业务视图 + graph（图谱）`，窄屏优先退化到 `workbench（工作台）`。
2. 资产图谱模式固定提供 `路径高亮`、`证据追踪`、`影响分析` 三种聚焦方式，不再只展示静态关系图。
3. 右侧工作台应收敛为检查面板主导，默认先讲对象概况，再讲覆盖检查，最后讲风险与影响；流程步骤和系统日志继续作为附属折叠区，不额外弹出打断式模态。
4. 已落地的治理过滤器至少包括对象类型、资产状态、关系类型、敏感级别和 `snapshot_id（快照标识）`；业务领域在一期由当前所选领域上下文固定，不做跨领域全局切换。
5. 右侧焦点详情必须同时覆盖节点详情和关系详情；当切到 `影响分析` 聚焦模式时，详情区继续保留当前节点信息，不与影响结果互斥。
6. 当数据地图通过 `Workbench Context Package（工作台上下文包）` 收到 `asset_ref（资产标识）` 或 `coverage_segment_id（覆盖分段标识）` 时，页面应自动进入 `lineage（资产图谱）` 模式，不要求用户先手动从 `browse（目录浏览）` 切换到图谱。
7. 当前阶段的高可见反馈固定遵循“节点先于路径”的视觉顺序：先以目标节点建立视觉锚点，再叠加覆盖分段或关系路径高亮；不以路径中点、统计卡或右侧面板作为第一落点。
8. 对话驱动进入图谱时，默认聚焦方式自动切到 `路径高亮`；工具栏继续保留手动切回 `证据追踪` / `影响分析` 的入口，并明确提示当前为自动路径模式。这里的层级固定为：`lineage（资产图谱）` 是图谱模式，`路径高亮` / `证据追踪` / `影响分析` 是图谱模式内部的子视角，不得把两层状态当作平级模式混用。

三个主视图的责任固定如下：

1. `业务视图`：默认主视图，展示业务领域、业务场景、方案资产、契约视图、覆盖声明、策略对象之间的结构关系。默认布局为 `graph（图谱）`，并采用“顶部摘要条 + 主图 + 底部场景带”的图谱主导结构：顶部摘要条负责领域级概况，主图负责局部业务关系骨架，底部场景带负责场景分发与落脚；切到 `split（双栏）` 时加强结构化对照，切到 `workbench（工作台）` 时强化解释与筛选。
2. `路径视图`：展示业务场景到来源契约、来源表、来源字段、路径模板之间的支撑链路。默认布局为 `graph（图谱）`，采用“路径 + 证据并置”的主结构：中央主图区负责主路径、候选路径与节点关系表达，右侧证据列负责证据片段、规则说明和来源契约摘要；切到 `split（双栏）` 时，左侧保留路径结构，右侧强化结构化对照。
3. `覆盖视图`：以时间带和覆盖分段展示完整覆盖、部分覆盖、存在缺口，以及对应回退动作。默认布局为 `workbench（工作台）`，中央主区改为时间带、覆盖矩阵和回退动作说明；`graph（图谱）` 不作为覆盖视图默认表达，避免用关系图误导时间覆盖判断。

三种主视图之间的切换规则固定如下：

1. 切换主视图时，默认保留当前选中的业务领域、当前焦点场景（如已通过底部场景带或图谱节点选中）、快照和治理过滤器，避免用户每次重选上下文。
2. 主视图切换优先触发中央主区内容替换，右侧检查面板保留焦点对象，只替换与当前主视图相关的页签内容。
3. 若当前布局不适合目标主视图，系统自动回落到推荐布局：
   - 从任意视图切到 `业务视图` 时，优先回落到 `graph（图谱）`
   - 切到 `路径视图` 时，优先回落到 `graph（图谱）`
   - 切到 `覆盖视图` 时，优先回落到 `workbench（工作台）`
4. 主视图切换只改变“当前看什么”，不重置搜索词、领域树滚动 / 选中状态和底部联动区的折叠状态。
5. 布局切换只改变“当前内容怎么摆”，不得篡改当前主视图的业务含义。

关键交互如下：

1. 支持直接输入业务问题，自动高亮相关业务场景、方案资产、覆盖分段、策略对象和证据片段。
2. 节点详情抽屉必须能看到定义、覆盖、策略、证据、版本和影响，不要求跳转到外部页面才能看清边界。
3. 当用户从左侧导航树选中某个业务领域后，中央主区默认直接进入该领域的局部业务图谱，不先把用户扔进全图状态。
4. `业务视图` 中，场景不再由左侧树承担首要分发责任，而由中区底部场景带和图谱节点共同承担；用户选中场景后，右侧检查面板延续当前焦点对象，不额外切页。
5. 右侧检查面板在 `业务视图` 中固定按 `对象概况`、`覆盖检查`、`风险与影响` 的顺序组织，在 `路径视图` 中优先展示来源与证据，在 `覆盖视图` 中优先展示覆盖分段、缺口和回退动作。
6. 默认态只展开一个主视图和一个详情区，底部联动区按需展开，避免信息墙。

#### 带上下文进入的高可见反馈

当数据地图作为跨工作台跳转的目标页时，除了沿用 §6.2 的锁定态提示条和快照校验规则，还必须提供可见、可感知的图谱反馈，避免用户误判为“只是多了一条提示”。

1. 当 `ctx` 中带 `asset_ref` 或 `coverage_segment_id`，且图谱数据可用时，中央主图区必须对目标节点执行一次短促的自动定位动画，并以该节点作为本次上下文的唯一视觉锚点。
2. 自动定位只在“新一轮 `ctx` 到达”时触发一次；用户随后手动平移或缩放画布后，同一轮上下文内不得再次强制把视口拉回目标位置，直到下一轮新 `ctx` 到达。
3. 当 `asset_ref` 与 `coverage_segment_id` 同时存在时，主图先强化目标节点，再叠加覆盖路径高亮；节点的描边、阴影和对比度权重必须高于路径描边，避免用户只看见一段动线却找不到锚点对象。
4. 当 `asset_ref` 在当前图谱中找不到时，页面不得把全图统一压暗，也不得执行空目标定位；必须保持当前视口不变，并以轻提示说明“该资产不在当前地图视野中”。
5. 当系统已识别到目标资产，但当前快照下暂无图谱数据时，页面应继续停留在 `lineage（资产图谱）` 模式，并显示“已定位到目标资产，但当前无图谱数据”的空态说明；只有在资产元信息本身都不存在时，才自动回退到 `browse（目录浏览）` 根目录。
6. 网络异常、超时或服务端错误只允许渲染错误横幅和重试入口，不得借由切回 `browse（目录浏览）` 掩盖错误来源。

#### 4.2.1 业务视图桌面端三区定稿

`业务视图` 的桌面端三区结构固定如下：

1. 左侧导航树只保留 `业务领域` 一级，不继续在树内展开场景、方案资产或治理对象。
2. 中央主区采用图谱主导，而不是场景卡片主导；用户选中某个领域后，默认看到该领域的局部业务图谱，而不是全局总图。
3. 中区排版固定为 `G1`：`顶部摘要条 + 主图 + 底部场景带`。顶部摘要条采用 `T2` 读法：先给一句领域判断，再给三个关键指标；摘要句承担场景总数和当前领域稳定度说明，指标区固定优先展示覆盖率、风险对象数和最近变更项数。主图承担结构理解，不被卡片墙挤压。
4. 底部场景带采用 `S2` 状态化场景卡带，默认展示 `6 张横向滚动卡`。排序采用 `3 + 3`：前 3 张为领域推荐场景，后 3 张吸收当前用户的最近访问与最近变更。
5. 场景卡第二行优先展示 `最近变更 + 最近访问`，不把覆盖等级和风险等级重复堆成第二道状态墙；点击场景卡后留在当前页，只让主图聚焦到该场景，不自动改写右侧检查面板。
6. 右侧检查面板固定以 `对象概况` 为第一页，随后是 `覆盖检查`，最后才是 `风险与影响`。
7. 阅读路径统一收敛为：先看领域关系骨架，再定位具体场景 / 对象，最后做覆盖与风险判断。

不采用“场景卡片主导”有两个原因：

1. `业务视图` 首先要回答的是“这个领域内部的知识结构怎么组织”，不是“有哪些卡片可以点”；如果首屏先铺场景卡，用户得到的是入口列表，不是结构理解。
2. 左侧导航树已经被收敛成“只保留领域”一级，场景卡如果再占据中央第一视觉，会把图谱降为二级信息，违背“先有图谱，才有结构化数据地图”的产品原则。

#### 4.2.2 路径视图桌面端定稿

`路径视图` 的桌面端结构固定如下：

1. 默认进入 `graph（图谱）` 布局，并采用 `P3`：`路径 + 证据并置`。中央主图区与右侧证据列宽度固定为 `7 : 3`，不采用整页表格或整页详情替代路径主图。
2. 页头左侧固定展示三项上下文信息：`当前主路径`、`当前快照 / 版本`、`候选路径数`。页头右侧独立放置 `路径版本 / 快照` 切换控件与 `主路径 / 候选路径对比` 入口，不把版本切换塞进主路径标题行。
3. 中央主图区默认同时显示 `当前主路径 + 2 条相邻候选路径`。主路径保持高亮，候选路径统一灰化。点击来源表 / 来源字段节点时，只高亮相关路径，不联动改写右侧证据列。
4. 点击灰色候选路径后进入临时预览态：候选路径被高亮，页头出现“正在预览候选路径”提示，但当前主路径默认不立刻切换。用户可通过页头 `设为主路径` 按钮升格，也可再次点击同一候选路径直接升格。切换 `路径版本 / 快照` 时，系统自动退出预览态。
5. 右侧证据列采用页签式切换，默认阅读顺序固定为：`证据片段` -> `规则说明` -> `来源契约摘要`。切换页签只改右侧内容，不改变主图结构，并清空当前页签带来的临时高亮。
6. `路径版本 / 快照` 切换时，中央主图与右侧证据列一起刷新；右侧强制回到默认的 `证据片段` 页签，并将当前展开态全部收起，避免旧版本证据与新版本路径混读。
7. `证据片段` 页签采用“双段信息型”结构：上半段先展示 `证据结论 -> 最近更新时间 -> 来源位置`，下半段展示 `证据原文 + 高亮命中句`。原文默认使用 `6-10 行` 的中摘录，不直接铺满完整原文。
8. `规则说明` 页签首屏采用“`一句话总规则 + 全部关键命中规则`”结构，第一眼先回答“它命中了哪些规则”。单条规则卡顺序固定为 `规则结论 -> 命中条件 -> 影响对象`，并附 `高 / 中 / 低` 的命中强度。点击规则卡后，在右侧展开规则明细，同时在主图轻量高亮对应规则节点；规则明细顺序固定为 `规则解释 -> 影响对象 -> 命中条件原文`。
9. `来源契约摘要` 页签默认按 `约束重要性` 排序，保持纯列表并 `全部展开`。单条约束卡顺序固定为 `约束结论 -> 影响范围 -> 来源系统名称 + 契约名 -> 生效状态`；生效状态口径固定为 `生效中 / 待确认 / 已失效`，并附 `阻断 / 高影响 / 一般` 的风险标识。点击约束卡后，只在右侧展开明细，不改写主图；明细顺序固定为 `一句话口径说明 -> 约束原文摘录`。
10. 进入 `主路径 / 候选路径对比` 后，路径视图切换为“上图下差异清单”结构：上方保留对比图，下方展示差异摘要，并允许切换查看两边证据。原浏览态下的三个页签在对比模式中先收起，替换为独立的对比面板。
11. 对比模式的默认对象固定为 `当前主路径 vs 第一条候选路径`。上方对比图采用 `主路径高亮 + 候选路径次亮` 的表达方式，不使用双主色抢夺视觉主次。
12. 下方差异清单默认按 `结构差异 -> 规则差异 -> 来源契约差异 -> 证据差异` 排序，不按来源系统或时间顺序混排。右侧独立对比面板默认先打开 `差异摘要`，而不是直接落到任一路径证据。
13. 点击下方某条差异项后，在下方展开该差异的结构化明细，并同步让上方对比图高亮对应的差异节点或差异边，形成“下方读差异、上方看位置”的联动。
14. 若用户在对比模式中将候选路径升格为主路径，系统应立即退出对比模式并回到普通浏览态；退出对比后仍必须回到进入对比前的浏览语境，而不是继续停留在候选路径预览态。
15. `差异摘要` 首屏先给一句总判断，再进入分项差异，不以数量统计开场。用户切到“两边证据”时，默认先打开 `主路径证据`，再允许切换到候选路径证据。
16. 对比模式的上方主图只保留 `差异相关局部子图`，不继续展示与当前差异无关的大量稳定节点，避免“为了证明没有变化而保留整张图”。
17. `结构差异` 组内先按 `新增 / 删除 / 变更` 分组，再在每组内按影响大小排序，不把三类结构变化混成一张线性列表。
18. 点击任何差异项时，下方先展开该项明细，上方再高亮对应差异节点；不允许只在图上闪烁而下方没有结构化解释。
19. 退出对比后，页头需保留“刚才比较过哪两条路径”的轻量记录，作为再次进入对比的快捷入口，但该记录只承载回看，不自动改变当前主路径状态。

#### 4.2.3 覆盖视图桌面端定稿

`覆盖视图` 的桌面端结构固定如下：

1. 默认进入 `workbench（工作台）` 布局，但中央主区采用 `左时间轴 + 右覆盖明细` 的桌面端骨架，不用整页图谱承担时间覆盖判断。
2. 中央第一视觉先看 `覆盖矩阵`，而不是先看时间带摘要。覆盖矩阵负责直接回答“哪些时间段 / 来源范围是完整覆盖、部分覆盖、还是存在缺口”。
3. 左侧时间轴承担时间定位和分段导航职责，右侧覆盖明细负责解释当前时间段对应的覆盖状态、缺口原因和回退动作，不把时间轴和解释明细混成单栏长页面。
4. 右侧检查面板默认顺序固定为：`覆盖总览 -> 缺口与回退 -> 发布影响`。阅读顺序先回答“整体覆盖如何”，再回答“哪里有洞、怎么回退”，最后回答“这会怎样影响发布”。
5. 点击某个覆盖分段后，中央主区只高亮该分段，不自动改写右侧检查面板当前页签，避免用户在按自己节奏阅读时被强制打断。
6. 缺口清单默认按 `阻断发布的缺口优先` 排序，不按时间最近或对象最近访问排序，确保发布阻断类问题先被识别和处理。
7. `覆盖视图` 的默认阅读路径固定为：先扫覆盖矩阵确定完整 / 部分 / 缺口，再沿左侧时间轴定位具体分段，最后到右侧覆盖明细核对缺口原因与回退动作。

### 4.3 知识生产台

知识生产台按生产链路组织，不按技术对象平铺。二级页面固定为：

1. 材料接入
2. 解析抽取
3. 资产建模
4. 推理校验
5. 元数据对齐
6. 复核
7. 缺口任务

页面整体采用“顶部阶段状态条 + 主任务区 + 右侧结果 / 约束摘要区 + 底部可折叠校验输出区”的工作台结构。生产台整体仍以符合业务习惯的向导式结构化流程为主；只有“解析抽取”阶段把图谱提升为左侧主画布，用于展示导入中的候选实体图谱，其余阶段继续以结构化表单、检查面板和折叠输出区为主。

知识生产台承接首个主流程链路，默认覆盖材料接入、候选实体图谱抽取与图谱辅助确认、资产建模、推理校验、元数据对齐（即确定业务要素后再找物理落位）。

#### 4.3.1 材料接入与解析主工作页重构约束（当前一期）

当前一期的 `/production/ingest` 不再定义为“四步并列编辑页”，而应重构为“当前导入任务工作页”。页面第一责任不是让用户阅读 `01-04` 四个步骤卡，而是让业务支持 / 口径维护人员在进入页面后的首个视野内回答三个问题：

1. 当前在处理哪一批材料。
2. 现在卡在哪个阶段。
3. 下一步应该处理什么。

重构后的页面信息层级固定如下：

1. `当前导入任务`：任务名称、来源类型、最近更新时间、整体进度、当前阶段、阻断状态、待处理场景数。
2. `任务主线区`：当前阶段、下一待办、推荐处理对象、主操作、影响提示。
3. `次级结果区`：场景队列、原文对照、候选图谱、导入明细。
4. `历史步骤摘要`：保留 `01-04` 作为状态机摘要和有限回跳入口，不再作为并列主编辑区。

首屏骨架固定为“三层布局”：

1. 顶部任务总览带：承载当前导入任务摘要与次级动作，不展开长说明文。
2. 中部主工作区：左侧固定为任务主线卡，右侧固定为风险与状态卡；首屏必须把“继续处理什么”放在中心，而不是把系统处理痕迹放在中心。
3. 下部结果区：以标签页或同级分段承接 `场景队列 / 原文对照 / 候选图谱 / 导入明细`，这些区块默认服务主线，不与主线争抢页面主舞台。

当前页的交互模型固定如下：

1. 页面默认焦点是 `当前导入任务`，而不是某个孤立场景，也不是四步清单本身。
2. 页面中心主操作优先级固定为：`继续当前待办` > `保存当前处理` > `查看阻断或补缺` > `跨工作台查看`。
3. 任一时刻只允许一个主按钮承担当前任务推进；“下载样例材料”“去数据地图”“去运行决策台”不得与主推进按钮并列竞争。
4. 任务主线按 `待导入 / 待质检 / 待对照 / 待发布整理 / 已完成` 五个任务态驱动界面，不要求用户先理解内部四步状态再继续操作。

跨工作台跳转采用“有上下文才出现”的半保留策略：

1. `查看数据地图` 仅在当前导入任务已生成可浏览场景或已有已确认对象时出现。
2. `查看运行决策台` 仅在当前导入任务存在可验证或已发布上下文时出现。
3. 两类跳转都只能放在顶部任务总览带或完成态区，属于次级动作，不得放进任务主线卡的主操作区。
4. 未满足上下文前提时，不显示跳转入口，也不显示“点进去才发现为空”的导航型按钮。

`01-04` 步骤条保留，但必须降级为历史步骤摘要区，约束如下：

1. 每条摘要只展示该阶段的核心结论，如来源、识别场景数、未确认数、待发布数，不在摘要层铺开长日志。
2. 用户可以查看历史步骤详情，但返回上游修改时，必须先看到影响提示，明确说明是否需要重新完成质检、对照或发布整理。
3. 不允许把“修改”按钮继续设计成无成本回改；若上游变更会使下游失效，必须显式提示“修改后需重新确认后续阶段”。
4. 四步摘要区服务“回看与有限回跳”，不再承担首屏主编辑职责。

业务语义与技术语义的曝光层级在当前页强制执行以下规则：

1. 主标题、推荐处理对象和场景队列首列优先展示业务名称；业务名称为空时，统一展示“待命名场景 + 序号”，不直接把数据库表名当成主标题。
2. 技术表名、`scene_id`、批次编号、原始来源表和调试辅助信息统一下沉到次级信息、折叠区或详情抽屉。
3. `置信度`、`字数`、`耗时` 等系统数值不再单独占据首屏高权重位置，必须和“这意味着什么风险 / 是否阻断下一步”一起展示。
4. 对业务支持 / 口径维护人员可见的首屏文案必须使用“当前阶段、当前待办、阻断原因、下一步动作”四类口径，不要求用户先理解技术处理流水线。

#### 材料接入

页面采用分步导入向导，固定四步：

1. 上传材料：支持口径文档、`SQL（结构化查询语言，Structured Query Language）` 样例、工单样例、截图附件。
2. 填写来源接入契约：标题、主对象、结果字段、默认时间语义、覆盖范围、敏感级别、来源表线索、材料出处。
3. 系统初判：展示候选术语、候选业务场景、候选字段、证据片段和解析疑点。
4. 人工补缺：缺少结果字段、默认时间不清、覆盖边界缺失、高敏等级不明确时，支持当场补齐或转缺口任务。

#### 解析抽取

页面展示机器抽取结果与人工确认动作，重点不是“解析成功”，而是“解析是否可信”。主区固定展示：

1. 候选业务场景与候选方案资产。
2. 标识对象、业务术语、字段概念与时间语义。
3. 来源表、来源字段、候选表间关联和证据片段。
4. 未确认关系、解析疑点与冲突说明。
5. 进入资产建模的下一步动作。

解析抽取阶段默认采用 `建图模式`。左侧优先展示任务级隔离的 `Candidate Entity Graph（候选实体图谱）`，节点类型固定为 `Candidate Scene（候选场景）`、`Candidate Plan（候选方案资产）`、`Candidate Evidence Fragment（候选证据片段）`、`Identifier（标识对象）`、`Field Concept（字段概念）`、`Business Term（业务术语）`、`Time Semantic（时间语义）`、`Source Table（来源表）`、`Column（字段）` 和候选表间关联；右侧面板负责当前节点 / 关系详情、置信度、证据片段、接受 / 驳回 / 合并 / 拆分等人工确认动作。

解析抽取页在桌面端固定为 `左侧活图谱主画布 + 右侧 Inspector（检查面板） + 底部 Console（控制台） / Trace（追踪）`。动态规则固定如下：

1. 导入开始后，左侧画布默认保持干净空态，只提示“正在等待首批实体”；图中的主角固定为候选实体和候选关系，不先摆大文档节点或总览卡片。
2. 后端通过 `SSE（服务端事件流，Server-Sent Events）` 按批次推送 `Graph Patch（图谱增量补丁）`，前端只对当前补丁新增或更新的节点、关系做局部更新，不允许整图重刷。
3. 右侧 `Inspector（检查面板）` 默认显示当前抽取进度与最近新增实体 / 关系；一旦用户点击节点或边，右侧立即完整切换为当前对象详情，后续新补丁不得抢占当前焦点。
4. 新节点进入时采用短时淡入与轻微放大回落，新关系进入时采用一次性路径描边或脉冲高亮；低置信度对象使用琥珀色与虚线表达“待确认”，不用红色失败态污染画布。
5. 图谱增长遵循“局部热启动、整图缓变”：新节点优先落在关联节点附近，旧节点只允许轻微让位；候选实体若后续被归一或合并，先标记为 `merged（已合并）` 或 `superseded（被替代）` 并弱化显示，直到导入完成后再统一收敛，不在导入中做硬删除。
6. 导入完成后不立刻跳离当前页；顶部阶段条保持“导入完成”，左侧保留稳定后的候选图，右侧默认态从“最近新增”切换为“最终摘要 + 待确认项”，再进入质量判断与资产建模。

#### 资产建模

页面负责把解析出来的结构落成业务场景、方案资产、输出契约、契约视图、覆盖声明和策略对象。页面不直接暴露原始 `JSON（JavaScript 对象表示法，JavaScript Object Notation）`，统一使用向导式结构化表单降低操作门槛。

资产建模阶段主动弱化全屏图谱，以**结构化表单作为第一交互界面**，重点编辑 `Scene（业务场景）`、`Plan（方案资产）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Output Contract（输出契约）` 等逻辑资产。在这里，用户定义的是“业务期望输出什么逻辑字段”。

页面固定区块如下：

1. **主编辑区**：按资产类型分页签展开结构化表单。
2. **右侧辅助视图（按需呼出）**：如果用户需查阅在“解析抽取”阶段得到的图谱事实或证据片段，可呼出浮动或折叠面板，不再强制占用屏幕空间。
3. **底部校验区**：展示当前建模结果的基础闭环检查。
4. **右上角状态提示**：实时反馈填写完整度。

关键交互如下：

1. 所填字段为业务维概念；下一步才会拿这些概念去与物理库碰撞。

#### 推理校验

推理校验页面承接主方案 §6.1 步骤 4 的推理构建结果，让治理人员在资产发布前确认推理资产的正确性、完整性和可解释性。该页面在知识生产链路中位于资产建模之后、元数据对齐之前，衍接的核心动作是：已经建好了逻辑资产和业务规则，现在要校验这些规则能否推出正确的推理结论。

页面采用双栏模式，固定三区：

1. **左侧推理结果区**：按推理规则分组展示候选 `Inference Assertion（推理结论）`，每条展示结论类型（标识收敛、时态收敛、关系补全、候选场景收敛）、主语、谓语、宾语 / 目标值、置信分值、适用范围和复核状态。支持按结论类型、置信分值和复核状态筛选。
2. **右侧推理链区**：选中某条推理结论后，右侧展示对应的 `Inference Chain（推理链）`，以步骤图方式展示输入事实 → 规则命中 → 中间节点 → 最终结论的解释链路。每个步骤可点击展开详情，查看证据引用和规则原文。
3. **底部冲突与门禁区**：展示推理冲突列表（确定性推理冲突直接阻断，置信度推理冲突降级为候选）、置信门禁命中结果（`Inference Confidence Record（推理置信记录）` 摘要）和推理适用边界摘要（`Inference Scope Profile（推理适用边界）` 的业务域范围、时间范围和禁止场景）。

推理校验操作固定为：

1. 确认推理结论（标记为已确认，允许进入下一步）
2. 驳回推理结论（标记为已驳回，需说明原因，退回资产建模或规则调整）
3. 标记为待人工复核（挂起当前结论，不阻断其他结论的确认）
4. 查看推理适用边界（展开 `Inference Scope Profile` 详情）

关键交互如下：

1. 左侧推理结果区按“阻断类冲突 → 低置信候选 → 已确认结论”排序，确保问题优先被看到。
2. 推理链步骤图中的证据节点可直接点击查看 `Evidence Fragment（证据片段）` 详情，不跳离当前页面。
3. 冲突检测结果为空时，底部冲突区显示“当前批次无推理冲突”并自动折叠。
4. 所有推理结论确认或挂起后，阶段状态条自动更新为“可进入元数据对齐”。

#### 元数据对齐

页面基于“资产建模”所沉淀的逻辑字段及“材料接入”登记的线索，直连底层数据字典进行“找表、找物理字段、看周期、对口径”操作，最终闭环生成正式来源契约。

页面核心交互形式为**两列表单或映射连线**：

1. **左侧（或上面部分）候选逻辑需求**：展示经过业务建模后必返的输出契约字段要求。
2. **右侧（或下面部分）物理落位区**：通过字典搜索或推荐，敲定关联的底层物理表名、字段名。
3. **缺口处理机制（核心交互）**：如果在底表中找不到对应的物理字段，交互上提供两个强阻断动作：
   - 必须强制退回“资产建模”步骤删除或修改此业务输出属性。
   - 或即刻派发一条 `Gap Task（缺口任务）` 提交给下游研发响应，页面状态进入“挂起 / 等待解决”。

#### 复核

复核页面承接主方案 §6.1 步骤 5 的业务、技术、合规三类复核，是材料接入到发布之间的必经环节，不归入发布中心。

页面采用 `工作台模式`，固定三区：

1. 左侧复核任务列表：按复核类型（业务复核、技术复核、合规复核）分组展示，每项显示任务编号、关联资产、责任人、截止时间、当前状态。支持按状态筛选（待分配、进行中、已通过、已驳回）。
2. 中部复核详情区：展示当前选中复核任务的完整上下文，固定包含以下区块：
   - 复核对象摘要：关联的业务场景、方案资产、覆盖声明、策略对象的核心信息。
   - 变更对比：当前草稿与上一已发布版本的差异视图，按字段逐项对比。
   - 证据支撑：当前资产关联的证据片段清单，支持点击查看原文锚点。
   - 缺口关联：与当前资产相关的未关闭缺口任务。
   - 复核意见区：历史复核意见时间轴和当前复核意见输入。
3. 右侧操作面板：当前复核任务的操作按钮和辅助信息。

复核操作固定为：

1. 通过
2. 驳回并说明原因
3. 转交其他复核人
4. 补充缺口任务

关键交互如下：

1. 业务、技术、合规三类复核可并行进行，但发布门禁要求三类全部通过后才能进入发布检查。
2. 驳回时必须填写原因，原因自动关联到对应资产的复核历史。
3. 复核完成后自动更新资产状态：三类全部通过则资产状态变更为"已复核"，任一驳回则保持"草稿"并标注驳回项。

#### 缺口任务

页面固定展示缺口任务列表、责任人、阻断等级、截止时间和当前处理状态。缺口任务必须能直接联动回材料、对象详情和发布阻断项。

知识生产台默认态从材料接入开始；空态显示“尚无待处理接入任务”；阻断态把缺口和阻断原因前置；无权限态不展示编辑按钮，只保留查看与追踪入口。

### 4.4 发布中心

发布中心是快照式发布与原子切换的界面化入口，不是单一发布按钮。页面固定为四区：

1. 顶部版本区：当前稳定 `snapshot_id（快照标识）` / `inference_snapshot_id（推理快照标识）` 配对、候选配对版本、发布状态、发布任务编号、发起人、创建时间、更新时间。快照配对关系必须以结构化方式展示（如“控制快照 v23 ↔ 推理快照 v23”），不允许只展示单一版本号。
2. 左侧变更清单：按新增、修改、替代、退役分组展示受影响治理对象；推理资产变更（`Inference Assertion（推理结论）`、`Inference Rule（推理规则）`、`Inference Scope Profile（推理适用边界）`）单独分组展示，不与控制资产混排。
3. 中部发布检查矩阵：逐项展示必需输出可兑现、契约视图完整、覆盖声明有效、策略对象可执行、推理资产可回溯、推理快照与控制快照配对一致、证据可追溯、来源契约可落位、元数据对齐通过、缓存刷新成功、索引刷新成功、版本切换可执行。
4. 右侧影响分析：展示会影响哪些业务场景、方案资产、契约视图、推理结论、审批模板和高敏场景，以及推荐动作。

底部标签区固定为：

1. 阻断问题
2. 版本差异
3. 灰度配置
4. 回滚预案
5. 发布说明

灰度配置标签页展示灰度发布范围（`gray_release_scope（灰度发布范围）`，可按领域、场景、角色、机构或流量分片配置）、灰度期间监控指标阈值（场景命中率、方案命中率、推理降级率、误路由率、高敏误放行风险）和灰度切换 / 冻结操作按钮。灰度期间任一核心指标超阈值时，该标签页前置红色警告并推荐冻结或回滚。

底部操作栏固定为：

1. 保存草稿
2. 发起复核
3. 执行发布
4. 执行回滚

执行发布与执行回滚必须使用明确的二次确认文案，说明“整体切换”和“整体回滚”的边界，不允许暗示局部回滚。

发布检查、快照切换和缓存 / 索引校验的运行输出统一放在底部可折叠区，默认不抢占主视线；只有出现阻断、失败或人工追查时再展开。

发布中心承接 Step 5 / 5。页面以 `工作台模式` 和 `双栏模式` 结合展示候选版本、发布检查矩阵、影响分析和回滚预案，不把发布操作做成单一按钮页。

### 4.5 运行决策台

运行决策台把一次请求的判断过程讲清楚，不直接把结果做成黑盒。

页面固定采用”左侧结果主体 + 右侧决策过程与辅助区 + 底部可折叠运行输出区”的结构：

1. 顶部输入区：支持输入业务请求，并补充角色、用途、是否导出、截止时点、请求字段等槽位。
2. 左侧结果主体：按业务场景、方案资产、契约、生效覆盖、策略结果、推理摘要、命中路径、证据摘要、风险说明和追踪信息结构化展示知识包。其中推理摘要区块固定展示：当前运行模式（`runtime_mode（运行模式）`，如完整推理 / 仅确定性推理 / 仅模板路径 / 仅澄清返回）、降级原因（`degrade_reason_codes（降级原因编码列表）`，以中文提示语展示）、命中推理结论数量及摘要、推理快照标识。若当前运行模式不是 `full_inference（完整推理）`，推理摘要区块必须前置橙色降级提示条，说明降级原因和对结果可靠性的影响。
3. 右侧决策过程区：以查询改写、槽位补齐、运行推理、场景召回、方案选择、覆盖判定、策略决策、路径解析、知识包输出 9 步卡片解释本次判断。其中“运行推理”步骤卡片独立展示推理断言视图摘要（断言类型、主对象、关联对象、置信分值、适用边界）、降级信息和推理链简要路径。
4. 底部运行输出区：展示调试输出、规则命中明细、推理链详细步骤、超时和回退说明，默认折叠。

页面默认不直接展示原始表字段和裸路径结果，而是先展示知识解释结果；只有在详情展开后，才显示来源约束、命中规则和原因编码。

运行决策台承接 Step 4 / 5，是运行样板问题、验证主链路和检查知识包输出的主工作台。右侧过程区的最终落点固定为 `Knowledge Package（知识包）` 的解释结果，不做”报告生成器”式表达。

右侧面板必须显式展示字段级安全结果：当前角色可见字段、默认脱敏字段、需审批字段和禁止返回字段，避免把安全门禁隐藏在结果之后。

#### 4.5.1 标识归一引导交互

顶部输入区必须对标识类输入提供结构化引导，不能把所有标识等价丢给后端检索。

1. 标识输入区固定提供标识类型下拉选择：客户号、户口号、证件号、协议号、批次号、产品号。用户选择标识类型后，输入框提示格式和示例。
2. 当用户直接在自然语言请求中包含标识但未选择类型时，系统在槽位补齐步骤自动识别并在右侧决策过程区的”槽位补齐”卡片内展示识别结果，包含：识别出的标识值、推断的标识类型、归一后的目标标识和归一依据。
3. 当同一请求中包含多个不同类型标识时，右侧面板高亮提示”检测到多入口标识”，并展示标识谱系映射关系，要求用户确认主入口标识。
4. 标识归一失败时（如证件号无法映射到客户号），结果区展示归一失败原因，并建议用户切换标识类型或联系支持人员。

#### 4.5.2 时间语义澄清交互

当请求涉及多时间语义冲突的场景时，系统必须在结果返回前触发澄清，不能静默选择默认时间。

1. 时间语义澄清以内联提示卡片形式出现在左侧结果主体的顶部，不使用模态弹窗打断操作流。
2. 澄清卡片固定展示：当前默认时间语义、可选备选时间语义（如申请日期、委托日期、资金日期、份额日期）、各时间语义对应的业务含义和覆盖差异。
3. 用户选择时间语义后，左侧结果区和右侧决策过程区同步刷新，覆盖判定和方案选择结果随之变化。
4. 若用户不主动选择，系统按默认时间语义返回结果，但在结果区顶部保留”当前使用默认时间：申请日期”提示条，支持随时切换。

#### 4.5.3 复合请求拆解交互

当系统识别出复合请求（多个主对象、多个不兼容时间语义或多个风险边界）时，不直接进入单一路径生成，必须返回拆解建议。

1. 复合请求被识别后，左侧结果区不展示知识包，而是展示 `Knowledge Decomposition Result（知识拆解结果）`，固定包含以下区块：
   - 拆解说明：解释为什么当前请求无法作为单一请求处理。
   - 子问题列表：每个子问题显示推断的业务场景、入口标识、时间语义和输出目标。
   - 场景候选：每个子问题对应的候选业务场景。
   - 合并提示：如果部分子问题可以合并，说明合并条件。
   - 澄清问题：需要用户进一步确认的问题。
2. 用户可以对子问题列表逐条操作：确认并分别执行、合并后重新提交、修改后重新提交。
3. 确认执行后，每个子问题独立进入运行主线，结果区按标签页分别展示每个子问题的知识包。
4. 右侧决策过程区在”查询改写”步骤明确标注”复合请求已拆解”，并展示拆解依据。

### 4.6 审批与导出

审批与导出面向审批人、申请人和合规角色，重点不是流程完整，而是让审批依据可见、导出结果可追踪。

页面固定包含三个层级：

1. 待审批列表：申请人、用途、涉及场景、风险等级、提交时间、超时状态。
2. 申请详情：申请摘要、命中策略、字段风险清单、机器建议、审批意见记录。
3. 导出记录：审批单号、文件指纹、导出状态、下载令牌状态、归档状态。

审批主操作固定为：

1. 同意
2. 拒绝
3. 要求补充说明
4. 改为脱敏导出

审批与导出继续保持结构化信息优先，不采用图谱主舞台。该页面重点是审批依据、字段风险、审批意见和导出追踪，不引导用户把审批页理解成图浏览器。

### 4.7 监控与审计

监控与审计不是运维附属页，而是治理风险前台。页面固定展示：

1. 运行指标：场景命中率、方案选择命中率、元数据新鲜度、发布通过率、审批队列积压量、图查询超时、接口错误率。
2. 告警列表：误路由、高敏字段暴露风险、元数据漂移、发布后契约失真、缓存与快照不一致。
3. 审计检索：支持按追踪编号、版本编号、审批任务、操作人、时间范围回查。
4. 链路回放：从一次请求、一次发布或一次审批回放整条路径。

监控与审计继续以指标、告警、时间轴和链路回放为主，不采用图谱优先布局；图谱只在特定异常的影响分析或证据回溯中按需嵌入。

### 4.8 评审原型映射

当前前端为评审和定型新增 3 张静态 `Prototype（原型）` 页面。它们不替代 7 个一级工作台，也不进入正式一级导航，只用于把“图谱驱动工作台”骨架先做对，再逐步拆回正式页面。

3 张原型与正式工作台的映射关系固定如下：

1. `/prototype/ingest-graph`：对应知识生产台的 Step 1 / 5 到 Step 2 / 5，覆盖材料接入、候选实体图谱抽取、关系确认、证据挂载和来源接入契约首轮校验。
2. `/prototype/modeling-dual-pane`：对应知识生产台的资产建模与元数据对齐步骤，重点覆盖资产建模、覆盖声明挂载、策略对象配置、输出契约和最终的元数据对齐校正。
3. `/prototype/runtime-publish`：对应运行决策台的 Step 4 / 5 和发布中心的 Step 5 / 5，页内切换运行验证与发布检查两种工作台视角，用于串起 `Knowledge Package（知识包）` 验证、发布检查矩阵、阻断项和回滚预案。

原型阶段统一遵循以下边界：

1. 只使用本地静态假数据，不接正式后端接口。
2. 只验证页面骨架、信息结构、视觉重心和工作流节奏，不替代现有正式页面功能。
3. 原型通过评审后，再拆回知识生产台、运行决策台和发布中心的正式实现。

### 4.9 全系统产品化升级原型

在既有 3 张结构评审原型之外，本轮新增一套面向业务演示、产品评审和后续高保真交互扩展的系统级静态 `Figma（设计协作工具，Figma）` 原型。该原型固定以“保留现有信息架构、升级为统一产品壳层和业务指挥台表达”为原则，不逐像素复刻当前 `React（前端组件框架，React）` 页面，也不额外发明并行业务链路。

本轮系统级原型固定覆盖 14 张高保真静态页面：

1. `首页总览 / 指挥页 / Default（默认态）`
2. `首页总览 / 风险聚焦 / Filtered（风险筛选态）`
3. `数据地图 / 模块入口 / Default`
4. `数据地图 / 资产图谱 / Default`
5. `知识生产台 / 模块入口 / Default`
6. `知识生产台 / 材料接入与解析 / Default`
7. `发布中心 / 模块入口 / Default`
8. `发布中心 / 发布检查与冻结快照 / Default`
9. `运行决策台 / 模块入口 / Default`
10. `运行决策台 / 查询验证与知识包解释 / Default`
11. `审批与导出 / 模块入口 / Default`
12. `审批与导出 / 审批详情与导出决策 / Default`
13. `监控与审计 / 模块入口 / Default`
14. `监控与审计 / 链路回放与审计检索 / Default`

页面分工固定如下：

1. 7 张模块入口页负责建立“整个系统如何分工、当前状态如何判断、建议从哪里进入”的全局认知。
2. 7 张代表深页负责体现真实操作深度，明确查询、接入、发布、审批和回放等核心动作如何落到产品工作台。
3. 首页没有独立二级导航时，允许用“风险聚焦态”承担代表深页角色，但它仍属于首页总览的正式表达，不视为额外业务模块。

本轮原型的视觉方向固定为“业务指挥台”而不是“研发样板页”：

1. 整体气质保持稳、清晰、可信，适合业务方和运营方演示，不使用宣传式品牌首屏。
2. 颜色体系以深蓝灰导航轨、冷白工作面、青绿色健康态、琥珀色待处理态、红色阻断态为统一语言。
3. 页面层级统一采用“模块页头 + 状态摘要条 + 主工作区 + 右侧判断区 + 证据区”的产品化骨架，减少碎片化卡片堆叠。
4. 每页只允许一个主舞台、一个辅助判断区和一个状态摘要区，避免多个视觉焦点竞争。
5. 首页、发布中心、审批与导出、监控与审计等页优先强化“当前状态与下一步”，图谱只在数据地图与影响分析相关场景承担主角色。

本轮原型的 `Figma（设计协作工具，Figma）` 文件结构固定如下：

1. `00 Cover`：封面页，说明原型范围、版本日期、适用评审场景。
2. `01 Principles`：设计原则、色彩、状态芯片、页面骨架和页面命名规则。
3. `02 Global Shell`：顶部横向轨道、状态摘要条、通用右侧信息面板和基础组件语言。
4. `03 Overview`：首页总览 2 张页面。
5. `04 Map`：数据地图 2 张页面。
6. `05 Production`：知识生产台 2 张页面。
7. `06 Publish`：发布中心 2 张页面。
8. `07 Runtime`：运行决策台 2 张页面。
9. `08 Approval`：审批与导出 2 张页面。
10. `09 Monitoring`：监控与审计 2 张页面。

`Figma` 画板命名与交付边界固定如下：

1. 命名统一采用“模块名 / 页面名 / 状态”的单一格式，不再并行维护“终版”“升级版”等过程性命名。
2. 本轮只交付桌面端 `1440` 宽度高保真静态页，不在同一轮里扩展移动端与点击连线原型。
3. 每张画板顶部需保留一条简短评审说明，说明“页面目标、主动作、评审重点”。
4. 每个模块页横向排版时，先放入口页，再放代表深页，保持系统级评审时的阅读顺序。
5. 页面内允许整理样例数据和状态标签，但不得改变现有业务语义、主任务顺序和正式中文口径。
6. 既有 `/prototype/*` 结构评审页继续保留，用于局部工作流定型；本轮系统级 `Figma` 原型负责统一产品壳层、信息层级和演示口径，两者不互相替代。

## 5. 关键深页设计

### 5.1 资产编辑页

资产编辑页面向业务场景和方案资产的维护，必须做到“带着门禁编辑”。页面固定四个页签：

1. 定义：名称、类型、主对象、默认时间、适用范围、不适用范围、责任人。
2. 运行约束：输入要求、时间语义、路由前置条件、回退动作、去重策略。
3. 输出与安全：必返字段、可选字段、脱敏字段、受限字段、禁止字段、契约视图、角色范围、审批模板。
4. 证据与来源：证据片段、来源契约、元数据对齐状态、证据可信度、复核状态。

页面右上角固定展示发布可行性提示：

1. 可发布
2. 可复核但不可发布
3. 阻断项数量

页面必须提供“当前草稿 vs 已发布版本”的差异视图，不允许盲改。

### 5.2 审批详情页

审批详情页是合规决策卡，固定三段：

1. 申请摘要：申请人、用途、角色、追踪编号、业务场景、方案资产、时间范围、导出诉求。
2. 字段风险清单：普通可见、默认脱敏、限制字段、禁止字段。
3. 审批建议：建议通过、建议拒绝、建议改为脱敏导出，以及触发策略、风险等级和审计要求。

审批动作区固定提供同意、拒绝、要求补充说明、改为脱敏导出四个按钮。

### 5.3 审计追踪页

审计追踪页用于回答“当时为什么这样判断”。页面以时间轴为主体，输入入口支持：

1. 追踪编号
2. 审批任务编号
3. 版本编号
4. 操作人

时间轴固定串起以下节点：

1. 请求进入
2. 场景命中
3. 方案选择
4. 覆盖判定
5. 策略决策
6. 审批动作
7. 导出动作
8. 审计归档

### 5.4 证据片段详情页

证据片段详情页回答"这个结论的依据从哪里来"。主方案要求所有正式输出具备证据可回溯能力，因此证据片段必须有独立的深页承载。

页面固定三区：

1. 证据元信息区：证据编号、证据类型（文档片段、`SQL（结构化查询语言，Structured Query Language）` 片段、工单记录、人工确认）、来源材料标题、来源页码或行号、提取时间、指纹值、确认状态（未确认、已确认、已驳回）、置信度分值。
2. 原文锚点区：展示证据片段的原始文本内容或 `SQL（结构化查询语言，Structured Query Language）` 代码，高亮关键语句。如果来源是文档，支持定位到原文上下文（前后各展示 3–5 行）；如果来源是 `SQL（结构化查询语言，Structured Query Language）`，以语法高亮方式展示完整代码段。
3. 关联对象区：展示引用此证据片段的所有治理对象（业务场景、方案资产、覆盖声明、策略对象等），每项显示对象名称、类型、当前状态和版本。

关键交互如下：

1. 点击关联对象可跳转到对应资产的编辑页或详情页。
2. 确认状态支持人工变更（未确认 -> 已确认 / 已驳回），变更时必须填写确认意见。
3. 证据片段被驳回时，关联对象自动收到"证据变更"提醒，影响其发布就绪状态。

### 5.5 覆盖矩阵页

覆盖矩阵页以二维矩阵展示样板场景的覆盖状况，回答"哪些场景覆盖到了、哪些还有缺口"。

页面固定三区：

1. 顶部筛选区：业务领域筛选、场景类型筛选、覆盖等级筛选（完整覆盖 / 部分覆盖 / 存在缺口）、仅看已发布。
2. 中部矩阵区：行为业务场景，列为覆盖维度（时间范围、产品范围、来源表可用性），单元格以颜色标识覆盖等级（绿色 = 完整覆盖，橙色 = 部分覆盖，红色 = 存在缺口，灰色 = 未评估）。点击单元格展开该分段的覆盖声明详情。
3. 底部统计区：覆盖完整率、缺口数量、最近变更的覆盖分段。

关键交互如下：

1. 点击矩阵单元格，在右侧展开对应覆盖声明的详情，包含覆盖分段列表、排除项、回退动作和证据引用。
2. 存在缺口的单元格支持直接创建缺口任务。
3. 矩阵支持按时间轴方向横向滚动，展示历史分段覆盖（如代发明细的 2004 前 / 2004–2013 / 2014+ 三段）。

### 5.6 影响分析页

影响分析页回答"如果这个对象变了，会影响什么"。在发布检查、元数据漂移和来源失效场景中承担核心解释职责。

页面采用 `建图模式`，固定三区：

1. 顶部输入区：选择影响分析起点（资产编号 / 版本编号 / 变更的来源表或字段），选择分析方向（正向影响 / 反向依赖）。
2. 中部影响图谱区：以变更起点为中心，按"字段 -> 关系 -> 方案资产 -> 业务场景 -> 契约视图"的级联方向展开影响路径。节点按影响严重程度着色（红色 = 必需输出不可落位、橙色 = 可选输出受影响、灰色 = 无直接影响）。
3. 右侧影响摘要面板：受影响资产数量、按严重程度分组的资产清单、推荐动作（阻断发布 / 降级运行 / 人工复核）。

关键交互如下：

1. 点击影响图谱中的节点，右侧面板展示该节点的具体受影响字段和原因。
2. 影响路径支持逐层展开和收起，避免一次性展开导致信息过载。
3. 从发布中心和监控与审计可直接跳转到影响分析页，并自动填入分析起点。

### 5.7 差异视图规范

资产编辑页（§5.1）和发布中心（§4.4）均要求差异视图。统一规范如下：

1. 差异视图采用并排双栏对比：左栏展示已发布版本，右栏展示当前草稿或候选版本。
2. 对比粒度为字段级：每个字段一行，变更字段高亮标识（新增 = 绿色背景、修改 = 橙色背景、删除 = 红色删除线）。
3. 结构化对象（如覆盖分段列表、输出字段列表）按列表项逐项对比，不做整体文本 diff。
4. 差异视图顶部固定展示变更统计：新增字段数、修改字段数、删除字段数、总变更项数。

## 6. 页面状态与交互原则

全站页面统一支持五类状态，并保持同一套视觉语言：

1. 默认态：展示当前最重要的信息，不把辅助技术信息铺满首屏。
2. 空态：说明当前没有数据、下一步可以做什么。
3. 阻断态：说明阻断原因、受影响对象、建议动作和责任人。
4. 部分覆盖态：明确说明覆盖边界、缺口位置和可选回退动作。
5. 无权限态：说明访问边界、受限原因和替代路径。

### 6.1 各工作台异常状态表现

五类状态必须在每个一级工作台有明确的表现定义，不允许只定义原则而不落地。

| 工作台 | 阻断态表现 | 部分覆盖态表现 | 空态表现 |
| --- | --- | --- | --- |
| 首页总览 | 页头下的一级状态卡红色高亮阻断发布数量和高敏告警；重点风险区前置到首屏可见区 | 不适用 | 局部业务区块显示"暂无数据"，一级状态卡和模块入口保持可用 |
| 数据地图 | 元数据漂移节点以红色边框标识，左侧领域树节点显示警告图标；右侧检查面板展示漂移原因和受影响字段 | 覆盖视图中部分覆盖分段以橙色时间带展示，缺口分段以红色虚线展示，回退动作内联提示 | 导航树显示"当前领域尚无已发布场景"，中央主区引导进入知识生产台 |
| 知识生产台 | 缺口任务数量徽标标注在侧栏；材料接入步骤被阻断时，阻断原因前置到步骤条上方，后续步骤置灰不可操作 | 不适用 | 材料接入显示"尚无待处理接入任务"，提供"新建接入"按钮 |
| 发布中心 | 发布检查矩阵阻断项以红色标识，执行发布按钮禁用并展示"存在 N 项阻断，无法发布"；缓存刷新失败时底部控制台自动展开，展示失败原因 | 不适用 | 当前无候选发布包时展示"暂无待发布版本" |
| 运行决策台 | 图查询超时时左侧结果区展示"图查询超时，系统未返回过期结果"提示卡片，右侧过程区对应步骤标红；策略拒绝时结果区以红色警告条展示拒绝原因 | 部分覆盖时结果区顶部展示橙色提示条，说明覆盖边界和缺口位置，提供"查看覆盖声明"链接 | 输入区正常展示，结果区显示"请输入业务请求" |
| 审批与导出 | 超时审批任务以红色高亮，排在列表顶部；超时时长在任务卡片上累计显示 | 不适用 | 待审批列表显示"暂无待审批任务" |
| 监控与审计 | 活跃告警以红色卡片前置到顶部；元数据漂移告警展示受影响场景数量和推荐动作 | 不适用 | 告警列表显示"系统运行正常，暂无告警" |

交互原则固定如下：

1. 永远先讲业务语义，再讲技术对象。
2. 风险和阻断前置，不藏在提交后。
3. 所有不能做的地方都要说明原因和下一步。
4. 所有详情页都要强化版本感，能看到当前版本、已发布版本、草稿版本和最近变更。
5. 图谱只在依赖、影响、证据回溯三类问题上承担主角色，其余情况优先结构化信息。

### 6.2 跨工作台上下文传递

7 个一级工作台之间必须支持”带上下文跳转”，避免用户在离开当前工作台后丢失焦点对象。所有跨工作台跳转必须通过主方案 §16 定义的 `Workbench Context Package（工作台上下文包）` 传递，不允许绕过上下文包直接拼 URL 参数。以下按跳转链路逐条固定规则。

#### 6.2.1 跳转链路规则总表

| 跳转链路 | `lock_mode` | 必填上下文字段 | 允许重算 | 目标工作台加载行为 |
| --- | --- | --- | --- | --- |
| 数据地图 → 运行决策台 | `latest` | `scene_code` 或 `plan_code`，可选 `asset_ref`、`edge_id` | 是 | 按 `scene_code` / `plan_code` 预填顶部输入区，使用后端最新已发布快照加载场景和方案数据，用户补充剩余槽位后提交 |
| 运行决策台 → 审批与导出 | `frozen` | `trace_id`、`scene_code`、`plan_code`、`snapshot_id`、`inference_snapshot_id`、`requested_fields`、`purpose` | 否 | 严格按 `snapshot_id` + `inference_snapshot_id` 加载审批依据数据，禁止自动刷新或版本漂移；审批单关联的数据必须与运行决策台提交时一致 |
| 运行决策台 → 数据地图 | `latest` | `scene_code` 或 `plan_code` 或 `asset_ref`，可选 `coverage_segment_id`、`evidence_refs` | 是 | 按标识定位到对应节点或覆盖分段，使用最新已发布快照加载图谱数据 |
| 发布中心 → 影响分析 | `frozen` | `asset_ref`、`snapshot_id`、`inference_snapshot_id` | 否 | 必须按 `snapshot_id` + `inference_snapshot_id` 锁定分析基线，展示候选版本相对于当前稳定版本的影响范围；禁止使用最新态替代候选态 |
| 监控与审计 → 运行决策台 | `replay` | `trace_id`、`snapshot_id`、`inference_snapshot_id` | 否 | 按 `trace_id` 加载历史请求的完整决策上下文，复现当时的场景命中、方案选择、推理断言和策略决策；只读查看，不允许重新提交 |
| 发布中心 → 数据地图 | `frozen` | `asset_ref`、`snapshot_id`、`inference_snapshot_id` | 否 | 按指定快照对加载图谱，展示候选版本的资产关系和覆盖状态 |
| 数据地图 → 知识生产台 | `latest` | `asset_ref`，可选 `scene_code`、`plan_code` | 是 | 按标识定位到对应资产的编辑页面，使用最新状态加载 |
| 监控与审计 → 数据地图 | `replay` | `asset_ref`、`snapshot_id`、`inference_snapshot_id` | 否 | 按历史快照加载图谱，展示告警发生时刻的资产关系状态 |

#### 6.2.2 重算与历史复现边界

1. **允许重算的链路**：`lock_mode` 为 `latest` 的链路允许目标工作台使用最新已发布数据重新计算和刷新，`snapshot_id` 和 `inference_snapshot_id` 仅作为参考。典型场景：从数据地图发起新查询、从运行决策台跳回数据地图浏览当前状态。
2. **必须复现历史态的链路**：`lock_mode` 为 `replay` 或 `frozen` 的链路禁止使用当前最新数据替代历史态。典型场景：审计回放必须还原当时决策上下文、审批流程必须锁定提交时版本、发布影响分析必须基于候选快照而非最新态。
3. 当 `lock_mode` 为 `replay` 或 `frozen` 时，`snapshot_id` 和 `inference_snapshot_id` 为必填字段，缺失时目标工作台必须展示错误提示，不得静默降级为 `latest` 模式。

#### 6.2.3 目标工作台加载规则

1. 目标工作台收到上下文包后，按 `lock_mode` 决定数据加载策略：`latest` 调用标准数据接口获取最新态；`replay` 调用历史快照接口按 `trace_id` 或快照对还原；`frozen` 调用版本锁定接口按快照对加载。
2. 加载成功后，按上下文包中的标识字段自动定位焦点对象：预填输入区、高亮图谱节点、展开检查面板对应页签或定位到指定覆盖分段。
3. 加载失败时（如快照已归档、资产已退役、`trace_id` 无法匹配），必须展示结构化错误提示，说明失败原因和替代路径（如”该快照已归档，可切换到最新版本查看”），不得静默回退。
4. 所有跨工作台跳转保持全局壳不变，只替换主内容区；跳转后浏览器历史记录支持正常回退。
5. 上下文包只携带标识和模式信息，不携带完整对象数据；目标工作台收到标识后必须自行从后端接口加载。
6. 第一阶段前端实现统一使用 `ctx` 查询参数承载 `Workbench Context Package（工作台上下文包）`；目标工作台先解析并校验 `ctx`，再决定是否加载数据，不允许页面各自散写查询参数名。
7. 当 `lock_mode` 为 `replay` 或 `frozen` 时，页面顶部固定展示锁定态提示条；若 `snapshot_id` / `inference_snapshot_id` 缺失或不成对，必须直接渲染结构化错误提示，不允许静默降级为 `latest`。
8. 数据地图作为目标工作台时，若 `ctx` 中存在 `asset_ref` 或 `coverage_segment_id`，目标页必须先切入 `lineage（资产图谱）` 模式，再执行焦点定位、高亮和路径叠加，不要求用户额外切换模式。
9. 数据地图的自动定位行为只允许在新一轮 `ctx` 首次生效时触发一次；一旦用户手动平移或缩放画布，同一轮上下文内后续刷新不得再次强拉视口回中心。
10. 数据地图在处理上下文定位失败时，按以下顺序降级：图中无目标节点时保留当前视口并给轻提示；资产存在但无图谱时保留 `lineage（资产图谱）` 空态；资产元信息不存在时才回退到 `browse（目录浏览）`；网络异常或超时始终保持当前模式并给重试入口。

### 6.3 推理资产前台可见规则

推理资产不应只在知识生产期的"推理校验"页面可见，而必须在发布后的运行消费和治理工作台中持续可见，支持回放、治理和持续监控。本节定义推理资产的前台稳定入口和展示规则，不新增一级导航。

#### 6.3.1 推理详情深页

新增一个稳定的 `推理资产详情页`（深页，非一级导航），作为推理资产在发布后持续可见的统一入口。该深页可从以下四个工作台进入：

| 入口工作台 | 进入方式 | 携带上下文 |
| --- | --- | --- |
| 数据地图 | 在业务视图图谱中启用推理节点过滤器后，点击 `Inference Assertion（推理结论）` 或 `Inference Chain（推理链）` 节点，右侧检查面板的"推理结论摘要"区块提供"查看推理详情"链接 | `asset_ref`（推理结论标识）、`snapshot_id`、`inference_snapshot_id`、`lock_mode`（默认 `latest`） |
| 运行决策台 | 左侧结果主体的"推理摘要"区块或右侧过程区的"运行推理"步骤卡片中，推理结论条目提供"查看推理详情"链接 | `asset_ref`、`trace_id`、`snapshot_id`、`inference_snapshot_id`、`lock_mode`（默认 `replay`，复现运行时推理上下文） |
| 发布中心 | 左侧变更清单的推理资产分组中，点击推理结论或推理规则条目提供"查看推理详情"链接 | `asset_ref`、`snapshot_id`、`inference_snapshot_id`、`lock_mode`（`frozen`，锁定候选版本） |
| 监控与审计 | 告警列表中的推理降级告警或审计追踪中的推理相关事件提供"查看推理详情"链接 | `asset_ref`、`trace_id`、`snapshot_id`、`inference_snapshot_id`、`lock_mode`（`replay`） |

#### 6.3.2 推理详情深页固定展示内容

推理资产详情页固定展示以下六个区块，不可省略：

1. **推理结论区（`Inference Assertion`）**：展示结论类型（标识收敛、时态收敛、关系补全、候选场景收敛）、主语、谓语、宾语 / 目标值、置信分值、适用范围、复核状态和关联的 `Inference Rule（推理规则）` 引用。
2. **推理链区（`Inference Chain`）**：以步骤图方式展示输入事实 → 规则命中 → 中间节点 → 最终结论的完整解释链路。每个步骤可点击展开详情，查看证据引用和规则原文。
3. **推理适用边界区（`Inference Scope Profile`）**：展示业务域范围、时间范围、来源范围、对象范围、禁止适用场景、`confidence_thresholds（置信阈值组）` 和 `gray_release_scope（灰度发布范围）`。
4. **运行模式区（`runtime_mode`）**：展示当前或历史运行时的推理模式（`full_inference（完整推理）` / `deterministic_only（仅确定性推理）` / `template_only（仅模板路径）` / `clarification_only（仅澄清返回）`）。当模式不是 `full_inference` 时，前置橙色降级提示条。
5. **降级原因区（`degrade_reason_codes`）**：以中文提示语逐条展示降级原因编码和对应的影响说明，复用 §3.3.1 的错误码映射规则。
6. **快照配对区（`snapshot pair`）**：展示当前推理资产所属的 `snapshot_id` / `inference_snapshot_id` 配对关系，并标注是否为当前生效版本。

#### 6.3.3 推理详情深页交互规则

1. 推理详情深页采用全页模式（非抽屉或弹窗），但保持全局壳层不变，浏览器历史记录支持正常回退到来源工作台。
2. 当 `lock_mode` 为 `replay` 或 `frozen` 时，页面顶部固定展示"历史回放态"或"版本冻结态"提示条，说明当前查看的是历史数据而非最新态。
3. 推理链步骤图中的证据节点可直接点击跳转到 §5.4 `证据片段详情页`，不跳离推理详情深页上下文。
4. 推理详情深页不承载编辑操作；编辑推理资产只能在知识生产台的"推理校验"页面进行。

## 7. 前端实现硬约束

### 7.1 文案与命名

1. 面向业务用户的主文案统一使用中文，不混用中英标题。
2. 前端状态机显示统一使用中文；英文状态编码只允许保留在调试、日志、编码复制区和开发实现层，不直接进入业务界面。
3. 技术缩写仅在详情、调试、编码复制区和辅助信息区保留。
4. 页面和文档中不使用“P0 / P1”一类内部优先级用语。
5. 页面与文档中不出现特定机构名称。

### 7.2 主流程交互

1. 主流程页面统一采用单列步骤流，不使用并排双主流程。
2. 草稿、方案、契约等对象以结构化表单编辑为主，不把原始 `JSON（JavaScript 对象表示法，JavaScript Object Notation）` 作为主入口。
3. 说明性文字弱化呈现，不盖过主任务控件、状态提示和风险提示。
4. 风险、阻断、部分覆盖和无权限信息必须前置显示，不能等提交后才提示。
5. 版本状态必须可见。涉及编辑、发布、审批和追踪的页面，都要能看到当前版本、已发布版本或相关版本边界。

### 7.3 导航与权限显隐

1. 一级导航固定为 7 个业务工作台全量常显，并固定放在顶部横向轨道中，不按角色隐藏导航项。
2. 二级导航固定放在左侧，只承担当前模块内导航职责，不承担跨模块切换职责。
3. 系统设置与个人协作入口进入全局工具区，不回流为一级业务导航。
4. 无权限工作台在顶栏置灰且不可点击，不采用页内再拦截。
5. 首页卡片和主操作允许按角色裁剪，但不得破坏“顶部一级导航 + 左侧二级导航”的结构稳定性。
6. 无权限时不展示误导性主操作按钮；如保留摘要卡片，必须提供边界说明和替代路径。

### 7.4 数据地图与可视化

1. 数据地图默认由顶部工具栏、页面内左侧导航树、中央主视图、右侧详情抽屉组成，并保持画布优先。
2. 图谱只在依赖、影响、证据回溯三类问题上承担主角色，其他场景优先结构化信息。
3. 节点与关系必须使用可区分的视觉编码，不允许只靠颜色表达状态或风险。
4. 数据地图的底部联动区默认折叠，按需展开，避免默认态信息墙。
5. 一期资产图谱画布必须支持节点点击高亮、边点击高亮、边详情面板、缩放平移、关系标签显隐、关系类型过滤和右侧详情联动；这些能力视为正式体验，不归类为原型特性。
6. 边（关系）必须在画布上与节点同等对待：点击边后右侧面板展示关系详情，关系类型可在顶部过滤器中筛选，不允许把关系退化为纯视觉装饰。
7. 推理结论和推理链在数据地图中作为可选展示节点类型出现在图谱画布中，不是默认展示，但用户可通过过滤器启用。
8. 当 `ctx` 携带 `asset_ref` 或 `coverage_segment_id` 进入数据地图时，页面必须自动进入 `lineage（资产图谱）` 模式，并默认启用 `路径高亮`，同时保留用户手动切回其他聚焦方式的入口。
9. 带上下文进入时，自动定位、自动居中和自动高亮必须以目标节点为锚点；若同时存在覆盖分段或关系路径，高亮路径只能作为节点后的第二层叙事，不得反客为主。
10. 若当前图谱中不存在上下文指定的目标节点，页面不得把全图统一压暗制造“伪高亮”；必须保留当前视口与正常对比度，并给出轻量提示说明未命中原因。
11. 自动定位只对单轮上下文触发一次；用户发生平移、缩放等主动操作后，当前轮次内后续刷新不得再次抢占画布方向盘。

### 7.5 前后端契约接入

1. 前端接口路径的正式真源固定为后端 `SpringDoc（接口文档生成框架，SpringDoc OpenAPI）` / `OpenAPI（开放接口描述规范，OpenAPI Specification）` 输出的 `/v3/api-docs`，不再允许以页面源码中的手写路径字符串充当事实契约。
2. 前端必须保留可再生的 `OpenAPI` 类型快照，并通过统一契约层向页面暴露路径常量或路径构造函数；页面、状态仓库和适配器不得继续散写 `/scenes`、`/graphrag/query` 一类裸路径。
3. 若后端接口路径、路径参数名或接口增删导致前端交互发生变化，必须先更新 `OpenAPI` 快照，再处理页面实现；未完成快照更新的改动不得视为完成。

## 8. 未实现能力标注规范

1. 未实现菜单或入口保留时，统一显示 `[后续建设]` 标签。
2. 未实现页面必须同时给出功能目标、预计阶段、当前替代方式。
3. 已实现页面不得出现“当前页面已实现”一类冗余提示。
4. 可选项中如包含未接入能力，必须显式标注“暂未接入”。

## 9. 文档同步与评审规则

前端界面方案、实现硬约束、未实现能力标注、评审规则和变更留痕统一以本文为准；主方案 [知识图谱与数据地图方案](system-design.md) 继续负责对象、边界、规则和运行主线。

以下情况必须同步更新本文：

1. 一级导航、左侧模块内导航、全局工具区或角色可用性规则变化。
2. 核心页面结构、关键页签、关键状态或主交互变化。
3. 中文命名词典、状态文案、按钮文案或拒绝提示语变化。
4. 数据地图、发布中心、运行决策台、审批与导出的主展示逻辑变化。
5. 前端实现硬约束、未实现能力标注、技能协作顺序或评审口径变化。
6. 系统级 `Figma（设计协作工具，Figma）` 原型的页集范围、页面命名、统一壳层或交付边界发生变化。

其他同步规则如下：

1. 任何新增接口或接口行为变更导致前端交互变化，必须同步 `SpringDoc（接口文档生成框架，SpringDoc OpenAPI）` / `OpenAPI（开放接口描述规范，OpenAPI Specification）` 输出或相关补充说明。
2. 任何新增前端 API 调用若未接入统一契约层，视为违反前端实现硬约束；评审时优先按“契约漂移风险”处理，而不是按普通代码风格问题处理。
3. 前端评审类请求优先使用 `web-design-guidelines`，再按项目口径执行 `frontend-design-governance`。
4. 外部技能接入、替换、升级或下线，属于前端治理决策，必须在本文留痕。
