# 一、特性概述

## 1.1 背景

代发批次结果与代发明细使用同一业务域，但主对象不同。业务侧在很多场景下只关心“某个代发批次成功了多少笔、总金额是多少、状态是否完成”，并不需要逐笔明细。如果仍然复用代发明细场景，会导致主对象、默认输出和策略门禁都发生漂移。

## 1.2 目标

本特性用于把“代发批次结果查询”独立成正式样板场景，支持按代发批次号、代发协议号和时间范围查询批次汇总结果，并返回批次级状态、总笔数、总金额和成功统计。

# 二、指标

1. 批次级查询不得误路由到明细场景，主对象识别准确率不低于 95%。统计口径：分母为人工标注为批次级问题的请求数，分子为正确命中批次场景且未误落到明细场景的请求数，统计窗口为首发回放集和灰度期间请求。
2. 成功笔数、成功金额、总笔数、总金额四项核心结果字段的契约兑现率达到 100%。统计口径：分母为应返回上述四项字段的批次结果请求数，分子为知识包中四项字段均按正式契约返回的请求数，统计窗口为首发回放集和灰度期间请求。
3. 批次级结果导出时必须区分普通汇总与高敏扩展字段，避免默认暴露付款方敏感信息。统计口径：分母为批次级导出请求数，分子为未经审批仍返回付款方敏感扩展字段的请求数，统计窗口为首发回放集与灰度期间导出请求，目标值为 0。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“代发批次结果查询”一个场景。该场景的主对象是代发批次，不复用代发明细的主对象与输出契约。

## 3.2 特性图示

本特性在运行决策台中以“批次汇总结果”知识包形式呈现，在数据地图中重点展示批次对象、批次事件来源、覆盖声明与策略对象的关系，不展开员工级明细路径。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为“运行决策台”；批次与来源关系的浏览承载在“数据地图”；批次覆盖差异查看复用 `frontend-workbench-design.md` §5.5 覆盖矩阵页，涉及导出审批时进入 §5.2 审批详情深页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：代发批次、批次汇总结果、批次来源事件、覆盖声明、策略对象和追踪信息。
- 默认时间语义：批次建立日期；如切到接收日期或完成日期，必须在结果中说明口径变化。
- 覆盖范围和缺口：覆盖批次级汇总结果与批次级覆盖说明；不覆盖员工逐笔明细与衍生财务核算。
- 策略或审批边界：普通汇总查询可直接返回；付款方账户、费用细项和大范围导出默认进入审批或字段裁剪。
- 前端入口、详情页和跳转链路：主入口为运行决策台；批次对象和来源事件关系在数据地图下钻；导出审批跳转审批详情页。
- 接口或数据边界：依赖 `Scene Search API（场景搜索接口）` 与 `Knowledge Package API（知识包接口）`；不得复用明细场景的输出集合。
- 存储落位边界：批次级场景、方案、覆盖声明和策略对象落关系型控制库；批次与来源事件关系投影到数据地图；导出审计和访问日志落审计 / 日志系统。

1. 输入入口以代发批次号为主，允许补充代发协议号、付款人客户号和建立日期范围；若用户只提供代发协议号，系统可以先召回候选批次，再进入本场景。
2. 主方案优先使用 `PDM_VHIS.T05_AGN_BCH_EVT` 作为批次级来源，返回总笔数、总金额、成功笔数、成功金额、代发状态、业务状态、完成时间和付款方信息摘要。
3. 默认时间语义建议使用批次建立日期；如业务明确要求接收日期或完成日期，系统要显式切换时间语义并说明口径变化。
4. 输出契约以批次级汇总字段为主，不直接承诺员工户口号、员工姓名等明细级字段；若用户要求逐笔信息，应引导切换到代发明细场景。
5. 覆盖声明按“批次表现有来源覆盖”表达，当前阶段不直接承诺沿用明细场景的历史分段口径；当底层来源不足以支持早期历史批次时，应按部分覆盖或缺口返回。
6. 策略对象允许普通汇总查询直接返回，但付款方账户、费用细项和大范围导出默认进入审批或字段裁剪。
7. 接口边界以 `Scene Search API（场景搜索接口）` 和 `Knowledge Package API（知识包接口）` 为主，知识包最少需要返回批次级汇总结果、覆盖说明、策略结果和追踪信息，不复用明细场景的输出集合。
8. 运行决策台应解释为何命中批次场景而非明细场景，并在结果中附带主对象说明，减少业务误解。
9. 数据地图需要支持展示批次对象与批次来源事件、代发协议对象、批次级策略对象和批次级覆盖声明的关系。
10. 本场景的边界是“批次级汇总结果”。它不负责逐笔员工发放明细，也不负责代发增值税等衍生财务核算场景。
