# 一、特性概述

## 1.1 背景

公司代发是当前样板域中最典型的历史分段场景。业务侧既可能按客户查询代发明细，也可能先查代发协议号再查明细；同时又存在“2004 年前不可查、2004-2013 多历史源、2014 年后主表稳定”的明显覆盖分层。如果不把这些规则单独固化，本项目无法验证 `Coverage Declaration（覆盖声明）`、历史 `Plan（方案资产）` 拆分和路径解释是否成立。

## 1.2 目标

本特性用于把“代发明细查询”落成正式样板场景，支持从客户号、公司户口号或代发协议号进入，按交易日期返回代发明细，并明确区分当前明细、历史明细和不可查询区间。

# 二、指标

1. 2014 年及以后代发明细命中主表路径的路由准确率不低于 95%。统计口径：分母为人工标注属于代发明细场景且查询区间完全落在 2014 年及以后的请求数，分子为正确命中本场景且选择 2014 年后正式明细方案的请求数，统计窗口为首发回放集与灰度期间真实请求。
2. 2004-2013 年历史代发明细必须全部带出“部分覆盖或历史补齐”说明，不允许伪装成完整覆盖。统计口径：分母为查询区间命中 2004-2013 年历史分段的请求数，分子为结果中明确返回历史来源、覆盖等级和补齐提示的请求数，统计窗口为首发回放集与灰度期间真实请求。
3. 2004 年前请求必须稳定返回缺口说明或替代处理建议，误报为可查的比例为 0。统计口径：分母为查询区间触达 2004 年前的请求数，分子为错误返回“可直接查询”或伪装成完整结果的请求数，统计窗口为首发回放集与灰度期间真实请求，目标值为 0。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“代发明细查询”一个场景。该场景的主对象是代发明细，核心运行主线是“先收敛协议号，再按时间分段选择明细方案，再根据覆盖结果和策略结果返回知识包”。

## 3.2 特性图示

本特性沿用运行决策台与数据地图双承载：运行决策台负责解释本次查询为什么走当前或历史明细方案，数据地图负责展示协议号、覆盖分段、来源契约和证据片段之间的关系。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为“运行决策台”；关系追踪和覆盖解释承载在“数据地图 -> 路径视图 / 覆盖视图”；覆盖分段下钻复用 `frontend-workbench-design.md` §5.5 覆盖矩阵页，证据定位复用 §5.4 证据片段详情，命中审批时进入 §5.2 审批详情深页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：代发明细事件、代发协议号、来源分段说明、覆盖声明、策略对象和追踪信息。
- 默认时间语义：交易日期；不把接收日期、起息时间或批次完成时间作为默认值。
- 覆盖范围和缺口：覆盖 2014 年后正式明细、2004-2013 年历史补齐与 2004 年前缺口提示；不覆盖批次级汇总结果。
- 策略或审批边界：普通明细查询可直接返回；全历史、多员工或大批量导出默认转审批。
- 前端入口、详情页和跳转链路：主入口为运行决策台；覆盖分段和证据定位下钻到数据地图与证据详情；命中审批时跳转审批详情页。
- 接口或数据边界：依赖 `Scene Search API（场景搜索接口）` 与 `Knowledge Package API（知识包接口）`；错误态必须返回覆盖或策略类原因编码。
- 存储落位边界：场景、方案、覆盖声明、策略对象和输出契约落关系型控制库；历史来源关系与覆盖分段投影到数据地图；原始历史补齐日志和回放记录落审计 / 日志系统。

1. 输入入口支持公司客户号、公司户口号、代发协议号、员工户口号和交易日期范围；系统优先通过协议出款户口表或合作方协议信息快照把客户或公司户口号收敛成代发协议号。
2. 方案拆分至少包括两类：2014 年及以后当前明细方案、2004-2013 年历史明细方案；2004 年前只返回缺口说明，不建立可运行方案。
3. 当前明细优先来源于 `PDM_VHIS.T05_AGN_DTL`；历史明细可按年份和来源选择 `UNICORE_EPHISTRXP`、`ODS_C3A3_A2DPADTLP_YEAR`、`TMP_LLJ_AGN_TRX_DATA_0308_RES_PRM` 等历史来源，并明确数据可能缺数。
4. 默认时间语义固定为交易日期，不把接收日期或起息时间作为默认值；若用户明确要求其他日期语义，系统必须显式切换并说明影响。
5. 输出契约至少覆盖代发批次号、员工户口号、客户编号、交易金额、交易日期、交易时间、状态代码、处理结果错误码和来源分段说明。
6. 覆盖声明必须明确三段：2014+ 完整覆盖、2004-2013 部分覆盖、2004 前存在缺口；跨段查询时需要返回分段命中说明和缺口提示。
7. 策略对象应允许普通明细查询返回，但对“全历史、多员工、大批量导出”默认转审批，避免把历史补齐结果直接用于大规模外传。
8. 接口边界以 `Scene Search API（场景搜索接口）` 和 `Knowledge Package API（知识包接口）` 为主：前者负责召回代发明细场景，后者至少返回 `scene`、`plan`、`contract`、`coverage`、`policy`、`path`、`risk`、`trace` 八个字段块，并在错误时使用覆盖或策略类原因编码。
9. 运行决策台必须解释当前命中的方案、当前覆盖等级、当前来源分段和为什么没有走其他历史表；数据地图要能显示协议号、历史来源和覆盖分段的关联。
10. 本场景不负责代发批次汇总结果；批次级查询另立“代发批次结果查询”特性文档。
