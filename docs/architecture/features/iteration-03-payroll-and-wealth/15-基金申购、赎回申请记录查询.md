# 基金申购、赎回申请记录查询特性文档

> 迭代归属：迭代三公司代发与财富样板域  
> 来源主文档：`system-design.md` §14；`frontend-workbench-design.md` §4.2、§4.5、§5.2、§5.4；研究材料：`research/source-materials/sql-samples/06-口径文档现状-基金理财保险交易查询.sql`

# 一、特性概述

## 1.1 背景

财富类场景里最容易混淆的不是产品名，而是时间语义。基金申购、赎回申请记录关注的是“客户什么时候发起申请”，而不是“资金什么时候清算”或“份额什么时候成交”。若不把申请记录单独成场景，基金资金交易、分红、定投等动作会混进同一口径，导致查询结果前后不一致。

## 1.2 目标

本特性用于把“基金申购、赎回申请记录查询”沉淀成独立样板场景，按申请日期返回基金申请级记录，并明确与资金交易、分红记录场景分离。

# 二、指标

1. 申请记录场景与资金交易场景的误切换率控制在 5% 以下。
2. 申请日期作为默认时间语义的命中解释覆盖率达到 100%。
3. 产品代码、产品名称、申请金额、申请份额、申请渠道等核心字段的契约兑现率达到 100%。
4. 统计口径：以上指标以基金样板回放集与灰度期间请求统计；误切换率分母为人工标注为基金申请或基金资金场景的请求数，分子为错误命中相邻场景的请求数。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“基金申购、赎回申请记录查询”一个场景。场景主对象是基金申请事件，主入口是客户号、账户号和产品号，主时间语义是申请日期。

## 3.2 特性图示

运行决策台负责呈现申请记录知识包与时间语义澄清；数据地图负责展示基金申请场景、输出契约、时间语义选择器和基金定义来源之间的关系。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为“运行决策台”；多时间语义解释承载在左侧结果区的内联澄清卡片；证据与口径核对复用 `frontend-workbench-design.md` §5.4 证据片段详情，命中审批时进入 §5.2 审批详情深页。

## 4.2 功能说明

### 4.2.1 特性场景1

1. 输入入口支持账户号、客户号、产品号和日期范围；若用户给出“开户以来”之类宽范围请求，系统要先提示历史覆盖可能不完整。
2. 默认时间语义固定为申请日期，不把资金交易日期、资金清算日期或份额日期作为默认值；若用户明确要求其他日期，必须切换到其他场景或澄清后重算。
3. 主来源应优先使用基金申请事件相关表，返回申请日期、申请时间、基金代码、产品名称、基金管理公司、委托金额、委托份额、资金交易金额、费用金额、交易代码和交易渠道。
4. 输出契约只承诺申请记录相关字段，不承诺分红、成交净值和资金清算结果字段；这些字段另由“基金资金交易与分红记录查询”场景承接。
5. 对于定投、撤单、基金转换等申请类动作，系统可以通过交易代码进一步筛分子类型，但仍属于本场景内部的申请事件集合。
6. 覆盖声明应明确说明：当前阶段优先承接已重构和已确认来源的申请记录；历史库待补齐部分不宣称完整覆盖。
7. 策略对象允许普通查询返回，但大范围导出、推荐人等敏感或边界不清字段默认转审批或裁剪。
8. 接口边界以 `Scene Search API（场景搜索接口）` 和 `Knowledge Package API（知识包接口）` 为主，知识包必须保留申请日期口径说明，并通过原因编码区分未命中、部分覆盖和需审批情况。
9. 运行决策台必须显式提示“当前按申请日期口径返回”，并提供切换到资金交易场景的入口，减少日期口径混淆。
10. 本场景的边界是“申请级事件”。它不负责资金清算结果、不负责分红流水，也不负责 SA 理财或受托理财场景。
