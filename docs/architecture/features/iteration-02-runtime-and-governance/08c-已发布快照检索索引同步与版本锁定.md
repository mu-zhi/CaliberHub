# 一、特性概述

## 1.1 背景

运行侧如果要引入实验检索引擎，就必须先解决“实验索引从哪里来、按什么版本读、如何随发布和回滚同步”的问题。若实验索引直接读取草稿对象、直接读当前可变内容，或者在回滚时仍指向新版本，就会破坏当前项目最重要的版本边界：运行面只能消费 `PUBLISHED（已发布）` 快照，而且必须按 `snapshot_id（快照标识）` 锁定版本。

本专题只解决一个问题：如何把已发布快照摘要同步成只读 `Experimental Retrieval Index（实验检索索引）`，并让运行请求按 `snapshot_id（快照标识）` 读取对应索引版本。它不负责运行排序策略，也不负责实验评测结论。

## 1.2 目标

本特性用于建立“已发布快照摘要导出 -> 实验索引构建 -> `snapshot_id（快照标识）` 与索引版本绑定 -> 回滚 / 退役同步”的正式链路，保证实验检索能力始终服从正式发布边界。

# 二、指标

1. 门禁型指标：非 `PUBLISHED（已发布）` 对象进入实验索引次数为 `0`。统计口径：分子为进入实验索引的 `DRAFT（草稿）`、候选态、待复核态或未通过发布门禁对象数，分母为实验索引同步涉及的对象总数，统计窗口为首轮发布任务与回放集。
2. 门禁型指标：`snapshot lock success（快照锁定成功率）` 达到 `100%`。统计口径：分母为启用实验检索的运行请求数，分子为成功命中与请求 `snapshot_id（快照标识）` 一致索引版本的请求数，统计窗口为首轮灰度期。
3. 门禁型指标：回滚后索引版本同步成功率达到 `100%`。统计口径：分母为触发实验索引回滚或退役的发布任务数，分子为同步完成索引版本切换且运行请求未读到旧版本的任务数，统计窗口为首轮灰度期与回放演练。
4. 效果型指标：已发布快照可实验检索覆盖率达到 `100%`。统计口径：分母为标记为允许实验检索的 `PUBLISHED（已发布）` 快照数，分子为已经生成并通过版本映射校验的实验索引快照数，统计窗口为首轮样板域发布集。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“已发布快照索引同步与版本锁定”一个场景。它负责把正式快照导出成只读实验索引，并为运行链路、回放链路和影子模式提供版本一致的只读基础。它不改变正式发布流程的真源，也不负责实验检索结果如何参与排序。

## 3.2 特性图示

结构固定为“正式 `PUBLISHED（已发布）` 快照 -> 摘要导出 -> `Experimental Retrieval Index（实验检索索引）` 构建 -> 索引版本清单绑定 `snapshot_id（快照标识）` -> 运行请求按版本读取 -> 发布 / 回滚 / 退役时同步刷新”。

# 四、特性说明

## 4.1 原型（非必要）

前端不新增独立业务工作台。相关状态继续由“发布检查、灰度发布与回滚”工作台和“监控审计与影响分析”工作台承载；当前页面入口分别为发布中心相关页面和 `frontend/src/pages/MonitoringAuditPage.jsx`。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Version Snapshot（版本快照）`、`Experimental Retrieval Index（实验检索索引）`、`Index Manifest（索引清单）`、`snapshot_id（快照标识）`、`index_version（索引版本）`、`Canonical Entity（统一实体）`、`Evidence Fragment（证据片段）`。
- 默认时间语义：本场景只解释发布版本时间与索引生效时间，不改写业务查询时间语义。
- 覆盖范围和缺口：首轮只覆盖 `代发 / 薪资域` 已发布场景、已发布统一实体和已发布证据摘要的实验索引构建；不覆盖草稿索引、不覆盖候选态索引、不覆盖跨业务域一次性全量重建。
- 策略或审批边界：只有已通过发布门禁并标记允许实验检索的快照才能进入同步；回滚和退役必须由正式发布动作驱动，不能由实验引擎自行切换。
- 前端入口、详情页和跳转链路：发布时的索引状态由“发布检查、灰度发布与回滚”查看；运行后的索引命中状态和版本错配告警由“监控审计与影响分析”查看；不新增业务详情页。
- 接口或数据边界：输入为已发布快照摘要、统一实体摘要、证据摘要和索引目标配置；输出为索引版本清单、快照到索引版本映射、构建状态和回滚结果；不输出业务知识包。
- 存储落位或持久化边界：正式真源继续落关系型控制库与已发布快照；实验索引使用独立索引存储；`snapshot_id（快照标识）` 到 `index_version（索引版本）` 的映射和同步台账落关系型控制库；任何实验索引都不得反向改写控制库。

1. 实验索引的数据来源固定为正式 `PUBLISHED（已发布）` 快照摘要，而不是当前可变内容。进入索引的数据至少包括 `Scene（业务场景）` 摘要、`Plan（方案资产）` 摘要、`Evidence Fragment（证据片段）` 摘要、`Canonical Entity（统一实体）` 摘要和必要的引用锚点。
2. 索引导出时必须显式裁剪掉草稿态、候选态、待复核态和未通过发布门禁的对象；实验索引没有“先写进去、后面再删除”的宽松模式。
3. 每个 `snapshot_id（快照标识）` 只能绑定一个当前有效的 `index_version（索引版本）`。运行请求若未显式传入 `snapshot_id（快照标识）`，系统不得因为实验索引而自行猜测版本。
4. 发布动作、回滚动作和退役动作都必须同步刷新 `snapshot_id（快照标识）` 与 `index_version（索引版本）` 的绑定关系。正式发布成功但索引构建失败时，正式发布结果仍然有效，但该快照不得进入实验检索路径。
5. 索引构建链路必须记录“导出批次、构建批次、构建人、构建时间、源快照、索引版本、状态、失败原因”。这些记录属于运维与审计对象，不属于业务知识对象。
6. 统一实体层只允许以已发布成员归属结果进入实验索引。实验索引可以保存 `Canonical Entity（统一实体）` 与场景成员归属的只读摘要，但不能反向修正 `Canonical Membership（统一成员归属）` 或 `Canonical Relation（统一实体关系）` 真源。
7. 运行链路读取实验索引时，必须先锁定请求 `snapshot_id（快照标识）`，再按映射读取唯一有效的 `index_version（索引版本）`。若映射缺失、索引退役、构建失败或版本不匹配，运行面只能降级回正式主链路。
8. 本场景不负责运行候选合并，不负责离线评测与 `Shadow Mode（影子模式）`，也不负责知识生产期候选图回写；这些职责分别由 `08b`、`08d` 和 `02f` 专题承接。
