# 一、特性概述

## 1.1 背景

知识底座不是静态文档仓库，而是有发布、运行、审批、导出和回滚的动态系统。若没有独立场景承接告警、审计追踪和影响分析，系统出现元数据漂移、推理降级、误路由或高敏误放行时，只能依赖离线排查，无法形成可运维、可问责的治理闭环。

## 1.2 目标

本特性用于把运行告警、审计追踪、历史回放和影响分析整合到统一工作台中，支持从问题发现、链路回放到受影响资产定位的完整追查路径。

# 二、指标

1. 关键告警定位完整率达到 100%。统计口径：分母为进入监控总览的活跃高优先级告警数，分子为同时带有 `trace_id（追踪编号）`、快照对和受影响资产引用的告警数，统计窗口为近 30 天样板环境告警记录。
2. 影响分析结果结构化输出率达到 100%。统计口径：分母为人工发起的影响分析任务数，分子为已返回严重程度分组、影响资产数量和推荐动作的任务数，统计窗口为首轮样板域回放和真实诊断任务。
3. 审计回放历史态复现成功率达到 100%。统计口径：分母为以 `replay（回放态）` 或 `frozen（冻结态）` 进入的审计回放次数，分子为成功按原快照对还原上下文且未退化为最新态的回放次数，统计窗口为当前阶段审计抽查与缺陷复盘样本。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“监控审计与影响分析”一个场景。它面向运维、治理、审计和研发角色，负责回答“出了什么问题、为什么会这样、影响到谁、现在该怎么处理”。

## 3.2 特性图示

本场景由三类页面组成：监控总览、审计追踪页、影响分析页。监控总览回答现状，审计追踪页回答历史链路，影响分析页回答范围和动作。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“监控与审计”一级工作台，深页包括审计追踪页与影响分析页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：活跃告警、审计事件时间轴、影响分析任务、受影响资产集合、快照对与运行追踪编号。
- 默认时间语义：默认以事件发生时间、告警触发时间和回放时间组织本场景，不直接解释业务查询时间；涉及历史复盘时以传入快照对和事件时间为准。
- 覆盖范围和缺口：覆盖监控总览、审计追踪、历史回放、影响分析与推荐动作；不覆盖资产修复动作本身，也不替代发布中心或审批工作台执行正式操作。
- 策略或审批边界：本场景可以解释高敏风险和审批链路，但不得在此场景直接放宽字段可见范围或绕过审批。
- 前端入口、详情页和跳转链路：入口为“监控与审计”一级工作台；深页包括审计追踪页和影响分析页；可从“发布中心”“运行决策台”“数据地图”带 `replay / frozen` 上下文进入。
- 接口或数据边界：本场景消费告警摘要接口、审计追踪接口、影响分析接口与异步事件查询接口；所有历史态回放必须通过正式上下文包和后端快照查询接口完成。
- 存储落位边界：告警事件、审计事件和影响分析结果摘要落关系型控制库或事件库，运行日志保存在日志系统，图谱仅承接受影响关系的浏览投影。

1. 监控总览必须前置展示活跃告警、元数据漂移、推理降级、发布冻结和高敏风险，不允许埋在深层分页里。
2. 每条告警至少要关联追踪编号、快照对、受影响场景、受影响方案或来源对象，保证后续可以直接跳转。
3. 审计追踪页固定用时间轴串联请求进入、场景命中、方案选择、覆盖判定、策略决策、审批动作、导出动作和归档结果八类节点。
4. 影响分析页必须支持以资产编号、版本编号或变更的来源表字段作为起点，按正向影响或反向依赖展开路径。
5. 影响图谱中的节点要按严重程度着色，右侧摘要面板必须给出受影响资产数量、影响等级和推荐动作，避免只有图没有结论。
6. 从发布中心、数据地图和运行决策台进入本场景时，必须支持 `replay（回放态）` 或 `frozen（冻结态）` 上下文，确保看到的是问题发生时的真实版本。
7. 审计与监控边界固定为“解释问题和影响”。本场景不直接修复资产、不直接改策略，也不跳过复核私自发布。
8. 所有异步事件至少要保留事件名、发生时间、操作人、任务编号、状态和原因编码，供本场景消费。
9. 本场景必须能区分“当前仍在发生的问题”和“已经归档的历史问题”，不把历史异常当成实时异常继续报警。
10. 当影响分析从来源字段或快照对象发起时，系统必须明确区分“正向影响链”和“反向依赖链”，不允许在同一张图里混合解释而无分组。
11. 本场景的边界是“可观测性、回放与影响解释”。它不直接执行发布回滚、不直接重跑知识包，也不直接修改上游治理对象。
