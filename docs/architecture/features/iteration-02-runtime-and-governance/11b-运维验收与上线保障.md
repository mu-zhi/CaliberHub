# 一、特性概述

## 1.1 背景

当前 `NFR（非功能需求，Non-Functional Requirements）` 基线、测试矩阵、样板回放集、迁移切换与上线回退规则已经在主方案中明确，但还没有独立特性文档承接。若继续把这些约束拆散到发布中心、监控审计和样板场景各自的描述里，团队只能知道“怎么发版”，却无法稳定回答“什么时候算达到上线资格、出问题时按什么标准回退”。

## 1.2 目标

本特性用于把上线前门禁、样板回放、运行保障、切换步骤和回退规则收口到同一个横切治理场景中，让候选快照、候选推理快照和运行面都能在统一的验收与回退标准下推进。

# 二、指标

1. 首发强制 `NFR` 达标率达到 `100%`。统计口径：分子为通过全部强制 `NFR` 项的候选版本数，分母为进入上线评估的候选版本总数，统计窗口为单次发布窗口。
2. 样板回放集通过率达到 `100%`。统计口径：分子为回放结果与预期一致的核心样板数，分母为纳入本次发布门禁的核心样板总数，统计窗口为单次发布前验证批次。
3. 上线回退演练或真实回退执行成功率达到 `100%`。统计口径：分子为完成成对快照回退且验证通过的回退次数，分母为触发的回退总次数，统计窗口为当前迭代上线准备周期。
4. 未授权高敏暴露次数为 `0`。统计口径：分子为灰度、正式发布与回放验证中发现的高敏误放行次数，分母为全部高敏相关验证样本数，统计窗口为单次发布窗口。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“运维验收与上线保障”一个横切治理场景。它不替代发布中心的执行交互，也不替代监控与审计的历史追踪，而是负责定义统一上线资格、统一验证动作、统一回退边界和统一责任分工。

## 3.2 特性图示

前端承载采用“发布中心上线准备区 + 回退预案区 + 监控与审计回放入口”的结构。主图示建议同时包含三类图：上线门禁矩阵、样板回放覆盖图、切换 / 回退流程图。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载页面为“发布中心 -> 上线准备 / 回退预案”标签区；“监控与审计”提供运行期回放、告警和回退后验证入口。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`NFR Baseline（非功能需求基线）`、`Test Matrix（测试矩阵）`、`Replay Set（样板回放集）`、`Alert Rule（告警规则）`、`Rollback Trigger（回退触发条件）`、`Migration Plan（迁移与切换方案）`。
- 默认时间语义：不适用；本场景定义发布前后统一门禁和回退规则，不直接执行业务查询。
- 覆盖范围和缺口：覆盖上线资格判断、样板回放、运行保障、切换步骤和回退验证；不覆盖单次发布按钮交互、不覆盖单个业务场景字段设计。
- 策略或审批边界：本场景定义上线与回退门禁，不替代业务审批；高敏验证是门禁的一部分，但高敏审批流程仍由业务场景和审批链路承接。
- 前端入口、详情页和跳转链路：入口为“发布中心 -> 上线准备 / 回退预案”；监控与审计中的异常链路可回跳到本场景查看触发门禁和回退标准。
- 接口或数据边界：本场景输入为候选快照、候选推理快照、验证结果和运行基线，输出为上线资格结论、阻断原因和回退动作要求。
- 存储落位边界：门禁结果、回放记录、切换记录和回退记录落关系型控制库与审计链路；运行监控结果由监控系统和审计事件提供，本场景消费但不替代其真源。

1. `NFR（非功能需求，Non-Functional Requirements）` 基线必须在正式上线前给出可验证门槛，至少覆盖性能、可用性、安全、审计完整性和降级可解释性。任何一项强制 `NFR` 未达标，都不得进入正式切换。
2. 测试矩阵是上线资格的一部分，不是附加说明。单元、契约、集成、权限 / 高敏、回放和灰度验证都必须明确各自覆盖范围；未进入矩阵的能力不能宣称已具备上线保障。
3. 样板回放集必须固定首批核心样板，包括但不限于代发明细、零售基础画像、密码修改日志和断卡排查。回放时至少比较场景命中、方案选择、覆盖判定、策略结果、推理降级和高敏控制；任何未解释差异都不得直接放行。
4. 告警必须按“阻断发布、灰度观察、运行告警”三层分级。涉及误路由、高敏误放行、审计链路缺失或关键接口异常的告警，必须能够直接联动冻结、阻断或回退判断，而不是只在监控页提示。
5. `Runbook（运行手册）` 必须为每类关键问题定义响应角色、定位步骤、升级路径和关闭要求，不能只依赖口头经验。`Runbook` 需要随版本一起维护，确保同一类问题在灰度和正式发布窗口内有一致处置方式。
6. 上线前准备必须明确主键生成、历史资产补齐、候选快照配对、灰度样本准备和切换前检查项。少一项都不得进入正式切换，以避免把运行保障问题延后到线上暴露。
7. 上线切换步骤必须写清执行顺序、观察窗口和成功判据。回退单位固定为成对快照，不允许只切控制资产或只切推理资产；切换成功必须同时满足核心接口、样板回放和审计链路三类验证通过。
8. 上线回退触发条件必须结构化定义，至少覆盖误路由率、推理降级率、高敏误放行、关键接口异常和审计链路缺失。每项触发条件都要明确对应动作是冻结灰度、维持旧版本还是直接回退，以及谁有权拍板。
9. 对检索实验侧车的回退顺序必须固定为“停灰度 -> 停影子模式 -> 禁用适配器”，并且要求 `snapshot mismatch（快照错配）`、`Policy false allow（策略误放行）`、实验索引版本错配三类问题都能直接触发该顺序，不允许实验链路继续带着不一致版本在线运行。
10. 回退执行后必须进行回退后验证，至少确认正式接口恢复、核心样板回放恢复、审计链路完整、运行读源与版本显示一致。未完成验证不得宣布事故处置结束。
11. 本场景的边界是“统一定义上线资格、回退标准和运维保障门禁”。它不替代“发布检查、灰度发布与回滚”中的页面执行动作，不替代“监控审计与影响分析”中的运行期解释，也不定义单个业务场景的字段、审批或知识包输出。
