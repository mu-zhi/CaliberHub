# 一、特性概述

## 1.1 背景

数据地图是本项目最核心的治理解释界面。当前高可见层并不是从零开始构思，而是已经存在前后端先行实现：前端已有图谱画布、详情面板、影响分析、日志与运行决策台跳转骨架，后端同时存在 `/api/assets/*` 与 `/api/datamap/*` 两条接口路径。当前最需要解决的问题不是“是否可做”，而是把“代码已经承载的事实”与“正式文档承诺的边界”重新对齐，避免后续计划、测试与评审继续建立在失真的前提上。

## 1.2 目标

本特性用于把正式治理对象稳定投影到数据地图工作台，支持业务视图、路径视图、覆盖视图三种主视图，并让节点详情、关系详情、覆盖分段和证据回溯在同一上下文内完成。当前首轮以 `代发 / 薪资域` 已发布场景为样板，数据地图只承接“已发布快照投影、覆盖追踪、运行定位”三类职责；同时需要把现有实现收口为一条可治理、可验收的正式主路径。

1. 统一数据地图高可见层的正式场景定义，使文档能够承载当前已实现能力。
2. 明确上下文带入后的自动切图谱、自动定位、单轮抢占与降级规则。
3. 明确节点详情、关系详情、影响分析和跨工作台跳转之间的责任边界。
4. 明确 `/api/datamap/*` 是高可见层正式主路径，`/api/assets/*` 仅作为兼容过渡输入。
5. 明确本文件只承接“数据地图浏览与覆盖追踪”的工作台行为；跨场景统一实体真源、领域级图谱融合与运行复用边界由 `10b-跨场景统一实体层与领域级图谱融合.md` 单独承接，避免在本文件内混入另一套治理主线。

# 二、指标

1. 图谱返回体必须始终包含当前快照口径和关系分组口径。
2. 业务视图、路径视图、覆盖视图之间的切换必须保留当前焦点场景与当前过滤条件。
3. 所有进入右侧检查面板的节点和关系都必须能回到对应正式治理对象。
4. 图谱投影与关系型控制库中的已发布快照一致性必须达到 100%，不接受图谱侧单独维护真源。
5. 门禁型指标：带 `ctx（上下文）` 进入数据地图时，系统必须自动切到 `lineage（资产图谱）` 视图，并在单轮上下文内完成一次自动定位；若图中未命中目标节点，必须退化为明确提示，不允许出现伪高亮。
6. 效果型指标：首轮验收至少覆盖“上下文进入 -> 自动图谱定位 -> 节点或关系详情 -> 影响分析 -> 跳转运行决策台”这一条闭环，闭环样本通过率达到 100%。
7. 提前接入 `Neo4j（图数据库产品，Neo4j）` 后，数据地图的图读结果必须显式返回 `readSource（读源，设计口径 read_source）` 与图投影校验状态；当图投影未通过发布时一致性校验、图数据库不可达或目标快照未投影时，页面必须稳定退回关系库聚合结果，不得展示过期图。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“数据地图浏览与覆盖追踪”一个场景。它是治理产品面的核心解释界面，承载业务场景、方案资产、覆盖声明、策略对象、来源契约、证据片段和版本快照的前台投影，并负责把运行决策台带入的对象上下文转成可见、可追踪、可解释的图谱反馈。当前首轮的验收对象固定为 `代发 / 薪资域` 多场景已发布资产，不把全域图谱一次性铺开作为前置门槛。

## 3.2 特性图示

桌面端骨架固定为顶部工具栏、主视图切换条、页面内左侧领域树、中央主区、右侧检查面板和底部联动区六部分。主视图先决定“看什么”，布局再决定“怎么摆”；其中高可见反馈以中央图谱和右侧详情面板为主承载，底部联动区只承担日志与补充反馈，不反向接管主场景。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“数据地图”一级工作台，当前实现主容器为 `frontend/src/components/datamap/DataMapContainer.jsx`，页面适配层为 `frontend/src/pages/datamap-adapter.js`；深页包括影响分析、证据片段详情和运行决策台联动入口。

## 4.2 功能说明

### 4.2.1 特性场景1

#### 通用补充项

1. 主对象：已发布 `Scene（业务场景）`、`Plan（方案资产）`、`Contract View（契约视图）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Source Contract（来源契约）`、`Evidence Fragment（证据片段）`、`Version Snapshot（版本快照）`，以及运行决策台带入的图谱节点、图谱关系、覆盖分段与上下文包。
2. 默认时间语义：图谱本身不改写业务时间语义，默认按当前快照与覆盖声明展示“在何时、覆盖到哪里”；若上下文包显式带入快照或覆盖分段时间点，以该轮上下文为准。
3. 覆盖范围和缺口：首轮覆盖已发布资产的图谱浏览、覆盖追踪、运行上下文定位、节点与关系详情、影响分析和跨工作台跳转；不覆盖知识生产编辑、发布执行或审批决策。
4. 策略或审批边界：数据地图只解释策略对象与覆盖边界；若对象本身受审批或高敏控制约束，只展示正式状态与限制提示，不直接执行审批或放宽字段可见范围。
5. 前端入口、详情页和跳转链路：入口为“数据地图”一级工作台；深页包括影响分析与证据详情；支持从“运行决策台”带锁定态 `ctx` 进入并聚焦目标对象，必要时再按需跳回运行决策台。
6. 接口或数据边界：图谱接口只读取已发布快照投影，不反向写回控制资产；高可见层正式主路径收口到 `/api/datamap/*`，`/api/assets/*` 仅保留为旧列树浏览和兼容过渡输入。
7. 存储落位边界：关系型控制库是正式治理资产真源，图谱存储只承接投影、检索与浏览，不新增独立真源模型。

1. 主视图固定为业务视图、路径视图、覆盖视图三类；默认进入业务视图，默认布局为图谱。
2. 业务视图负责展示业务领域、业务场景、方案资产、契约视图、覆盖声明和策略对象的结构关系；路径视图负责展示主路径、候选路径和证据列；覆盖视图负责展示时间带、覆盖矩阵和回退动作。
3. 左侧领域树只保留业务领域一级，不把业务场景和治理对象继续塞回树里，避免站点导航与页面导航混用。
4. 带 `ctx（上下文）` 进入数据地图时，系统必须先判断是否命中高可见反馈规则：若命中，则自动切换到 `lineage（资产图谱）` 视图、执行单轮自动定位，并以节点优先于路径的方式表达高亮；该自动抢占只在当前上下文轮次生效一次，用户手动切换视图或手动重新检索后，自动抢占应让位给手动操作。
5. 若图谱未命中目标节点或路径，页面必须退化为“未命中但已收到上下文”的明确提示，并保留排障信息；不允许继续保留伪高亮或误导性的默认聚焦。
6. 右侧检查面板必须同时支持节点详情和关系详情，并按对象概况、覆盖检查、风险与影响等固定顺序组织，不允许因为点击关系就跳出当前上下文。
7. 节点详情、关系详情和影响分析的职责边界如下：节点详情负责对象概况与当前关系摘要；关系详情负责解释边的语义与来源；影响分析负责在当前焦点对象之上展开下游影响，不重复承载节点基础概况。
8. 与运行决策台联动时，本场景优先承接焦点对象定位与证据追踪，而不是重新跑一遍运行主线；跳转回运行决策台时，应保留足以复原当前焦点对象与来源路径的最小上下文包。
9. 顶部过滤器至少支持对象类型、资产状态、关系类型、敏感级别和快照范围，便于控制画布复杂度；视图切换时必须保留当前焦点场景与过滤条件。
10. 顶部过滤器中的资产状态、画布或详情中的状态徽标，以及由上下文带入的状态摘要统一显示中文；筛选逻辑和接口传输仍使用英文状态编码，不在界面直接展示 `DRAFT / REVIEWED / PUBLISHED / RETIRED` 等英文枚举。
11. 覆盖视图中的覆盖矩阵与时间轴必须能说明完整覆盖、部分覆盖、存在缺口三类状态，并支持从缺口直接进入缺口处理或发布影响判断。
12. 图谱接口返回的节点和边必须带上 `node_type（节点类型）`、`relation_group（关系分组）`、快照标识和必要投影提示，确保前端不会误读不同版本或不同关系分组的数据；当前首轮所有图谱数据都必须能回到同一批已发布快照。
13. 当前首轮的 `代发 / 薪资域` 图谱至少覆盖“代发明细查询”“代发批次结果查询”等相邻场景，支持从运行结果定位到正确场景、方案、覆盖分段和证据片段。
14. 图谱布局、节点着色、边的视觉分层、首次加载留白比例和右侧详情面板交互，需要遵循 `frontend-workbench-design.md` §3.5.1 的 `MiroFish（开源项目）` 吸收细则：吸收工作台表达方式，不吸收默认全图铺开和高密度彩虹边风格。
15. 本场景的边界是“让正式治理对象可被浏览和追踪”。它不直接执行发布、不直接审批，也不承担知识生产中的对象编辑职责，也不负责继续扩张新的多工作台总览表达。
16. 当前 Phase 1 提前接入 `Neo4j` 的范围只覆盖 `Data Map Graph API（数据地图图谱接口）` 与 `Impact Analysis API（影响分析接口）` 的图拓扑读取；`/api/graphrag/query`、`Knowledge Package（知识包）`、发布门禁、覆盖判定和策略决策不在本阶段切换到图数据库主路径。
17. 图数据库内只允许存在已发布快照子图，不允许写入草稿态、候选态或未复核对象；发布成功后必须按 `scene_id（场景标识） + snapshot_id（快照标识）` 同步完成图投影和发布时一致性校验，校验失败时该快照不得进入图读路径。
18. 数据地图是否读 `Neo4j` 的判断顺序固定为：先锁定当前请求实际命中的 `snapshot_id`，再检查该快照最近一次图投影校验状态；只有校验状态为 `PASSED（已通过）` 时才允许返回 `readSource=NEO4J`，否则统一回退 `readSource=RELATIONAL`。
19. 节点详情、覆盖检查、策略说明等属性型细节在 Phase 1 允许继续回读关系型控制库补充；图数据库优先承接节点、关系、版本锚点、拓扑遍历和影响分析子图，不要求把所有治理文本全文冗余进图节点。
