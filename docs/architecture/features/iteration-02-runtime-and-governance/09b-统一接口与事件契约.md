# 一、特性概述

## 1.1 背景

当前正式接口、错误码、决策码和异步事件都已在主方案中给出统一基线，但这些约束还没有被拆成独立特性文档。若继续把 `OpenAPI（开放接口描述规范，OpenAPI Specification）` 规则、错误码语义、决策码语义和事件顺序散落在运行决策、审批导出、监控审计和前端页面各自的实现里，后续任何契约变更都会演变为“谁都在消费、谁都不定义”的失控状态。

## 1.2 目标

本特性用于把正式接口发布与消费规则、错误码与决策码注册基线、异步事件契约、幂等边界和契约变更门禁收口到同一个横切治理场景中，确保运行面、审批链路、监控链路和前端工作台基于同一套正式契约工作。

# 二、指标

1. 正式接口契约登记覆盖率达到 `100%`。统计口径：分子为已进入 `OpenAPI` 正式登记的稳定消费接口数，分母为当前被前端、`BFF（前端聚合层，Backend For Frontend）` 或下游服务稳定消费的正式接口总数，统计窗口为当前发布版本。
2. 未登记错误码或未登记决策码对外暴露次数为 `0`。统计口径：分子为运行或审批链路返回未登记码值的次数，分母为外部可见的接口调用总数，统计窗口为单次灰度或正式发布窗口。
3. 事件顺序、幂等或去重违规次数为 `0`。统计口径：分子为事件流回放、灰度观测或前端消费中发现的契约违规次数，分母为已纳管异步事件流总数，统计窗口为单次版本验证周期。
4. 契约变更未经评审直接发布次数为 `0`。统计口径：分子为绕过正式评审的契约变更次数，分母为当前版本全部契约变更次数，统计窗口为单次版本迭代。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“统一接口与事件契约”一个横切治理场景。它不定义单个业务场景的字段集合，而是定义所有正式接口与正式事件都必须遵守的基线：哪些接口必须纳管、哪些字段块必须稳定返回、错误和决策如何编码、事件如何排序、契约变化如何评审。

## 3.2 特性图示

前端承载采用“发布中心契约检查区 + 消费方引用说明 + 事件时序面板”的结构。主图示建议同时包含三类图：同步接口统一返回契约图、异步事件顺序图、错误码 / 决策码分层图。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载页面为“发布中心 -> 契约检查”标签区；运行决策台、审批与导出、监控与审计是本场景的正式消费方。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`OpenAPI Contract（正式接口契约）`、`Error Code Registry（错误码注册表）`、`Decision Code Registry（决策码注册表）`、`Event Contract（事件契约）`。
- 默认时间语义：不适用；本场景定义跨场景统一契约，不直接承接某个业务查询的时间解释。
- 覆盖范围和缺口：覆盖正式接口登记、同步最小返回契约、错误 / 决策编码、事件顺序、幂等边界和契约变更门禁；不覆盖单个业务场景的字段设计和页面交互细节。
- 策略或审批边界：本场景定义契约门禁，不直接替代业务审批；契约变更必须过评审，但评审通过不等于高敏业务边界自动放行。
- 前端入口、详情页和跳转链路：入口为“发布中心 -> 契约检查”；运行决策台、审批与导出、监控与审计中的相关深页可回跳到契约检查区查看统一定义。
- 接口或数据边界：本场景输入为已纳入正式链路的接口与事件定义，输出为统一契约基线和消费规则；若依赖 `BFF`，必须明确它消费正式契约而不是自造私有协议。
- 存储落位边界：正式契约真源由后端 `OpenAPI` 输出和注册基线组成，前端只消费检查结果和引用关系，不维护独立契约真源。

1. 凡被前端、`BFF`、外部服务或自动化流程稳定消费的正式接口，都必须进入 `OpenAPI` 契约管理范围。页面内部临时拼装对象、调试态字段和未发布实验字段不得被视为正式契约，也不得被长期消费。
2. 正式同步接口必须稳定返回统一公共块，至少包括结果状态、追踪信息、必要的版本或快照上下文，以及错误码或决策码。业务字段可以扩展，但不得删掉统一公共块，也不得要求消费方从业务字段里反推系统行为。
3. 错误码必须统一命名、唯一登记、可追溯到产生方和处置建议。错误码主要服务日志、控制台、审计链路和稳定消费，不等于直接给用户看的提示文案；未经登记的错误码不得进入正式发布。
4. 决策码负责表达系统如何处理请求，例如 `allow`、`need_approval`、`deny`、`clarification`、`partial` 等最小决策集合。运行结果需要审批、需要澄清、部分覆盖或被拒绝时，必须返回正式决策码，而不是伪装成普通成功或普通失败。
5. 异步事件契约必须显式定义事件类型、事件顺序、结束规则和失败替代规则。开始、过程、结果和失败事件的职责必须分开，事件顺序是正式契约的一部分，不允许由页面体验倒推。
6. 每条事件流都必须定义幂等边界、重复消费去重规则和重放处理方式。消费方收到重复事件时必须可安全去重，不得重复入库、重复加节点或重复触发副作用；历史重放事件不得被误当作新任务事件。
7. `BFF` 与前端只能消费正式契约，不能把未登记字段、私有返回块或页面临时约定提升为长期依赖。前端事件消费也必须明确哪些状态可以显示、哪些状态必须以后端事件为准，不得在主提示区擅自创造新语义。
8. 契约变更属于跨场景影响项。字段删除、语义变化、错误码替换、事件顺序调整、幂等键变化等都必须进入正式评审，并至少补齐契约测试、消费方兼容性检查；必要时还要补回放或灰度验证。未经评审或验证未通过的契约变更不得进入正式发布。
9. 本场景的边界是“统一定义和门禁正式接口与正式事件契约”。它不定义单个业务场景的输出字段集合，不替代“运行决策与知识包生成”或“审批与导出”中的业务边界，也不替代前端主文档中的页面交互与中文文案规范。

### 4.2.2 `OpenAPI` 文档质量门禁

1. 正式接口契约不仅要“存在”，还必须同时满足“人可读、机可消费、可回归验证”三项要求。若 `/v3/api-docs` 只能靠消费方阅读源码才能理解，则该契约视为未达标。
2. 正式分组必须使用业务域 `tag`，例如“领域管理”“场景设计”“图谱检索与知识包”“导入与质检”等；`domain-controller`、`graph-rag-controller` 一类自动推断技术标签不得作为正式对外分组。
3. 每个正式接口都必须具备稳定 `summary`，并在需要时补 `description`；`operationId` 必须可预测、可检索、可长期引用，不允许继续使用 `list_2`、`query_1` 之类自动兜底命名。
4. 成功响应码、错误响应码与公共错误模型必须同步进入正式契约。创建类接口要显式暴露 `201`；校验失败、资源缺失、状态冲突和系统内部错误至少要有统一的 `ApiErrorDTO` 契约出口。
5. 模型说明采用“核心对象先达标、长尾对象逐步补齐”的治理策略。首批强制覆盖公共错误模型、前端直接消费对象、Swagger UI 首屏高频对象以及字段名难以自解释的核心对象。
6. 契约质量门禁必须落成可执行检查：集成测试负责锁定关键路径的 `tag / summary / operationId / response / schema description`，真实服务探活负责确认运行时 `/v3/api-docs` 未退化为自动推断结果。未通过门禁的接口契约，不得视为完成收口。
