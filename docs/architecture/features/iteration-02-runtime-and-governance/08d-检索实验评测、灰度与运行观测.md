# 一、特性概述

## 1.1 背景

引入实验检索引擎之后，系统不仅要知道“能不能接”，还要知道“值不值得开、什么时候开、开了之后怎么发现风险”。如果缺少统一评测、`Shadow Mode（影子模式）`、灰度门禁和运行观测，检索实验就会长期停留在“命中看起来不错”的主观判断上，无法进入正式工程判断。

本专题只解决一个问题：如何用统一指标、统一影子模式、统一灰度开关和统一审计事件来评估检索实验能力，并保证实验能力在未通过门禁前不会影响正式业务结果。

## 1.2 目标

本特性用于建立“离线评测 -> `Shadow Mode（影子模式）` -> 灰度门禁 -> 运行观测 -> 回退规则”的完整实验治理链路，让检索实验从研究结论变成可追溯、可回滚、可停止的工程能力。

# 二、指标

1. 门禁型指标：`Policy false allow（策略误放行）` 次数始终为 `0`。统计口径：分子为灰度或影子模式期间因实验链路导致的高敏误放行请求数，分母为启用检索实验的运行请求数，统计窗口为首轮回放集与灰度流量。
2. 门禁型指标：实验观测字段完整率达到 `100%`。统计口径：分母为启用实验适配器的运行请求数，分子为同时记录 `trace_id（追踪编号）`、`snapshot_id（快照标识）`、`adapter_name（适配器名称）`、`index_version（索引版本）`、`latency_ms（耗时毫秒）` 与引用数量的请求数，统计窗口为首轮灰度期。
3. 效果型指标：`scene hit@5（前 5 命中率，Hit at 5）` 和证据 `precision@10（前 10 精确率，Precision at 10）` 不低于现有正式基线。统计口径：分母为进入离线评测集的回放请求数，分子分别为真实目标场景进入前 5 候选的请求数、前 10 候选证据中的正确证据数，统计窗口为首轮 `代发 / 薪资域` 回放集。
4. 效果型指标：启用 `Shadow Mode（影子模式）` 后，正式响应 `p95（第 95 百分位延迟，95th Percentile Latency）` 不突破当前知识包生成预算。统计口径：分母为影子模式期间正式运行请求数，分子为延迟小于或等于预算阈值的请求数，统计窗口为首轮灰度期，目标值 `100%`。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“检索实验评测、灰度与运行观测”一个场景。它不负责检索结果本身如何生成，而是负责决定实验是否通过离线评测、是否允许进入影子模式、是否允许进入灰度，以及一旦出现风险如何停机或回退。

## 3.2 特性图示

结构固定为“离线评测集 -> `Shadow Mode（影子模式）` -> 灰度开关 -> 运行观测与告警 -> 回退 / 停机动作”。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载继续沿用“监控审计与影响分析”工作台和“系统管理”页，当前页面入口为 `frontend/src/pages/MonitoringAuditPage.jsx` 与 `frontend/src/pages/SystemPage.jsx`。本专题不新增业务工作台，不新增面向业务用户的调试页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Evaluation Corpus（评测语料集）`、`Shadow Mode（影子模式）`、`Gray Release Scope（灰度发布范围）`、`Experiment Audit Event（实验审计事件）`、`scene hit@5（前 5 命中率）`、`precision@10（前 10 精确率）`、`p95（第 95 百分位延迟）`。
- 默认时间语义：本场景关注评测窗口、灰度窗口和运行观测时间，不改写业务查询时间；涉及运行请求时仍以请求自带 `snapshot_id（快照标识）` 为准。
- 覆盖范围和缺口：首轮只覆盖 `代发 / 薪资域` 样板域的离线评测、影子模式、灰度开关、监控指标和回退规则；不覆盖跨业务域统一评测平台，不覆盖知识生产 Prompt 质量评测。
- 策略或审批边界：实验链路未通过评测和灰度门禁前，只允许进入影子模式；即使进入灰度，正式响应也必须继续服从现有 `Policy Decision（策略决策）` 和审批链路。
- 前端入口、详情页和跳转链路：配置入口为“系统管理”，观测入口为“监控审计与影响分析”；告警需能带 `trace_id（追踪编号）` 跳回运行决策台复盘；不新增业务详情页。
- 接口或数据边界：输入为回放集、灰度范围、实验运行记录和审计事件；输出为评测报告、灰度状态、告警事件、停机 / 回退建议；不输出业务知识包。
- 存储落位或持久化边界：评测结果、灰度状态、实验审计事件和告警摘要落监控 / 审计链路；不新增正式业务真源对象。

1. 离线评测必须同时对比“现有正式召回基线”“普通向量检索基线”和“统一 `Retrieval Experiment Adapter（运行检索实验适配器）` 实现”。评测结果不能只给总分，至少要拆到 `scene hit@5（前 5 命中率）`、证据 `precision@10（前 10 精确率）`、`snapshot mismatch（快照错配）`、`Policy false allow（策略误放行）` 与 `latency_ms（耗时毫秒）`。
2. `Shadow Mode（影子模式）` 必须与正式请求并行运行，但默认不影响正式结果。它只记录实验候选、引用、延迟和分数组成，用于比较实验结果与正式结果的差异。
3. 灰度开关只能在离线评测通过且 `Shadow Mode（影子模式）` 差异可解释后开启。灰度范围至少支持业务域、场景、角色、机构或固定回放样本集合，不允许“一键全量开启”。
4. 运行观测必须同时记录 `trace_id（追踪编号）`、`snapshot_id（快照标识）`、`adapter_name（适配器名称）`、`index_version（索引版本）`、`latency_ms（耗时毫秒）`、引用数量、候选数量和失败原因；缺少任一核心字段时，不允许把该批请求计为“已观测”。
5. 回退规则固定为三层：先停灰度、再停 `Shadow Mode（影子模式）`、最后禁用实验适配器。任何回退动作都不能影响正式 `snapshot_id（快照标识）` / `inference_snapshot_id（推理快照标识）` 绑定对。
6. 监控与审计工作台必须能直接看见实验链路的失败类型、版本错配、引用为空、延迟异常和误放行风险，不允许把实验观测埋在通用日志深处。
7. 首轮默认以 `LightRAG（开源 GraphRAG 检索框架，LightRAG）` 作为首个适配器和评测基线，但评测框架、灰度门禁和审计字段必须对后续其他实验引擎复用，不因首个产品名固化。
8. 本场景不新增业务工作台，不定义知识生产 Prompt 语义，也不代替正式上线保障总纲；它只负责“检索实验是否达标、如何观察、何时停用”的实验治理动作。正式上线切换与成对快照回滚仍由 `11b-运维验收与上线保障.md` 约束。
