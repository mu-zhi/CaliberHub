# 一、特性概述

## 1.1 背景

当前正式治理资产采用“一场景一图谱、场景内全量替换”的组织方式。即使两个场景依赖同一张物理表、同一条策略、同一份法规出处或同一份输出契约，只要它们分别在不同场景中生成，就会以两套彼此隔离的正式对象存在。这个结构在单场景浏览阶段还能工作，但一旦进入 `Domain（业务领域）` 级数据地图、跨场景影响分析和运行检索复用，就会暴露三个核心问题：第一，控制库中缺少长期稳定的共享身份，系统只能看到“两个长得像的场景资产”；第二，发布快照无法冻结“同一个统一实体在本次发布中被哪些场景成员引用”；第三，`Data Map（数据地图）`、`Impact Analysis（影响分析）`、`Runtime Retrieval（运行检索）` 三条链路只能各自维护一套跨场景拼接逻辑，难以保证解释一致性。

## 1.2 目标

本特性用于在 `MySQL（关系型数据库，MySQL）` 控制库内引入跨场景统一实体真源层，把 `Source Contract（来源契约）`、`Policy（策略对象）`、`Evidence（证据出处）`、`Output Contract（输出契约）` 四类对象升级为长期稳定身份，并让场景资产变成统一实体的成员实例。首轮只解决 `代发 / 薪资域` 已发布场景内的多场景共享识别、发布冻结、领域图谱展示和运行复用边界，不把 `Neo4j（图数据库产品，Neo4j）` 升级为真源，也不在本轮扩展到知识生产编辑台的全量重做。

1. 在关系型控制库中建立 `Canonical Entity（统一实体）`、`Canonical Membership（统一成员归属）`、`Canonical Relation（统一实体关系）` 与 `Resolution Audit（解析审计记录）` 四类正式对象。
2. 让 `Scene Asset（场景资产）` 继续承担场景内语义、编辑和审核过程，但其跨场景共享身份统一落到统一实体层，不再依赖场景间临时横向连边。
3. 把 `publish（发布）`、`Data Map（数据地图）`、`Impact Analysis（影响分析）`、`Runtime Retrieval（运行检索）` 统一收口到“统一实体真源 + 场景成员实例 + 发布冻结归属”的同一条主线。
4. 明确四类对象的 `canonical_key（统一键）`、自动归并条件、人工复核条件和快照冻结边界，禁止继续使用场景派生 `code` 充当全局身份。

# 二、指标

1. 门禁型指标：进入 `PUBLISHED（已发布）` 快照的 `Source Contract / Policy / Evidence / Output Contract` 四类正式资产，必须 100% 具备以下两种结果之一：要么成功挂到唯一 `Canonical Entity（统一实体）`，要么明确落入 `NEEDS_REVIEW（待复核）` 状态；分子为当次发布涉及的四类正式资产数，分母为当次发布进入快照绑定范围的四类正式资产总数，统计窗口为单次发布任务。
2. 门禁型指标：发布时冻结的 `Canonical Membership Snapshot（统一成员归属快照）` 必须与该次 `snapshot_id（快照标识）` 内的正式场景资产集合完全一致，错绑、漏绑和跨快照串绑次数必须为 0；分子为冻结错误次数，分母为发布任务数，统计窗口为回放集与真实发布任务。
3. 门禁型指标：`Policy（策略对象）`、`Evidence（证据出处）`、`Output Contract（输出契约）` 在缺少显式统一键时，不允许自动强归并；分子为“缺键仍被自动归并”的对象数，分母为缺少显式统一键的相关对象数，统计窗口为首轮样板域发布回放，目标值为 0。
4. 效果型指标：`GET /api/datamap/graph?root_type=DOMAIN&root_id=<domainId>` 返回的领域级图谱，必须能稳定展示“共享统一实体 + 场景成员实例 + 正式归属关系”，并覆盖 `代发明细查询 / 代发批次结果查询` 至少两个相邻场景；分子为正确呈现共享关系的样板场景对数量，分母为样板域计划覆盖的场景对数量，统计窗口为首轮验收回放集，目标值 100%。
5. 效果型指标：运行检索在样板域的多场景召回中，允许先用统一实体层做跨场景缩域与扩展，但最终选中的 `scene_id + snapshot_id + plan_id` 必须与正式已发布成员实例一致；分子为“统一实体层参与召回且最终命中正式场景实例”的请求数，分母为启用统一实体缩域的请求数，统计窗口为首轮验收回放集，目标值 100%。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“跨场景统一实体层与领域级图谱融合”一个治理场景。它不是新的独立工作台，而是给现有控制库增加一层正式共享身份真源，并让 `Data Map（数据地图）`、`Impact Analysis（影响分析）` 与 `Runtime Retrieval（运行检索）` 可以在同一套共享身份之上工作。当前首轮仅以 `代发 / 薪资域` 已发布多场景为样板，重点验证“同一物理表 / 同一策略语义 / 同一证据出处 / 同一输出契约”在控制库、发布快照、图投影和运行检索中的统一口径。

## 3.2 特性图示

系统结构固定为三层：底层为 `Scene Asset（场景资产实例层）`，承载每个场景内的正式对象、场景语义和审核历史；中间为 `Canonical Entity（统一实体层）`，承载长期稳定身份、统一键、实体画像和统一实体之间的正式关系；顶层为 `Published Snapshot（已发布快照）`，冻结“本次发布有哪些场景成员实例引用了哪些统一实体”的归属结果，并把这批结果投影到 `Data Map（数据地图）`、`Neo4j（图数据库产品，Neo4j）` 读模型和运行检索辅助索引中。`Neo4j` 只承接已发布快照投影，不反向写回统一实体真源。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载仍为“数据地图”一级工作台中的 `业务视图`，领域树选中 `Domain（业务领域）` 后，中央主图允许显示统一实体节点与场景成员实例；深页消费者包括“影响分析”与“运行决策台”的跨页上下文定位，但本特性不新增独立页面壳层。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Canonical Entity（统一实体）`、`Canonical Membership（统一成员归属）`、`Canonical Relation（统一实体关系）`、`Canonical Resolution Audit（统一解析审计记录）`、`Scene Asset（场景资产实例）`、`Published Snapshot（已发布快照）`。
- 默认时间语义：统一实体层本身不重写业务时间语义；业务时间、覆盖范围和时态解释仍以场景实例和发布快照为准。统一实体只提供跨场景共享身份和领域级连接骨架。
- 覆盖范围和缺口：当前首轮覆盖四类正式对象的统一实体化、领域级图谱查询、发布冻结归属、图投影复用和运行检索辅助缩域；不覆盖知识生产阶段的候选对象统一化，不覆盖跨业务域的统一主数据治理，也不覆盖图侧反向修复关系型真源。
- 策略或审批边界：自动归并不得绕过人工复核门禁。`Source Contract（来源契约）` 可按稳定物理锚点自动归并；`Policy / Evidence / Output Contract` 在缺少显式统一键时只能进入 `NEEDS_REVIEW（待复核）`，不得自动合并。
- 前端入口、详情页和跳转链路：入口为“数据地图”工作台的领域业务视图；详情仍通过右侧检查面板展示统一实体画像与成员场景摘要；运行决策台和影响分析可带 `asset_ref（资产引用）` 或 `canonical_ref（统一实体引用）` 进入数据地图做定位。
- 接口或数据边界：`Data Map Graph API（数据地图图谱接口）` 新增 `root_type=DOMAIN` 下的领域级统一实体返回；`Impact Analysis API（影响分析接口）` 允许先在统一实体层扩散再映射回场景资产；运行检索可以读统一实体层做召回辅助，但最终执行仍落回 `scene_id + snapshot_id + plan_id` 的正式运行边界。
- 存储落位边界：统一实体层、成员归属、归属冻结和解析审计全部落在 `MySQL` 控制库；`Neo4j` 只投影已发布快照内的统一实体节点、场景实例节点与正式关系，不承载真源写入。

1. 关系型控制库必须新增统一实体核心对象：`canonical_entity`、`canonical_entity_membership`、`canonical_entity_relation`、`canonical_resolution_audit`，并为四类对象补齐类型专属 profile 或结构化扩展字段，确保统一层既有统一骨架，也能表达类型差异。
2. 四类统一实体的 `canonical_key（统一键）` 必须按“显式业务键优先，物理身份次之，结构化来源再次，指纹只做建议”的规则定义：`Source Contract` 以 `sourceSystem + normalizedPhysicalTable` 为正式锚点；`Policy` 需要 `policySemanticKey（策略语义键）`；`Evidence` 需要 `originType + originRef + originLocator（证据出处定位键）`；`Output Contract` 需要 `contractSemanticKey（输出契约语义键）`。场景派生 `code` 不再允许作为全局统一身份。
3. `Scene Graph Asset Sync（场景图谱资产同步）` 必须从“全量删除后重建”改为“按场景实例稳定键 `upsert（插入或更新）` + `deprecate（标记废弃）`”。场景资产实例继续保留本地身份和历史痕迹，但不再因为一次同步重建就丢失与统一实体层之间的长期归属。
4. 在场景资产同步完成后，系统必须自动进入 `Canonical Entity Resolution（统一实体解析）` 阶段：根据四类对象的统一键、证据来源、结构化字段和人工覆盖规则，决定是挂到已有统一实体、创建新的统一实体，还是落入 `NEEDS_REVIEW（待复核）` 并记录审计理由。
5. 发布时不得复制新的统一实体副本。统一实体必须跨快照长期存在；发布动作只冻结“哪些场景资产实例在当前 `snapshot_id` 下属于哪些统一实体”以及“哪些统一实体关系在当前快照中可见”，形成可追溯的归属快照结果。
6. `Data Map Graph API` 在 `root_type=DOMAIN` 下必须返回领域级聚合图：图中允许同时包含 `Domain（业务领域）`、`Scene（业务场景）`、场景资产实例节点、统一实体节点，以及至少三类正式关系边：`INSTANCE_OF（实例属于统一实体）`、`CANONICAL_RELATION（统一实体之间的正式关系）`、`SCENE_MEMBERSHIP（场景与统一实体的归属关系）`。领域级返回允许覆盖多个场景，但同一响应内必须显式标注快照范围和读源，不允许混淆不同发布口径。
7. `Impact Analysis（影响分析）` 必须支持从统一实体层出发：先判断某个统一实体或统一关系会影响哪些场景成员，再映射回具体 `Scene Asset（场景资产实例）`、`Scene（业务场景）` 和 `Published Snapshot（已发布快照）`，避免继续只在单场景图内做局部广度遍历。
8. `Runtime Retrieval（运行检索）` 和 `Knowledge Package（知识包）` 允许复用统一实体层做召回缩域、跨场景扩展和共享对象解释，但最终运行边界仍然固定在正式已发布的场景实例集合；统一实体层增强检索，不直接替代场景执行对象。
9. `Neo4j` 投影层必须跟随统一实体层一起调整：统一实体节点使用 `canonical_key` 做 `MERGE（合并写入）` 主键，场景实例节点使用稳定场景实例键做 `MERGE`，快照内归属边必须带 `snapshot_id`。`Neo4j` 继续是已发布快照读模型，不得承担统一实体真源。
10. 统一实体层必须显式支持人工覆盖：允许人工确认“两个对象属于同一统一实体”“两个对象必须拆分”“当前对象仅建议关联不可自动合并”等操作，并把最终决议写入 `Canonical Resolution Audit（统一解析审计记录）`。自动规则不得覆盖人工决议。
11. 本场景不负责把所有存量对象一次性回填成完美统一实体，也不负责在本轮引入跨业务域公共主数据平台。首轮只要求在 `代发 / 薪资域` 样板中把四类对象的统一身份、归属冻结和多链路复用口径打通，并对未具备自动归并条件的对象维持显式待复核状态。

## 4.2.2 当前首轮最终收口规则

1. 正式链路固定为：`Candidate Graph(task) -> Formal Scene Subgraph(scene) -> Canonical Layer(domain) -> Snapshot Freeze(snapshot) -> Neo4j Projection(snapshot)`。
2. 真源边界固定为三层：`Candidate（候选）`、`Formal Control（正式真源）`、`Snapshot（冻结可见边实例）`。不得把快照前正式关系真源压缩到 `Snapshot` 层，也不得让 `Neo4j` 反向承担控制库真源。
3. `Formal Scene Subgraph（场景正式子图）` 的首轮资产范围固定为：`Scene（业务场景）`、`Plan（方案资产）`、`Coverage Declaration（覆盖声明）`、`Source Contract（来源契约）`、`Policy（策略对象）`、`Evidence Fragment（证据片段）`、`Output Contract（输出契约）`、`Contract View（契约视图）`。实现时不得把“统一实体首轮范围”误替换为“正式场景资产首轮范围”。
4. 场景正式资产锚点固定使用 `scene_asset_ref = scene-asset:{scene_id}:{asset_type}:{asset_id}`。同一资产在发布冻结、领域级聚合图和图投影中的主语标识必须复用同一引用格式。
5. 最小关系骨架固定为：`Scene -> SceneAsset` 使用 `SCENE_MEMBERSHIP`，`SceneAsset -> CanonicalEntity` 使用 `INSTANCE_OF`，`CanonicalEntity -> CanonicalEntity` 使用 `CANONICAL_RELATION`；`Scene -> CanonicalEntity` 不直接连边，`HAS_ASSET` 废弃。
6. `SCENE_MEMBERSHIP（场景成员归属）` 在 `root_type=DOMAIN` 返回中的来源必须固定为“已发布 `scene_id + snapshot_id` 对应的场景资产成员集”，不得从当前可变场景内容现算。
7. `Canonical Layer（统一实体层）` 首轮只对 `Source Contract（来源契约）`、`Policy（策略对象）`、`Evidence（证据出处）`、`Output Contract（输出契约）` 四类对象做跨场景身份融合。
8. `Evidence Fragment（证据片段）` 只表示场景内可回溯证据片段；`Canonical Evidence（统一证据出处）` 只表示跨场景共享证据出处身份。只有正式 `Evidence Fragment` 能稳定解析 `originType + originRef + originLocator` 时，才允许建立 `INSTANCE_OF`；否则只能停在 `fragment` 或 `NEEDS_REVIEW（待复核）`。
9. `Canonical Relation（统一实体关系）` 首轮最小枚举固定为：`MAPS_TO_SOURCE`、`APPLIES_POLICY`、`SUPPORTED_BY`。首轮端点固定为 `CanonicalEntity -> CanonicalEntity`；`SceneAsset -> CanonicalEntity` 只能用 `INSTANCE_OF`，不得伪装成统一实体关系。
10. `relation_version（关系语义版本）` 用于跨快照判断是否仍为同一条语义关系，不得包含 `snapshot_id`；`edge_id（快照内边实例标识）` 用于标识某条边在特定 `snapshot_id` 下的可见实例，必须包含 `snapshot_id`。
11. 第一轮最小真源表集固定为：`canonical_entity`、`canonical_entity_membership`、`canonical_relation`、`canonical_resolution_audit`、`canonical_snapshot_membership`、`canonical_snapshot_relation_visibility` 六张表；`graph_statement_ledger` 首轮不建表，但必须保留按 `snapshot_id` 导出标准化 statement 的统一接口或内部服务。
12. `relation_group（关系分组）` 的首轮正式映射固定为：`SCENE_MEMBERSHIP -> control`、`INSTANCE_OF -> control`、`MAPS_TO_SOURCE -> metadata`、`APPLIES_POLICY -> control`、`SUPPORTED_BY -> evidence`。关系库聚合与 `Neo4j` 投影必须使用同一映射，不得各自生成不同分组口径。
13. `DOMAIN` 领域级图读第一阶段默认走 `RELATIONAL（关系库聚合回退）`。只有在控制库已补齐“统一实体归属冻结 -> 图投影 -> 发布时一致性校验”的闭环后，才允许切换到 `NEO4J（图数据库读模型）`。

## 4.3 进入开发队列的实施约束

1. 本特性作为当前首个 `Depth-First Working（纵深优先执行）` 试点闭环，进入开发队列前必须同时满足四项前置条件：主方案已写明六层边界和试点闭环、特性文档已收口本场景的实施约束、实施计划已对齐试点任务拆分、`docs/engineering/current-delivery-status.md` 已把该工作项标记为正式在途事项。四项任意缺失，都不得直接编码。
2. 实施边界固定为六层：`Types（类型与契约层）` 只定义统一实体、成员归属、冻结归属和领域级返回契约；`Config（规则与路由层）` 只维护归并策略、门禁开关和固定角色智能体路由；`Repo（持久化访问层）` 只处理控制库与图读源的正式读写；`Service（领域服务层）` 负责统一实体解析、成员归属维护、发布冻结与领域级聚合；`Runtime（运行时消费层）` 只消费已发布、已冻结的统一身份结果；`UI（前端表达层）` 只渲染接口返回的正式口径，不自行推断统一实体归属。
3. 本特性的开发任务固定拆为 8 个构件：`Task 0` 先固化测试文档骨架、边界检查、契约断言与 `Golden Path（黄金路径样例）` 烟测；`Task 1` 控制库迁移；`Task 2` 统一实体仓储；`Task 3` 归并决策服务；`Task 4` 成员归属维护服务；`Task 5` 发布冻结服务；`Task 6` `DOMAIN` 领域级图读服务；`Task 7` `Impact Analysis（影响分析）` / `Runtime Retrieval（运行检索）` 复用适配。每个构件只允许触碰一层或相邻两层，不允许跨层顺手扩展范围。
4. `Build Agent（实现智能体）` 的执行输入必须是结构化任务卡，至少写明：任务编号、所属层级、可修改目录、禁止触碰目录、输入契约、输出契约、必跑测试、`Review Agent（评审智能体）` 关注点。任务卡之外的新增实现，不纳入本特性的首轮开发范围。
5. 首轮必须先机械化 5 类门禁：分层越界检查、设计/计划/测试文档联通性检查、统一实体契约快照检查、发布“只冻结归属不复制真源”回归检查、`Golden Path` 烟测。第一轮允许以目录约束、导入规则、关键字扫描、回归测试和 `CI` 组合拦错，但不得只停留在聊天约定或人工提醒。
6. 半自动多智能体编排按状态路由：`planning（计划中）` 优先进入 `Plan Agent（计划智能体）`；`implementing（开发中）` / `fixing（修复中）` 进入 `Build Agent（实现智能体）`；`reviewing（评审验证中）` 进入 `Review Agent（评审智能体）`；只有当统一键规则、发布边界或运行时消费口径发生变化时，才回到 `Design Agent（设计智能体）`。`automation（自动化巡检）` 只负责推荐下一角色和下一动作，不直接改状态、不直接提交业务代码。
7. 试点是否跑通，不以“代码已写完”为判断依据，而以 5 个闭环同时成立为准：统一实体真源与发布冻结的功能闭环、六层边界未被越界的边界闭环、自动化门禁和回归测试可复验的测试闭环、固定角色智能体按状态接力的协作闭环、主方案 / 特性文档 / 实施计划 / 状态文档口径一致的文档闭环。
