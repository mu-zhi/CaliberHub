> 迭代归属：迭代二运行与治理工作台
> 来源主文档：`system-design.md` §5.5、§6.1、§9；`frontend-workbench-design.md` §4.4
> 对应实施计划：`docs/plans/2026-03-29-payroll-publish-datamap-implementation-plan.md`

# 一、特性概述

## 1.1 背景

本项目的运行面只能读取已发布快照，且控制资产快照与推理快照必须成对切换。若发布中心没有独立场景承接发布检查、灰度配置、冻结和回滚，系统将无法证明“当前运行版本为什么可信、为什么能切、出了问题如何回”，高敏场景也无法接受这种黑盒发布方式。

## 1.2 目标

本特性用于建立发布中心的一整套场景能力：对候选版本执行发布检查、展示快照配对关系、配置灰度范围、监控灰度指标，并在超阈值时冻结或回滚到上一稳定快照对。

# 二、指标

1. 发布前检查项覆盖率达到 100%。统计口径：分母为进入发布中心候选包的必检项总数，分子为已被结构化展示并给出通过 / 阻断结论的检查项数，统计窗口为首轮 `代发 / 薪资域` 回放包与灰度包联合样本。
2. 控制资产快照与推理快照配对一致率达到 100%。统计口径：分母为执行发布、冻结或回滚的快照切换动作数，分子为快照对字段完整且无单边切换的动作数，统计窗口为本阶段全部候选发布任务。
3. 灰度期间关键指标超阈值时，冻结或回滚动作的触发与记录成功率达到 100%。统计口径：分母为命中冻结 / 回滚阈值的灰度任务数，分子为已生成结构化告警、执行动作并完成审计留痕的任务数，统计窗口为当前阶段灰度演练与真实灰度任务。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“发布检查、灰度发布与回滚”一个场景。它位于治理产品面的发布中心，承接知识生产链路输出的候选资产与推理资产，并决定何时让它们进入运行面。

## 3.2 特性图示

页面固定包含顶部版本区、左侧变更清单、中部发布检查矩阵、右侧影响分析和底部标签区。灰度配置与回滚预案都在底部标签区显式展开，不隐藏在二次弹窗里。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“发布中心”一级工作台。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：候选 `Version Snapshot（版本快照）`、候选 `inference_snapshot_id（推理快照标识）`、发布检查矩阵、灰度任务、冻结动作、回滚动作。
- 默认时间语义：默认以候选快照生成时间、灰度生效时间和动作执行时间组织页面，不承接业务查询时间语义。
- 覆盖范围和缺口：覆盖发布检查、快照配对、灰度范围定义、冻结 / 回滚执行与操作留痕；不覆盖知识生产链路中的资产修正，也不替代运行态请求解释。
- 策略或审批边界：本场景执行版本进入运行面的正式门禁，但不替代高敏业务审批；任何阻断项未关闭前不得执行发布。
- 前端入口、详情页和跳转链路：入口为“发布中心”一级工作台；可下钻影响分析、差异视图和审计追踪，必要时带 `frozen（冻结态）` 上下文跳到“监控与审计”。
- 接口或数据边界：本场景消费候选发布包详情接口、发布检查结果接口、灰度配置接口、发布 / 回滚动作接口和状态查询接口；若存在 `BFF（前端聚合层，Backend For Frontend）`，也必须进入正式 `OpenAPI（开放接口描述规范，OpenAPI Specification）` 契约。
- 存储落位边界：快照元数据、发布任务、灰度任务和动作审计留痕落关系型控制库，缓存 / 索引刷新结果进入运行日志与事件链，图谱侧只消费已生效快照投影而不反向决定发布结论。

1. 顶部版本区必须同时展示当前稳定 `snapshot_id（快照标识）`、候选 `snapshot_id（快照标识）`、当前稳定 `inference_snapshot_id（推理快照标识）`、候选 `inference_snapshot_id（推理快照标识）`，并明确它们的配对关系。
2. 发布检查矩阵至少检查以下事项：必需输出可兑现、契约视图完整、覆盖声明有效、策略对象可执行、推理资产可回溯、快照配对一致、来源契约可落位、缓存刷新成功、索引刷新成功。
3. 左侧变更清单必须把控制资产变更与推理资产变更分组展示，避免把 `Plan（方案资产）` 变更与 `Inference Assertion（推理结论）` 变更混在一张列表里。
4. 灰度配置至少支持按业务领域、业务场景、角色、机构和流量分片定义范围，并为每个灰度任务记录监控阈值。
5. 关键灰度指标至少包括场景命中率、方案命中率、推理降级率、误路由率和高敏误放行风险；任一指标超阈值时，页面必须前置红色告警并推荐冻结或回滚。
6. 回滚单位固定为“成对快照”，不允许只回滚控制资产或只回滚推理资产。
7. 底部操作栏固定提供保存草稿、发起复核、执行发布、执行回滚四个动作；执行发布和执行回滚都必须使用说明边界的二次确认文案。
8. 发布中心的边界是“决定版本能否进入运行面”。它不替代知识生产链路修问题，也不替代运行决策台解释单次请求。
9. 所有发布、冻结、回滚动作都必须留下操作人、时间、原因、影响范围和最终结果，供审计与影响分析复盘。
10. 本场景的接口必须显式区分“候选包读取”“检查结果刷新”“动作执行”“动作状态查询”四类边界，不允许页面自行拼装隐式动作路径。
11. 本场景的边界是“发布门禁与版本切换”。它不负责直接编辑治理资产、不负责审批详情裁决，也不负责在运行面重新解释单次知识包结果。
