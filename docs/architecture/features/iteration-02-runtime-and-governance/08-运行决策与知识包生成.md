# 运行决策与知识包生成特性文档

> 迭代归属：迭代二运行与治理工作台  
> 来源主文档：`system-design.md` §6.2、§6.2.1、§6.2.2、§9.2、§9.3；`frontend-workbench-design.md` §4.5

# 一、特性概述

## 1.1 背景

本项目不是直接返回裸表和裸字段，而是返回带有场景、方案、覆盖、策略、路径、证据和追踪信息的 `Knowledge Package（知识包）`。如果运行面没有独立的决策场景，用户只能看到“命中了什么”而看不到“为什么命中、为什么拒绝、为什么需要审批”，系统就无法承担治理型产品的解释责任。

## 1.2 目标

本特性用于把一次请求的运行主线显式展开：查询改写、槽位补齐、运行推理、场景召回、方案选择、覆盖判定、策略决策、路径解析和知识包输出九步缺一不可，且结果必须可回溯、可解释、可降级。当前首轮只验收 `代发 / 薪资域` 多场景运行链路，不在本阶段扩展为跨业务域通用召回。

# 二、指标

1. 运行态返回必须始终携带追踪编号、快照标识和决策结果。
2. 降级到非完整推理模式时，降级原因展示覆盖率达到 100%。
3. 对于成功生成的知识包，场景、方案、覆盖、策略、路径、证据六类关键信息不得缺块。
4. `代发 / 薪资域` 首轮回放集中，`Scene（业务场景）` 与 `Plan（方案资产）` 命中正确率必须达到 100%。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“运行决策与知识包生成”一个场景。它是运行消费工作台的主场景，也是 `Knowledge Package API（知识包接口）` 的前台表达。当前首轮先以 `代发 / 薪资域` 的 2 至 3 个已发布场景为样板，验证单业务域多场景的运行闭环。

## 3.2 特性图示

页面采用“顶部输入区 + 左侧结果主体 + 右侧九步决策过程区 + 底部运行输出区”的固定结构。左侧讲结果，右侧讲为什么，底部只在追查时展开细节。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“运行决策台”一级工作台。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：运行请求、`Scene（业务场景）`、`Plan（方案资产）`、`Contract View（契约视图）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Knowledge Package（知识包）`。
- 默认时间语义：以输入中的查询时间和已发布 `Coverage Declaration（覆盖声明）` 为准；若冲突则返回澄清或知识拆解结果，不允许静默兜底。
- 覆盖范围和缺口：当前首轮只覆盖 `代发 / 薪资域` 多场景运行；跨业务域复合问题、跨域统一召回和多知识包并行输出不在本轮范围内。
- 策略或审批边界：高敏字段、导出诉求和专审场景必须先过 `Policy（策略对象）` 决策，不允许运行时绕过审批。
- 前端入口、详情页和跳转链路：入口为“运行决策台”；输出必须能携带上下文跳到“数据地图”“审批与导出”“监控与审计”。
- 接口或数据边界：本场景以已发布快照为唯一输入边界，只消费正式 `Scene / Plan / Contract / Coverage / Policy / Source Contract` 资产；不读取草稿对象。
- 存储落位边界：运行态只读关系型控制库中的已发布快照与图谱投影，不回写治理资产主数据。

1. 顶部输入区必须允许输入业务问题，并补充角色、用途、是否导出、截止时点和请求字段等槽位，不能只给一个纯文本输入框。
2. 右侧过程区固定展示九步：查询改写、槽位补齐、运行推理、场景召回、方案选择、覆盖判定、策略决策、路径解析、知识包输出。
3. 左侧结果主体必须结构化展示业务场景、方案资产、契约视图、生效覆盖、策略结果、推理摘要、主路径、证据摘要、风险说明和追踪信息。
4. 结果主体、覆盖列表、图谱投影卡片、策略决策徽标和概览卡中的状态类信息必须统一显示中文；英文状态编码只用于逻辑判断、上下文跳转和调试信息，不直接暴露给业务用户。
5. 若运行推理没有走 `full_inference（完整推理）`，页面必须前置橙色提示条，并说明当前 `runtime_mode（运行模式）` 与 `degrade_reason_codes（降级原因编码列表）`。
6. 标识归一、时间语义澄清和复合请求拆解是本场景的硬能力，不允许把多入口标识冲突、多时间语义冲突和多主对象复合请求 silently 默认成单一路径。
7. 返回结果必须符合知识包最小返回契约，至少包含场景、方案、契约、覆盖、策略、路径、推理、证据、风险和追踪十个字段块。
8. 若识别到复合请求、跨场景多意图或时间语义冲突，本场景必须返回 `Knowledge Decomposition Result（知识拆解结果）`，至少给出子问题、候选场景和澄清提示，而不是伪造一个混合知识包。
9. 本场景与审批场景的边界固定为：一旦策略决策为“需要审批”，运行决策台只解释原因与需要的字段，不直接绕过审批返回完整高敏结果。
10. 本场景的输出必须能带着锁定态上下文跳转到审批与导出、数据地图或审计回放，不允许目标页重新猜测当前快照和当前场景。
11. 当前首轮对 `代发 / 薪资域` 的已发布场景，场景召回与方案选择必须显式区分“代发明细查询”“代发批次结果查询”等相邻场景，不能继续以单样例关键词命中代替正式多场景收敛。
12. 若运行推理依赖不可用或适用边界不完整，本场景必须严格遵循 `full_inference（完整推理） -> deterministic_only（仅确定性推理） -> template_only（仅模板路径） -> clarification_only（仅澄清返回）` 的降级序列，并返回结构化降级原因编码。
