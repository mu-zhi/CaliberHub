# 一、特性概述

## 1.1 背景

客户密码修改日志属于典型 `S4` 高敏场景。主方案已经明确：密码修改日志默认不进入普通允许返回，而是拒绝或专审。如果没有专门特性文档，这类场景最容易在实现时被“为了查问题方便”弱化成普通日志查询，直接突破安全边界。

## 1.2 目标

本特性用于把“客户密码修改日志查询”定义为独立高敏审计场景，明确默认拒绝、专审触发、可返回信息最小集和审计链路要求。

# 二、指标

1. 未经专审的密码修改日志原始记录暴露次数为 0。统计口径：分母为命中本场景的请求数，分子为未经专审即返回原始密码修改日志的请求数，统计窗口为灰度期间和正式运行双窗口，目标值为 0。
2. 高敏拒绝或专审原因必须全部可追溯到策略对象与审批模板。统计口径：分母为本场景返回拒绝或专审结论的请求数，分子为结果中能回溯到正式策略对象和审批模板编号的请求数，统计窗口为灰度期间和正式运行双窗口。
3. 所有进入本场景的请求都必须形成完整审计事件链。统计口径：分母为命中本场景的请求数，分子为在审计链路中同时记录访问、审批和导出节点的请求数，统计窗口为灰度期间和正式运行双窗口。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“客户密码修改日志查询”一个场景。主对象是密码修改审计记录，而不是普通客户信息或普通操作日志。

## 3.2 特性图示

运行决策台在本场景中优先返回策略决策和最小化解释结果；审批与导出页承接专审；监控与审计页承接所有访问留痕与追踪回放。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为“运行决策台”、“审批与导出”和“监控与审计”；专审流程下钻 `frontend-workbench-design.md` §5.2 审批详情深页，完整访问链路复用 §5.3 审计追踪页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：密码修改审计记录、最小化解释结果、审批任务、导出任务、策略对象和追踪信息。
- 默认时间语义：不适用；本场景默认关注密码修改审计事件时间和审批 / 导出时间，不解释业务查询时间。
- 覆盖范围和缺口：覆盖密码修改审计、专审申请和导出留痕；不覆盖普通客户画像、账户机构变更轨迹和名单核验。
- 策略或审批边界：默认拒绝或专审；不允许通过角色配置临时下放为普通 `allow（允许）` 场景。
- 前端入口、详情页和跳转链路：主入口为运行决策台；专审流程进入审批与导出；完整访问链路在监控与审计页回放。
- 接口或数据边界：依赖 `Scene Search API（场景搜索接口）`、`Knowledge Package API（知识包接口）`、审批状态接口和导出状态接口；运行态只返回最小解释信息。
- 存储落位边界：场景、方案、输出契约、策略对象、审批模板和导出任务落关系型控制库；访问、审批与导出事件链落审计事件表与日志系统；导出文件指纹落导出审计记录。

1. 输入入口支持用户号、客户号和户口号，但系统在场景命中后默认不直接返回原始日志明细。
2. 策略对象默认输出为拒绝或需要专审，不允许把本场景放进普通 `allow（允许）` 路径。
3. 对于普通查询，只允许返回“是否存在密码修改记录、最近修改时间范围、可申请字段说明”等最小化解释信息，不返回原始密码审计内容。
4. 只有在专审通过后，系统才能生成受控导出任务，且导出内容必须带审批单号、操作人、导出时间和文件指纹。
5. 输出契约必须区分“普通可见说明字段”和“专审后可见日志字段”，不能只维护一套字段集合。
6. 运行决策台必须明确提示当前命中高敏规则、当前默认拒绝或专审，并说明下一步只能去审批链路。
7. 接口边界以 `Scene Search API（场景搜索接口）`、`Knowledge Package API（知识包接口）`、审批状态接口和导出状态接口为主；运行态只返回最小解释信息，审批和导出接口才承接专审后的后续动作。
8. 监控与审计页必须能按追踪编号、审批任务编号和操作人完整回放本场景的访问、审批和导出过程。
9. 本场景不支持通过修改角色配置临时下放权限；所有例外都必须通过正式审批模板和审计记录处理。
10. 本场景的边界是“密码修改审计”。它不承载普通客户画像、不承载账户变更轨迹，也不承载名单核验。
