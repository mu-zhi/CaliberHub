# 断卡排查账户管控核验特性文档

> 迭代归属：迭代四零售画像与高风险样板域  
> 来源主文档：`system-design.md` §3.4、§7.6、§10.2、§14；`frontend-workbench-design.md` §4.2、§4.5、§4.6、§4.7、§5.2、§5.3、§5.4；研究材料：`research/source-materials/sql-samples/02-工单样例集.sql`

# 一、特性概述

## 1.1 背景

断卡排查场景是本项目最能验证“多入口标识收敛 + 截止时点回看 + 高敏策略控制”三项能力是否真实可用的样板场景。研究材料已经表明，同一请求可能从客户号、户口号、证件号或内部 `UID（统一身份标识，Unique Identifier）` 进入，并且需要按投诉日期向前回看最近的管控记录。如果没有专门特性文档，这类复杂规则只会停留在工单经验里，无法成为正式治理资产。

## 1.2 目标

本特性用于把“断卡排查账户管控核验”沉淀成高风险样板场景，支持多入口标识收敛、按 `as_of_time（截止时点）` 回看最近管控记录、返回最小可见核验结果，并通过审批或专审控制高敏明细输出。

# 二、指标

1. 多入口标识收敛到统一主对象的准确率不低于 95%。
2. 按投诉日期回看的最近管控记录命中解释覆盖率达到 100%。
3. 未经审批的高敏管控明细暴露次数为 0。
4. 统计口径：以上指标以断卡排查样板回放集和灰度期间请求统计；标识收敛准确率分母为人工标注属于断卡核验场景的请求数，分子为正确收敛到统一主对象并命中最近有效记录的请求数。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“断卡排查账户管控核验”一个场景。主对象是账户管控记录，主时间语义是 `as_of_time（截止时点）`，主风险边界是名单、冻结和管控明细默认不能普通放行。

## 3.2 特性图示

运行决策台负责展示标识收敛过程、截止时点解释和核验结论；数据地图负责展示标识谱系、推理结论、策略对象和证据片段的关系；审批与导出页负责承接高敏明细申请。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为“运行决策台”，审计与审批承载在“监控与审计”和“审批与导出”；专审流程下钻 `frontend-workbench-design.md` §5.2 审批详情深页，审计复盘复用 §5.3 审计追踪页，证据核对复用 §5.4 证据片段详情。

## 4.2 功能说明

### 4.2.1 特性场景1

1. 输入入口支持工单号、客户号、户口号、证件号和 `UID（统一身份标识，Unique Identifier）`；系统必须先做标识谱系收敛，再进入正式查询。
2. 默认时间语义固定为 `as_of_time（截止时点）`，通常来自投诉日期或业务指定核验日期；系统需要向前回看该时点及之前最近的一条有效管控记录。
3. 输出契约对普通查询只承诺“是否命中管控、命中类型、命中时间、是否需要进一步审批”等最小核验结果；完整管控明细、名单来源和高敏原因默认不直接返回。
4. 场景至少要建立两类推理资产：多入口标识到统一主对象的收敛推理，以及按截止时点选择最近有效记录的时间收敛推理。
5. 策略对象必须把名单命中、冻结原因、强管控明细等字段纳入高敏控制，默认降级返回或转专审。
6. 运行决策台要显式展示本次标识收敛链、当前截止时点、命中的最近记录和当前返回级别，避免业务误把“未展示明细”理解成“没有记录”。
7. 接口边界以 `Scene Search API（场景搜索接口）`、`Knowledge Package API（知识包接口）`、审批提交接口和审批状态接口为主，知识包只能返回核验级结果，不直接返回完整高敏管控明细。
8. 数据地图需能展示 `Identifier Lineage（标识谱系）`、推理结论、策略对象和证据片段的关系，支撑治理复盘。
9. 审批通过后的导出必须绑定追踪编号、审批号、导出字段范围和文件指纹，满足高风险审计要求。
10. 本场景的边界是“账户管控核验”。它不代替普通客户画像，也不代替密码日志专审。
