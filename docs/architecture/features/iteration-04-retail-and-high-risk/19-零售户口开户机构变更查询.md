# 零售户口开户机构变更查询特性文档

> 迭代归属：迭代四零售画像与高风险样板域  
> 来源主文档：`system-design.md` §14；`frontend-workbench-design.md` §4.5、§5.6；研究材料：`research/source-materials/sql-samples/04-口径文档现状-零售客户信息变更.sql`

# 一、特性概述

## 1.1 背景

零售客户信息变更类材料中，开户机构变更是典型的“有业务诉求、但资料不完整”的场景。主方案已明确：对于结果字段未提供、场景描述不充分的材料，不能强行发布，而要通过 `Gap Task（缺口任务）` 与复核链路治理。如果没有单独特性文档，这一类半成品场景很容易被误当成已可交付功能。

## 1.2 目标

本特性用于定义“零售户口开户机构变更查询”这一条变更轨迹场景的正式口径，明确当前阶段能承诺哪些字段、缺哪些材料、哪些条件下需要先补缺再发布。

# 二、指标

1. 未确认结果字段不得进入已发布契约，阻断率达到 100%。
2. 已确认的变更轨迹字段必须能够说明变更前机构、变更后机构和变更时间。
3. 因材料不完整产生的阻断项必须全部转缺口任务，不允许以自由文本备注代替。
4. 统计口径：以上指标以本场景候选发布请求和样板回放集统计；阻断率分母为结果字段或变更判定规则不完整的候选发布请求数，分子为被成功阻断并转缺口任务的请求数。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“零售户口开户机构变更查询”一个场景。主对象是开户机构变更轨迹，重点不是当前画像，而是某个户口在不同时间点的归属机构变化。

## 3.2 特性图示

运行决策台负责展示变更轨迹知识包与材料缺口提示；知识生产台中的复核与缺口任务页负责承接当前阶段仍未补齐的字段定义。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为“运行决策台”；场景补缺与复核承载在“知识生产台 -> 复核 / 缺口任务”；若变更定义调整影响其他画像口径或发布判断，需要下钻 `frontend-workbench-design.md` §5.6 影响分析页。

## 4.2 功能说明

### 4.2.1 特性场景1

1. 输入入口支持户口号和客户号；若业务只提供客户号，系统需要先收敛到目标户口，再判断是否存在开户机构变更记录。
2. 当前阶段可承诺的输出字段应至少包括户口号、客户号、原开户机构、现开户机构、变更生效日期和记录来源；其他尚未确认字段不得先写进正式输出契约。
3. 若材料中缺少关键结果字段或缺少变更判定规则，系统必须创建阻断级缺口任务，并禁止该场景进入已发布状态。
4. 时间语义默认使用变更生效日期；如果业务要求按快照日期看当前归属，应引导切换到客户基础画像场景。
5. 覆盖声明必须说明当前场景是否只覆盖已确认的变更记录来源，还是已形成全量轨迹；若仍在补口径阶段，返回结果里必须附带范围提示。
6. 策略对象允许普通变更轨迹查看，但批量导出、涉及内部机构映射说明或扩展敏感字段时需审批。
7. 接口边界以 `Scene Search API（场景搜索接口）` 和 `Knowledge Package API（知识包接口）` 为主，知识包必须返回轨迹主对象、变更时间语义、覆盖说明和当前缺口提示，不能复用客户画像的返回口径。
8. 运行决策台需要明确提示“当前场景返回的是变更轨迹，不是当前快照画像”，避免业务误把本场景当画像查询。
9. 本场景与客户基础信息查询共用客户入口与部分来源，但主对象和默认时间语义不同，不得合并为同一个场景。
10. 本场景的边界是“开户机构变更轨迹”。它不负责密码修改、名单核验或其他账户属性变更。
