# 解析抽取与证据确认特性文档

> 迭代归属：迭代一知识生产链路
> 来源主文档：`system-design.md` §5、§6.1；`frontend-workbench-design.md` §4.3、§5.4
> 对应实施计划：`docs/plans/2026-03-29-payroll-knowledge-production-implementation-plan.md`、`docs/plans/2026-03-30-import-live-graph-implementation-plan.md`

# 一、特性概述

## 1.1 背景

来源接入契约只解决“材料能不能进入系统”，并不解决“材料里到底抽出了哪些术语、场景、规则、字段和证据”。如果解析结果不能在进入建模前被人工确认，系统会把噪声文本、错误字段或无依据的经验口径带入正式治理对象，后续发布和运行都会失真。

## 1.2 目标

本特性用于把已登记材料解析成候选业务场景、候选方案资产、候选实体图谱、候选证据片段和解析疑点，并提供人工确认动作。只有被确认的证据和结构化候选结果，才能继续进入资产建模；这些候选结果必须以真实关系型控制库持久化，并能通过 `material_id（材料标识）` 与 `task_id（任务标识）` 回到来源材料。

# 二、指标

1. 进入资产建模前，所有正式引用的证据片段都必须能回到原文锚点。
2. 解析结果中的高风险冲突项必须做到 100% 人工确认，不允许自动跳过。
3. 候选证据片段的原文定位成功率不低于 95%。
4. 首轮样板材料的候选场景、候选方案和证据确认结果落库成功率达到 100%，不得只停留在瞬时预处理结果中。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“解析抽取与证据确认”一个场景。它承接材料接入后的已登记材料，把非结构化文本、`SQL（结构化查询语言，Structured Query Language）` 片段和工单样例转成候选知识要素，并在知识生产台里完成机器结果与人工确认的闭环。当前首轮以 `代发 / 薪资域` 多场景作为样板，允许同一材料同时产出多个候选场景，但不允许在本场景内直接把候选集伪装成已发布资产。

当前实体抽取采用“混合方案”：沿用现有导入任务、候选持久化和人工确认主链路，并在解析阶段新增按 `material_id（材料标识）` / `task_id（任务标识）` 隔离的 `Candidate Entity Graph（候选实体图谱）`。流程固定为“候选本体生成 -> 候选图构建 -> 实体与关系回读 -> 人工确认”，但实体类型必须适配本项目图谱，不复用 `mirofish` 中面向社媒模拟的 `Person（个人）`、`Organization（组织）` 等主体本体。

## 3.2 特性图示

页面结构固定为左侧活图谱主区、右侧当前节点或关系详情区、底部疑点与校验输出区。导入开始后，左侧先保持轻量空态并等待首批候选实体；随着候选实体和关系被逐批抽取，画布持续增长。用户在左侧选中候选场景、候选方案、标识对象、字段概念、时间语义、来源表 / 字段、候选关系或证据片段后，右侧展示对应的原文、置信信息和确认动作。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“知识生产台 -> 解析抽取”子页；证据片段深页复用 [`frontend-workbench-design.md`](../../frontend-workbench-design.md) §5.4 的证据片段详情结构。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：已确认候选集、`Candidate Entity Graph（候选实体图谱）`、`Evidence Fragment（证据片段）` 候选对象、解析疑点。
- 默认时间语义：默认继承原始材料中的时间表达，不在本场景内替用户裁定业务默认时间；冲突时间语义只以候选项和疑点形式保留。
- 覆盖范围和缺口：只覆盖“证据是否可定位、候选结构是否可信”；对覆盖声明、正式来源契约和发布可用性不作承诺。
- 策略或审批边界：本场景的门禁是人工确认，不是业务审批；高风险冲突必须人工确认后才能进入下一步。
- 前端入口、详情页和跳转链路：入口为“知识生产台 -> 解析抽取”；证据下钻进入证据片段详情深页，确认完成后跳转“资产建模”，存在阻断冲突时可转“缺口任务”。
- 接口或数据边界：本场景输入为已登记的 `material_id（材料标识）` / `task_id（任务标识）` 与来源接入契约，输出为候选场景、候选方案、候选实体对象、候选关系、证据片段和疑点，不直接生成正式 `Scene（业务场景）` 发布结果。
- 存储落位边界：候选解析结果、候选实体图谱、确认动作和疑点状态落关系型控制库与任务级隔离图空间，原文锚点和证据摘录指向对象存储原文；后续可投影为图谱证据关系，但不在本场景内直接发布到已发布主图。

1. 输入对象固定为已经完成来源接入契约登记的材料；未登记完整的材料不得进入本场景。
2. 系统需要输出六类候选结果：候选业务场景、候选方案资产、候选实体对象、候选关系、候选证据片段、解析疑点与冲突说明。对 `代发 / 薪资域` 首轮样板，候选业务场景至少允许同时覆盖“代发明细查询”“代发批次结果查询”等相邻场景，不得强制压缩成单一路径。
3. `Candidate Entity Graph（候选实体图谱）` 必须按 `material_id（材料标识）` / `task_id（任务标识）` 隔离构建。首轮固定 10 类节点：`Candidate Scene（候选场景）`、`Candidate Plan（候选方案资产）`、`Candidate Evidence Fragment（候选证据片段）`、`Identifier（标识对象）`、`Field Concept（字段概念）`、`Business Term（业务术语）`、`Time Semantic（时间语义）`、`Source Table（来源表）`、`Column（字段）`、候选表间关联。
4. 同一材料内若多个 `SQL（结构化查询语言）` 片段、本地文档中的 `Step（步骤）`、`方法1/方法2`、时段切换或来源路径差异，指向的是同一个业务问题，则系统必须优先归一为一个共同的 `Candidate Scene（候选场景）`，再把差异下沉为多个 `Candidate Plan（候选方案资产）`；不得因为存在多种取数方案就默认拆成多个场景节点。
5. 只有当以下边界发生实质变化时，才允许拆成多个 `Candidate Scene（候选场景）`：主对象不同、适用范围不同、不适用范围不同、标准输出承诺明显不同。来源表、过滤条件、适用时段、历史表切换、`SQL` 写法和中间路由差异，默认都属于 `Candidate Plan（候选方案资产）` 层变化。
6. 候选图节点和候选关系不能直接转成正式发布对象。只有经过人工确认后，才允许按既定映射落到 `Scene（业务场景）`、`Plan（方案资产）`、`Evidence Fragment（证据片段）`、`EntityAlias（实体别名）`、`Input Slot Schema（输入槽位模式）`、`Time Semantic Selector（时间语义选择器）`、`Source Contract（来源契约）` 或 `Join Relation Object（表间关联关系对象）`。
7. 证据片段必须同时保存证据类型、来源材料标题、原文锚点、内容摘录、指纹值、关联 `material_id（材料标识）` 和当前确认状态，确保后续 `Evidence Fragment（证据片段）` 可以成为正式治理对象。
8. 人工确认动作至少包括接受、驳回、合并、拆分四类；每次动作都要记录操作者、动作时间和理由。对标识归一、时间语义、来源表 / 字段和表间关联这四类高风险对象，默认要求逐条确认，不允许批量自动放行。
9. 对于存在多义性的字段、时间表达或关联键，系统只能产出“候选项 + 疑点说明”，不能直接替用户定主口径。
10. 右侧详情区展示顺序固定为“当前候选摘要 -> 原文锚点 -> 置信信息 -> 确认动作 -> 关联候选对象”，避免用户脱离上下文确认。
11. 底部疑点区必须集中展示会阻断后续建模的冲突，例如同一材料同时出现多个主对象、结果字段名称不一致、时间语义相互矛盾、敏感级别描述冲突、候选关联键前后不一致。
12. 本场景结束后，系统得到的是“可进入建模的已确认候选集”，而不是最终发布态资产；`Scene（业务场景）`、`Plan（方案资产）`、`Policy（策略对象）` 等正式对象仍在下一场景创建。
13. 本场景的边界是“让证据和候选结构变可信”。它不负责判断覆盖声明、来源契约或前端页面文案是否成立，也不把候选实体图谱直接暴露给运行时消费。

### 4.2.2 导入中活图谱与事件流

1. 本场景固定通过 `Import Preprocess Stream API（导入预处理事件流接口）` 驱动前端导入工作台；接口形态采用 `SSE（服务端事件流，Server-Sent Events）`，不额外再造第二套轮询任务机制。
2. 事件流固定为 `start`、`stage`、`graph_patch`、`draft`、`done`、`error` 六类事件。`stage` 负责阶段进度，`draft` 负责候选草稿状态，`graph_patch` 负责活图谱增量，不允许把三类语义混在一个事件体里。
3. `graph_patch` 固定为“增量补丁”而不是“整图快照”，至少包含 `patchSeq（补丁序号）`、`stageKey（阶段键）`、`chunkIndex（分块序号）`、`chunkTotal（分块总数）`、`addedNodes（新增节点列表）`、`updatedNodes（更新节点列表）`、`addedEdges（新增边列表）`、`updatedEdges（更新边列表）`、`focusNodeIds（焦点节点列表）` 和 `summary（本批次摘要）`。
4. 左侧活图谱默认空态只显示“正在等待首批实体”；图中的主角固定是候选实体和候选关系，不在首屏预摆文档节点、总览卡或静态背景图。
5. 右侧详情区默认态先显示“当前抽取进度 + 最近新增实体 / 关系”；一旦用户点中节点或关系，右侧立即完整切换为该对象详情，后续新补丁不得抢焦点。只有用户主动清空焦点后，右侧才回到默认态。
6. 活图谱更新必须遵循“局部补丁、整图缓变”原则：新节点优先落在关联节点附近，新关系做一次性描边出现，旧节点只允许轻微让位；前端不得因收到新补丁而整图清空重画。
7. 导入过程中不做硬删除。若后续抽取发现两个实体应合并、某条关系应降级或候选应驳回，旧对象先弱化为 `merged（已合并）`、`superseded（被替代）` 或 `rejected（已驳回）` 状态，直到完成态再统一收敛。
8. 导入完成后，页面继续停留在当前活图谱工作台，不跳回普通表单页；顶部阶段条标记为“导入完成”，右侧默认态切换为“最终摘要 + 待确认项”，作为进入资产建模前的最后确认界面。
