# 一、特性概述

## 1.1 背景

当前导入预处理链路的 `LLM（大语言模型，Large Language Model）` 调用实现基于兼容型 `chat/completions` 结构，默认模型和 endpoint 也仍是非 OpenAI 原生预设。这种实现可以继续工作，但不能把 OpenAI 官方当前推荐的 `Responses API（响应式接口）` 能力直接纳入正式链路；如果只替换 URL 或模型名，会导致请求体、响应解析、错误处理和回退逻辑同时失真。

## 1.2 目标

本特性用于把“导入预处理”这条链路中的模型调用适配层正式拆出来：在不破坏现有兼容型 provider 的前提下，为 OpenAI 路径增加独立的 `Responses API` 请求构造、响应解析、错误映射和能力开关，并确保导入预处理接口、预览测试接口和回退逻辑仍保持现有业务行为。

# 二、指标

1. OpenAI 路径请求成功率达到 100%。统计口径：分母为使用 OpenAI provider 预设触发的预处理测试请求数，分子为返回结构化结果或受控错误码的请求数，统计窗口为首轮回放集与系统管理测试入口。
2. 兼容型 provider 回归通过率达到 100%。统计口径：分母为现有 `chat/completions` 兼容 provider 的回放用例数，分子为迁移后仍保持既有输出结构与错误语义的用例数，统计窗口为当前导入预处理回放集。
3. 解析层字段兼容率达到 100%。统计口径：分母为导入预处理主链路消费的关键字段数，分子为在 OpenAI 路径与兼容型路径上都能稳定映射到统一域结果的字段数，统计窗口为当前 `CALIBER_PREP_V1` 输出对象。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“OpenAI Responses 预处理适配”一个场景。它位于导入预处理的 provider 适配层，负责把统一的预处理域请求转换成 OpenAI `Responses API` 所需结构，并把 OpenAI 响应重新映射回系统当前消费的统一结果对象。它不改变预处理业务语义，不直接改 Prompt 内容，也不负责 provider 预设治理页面本身。

## 3.2 特性图示

结构固定为“统一预处理域请求 -> provider 选择 -> OpenAI Responses 适配器 / 兼容型 Chat Completions 适配器 -> 统一结果解析器 -> 现有导入预处理链路”。

# 四、特性说明

## 4.1 原型（非必要）

前端主入口为系统管理中的大模型配置测试区；业务主入口为导入预处理与导入流式预处理接口。前端不新增独立页面。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：统一预处理域请求、OpenAI Responses 适配器、兼容型 Chat Completions 适配器、统一结果解析器、provider 能力标记。
- 默认时间语义：不适用；本场景只负责模型调用适配，不裁定业务时间语义。
- 覆盖范围和缺口：覆盖 OpenAI 请求构造、响应解析、错误映射、与现有兼容 provider 并存；不覆盖 Prompt 治理、结构化输出 Schema 重构、系统管理页面布局优化。
- 策略或审批边界：OpenAI 路径只允许消费正式配置或测试草稿，不允许在没有能力标记的情况下透传 OpenAI 专属字段到兼容 provider。
- 前端入口、详情页和跳转链路：入口为“系统管理 -> 大模型配置 -> 测试”，业务链路入口为导入预处理接口；不新增详情页。
- 接口或数据边界：输入为统一预处理域请求与 provider/model 配置，输出为现有 `CALIBER_PREP_V1` 结果或受控错误；不改动上游导入页面请求协议。
- 存储落位或持久化边界：provider 选择结果、错误码与测试结果沿用现有配置存储与审计路径；不新增业务主数据对象。

1. OpenAI 路径必须通过独立适配器调用 `Responses API`，不得通过“保留旧请求体只替换 endpoint”的方式硬迁移。
2. OpenAI 适配器必须把统一域请求映射为 `instructions`、结构化文本输出约束、必要的输入消息和受控参数；兼容型 provider 继续走 `messages` 结构。
3. 响应解析必须回到统一域结果，确保导入预处理、预处理测试、提示词预览和后续解析持久化逻辑不需要识别 provider 差异。
4. OpenAI 路径必须把响应失败、结构化输出失败、字段缺失和 provider 能力不匹配映射为受控错误码；不得把原始 provider 错误直接泄露到前端。
5. 若当前 provider 不支持 OpenAI 专属能力，系统必须拒绝发起对应请求，而不是静默忽略参数后继续执行。
6. 本场景与 [02-解析抽取与证据确认.md](02-解析抽取与证据确认.md) 的边界是：`02` 负责候选结果和证据确认主链路，本场景只负责模型调用适配层。
7. 本场景与“结构化输出与 Prompt 治理”专题的边界是：本场景只承接 transport 与 parse，不定义新的 Prompt 版本规则，也不负责系统管理页上的 Prompt 可视化治理。
8. 本场景结束后，系统得到的是“可切换的 provider 适配层”，而不是正式的 Prompt 治理闭环或 provider 发布门禁。
