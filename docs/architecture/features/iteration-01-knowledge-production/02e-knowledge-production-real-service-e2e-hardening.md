# 知识生产台真实服务 E2E 收口特性文档

> 迭代归属：迭代一知识生产链路
> 来源主文档：`frontend-workbench-design.md` §4.3、§4.3.1；`current-delivery-status.md`
> 对应实施计划：`docs/plans/2026-04-02-knowledge-production-real-service-e2e-hardening-implementation-plan.md`

# 一、特性概述

## 1.1 背景

知识生产台任务主线页和导入中活图谱首轮实现都已进入 `reviewing（测试与评审中）`，现有前端测试、浏览器级快照和真实服务 smoke 已经覆盖主链路，但仍缺两条最小必要、能直接支撑方案收口的真实服务 `Playwright（浏览器自动化框架）` 专项用例：空态门禁，以及完成态 `CTA（行动按钮，Call To Action）` 跳转到 `/map/scenes` 并看到真实场景数据。

## 1.2 目标

本特性只负责把“知识生产台真实服务 E2E 收口”正式化为可实施、可验收的最小专项范围：不扩成大而全的交互套件，不新增第三条用例，只把两条关键真实服务用例写清楚，并固定优先采用现有文案、标题和路由的无侵入选择器。

# 二、指标

1. 门禁型指标：两条真实服务 `Playwright` 用例必须进入稳定回归集合，并在无 `page.route / route.fulfill` 的条件下通过。
2. 门禁型指标：空态用例必须证明 `/#/production/ingest` 下 `Step 02-04` 不提供可进入动作。
3. 门禁型指标：完成态用例必须证明从“恢复处理”到“查看已处理场景”再到 `/#/map/scenes` 的真实服务跳转可稳定落页，并看到至少一条真实场景标题。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“知识生产台真实服务 E2E 收口”一个专题能力。它不重写页面结构，不改业务状态机，不改活图谱事件流，只把当前收口所必需的两条真实服务专项回归固定下来。

## 3.2 特性图示

测试闭环固定为两条：

1. `空态任务 -> 只能停留在 Step 01 -> 02-04 无可进入动作`
2. `已完成导入任务 -> 恢复处理 -> 查看已处理场景 -> /map/scenes -> 真实场景已显示`

# 四、特性说明

## 4.1 原型（非必要）

前端入口固定为 `/#/production/ingest`，目标落点固定为 `/#/map/scenes`。测试承载工具固定为仓库现有真实服务 `Playwright` 回归。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`KnowledgePage（知识生产页）`、历史任务恢复链路、主线完成态 `CTA`、`/map/scenes` 真实落页。
- 默认时间语义：不适用；本特性只验证前端交互闭环与真实后端数据联动。
- 覆盖范围和缺口：只覆盖两条最小必要真实服务用例；不扩成更多浏览器级交互，不覆盖 Mock 用例层。
- 策略或审批边界：测试过程中禁止通过 `page.route`、`route.fulfill` 伪造接口结果，必须以真实后端驱动。
- 前端入口、详情页和跳转链路：入口为 `/#/production/ingest`，完成态跳转落点为 `/#/map/scenes`。
- 接口或数据边界：依赖真实导入任务、真实恢复处理链路、真实场景列表读取接口；选择器优先使用现有标题、文案、路由和已存在的 `data-testid`，不把新增专用测试钩子作为方案前提。
- 存储落位或持久化边界：导入任务和场景数据必须来自真实控制库，不允许只在浏览器内存中构造。

1. 用例 1 固定为“空态任务只允许停留在 `Step 01`，`Step 02-04` 不提供可进入动作”。前置数据为：真实后端无有效导入任务、`draftCount=0`、无 `preprocess` 结果、无 `candidateGraph`、无 `scenes`。
2. 用例 1 的断言固定优先使用现有页面标题、现有按钮文案和路由保持，不依赖尚未确认存在的新测试钩子。至少要验证：空态标题可见、`导入并生成草稿` 按钮可见、`回到此阶段` 只存在一处、`确认质检，进入对照` / `确认对照，进入发布` / `保存当前场景` 不存在。
3. 用例 2 固定为“完成态主线 `CTA` 必须跳转到 `/#/map/scenes` 并加载真实场景数据”。前置数据为：真实后端存在 1 个可恢复的已完成导入任务；恢复后 `draftCount=0`，`publishedCount>=1`；`/map/scenes` 至少返回 1 条真实场景标题，样板标题固定为 `代发协议查询`。
4. 用例 2 固定沿用现有历史任务恢复链路：从 `/#/production/ingest` 点击 `恢复处理`，等待 `当前导入任务已处理完成`，再点击 `查看已处理场景`，最后验证 `/#/map/scenes` 已落页且真实场景标题可见。
5. 如果页面已有 `data-testid^="knowledge-restore-task-"` 之类的现成选择器，可以作为实现层优化；但本特性的正式口径不再把未确认选择器写成前置约束。
6. 本特性不负责扩展第三条真实服务用例，不负责把组件级测试替换掉，也不负责新增 Mock 版本验证。

# 五、最小完整性检查清单

1. 是否已写明迭代归属、来源主文档和对应实施计划。
2. 是否坚持只写“真实服务 E2E 收口”一个专题能力，没有混入页面结构或图谱流式协议。
3. 是否明确这轮只保留两条真实服务用例，没有扩成第三条。
4. 是否写清“禁止 `page.route / route.fulfill`”这一真实服务门禁。
5. 是否明确优先使用现有文案、标题和路由，而不是把新增测试钩子当作方案前提。
6. 是否写清入口、落点、真实后端数据前置条件和不负责范围。
