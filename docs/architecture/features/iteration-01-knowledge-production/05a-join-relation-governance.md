# 表间关联关系治理特性文档

> 迭代归属：迭代一知识生产链路  
> 来源主文档：`system-design.md` §5.1.3、§5.2、§8；`frontend-workbench-design.md` §4.3、§5.1

# 一、特性概述

## 1.1 背景

大量样板场景依赖多表关联，但当前特性文档主要覆盖“字段落位”和“来源契约”，尚未把“表与表之间为什么这样关联、标准连接键是什么、过滤条件是什么”提升为独立治理对象。没有专门场景时，多表关联知识仍会藏在 `SQL` 片段和研发记忆中。

## 1.2 目标

本特性用于把 `Join Relation Object（表间关联关系对象）` 作为一等治理资产沉淀，明确关联关系的建档、证据、适用边界、差异对比和与 `Plan（方案资产）` / `Source Contract（来源契约）` 的引用规则。

# 二、指标

1. 多表场景关联治理覆盖率达到 100%。统计口径：分母为已建模多表 `Plan` 数，分子为已引用至少一个正式 `Join Relation Object` 的 `Plan` 数，统计窗口为首轮样板域已建模场景。
2. 关联键与过滤条件可追溯率达到 100%。统计口径：分母为已建档表间关联关系数，分子为带有连接键、关联类型、业务语义和证据引用的关系数，统计窗口为知识生产链路已复核对象。
3. 关联差异可见率达到 100%。统计口径：分母为进入候选发布包的表间关联变更数，分子为能在差异视图中展示连接键或过滤条件差异的变更数，统计窗口为当前阶段候选发布包。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“表间关联关系治理”一个场景。它位于元数据对齐与来源契约固化场景旁侧，负责把多表查询所需的连接知识结构化，避免运行面继续依赖隐式 `SQL join` 经验。

## 3.2 特性图示

页面采用“左侧候选关联列表 + 中部关联编辑区 + 右侧证据与引用区 + 底部校验区”的结构。中部聚焦连接键、关联类型、过滤条件和业务语义，右侧聚焦该关系被哪些方案使用。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“知识生产台 -> 元数据对齐”中的表间关联治理分区，深页编辑入口复用 [`frontend-workbench-design.md`](../../frontend-workbench-design.md) §5.1 资产编辑页；差异展示复用 §5.7 差异视图规范。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Join Relation Object`、连接键、关联类型、过滤条件、业务语义、证据引用、引用该关联的 `Plan`。
- 默认时间语义：默认按关联关系版本时间和来源快照有效期组织本场景，不直接承接业务查询时间。
- 覆盖范围和缺口：覆盖表间关联关系建档、引用、校验和差异对比；不覆盖运行期路径命中算法，也不替代字段级来源落位。
- 策略或审批边界：本场景不直接做业务审批，但缺少证据、缺少连接键或过滤条件冲突的关联关系不得进入已复核状态。
- 前端入口、详情页和跳转链路：入口为“知识生产台 -> 元数据对齐”；可从关联详情跳到相关 `Plan`、`Source Contract` 和证据片段。
- 接口或数据边界：本场景输入为来源契约和方案对多表关联的需求，输出为正式表间关联对象；运行态只读已发布关联关系，不在运行中临时创建新关联。
- 存储落位边界：关联关系主数据、连接键、过滤条件和引用关系落关系型控制库，图谱可投影该关系用于数据地图和影响分析展示，证据锚点仍指向原始材料与证据片段。

1. 每个 `Join Relation Object` 必须具备稳定的 `join_code（关联编码）`、来源表、目标表、连接键、关联类型、业务语义和发布状态，不允许只在 `Plan` 备注里临时描述。
2. 若一个多表 `Plan` 依赖两个以上关联关系，系统必须分别建档，而不是把整段多表 `join` 逻辑打包成一条不可拆解说明。
3. 连接键和过滤条件必须可分别编辑、可分别差异对比，避免把“连接键变化”和“过滤条件变化”混在一段文本里。
4. 右侧引用区必须展示该关联关系被哪些场景和方案引用，防止关联关系变更影响面不明。
5. 若相同来源表与目标表之间存在多个业务语义不同的关联关系，必须通过 `join_code` 和适用边界显式区分，不允许互相覆盖。
6. 缺少证据引用、缺少连接键或过滤条件与来源契约冲突时，系统必须阻断并要求修正，不允许继续流向发布检查。
7. 本场景与 [05-metadata-alignment-and-source-contract.md](05-metadata-alignment-and-source-contract.md) 的边界是：`05` 负责字段与来源契约固化，本场景负责表和表之间的正式业务关联治理。
8. 本场景与样板场景文档的边界是：样板场景只消费已治理的表间关联对象，不再在单场景文档里重复定义隐式 join 规则。
9. 本场景的边界是“多表关联知识的正式治理”。它不负责业务场景召回、不负责运行时图查询执行，也不替代推理资产管理。
