# 元数据对齐与来源契约固化特性文档

> 迭代归属：迭代一知识生产链路
> 来源主文档：`system-design.md` §6.1、§8；`frontend-workbench-design.md` §4.3

# 一、特性概述

## 1.1 背景

资产建模阶段定义的是业务语义字段和治理边界，不能保证这些字段已经在物理库中找到稳定来源。若逻辑输出不经过元数据对齐就直接进入发布检查，系统会出现“契约承诺了字段，但底层无表可查”“覆盖声明已定义，但来源快照不稳定”等问题。

## 1.2 目标

本特性用于把逻辑建模结果对齐到底层物理表、物理字段和来源系统，形成正式 `Source Contract（来源契约）`。找不到物理落位的字段必须强制退回建模修改，或立即生成缺口任务。

# 二、指标

1. `required_outputs`（必返字段）物理落位校验覆盖率达到 100%。
2. 无法落位的逻辑字段必须全部转成阻断项，不允许带病进入发布检查。
3. 已固化的来源契约必须都包含表、字段、快照周期和可用性条件四类核心信息。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“元数据对齐与来源契约固化”一个场景。它承接逻辑建模后的结果，把业务字段、来源线索和覆盖要求真正落到可运行来源约束上。

## 3.2 特性图示

页面采用“双列表单或映射连线”结构。左侧展示逻辑输出需求，右侧展示候选物理表、物理字段、快照周期和来源状态，中间完成正式映射。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“知识生产台 -> 元数据对齐”子页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Source Contract（来源契约）`、物理落位映射、来源可用性约束。
- 默认时间语义：默认围绕快照周期、生效日期和新鲜度解释来源时间，不直接承接业务请求的查询时间语义。
- 覆盖范围和缺口：覆盖逻辑字段到物理表字段的正式落位；找不到落位的字段必须暴露为阻断或缺口，不允许伪装成已对齐。
- 策略或审批边界：本场景不做业务审批，但来源不稳定、快照不一致或新鲜度不满足时必须阻断后续发布检查。
- 前端入口、详情页和跳转链路：入口为“知识生产台 -> 元数据对齐”；对齐完成后可进入“复核”，阻断项直接跳转“缺口任务”。
- 存储落位边界：来源契约、字段映射和可用性状态落关系型控制库，表字段血缘关系投影到图存储，原始来源描述和扫描结果归档到对象存储或日志库。

1. 输入来源包括已建模的输出契约、覆盖声明、来源表线索和已确认推理资产；未完成逻辑建模的对象不得进入本场景。
2. 左侧逻辑需求区至少展示字段名称、业务含义、必返或可选、时间语义要求、覆盖分段要求和敏感等级。
3. 右侧物理落位区至少展示来源系统、表名、字段名、快照周期、新鲜度、有效条件和当前可用性。
4. 一旦必返字段找不到物理落位，系统只能执行两类动作：退回资产建模删除或修改该字段，或生成缺口任务等待补齐；不得人为勾选“暂时跳过”。
5. 当逻辑声明与物理快照不一致时，例如主文档要求历史全量，但来源只覆盖某一时间段，系统必须反向提示覆盖声明调整，不允许靠页面备注掩盖。
6. 对齐成功后，需要生成正式 `Source Contract（来源契约）`，并与 `Plan（方案资产）`、`Coverage Declaration（覆盖声明）` 建立引用关系。
7. 页面输出应区分“已对齐、待确认、已阻断”三类状态，避免用户误把候选物理落位当成正式来源契约。
8. 底部校验区必须明确展示哪些字段已经落位、哪些字段阻断发布、哪些字段只影响可选输出。
9. 本场景的边界是“确认真实来源”。它不负责最终业务复核与合规复核，也不负责直接执行快照切换。
