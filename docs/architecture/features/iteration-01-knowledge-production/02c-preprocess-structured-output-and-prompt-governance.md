> 对应实施计划：`docs/plans/2026-04-01-preprocess-structured-output-and-prompt-governance-implementation-plan.md`

# 一、特性概述

## 1.1 背景

当前预处理链路依赖文本 Prompt 与 `json_object` 风格约束来驱动结构化输出，系统管理页也已经承载了默认 Prompt、用户 Prompt 模板和 Schema 的管理入口。但随着 OpenAI 推荐模型与 API 发生变化，现有 Prompt 已经需要更明确的输出契约、完成标准与冲突规则，同时结构化输出治理也不能继续只停留在“有一个 JSON 字符串”。

## 1.2 目标

本特性用于把导入预处理链路中的 Prompt 版本、结构化输出 Schema、回退语义、Prompt 预览 / 测试 / 重置规则和人工审核标记收口成正式治理对象，确保后续无论底层 provider 如何切换，Prompt 与输出契约都能被审计、回放和验证。

# 二、指标

1. Prompt 版本可追溯率达到 100%。统计口径：分母为进入系统管理保存、重置或上线的 Prompt 变更次数，分子为带有版本、哈希、审核状态和变更摘要的次数，统计窗口为迁移实施周期。
2. 结构化输出校验通过率达到 100%。统计口径：分母为系统管理测试入口和回放集中返回的结构化结果数，分子为通过 Schema 校验与关键字段校验的结果数，统计窗口为迁移回放集。
3. 人工审核覆盖率达到 100%。统计口径：分母为涉及 Prompt 主语义、Schema 结构或 provider 专属能力的变更数，分子为被显式标记“需人工审核”后再进入上线或默认配置的变更数，统计窗口为迁移实施周期。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“预处理结构化输出与 Prompt 治理”一个场景。它负责 Prompt 契约、Schema 校验、预览 / 测试回放、人工审核标记和默认值治理，不负责底层 `Responses API` 适配，也不负责 provider 预设与灰度发布。

## 3.2 特性图示

结构固定为“系统默认 Prompt / Schema -> 管理员草稿 -> 预览与结构化校验 -> 人工审核标记 -> 默认配置 / 测试调用 -> 回放验证”。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“系统管理 -> 大模型配置 / Prompt 配置”区域；后端承载为现有 LLM 配置查询、更新、重置、预览与测试接口。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：默认 Prompt、Prompt 草稿、Prompt 指纹、结构化输出 Schema、人工审核标记、预览结果、测试回放结果。
- 默认时间语义：不适用；本场景关注 Prompt 与结构化输出，不解释业务时间语义。
- 覆盖范围和缺口：覆盖 Prompt 版本治理、Schema 校验、测试与审核标记；不覆盖 provider transport、模型预设和灰度发布策略。
- 策略或审批边界：涉及 Prompt 主规则、Schema 字段和 provider 专属能力的变更必须标记人工审核；未审核草稿不得直接成为默认配置。
- 前端入口、详情页和跳转链路：入口为系统管理页 Prompt 配置区；无独立详情页；测试结果停留在当前页展示。
- 接口或数据边界：输入为管理员维护的 Prompt / Schema 草稿和测试材料，输出为预览结果、结构化校验结果、测试结果和审核标记；不改变导入主页面提交协议。
- 存储落位或持久化边界：Prompt 正式配置、版本号、哈希、审核状态和摘要落关系型控制库；测试样例可沿用现有临时输入，不新增正式业务资产。

1. Prompt 治理必须把系统 Prompt、用户 Prompt 模板和结构化输出 Schema 视为一组联动契约，不允许只改其中一项就默认认为配置仍可上线。
2. 结构化输出校验必须同时覆盖“可解析 JSON”“满足 Schema”“关键业务字段齐全”“回退字段不越权”四类门禁。
3. 系统管理页必须显式区分“草稿”“默认值”“最近一次通过审核的版本”，避免管理员把未审核草稿当正式默认。
4. 涉及 OpenAI 专属结构化能力、Schema 关键字段新增删除或 Prompt 主规则变化时，必须标记为“需人工审核”；未审核时只能用于草稿测试，不得成为默认值。
5. Prompt 预览与测试结果必须包含版本、哈希、指纹和使用的 provider/model 摘要，保证后续回放可追溯。
6. 本场景与 [02b-openai-responses-preprocess-adapter.md](02b-openai-responses-preprocess-adapter.md) 的边界是：`02b` 负责 transport / parse 适配，本场景只负责 Prompt 与输出契约治理。
7. 本场景与 provider 路由治理专题的边界是：本场景不决定模型预设、endpoint 白名单和灰度发布，只消费它们暴露出来的能力矩阵。
8. 本场景结束后，系统得到的是“可审计的 Prompt / Schema 治理闭环”，而不是 provider 发布闭环。
