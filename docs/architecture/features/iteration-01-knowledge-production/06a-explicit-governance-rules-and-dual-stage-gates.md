# 显式治理规则与双阶段发布门禁特性文档

> 迭代归属：迭代一知识生产链路  
> 来源主文档：`system-design.md` §5.4、§5.5、§6.1；`frontend-workbench-design.md` §4.3、§6.1  
> 对应实施计划：`docs/plans/2026-04-02-explicit-governance-rules-dual-stage-gates-implementation-plan.md`

# 一、特性概述

## 1.1 背景

当前 `Import Confirm（导入确认）` 会在候选场景确认时隐式补齐最小 `Dictionary（字典）`、`Identifier Lineage（标识链）` 与 `Time Semantic Selector（时间语义选择器）`。这能临时打通 `MVP（最小可行产品，Minimum Viable Product）` 主链路，但会把“导入动作”和“治理动作”混在一起，导致用户看不到真正缺了什么，也无法审计“为何能够发布”。

## 1.2 目标

本特性用于把治理完整性从隐式补种子改为显式规则闭环：

1. `Import Confirm（导入确认）` 只负责把候选场景落成 `Scene Draft（场景草稿）`，并显式产出治理缺口。
2. 正式治理对象只能通过对应治理入口创建或补齐，不能由导入流程偷偷兜底。
3. `Publish Gate（发布门禁）` 只消费结构化规则评估结果与缺口状态，不再做隐式修复。

# 二、指标

1. 候选场景确认后，失败治理规则必须全部可见，并能定位到具体缺口任务。
2. 阻断级治理缺口未关闭前，发布检查通过率必须为 `0%`。
3. 发布阻断原因必须可结构化返回到前端，而不是只返回拼接字符串。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“显式治理规则与双阶段发布门禁”一个专题能力。它位于“候选场景确认 -> 治理对象补齐 -> 发布检查”之间，负责定义治理缺口如何产生、如何展示、如何阻断发布。

## 3.2 特性图示

链路固定为：

1. `Import Confirm（导入确认）`：创建或更新草稿场景，并执行 `IMPORT_CONFIRM` 阶段规则评估。
2. `Governance Gap（治理缺口）`：对失败规则创建或刷新 `Gap Task（缺口任务）`。
3. `Governance Editing（治理补齐）`：通过字典、标识链、时间语义等正式治理入口补齐对象。
4. `Pre Publish Check（发布前检查）`：执行 `PRE_PUBLISH` 阶段规则评估，读取未关闭阻断缺口并决定能否发布。

# 四、特性说明

## 4.1 前端承载

前端承载页面固定为：

- “知识生产台 -> 场景建模 / 发布检查”显示治理缺口摘要和规则状态。
- “知识生产台 -> 缺口任务”承接阻断级缺口的处理与关闭。

## 4.2 功能说明

### 4.2.1 显式治理规则对象

通用补充项：

- 主对象：`Governance Rule Evaluation（治理规则评估结果）`、`Gap Task（缺口任务）`、`Publish Gate Summary（发布门禁摘要）`
- 默认时间语义：本场景默认关注规则评估时间、缺口任务更新时间和发布检查时间，不承接业务查询时间
- 覆盖范围和缺口：覆盖导入确认后的治理完整性判断、缺口任务生成与发布阻断摘要；不覆盖独立治理工作台，不覆盖通用规则 `DSL（领域特定语言，Domain-Specific Language）`
- 策略或审批边界：仅治理对象完整性进入本特性；高敏审批仍沿原审批链路处理
- 存储落位边界：规则评估结果、缺口任务状态和发布阻断摘要落关系型控制库；正式治理对象仍落现有治理资产表

1. `Governance Rule Evaluation（治理规则评估结果）` 必须记录 `rule_code（规则编码）`、`stage（阶段）`、`status（结果状态）`、`blocking_level（阻断等级）`、`detail_json（明细 JSON）` 与评估时间，用来表达“系统判断过什么”。
2. `Gap Task（缺口任务）` 必须引用具体 `scene_id（场景标识）`、`rule_code（规则编码）` 与缺失对象类型，不能创建无主缺口。
3. `Publish Gate Summary（发布门禁摘要）` 必须能返回 `failed_rules（失败规则）`、`open_blocking_gaps（未关闭阻断缺口）` 与可读摘要，供前端和发布命令复用。

### 4.2.2 双阶段规则执行

1. `Import Confirm（导入确认）` 只做三件事：创建或更新草稿场景、执行 `IMPORT_CONFIRM` 规则评估、创建或刷新治理缺口；不得再隐式补齐 `Dictionary（字典）`、`Identifier Lineage（标识链）`、`Time Semantic Selector（时间语义选择器）`。
2. 第一版显式规则固定只覆盖三条：
   - `GR-DICT-001`：场景至少存在一个活动中的 `Dictionary（字典）`
   - `GR-IDL-001`：场景至少存在一个活动中的 `Identifier Lineage（标识链）`
   - `GR-TIME-001`：场景至少存在一个活动中的 `Time Semantic Selector（时间语义选择器）`
3. `Publish Check（发布检查）` 必须在 `PRE_PUBLISH` 阶段重新评估同一组规则，并同时读取当前未关闭的阻断级缺口。
4. 发布命令只消费结构化摘要；若失败，只能返回失败规则和缺口摘要，不能执行自动补种子。

### 4.2.3 前端可见性

1. 知识生产台中的治理摘要区必须明确展示“失败规则 / 阻断等级 / 缺什么对象 / 当前缺口状态”。
2. 发布检查矩阵必须把治理规则失败与一般控制资产失败区分开来，避免用户误以为只是普通字段未填。
3. 页面必须支持从治理规则摘要或发布检查阻断项跳转到“缺口任务”或对应治理对象编辑页。

### 4.2.4 与相邻特性的边界

1. 与 [`03-资产建模与治理对象编辑.md`](03-资产建模与治理对象编辑.md) 的边界是：`03` 负责正式治理对象如何建模和编辑；本特性只负责何时认定这些对象缺失，以及如何阻断发布。
2. 与 [`06-复核与缺口任务协同.md`](06-复核与缺口任务协同.md) 的边界是：`06` 负责缺口任务与复核主线的协同状态机；本特性负责把“治理对象缺失”落成结构化规则与缺口来源。
3. 与 [`../iteration-02-runtime-and-governance/07-发布检查、灰度发布与回滚.md`](../iteration-02-runtime-and-governance/07-发布检查、灰度发布与回滚.md) 的边界是：`07` 负责完整发布流程与灰度切换；本特性只负责治理完整性门禁输入。
