# 推理资产详情与前台可见特性文档

> 迭代归属：迭代一知识生产链路  
> 来源主文档：`system-design.md` §4.4、§5.1、§15；`frontend-workbench-design.md` §5.4、§6.3

# 一、特性概述

## 1.1 背景

主方案已经把 `Inference Assertion（推理结论）`、`Inference Chain（推理链）` 和 `Inference Scope Profile（推理适用边界）` 定义为正式资产，但当前特性目录里还没有专门承接“推理资产在前台如何持续可见、如何下钻、如何在不同工作台复用”这一能力。没有独立场景时，推理资产容易退化成只在推理校验页短暂可见的中间结果。

## 1.2 目标

本特性用于把推理资产详情深页、跨工作台入口和前台展示口径固定下来，让推理结论在知识生产、运行决策、数据地图、发布中心和监控审计中都能被一致理解和引用。

# 二、指标

1. 推理资产详情可达率达到 100%。统计口径：分母为出现在数据地图、运行决策台、发布中心和监控与审计中的正式推理资产入口数，分子为可跳转到统一推理详情页的入口数，统计窗口为首轮已实现前台入口。
2. 推理详情结构化展示完整率达到 100%。统计口径：分母为进入详情页的正式推理资产数，分子为同时展示结论摘要、推理链、适用边界、运行模式和快照对信息的资产数，统计窗口为样板回放集。
3. 历史锁定态复现成功率达到 100%。统计口径：分母为以 `replay / frozen` 模式进入推理详情页的次数，分子为按原快照对成功加载且未退化为最新态的次数，统计窗口为审计回放与发布差异复盘样本。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“推理资产详情与前台可见”一个场景。它把 [推理校验与推理链复核](04-inference-validation-and-chain-review.md) 产生的正式推理资产延伸到运行与治理工作台，使推理资产不再只是知识生产过程中的临时校验对象。

## 3.2 特性图示

页面采用“左侧推理结论摘要 + 中部推理链步骤图 + 右侧适用边界与运行信息 + 底部证据联动区”的结构。不同工作台进入时使用统一详情骨架，只改变入口上下文和锁定模式。

# 四、特性说明

## 4.1 原型（非必要）

前端主承载为统一的推理资产详情深页，入口来自“知识生产台 -> 推理校验”、“运行决策台”、“数据地图”、“发布中心”和“监控与审计”；证据联动复用 [`frontend-workbench-design.md`](../../frontend-workbench-design.md) §5.4 证据片段详情页。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Inference Assertion`、`Inference Chain`、`Inference Scope Profile`、`Inference Confidence Record（推理置信记录）`、推理详情入口、推理详情深页。
- 默认时间语义：默认以推理资产所属快照对和运行时模式解释本场景，不直接改写业务查询时间。
- 覆盖范围和缺口：覆盖推理资产详情页、跨工作台可见规则、锁定态加载和证据联动；不覆盖推理规则建模本身，也不替代运行决策台的完整知识包解释。
- 策略或审批边界：本场景可展示推理资产的适用边界和降级原因，但不得据此绕过 `Policy（策略对象）` 或审批链路直接暴露高敏结果。
- 前端入口、详情页和跳转链路：入口来自 `04`、`08`、`10`、`11`、`07` 等场景；可继续跳到证据详情、审计追踪或数据地图焦点对象。
- 接口或数据边界：本场景只消费已发布或已复核的推理资产详情接口与快照查询接口，不直接修改推理资产；知识生产侧的确认动作仍由 `04` 承接。
- 存储落位边界：推理资产主数据落关系型控制库，推理链步骤关系和证据引用关系可投影到图存储，历史访问和跳转留痕进入审计事件链。

1. 推理详情页必须至少展示六个固定区块：推理结论摘要、推理链步骤图、适用边界、运行模式、降级原因编码、快照对信息。
2. 数据地图、运行决策台、发布中心和监控与审计中的推理资产入口必须全部收口到同一详情页，不允许各自维护不同版本的推理展示逻辑。
3. 若入口以 `latest（最新态）` 进入，页面按当前已发布快照加载；若入口以 `replay / frozen` 进入，页面必须锁定传入快照对，不允许静默重算。
4. 推理链步骤图中的每一步都必须能下钻查看对应证据和规则引用，确保“为什么得出这个结论”在前台可追溯。
5. 详情页必须显式区分“正式推理结论”“低置信候选”“已挂起待复核”三类状态，避免候选结果被误读为正式资产。
6. 运行态若采用 `deterministic_only（仅确定性推理）`、`template_only（仅模板路径）` 或 `clarification_only（仅澄清返回）`，详情页必须展示对应 `runtime_mode（运行模式）` 与 `degrade_reason_codes（降级原因编码列表）`。
7. 本场景与 [04-inference-validation-and-chain-review.md](04-inference-validation-and-chain-review.md) 的边界是：`04` 负责知识生产侧的复核动作，本场景负责推理资产被发布后的持续可见与统一下钻。
8. 本场景与 [08-runtime-decision-and-knowledge-package.md](../iteration-02-runtime-and-governance/08-runtime-decision-and-knowledge-package.md) 的边界是：`08` 负责解释一次请求的九步运行主线，本场景只负责解释某个推理资产对象本身。
9. 本场景的边界是“推理资产详情与可见性规则”。它不负责新建推理结论，不负责审批动作，也不替代数据地图总览。
