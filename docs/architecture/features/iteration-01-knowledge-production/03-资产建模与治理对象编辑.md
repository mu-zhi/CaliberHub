# 资产建模与治理对象编辑特性文档

> 迭代归属：迭代一知识生产链路  
> 来源主文档：`system-design.md` §5.1.1-§5.1.3、§6.1、§7.4-§7.7；`frontend-workbench-design.md` §4.3、§7.5

# 一、特性概述

## 1.1 背景

解析抽取得到的是候选知识要素，还不是系统可以运行和发布的正式治理资产。若没有专门的资产建模场景，`Dictionary（字典）`、`Input Slot Schema（输入槽位模式）`、`Identifier Lineage（标识谱系）`、`Time Semantic Selector（时间语义选择器）`、`Coverage Declaration（覆盖声明）`、`Contract View（契约视图）` 和 `Policy（策略对象）` 就会继续零散维护，无法形成统一主键、统一状态机、统一送复核门禁，也无法支撑后续运行决策、审批导出和数据地图解释链路。

## 1.2 目标

本特性用于把已确认的候选结果落成正式治理对象，统一建立 `Scene（业务场景）`、`Plan（方案资产）`、`Output Contract（输出契约）`、`Contract View（契约视图）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Dictionary（字典）`、`Input Slot Schema（输入槽位模式）`、`Identifier Lineage（标识谱系）` 与 `Time Semantic Selector（时间语义选择器）` 等对象，并提供结构化编辑、对象引用校验与差异对比能力。当前首轮交付要求这些对象以真实关系型控制库作为正式真源，并为后续 `Version Snapshot（版本快照）` 发布准备稳定逻辑主键与版本主键。

# 二、指标

1. 正式治理对象必填字段完整率达到 `100%`。统计口径：分子为通过建模完整性校验的正式治理对象数，分母为进入送复核流程的正式治理对象总数，统计窗口为单次候选建模批次。
2. 未绑定有效 `Contract View（契约视图）`、`Coverage Declaration（覆盖声明）` 或 `Policy（策略对象）` 的 `Plan（方案资产）` 送复核放行次数为 `0`。统计口径：分子为被错误放行的 `Plan` 数，分母为进入送复核校验的 `Plan` 总数，统计窗口为当前迭代样板域建模批次。
3. 码值字段引用已发布 `Dictionary（字典）` 覆盖率达到 `100%`。统计口径：分子为引用已发布字典的码值字段数，分母为正式 `Output Contract（输出契约）` 中所有码值字段数，统计窗口为单次送复核版本。
4. 存在多标识或多时间语义冲突的场景中，缺少 `Input Slot Schema`、`Identifier Lineage` 或 `Time Semantic Selector` 仍被送复核的次数为 `0`。统计口径：分子为校验漏放次数，分母为存在此类冲突的场景总数，统计窗口为当前样板域建模阶段。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“资产建模与治理对象编辑”一个场景。它是知识生产链路的正式建模中心，把机器解析结果转成可复核、可发布、可审计的治理资产。当前首轮以 `代发 / 薪资域` 多场景为样板，要求同域内不同场景在标识类型、时间覆盖、来源路径、输出契约或风险边界上具备真实差异。

## 3.2 特性图示

页面结构采用“主编辑区 + 右侧辅助视图 + 底部校验区 + 右上角完整度状态提示”的工作台模式。编辑主轴按治理对象分组展开，而不是按底层存储结构分页；差异视图、证据引用与校验输出始终围绕当前对象展示，不把资产编辑退化成原始 `JSON（JavaScript 对象表示法，JavaScript Object Notation）` 文本编辑器。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“知识生产台 -> 资产建模”子页，深页编辑入口复用资产编辑页结构。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Scene（业务场景）`、`Plan（方案资产）`、`Output Contract（输出契约）`、`Contract View（契约视图）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Dictionary（字典）`、`Input Slot Schema（输入槽位模式）`、`Identifier Lineage（标识谱系）`、`Time Semantic Selector（时间语义选择器）`。
- 默认时间语义：本场景本身不执行业务查询，因此默认时间语义对当前编辑动作“不适用”；但凡面向运行查询的对象，都必须在建模时显式定义默认时间语义或显式声明“必须澄清”，不允许以隐含默认值跳过。
- 覆盖范围和缺口：覆盖正式治理对象建模、引用关系建立、完整性校验和差异对比；不覆盖物理来源表字段落位、不覆盖推理链可信性校验、不覆盖运行期命中效果。
- 策略或审批边界：本场景定义 `Policy（策略对象）`、字段级裁剪和审批模板，但不执行正式审批；未定义决策三态、字段裁剪或高敏边界的对象不得进入已复核状态。
- 前端入口、详情页和跳转链路：入口为“知识生产台 -> 资产建模”；对象可下钻到资产编辑深页，建模完成后流转到“推理校验”或“元数据对齐”。
- 接口或数据边界：本场景输入为已确认候选集、材料引用和证据引用，输出为正式治理对象草稿；正式对外可见能力必须等待后续发布与快照切换完成。
- 存储落位边界：治理对象主数据落关系型控制库，对象间正式关系与引用边投影到图存储，差异对比和版本快照索引进入控制资产快照体系；图谱不能反向替代本场景的主数据真源。

1. 主编辑区必须支持以下正式对象的结构化编辑：`Scene（业务场景）`、`Plan（方案资产）`、`Output Contract（输出契约）`、`Contract View（契约视图）`、`Coverage Declaration（覆盖声明）`、`Policy（策略对象）`、`Dictionary（字典）`、`Input Slot Schema（输入槽位模式）`、`Identifier Lineage（标识谱系）`、`Time Semantic Selector（时间语义选择器）`。这些对象都是一等治理对象，不允许退化成 `Scene` 或 `Plan` 上的自由注释字段。
2. `Dictionary（字典）` 必须作为独立资产登记，至少具备 `dict_code`、分类、版本、发布状态、条目列表和引用方。凡 `Output Contract` 或 `Derivation Rule（派生规则）` 涉及码值翻译、枚举定义或业务映射时，都必须引用已发布 `Dictionary`；未引用字典、引用未发布字典或形成孤儿字典时，不得进入送复核或发布链路。
3. `Input Slot Schema（输入槽位模式）` 必须明确槽位名、类型、必填性、标准化规则、澄清提示和匹配失败回退动作。标识槽位、时间槽位和普通筛选槽位必须分开定义；未完成槽位建模的运行场景不得被标记为“可运行”。
4. `Identifier Lineage（标识谱系）` 只允许登记稳定、可审计、可回溯的显式映射关系。每条映射都必须绑定证据引用、来源说明和确认状态；来源不明的经验性映射不能直接作为运行时默认规则。运行时只能消费已发布谱系，不得反向把临时推测写回正式真源。
5. `Time Semantic Selector（时间语义选择器）` 必须定义默认时间、可选时间、触发澄清词和解释优先级。存在多时间语义冲突的 `Scene` 或 `Plan` 必须挂接时间语义选择器；若未定义默认语义或未声明“必须澄清”，则不得进入送复核状态。
6. `Coverage Declaration（覆盖声明）` 必须表达“查到哪里、缺哪里、缺了怎么办”。每个 `Plan` 至少要有一份正式覆盖声明，明确覆盖分段、完整度、排除项、缺口说明和回退动作；阻断级缺口不得被“先上线后补”替代。
7. `Contract View（契约视图）` 是字段级安全唯一真值来源，不是 `Output Contract` 的附注。它必须显式定义角色范围、字段裁剪、脱敏方式和审批模板；前端、导出任务和下游服务都只能消费 `Contract View` 给出的字段集，不得绕过它直接拼字段。
8. `Policy（策略对象）` 负责输出 `allow（允许）`、`need_approval（需要审批）`、`deny（拒绝）` 三态决策，并定义访问规则、脱敏规则、审批规则、导出规则和留痕要求。高敏日志、名单命中、冻结原因、`S3 / S4` 字段等不得落入普通 `allow` 路径；策略变更应按高风险对象进入影响分析和发布门禁。
9. `Plan（方案资产）` 进入送复核前，必须完成对象引用一致性校验：至少绑定有效的 `Coverage Declaration`、`Policy`、`Contract View`，必要时还要绑定 `Dictionary`、`Identifier Lineage`、`Time Semantic Selector` 和证据引用。缺对象、引用失效、字段未兑现或规则互相冲突时，一律阻断送复核并生成建模修正或缺口任务。
10. 资产编辑页必须支持“当前草稿 vs 已发布版本”差异视图，按字段逐项展示新增、修改、删除，不允许只给整段文本 diff（差异对比）。可直接修改的仅限当前草稿对象；若差异来自候选确认错误或来源定义错误，必须回到上游候选确认或元数据对齐场景处理。
11. “知识生产台 -> 场景编辑与发布”中的 `业务领域` 选择器在检测到领域表为空时，必须允许用户通过“重载业务领域”从 [`research/source-materials/01-当前业务侧的业务场景分类.md`](../../../../research/source-materials/01-当前业务侧的业务场景分类.md) 的一级分类初始化默认领域；运行时可通过由该文档派生的 `static/business_categories.json` 读取，但初始化结果必须与该文档的一级分类口径一致。初始化只创建一级 `Domain（业务领域）`，不把二级、三级或更细业务分类直接写入领域表，并保持幂等，避免重复创建同名或同编码领域。
12. 本场景的边界是“完成正式治理对象建模”。它不负责来源表和字段的物理落位，那是“元数据对齐与来源契约固化”场景的职责；不负责推理资产可信性和阈值校验，那是“推理校验与推理链复核”场景的职责；也不负责运行态知识包解释，那是“运行决策与知识包生成”场景的职责。
