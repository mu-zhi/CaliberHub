# 推理校验与推理链复核特性文档

> 迭代归属：迭代一知识生产链路
> 来源主文档：`system-design.md` §4.4、§6.1；`frontend-workbench-design.md` §4.3、§6.3

# 一、特性概述

## 1.1 背景

本项目的运行链路依赖 `Inference Rule（推理规则）`、`Inference Assertion（推理结论）`、`Inference Chain（推理链）` 和 `Inference Scope Profile（推理适用边界）`。如果这些对象只在后台自动生成、没有独立校验场景，运行时就会出现标识收敛错误、时间语义误判、关系补全失真和高风险场景误路由。

## 1.2 目标

本特性用于把推理资产从“自动生成结果”变成“可人工复核、可发布、可审计的正式资产”。在进入元数据对齐和发布前，治理人员必须能看见推理结论、推理链、冲突、置信门禁和适用边界。

# 二、指标

1. 确定性推理冲突必须做到 100% 阻断，不允许直接发布。
2. 每条进入发布态的推理结论都必须绑定至少一条可回放推理链。
3. 低置信候选与待人工复核结论必须能被独立筛出，不能混入已确认结论列表。

# 三、特性全景

## 3.1 特性全景描述

本特性只覆盖“推理校验与推理链复核”一个场景。它位于资产建模之后、元数据对齐之前，是知识生产台中专门负责推理资产质量门禁的环节。

## 3.2 特性图示

页面固定为左侧推理结果区、右侧推理链区、底部冲突与门禁区。用户先看结论，再沿推理链核对证据，最后处理冲突和置信门禁。

# 四、特性说明

## 4.1 原型（非必要）

前端承载页面为“知识生产台 -> 推理校验”子页；推理详情深页与运行消费侧的推理详情入口保持同一对象口径。

## 4.2 功能说明

### 4.2.1 特性场景1

通用补充项：
- 主对象：`Inference Rule（推理规则）`、`Inference Assertion（推理结论）`、`Inference Chain（推理链）`、`Inference Scope Profile（推理适用边界）`。
- 默认时间语义：默认使用推理适用边界中定义的时间范围，不以当前页面打开时间替代业务时间语义。
- 覆盖范围和缺口：覆盖推理资产可信度、冲突和适用边界复核；不覆盖运行期知识包组装和正式返回解释。
- 策略或审批边界：确定性冲突直接阻断，低置信候选只可挂起待复核；本场景不执行业务审批，但为后续发布和运行策略提供门禁输入。
- 前端入口、详情页和跳转链路：入口为“知识生产台 -> 推理校验”；可下钻推理详情，确认完成后跳转“元数据对齐”或“复核”。
- 存储落位边界：推理规则、结论和适用边界元数据落控制库，推理链关系和证据引用关系落图存储，复核动作进入审计事件链。

1. 左侧推理结果区按结论类型展示标识收敛、时间收敛、关系补全和候选场景收敛等结果，并显示置信分值、适用范围和复核状态。
2. 右侧推理链区必须能回放“输入事实 -> 规则命中 -> 中间节点 -> 最终结论”的完整结构，且每一步都能展开证据引用与规则原文。
3. 底部冲突与门禁区固定展示三类信息：确定性冲突、置信门禁命中结果、推理适用边界摘要。
4. 确认动作只有四类：确认、驳回、挂起待人工复核、查看适用边界；任一动作都要留下复核人和理由。
5. 确定性推理冲突必须直接阻断，不允许通过“暂时忽略”方式跳到下一步；置信度冲突可以挂为候选，但不能伪装成正式结论。
6. 推理适用边界至少要明确业务域范围、时间范围、来源范围、禁止场景和阈值组；缺任一项时不得进入可发布状态。
7. 页面排序固定为“阻断类冲突 -> 低置信候选 -> 已确认结论”，确保问题优先暴露，而不是把已确认项堆在最前面。
8. 本场景结束后，系统得到的是已确认或已挂起的推理资产集，供元数据对齐与后续发布门禁消费。
9. 本场景的边界是“推理资产可信不可信”。它不负责决定页面如何展示，也不直接返回运行态知识包。
