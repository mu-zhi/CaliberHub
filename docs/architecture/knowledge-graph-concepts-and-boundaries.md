# 知识图谱概念与边界

## 1. 文档定位

本文是数据直通车主方案旁的独立支撑文档，用于回答三个问题：

1. 知识图谱在本项目里到底指什么，不指什么。
2. 哪些概念会直接影响主方案建模与对象边界。
3. 哪些常见实现路线可以作为研究参考，但当前阶段不应直接写成正式承诺。

本文不是知识图谱技术教程，也不是某个图数据库、`RDF（资源描述框架，Resource Description Framework）`、`SPARQL（图查询语言，SPARQL Protocol and RDF Query Language）` 或图神经网络方案的选型文档。对外部实现路线的引用，只用于帮助裁剪项目边界，不作为当前仓库的正式实现约束。

## 2. 为什么数据直通车需要这份文档

数据直通车当前要解决的核心问题，不是“能不能把文档转成一张图”，而是“能不能把口径知识组织成可治理、可发布、可解释、可追溯的正式对象体系”。如果没有一份独立的概念边界文档，主方案在后续扩展时很容易出现三种偏差。

第一种偏差，是把知识图谱误写成存储技术决策。例如把“建设知识图谱”直接等同于“引入某个图数据库产品”。这会让方案过早绑定实现栈，却没有先说明图中到底有哪些对象、哪些关系、哪些证据和哪些约束。

第二种偏差，是把知识图谱误写成下游消费方式。例如把 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 当成上游治理本体，导致抽取、建模、发布和运行都围着问答链路组织，而不是围着口径治理组织。

第三种偏差，是把知识图谱误写成“一个大而全的技术总装系统”。很多外部介绍会把数据采集、清洗、抽取、融合、存储、推理、应用、运维一次性列成完整项目清单。这个视角适合做通用介绍，但对数据直通车不够精确。数据直通车当前阶段需要的是“哪些概念必须进入正式对象层”，而不是“把所有知识图谱常见模块都纳入首轮建设范围”。

因此，本文的作用不是扩充主方案篇幅，而是给主方案提供一套稳定的概念判断依据。

## 3. 数据直通车语境下的知识图谱定义

在本项目里，`Knowledge Graph（知识图谱）` 指的是一种受治理的知识组织方式：它把业务场景、治理资产、来源材料、证据、策略、时间边界、对象之间的关联关系组织成结构化关系网络，并要求这些关系能够被解释、被追溯、被发布、被校验。

这里有三个关键词必须同时成立。

第一，知识图谱不是只有“点和线”的可视化结构。图可视化只是展示方式，不是定义本身。真正重要的是：对象是否有稳定身份，关系是否表达明确业务语义，证据是否能回溯到正式来源，约束是否能限制错误连接和错误解释。

第二，知识图谱不是脱离治理的自由知识网络。数据直通车面对的是银行场景，任何进入正式图谱的对象和关系，都必须能说明其来源、适用边界、发布时间、责任角色和下游消费限制。

第三，知识图谱不是孤立存在的存储层。它要与关系型控制库中的正式治理对象、版本快照、发布流程和运行服务共同构成受控知识底座。也就是说，本项目里的知识图谱是“治理对象的图式表达与服务化投影”，而不是一套独立替代主数据真源的平行系统。

## 4. 对主方案有直接约束力的核心概念

### 4.1 `Entity（实体）`

实体是图中可被独立识别、独立治理、独立引用的对象。对数据直通车而言，实体不是“从文本里抽到的任何名词”，而是能够进入正式治理边界的业务对象、治理对象或共享锚点。

在当前方案里，实体至少包含三类：

1. 业务场景实体，例如 `Scene（业务场景，Business Scene）`。
2. 治理资产实体，例如 `Plan（方案资产，Plan Asset）`、`Output Contract（输出契约，Output Contract）`、`Policy（策略对象，Policy）`、`Source Contract（来源契约，Source Contract）`。
3. 共享归一实体，例如 [`Canonical Entity（统一实体）`](/Users/rlc/Code/CaliberHub/docs/architecture/system-design.md)，用于把跨场景共享对象收口到稳定身份。

这一定义直接约束主方案：不是所有抽取得到的候选名词都应成为正式实体。只有能够被治理、被发布、被解释、被审计的对象，才有资格进入正式实体层。

### 4.2 `Relation（关系）`

关系是两个实体之间具有明确业务语义的连接。关系的价值不在“连起来”，而在“为什么连、按什么口径连、何时成立、由什么证据支持”。

对数据直通车而言，关系至少要覆盖以下几类语义：

1. 归属关系，例如某个场景资产属于哪个统一实体。
2. 使用关系，例如某个场景使用哪条业务口径、哪条派生规则、哪份字典。
3. 依赖关系，例如某个方案依赖哪些输入、哪些来源契约、哪些上游对象。
4. 证据关系，例如某个结论、规则或对象由哪段证据支撑。
5. 版本关系，例如某个对象与哪个快照、哪个发布批次、哪个替代版本相关。

这一定义约束主方案不要把关系降格为“任意可连的边”。在本项目中，关系必须携带稳定语义，不能只因为文本共现、字段相似或向量相近就直接进入正式图谱。

### 4.3 `Attribute（属性）`

属性是附着在实体或关系上的描述性字段，用于补充对象状态、时间边界、来源说明、置信度、敏感等级和其他运行约束。

属性不是知识图谱的次要部分。对数据直通车来说，很多治理信息恰恰通过属性表达，例如：

1. 生效时间、失效时间和默认时间语义。
2. 敏感等级、审批要求和导出限制。
3. 证据出处、确认状态和人工覆盖结果。
4. 置信度、优先级和冲突处理结果。

这一定义约束主方案不能只写实体和关系，而忽略对象属性。没有属性，很多“关系为什么成立、是否现在仍成立、能否对外展示”的治理判断就无法落地。

### 4.4 `Ontology（本体）`

本体不是为了追求学术完备性而额外增加的一层抽象，而是用来规定“什么对象能进来、允许发生哪些关系、哪些约束不能被破坏”的概念边界。

对数据直通车而言，本体至少承担三件事：

1. 规定对象类型边界，例如哪些是业务场景，哪些是治理资产，哪些是证据对象，哪些只是候选抽取结果。
2. 规定关系合法性，例如什么对象之间允许存在“使用”“归属”“替代”“支撑”等关系。
3. 规定约束条件，例如敏感对象不能绕过审批直接暴露，失效对象不能参与发布快照，未确认候选对象不能直接成为正式真源。

这一定义约束主方案：当前阶段不必承诺引入完整的 `OWL（本体语言，Web Ontology Language）` 工具链，但必须承诺存在一套项目内有效的概念类型与关系约束。否则，知识图谱会退化成“结构化得不够彻底的连接表”。

### 4.5 `Evidence（证据）`

证据不是知识图谱的附件，而是正式事实进入治理边界的前提。数据直通车不是面向公开百科问答，而是面向银行口径治理。任何正式对象和正式关系，如果无法说明其证据来源，就不应进入已发布快照。

本项目中的证据至少要回答四个问题：

1. 事实来自哪份材料、哪段文本、哪个样例、哪条元数据或哪次人工确认。
2. 事实在什么上下文中成立。
3. 事实由谁确认、何时确认、是否已被后续版本替代。
4. 下游消费时，是否需要同时展示证据摘要或审计说明。

这一定义约束主方案继续把证据对象放在核心层，而不是把证据回溯当成可选增强项。

### 4.6 `Semantic Layer（语义层）`

语义层不是知识图谱的别名，也不是图谱的替代物。它更像是“把业务表达收敛为可治理概念、词元、规则和边界锚点”的组织层。

对数据直通车而言，语义层负责把“业务怎么说”转成“系统按什么正式对象理解”，它解决的是术语归一、口径定义、别名映射、时间语义、边界锚点和规则引用问题。知识图谱则负责把这些正式对象与它们之间的结构关系组织起来，并把关系网络服务给发布、浏览、检索和解释链路。

这一定义约束主方案不要把语义层和知识图谱混写成一个词。两者高度相关，但职责不同：语义层更偏概念组织与解释入口，知识图谱更偏正式对象和关系网络的结构化表达。

## 5. 四组最容易混淆的边界

### 5.1 知识图谱不等于图数据库

图数据库是存储与查询图结构的一类产品形态，知识图谱是知识组织与治理方式。没有图数据库，也可以存在知识图谱；用了图数据库，也不自动得到知识图谱。

对数据直通车的直接约束是：主方案可以承认图谱投影、图遍历、图查询和图展示的必要性，但不应把“选择某个图数据库产品”写成知识图谱成立的前提。

### 5.2 知识图谱不等于 `RAG（检索增强生成，Retrieval-Augmented Generation）`

`RAG（检索增强生成，Retrieval-Augmented Generation）` 解决的是“生成前如何召回上下文”，知识图谱解决的是“知识本身如何被组织、约束、解释和治理”。前者是消费方式，后者是知识底座。

如果把两者混为一谈，方案就会偏向“为回答问题临时拼接上下文”，而忽略正式对象、版本边界、证据链和发布机制。

### 5.3 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 不等于知识图谱上游治理

`GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 的价值，在于让图结构参与召回、路径约束、上下文组织和生成解释。它更适合放在知识图谱的下游消费层，而不是放在上游治理层充当建模总纲。

对本项目来说，`GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 可以是未来运行服务面的重要消费方式，但它不应该反向决定正式对象集、正式关系集和发布边界。上游治理必须先围绕口径治理、证据追溯和正式发布来设计。

### 5.4 语义层不等于图谱投影

语义层强调“业务表达如何映射为正式概念”，图谱投影强调“正式对象如何在图结构中展示、检索和遍历”。两者相互支撑，但不能互相替代。

如果只建设图谱投影，不建设语义层，系统会缺少术语归一、业务表达收口和规则锚点；如果只建设语义层，不建设图谱表达，系统又难以支持复杂关系浏览、路径解释和跨对象影响分析。

## 6. 这些概念在数据直通车里落到哪些正式对象

为了避免本文停留在通用理论层，下面把概念直接映射回项目对象。

| 概念 | 在数据直通车中的主要落点 | 作用 |
| --- | --- | --- |
| `Entity（实体）` | `Scene（业务场景）`、`Plan（方案资产）`、`Output Contract（输出契约）`、`Policy（策略对象）`、`Source Contract（来源契约）`、`Canonical Entity（统一实体）` | 定义哪些对象具备稳定身份并可进入治理边界 |
| `Relation（关系）` | `USES_*（使用类关系）`、归属关系、依赖关系、证据支撑关系、版本替代关系 | 定义对象之间为何连接、如何解释 |
| `Attribute（属性）` | 时间边界、敏感等级、审批要求、确认状态、置信度、来源信息 | 定义对象和关系成立的条件与限制 |
| `Ontology（本体）` | 类型约束、关系合法性、发布门禁、候选到正式的转化规则 | 防止图谱退化为随意拼接的连接网络 |
| `Evidence（证据）` | `Evidence Fragment（证据片段）`、来源材料定位、人工确认记录、审计说明 | 让正式对象和关系具备可追溯性 |
| `Semantic Layer（语义层）` | 术语、别名、口径解释、时间语义、规则锚点、字典治理对象 | 把业务表达收敛为受控概念入口 |

这张映射表的含义不是“这些对象已经全部实现完毕”，而是说明：主方案里哪些对象承担知识图谱概念的正式落点，后续设计与实现不能脱离这些对象另起一套概念体系。

## 7. 对过时实现路线的裁剪意见

外部知识图谱介绍常见一种写法：先按“需求分析 -> 数据收集 -> 抽取 -> 融合 -> 存储 -> 推理 -> 应用 -> 运维”列出完整项目模块，再给出一组固定工具清单。这种写法适合做通用认知入门，但对当前项目有四个风险。

第一，容易把“概念边界”写成“技术栈清单”。一旦把 `Neo4j（Neo4j 图数据库产品）`、`JanusGraph（JanusGraph 图数据库产品）`、`SPARQL（图查询语言，SPARQL Protocol and RDF Query Language）`、某类抽取模型或某套微服务拆分直接写进主方案，文档会很快过时。

第二，容易把“知识图谱项目”误解为“必须一次建设全链路平台”。而数据直通车当前阶段明确采取受控收敛策略：先围绕口径治理、正式对象、已发布快照、图谱投影和运行解释打穿主链路，不把所有通用图计算能力都纳入首轮目标。

第三，容易弱化证据和发布门禁。很多通用文章默认目标是搜索、推荐、问答或百科组织，因此更强调抽取率和召回率。数据直通车则必须把证据、审计、适用边界和发布约束放在同等甚至更高的位置。

第四，容易把“推理”写成一个泛化技术篮子。当前项目确实需要推理能力，但更应优先落地“基于规则、边界和正式对象关系的受控推理”，而不是在主方案里提前承诺复杂的图学习、自动补全或开放世界推断能力。

因此，本文对外部路线的裁剪意见固定如下：

1. 保留“知识图谱是由对象、关系、证据、约束、消费方式共同构成”的整体视角。
2. 不把固定图库、固定查询语言、固定抽取模型、固定推理框架写成当前主方案承诺。
3. 不把开放问答、推荐系统、百科搜索类目标直接迁移为本项目目标。
4. 把“可治理、可发布、可追溯、可解释”作为比“可自动抽取、可多跳问答”更高的优先级。

## 8. 对主方案写作的直接约束

为了让本文真正服务主方案，后续涉及知识图谱的方案描述应遵守以下写法边界。

第一，凡是写进 [`system-design.md`](/Users/rlc/Code/CaliberHub/docs/architecture/system-design.md) 的知识图谱内容，都应优先回答对象、关系、证据、时间边界、发布约束和下游消费边界，而不是先回答选哪种图库。

第二，凡是写进正式方案的“图关系”，都要能回到项目正式对象，而不是停留在通用概念图层。也就是说，不能只写“实体之间有关联”，必须说明是哪些项目对象通过什么语义关系相连。

第三，凡是写进正式方案的“推理能力”，都要说明推理前提、输入边界、证据要求和错误兜底，不能把推理写成默认可信的自动补全黑盒。

第四，凡是写进正式方案的“`GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 能力”，都应归入运行消费层，不应反向取代上游治理设计。

第五，凡是来自外部研究或旧技术路线的内容，若只具备参考价值而未形成当前项目承诺，应继续留在 `research/`，由正文文档引用，不直接混入正式对象定义。

## 9. 当前阶段建议

结合项目现状，当前阶段最稳妥的路线不是先追求“大而全的图能力”，而是先把以下四件事做稳。

第一，把哪些对象能成为正式实体、哪些关系能进入正式图谱、哪些证据能支撑发布说清楚。

第二，把语义层与知识图谱的分工说清楚，避免术语归一、规则锚点、边界解释和图谱投影混在一起。

第三，把图谱存储和关系型控制库的边界说清楚，继续坚持“关系型控制库是真源，图谱是已发布快照投影与服务化表达”的当前约束。

第四，把 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 放在未来可选消费层位置，而不是让它主导上游对象建模。

如果这四件事没有先稳定，越早引入复杂图库、复杂推理框架和复杂消费链路，后续返工成本越高。
