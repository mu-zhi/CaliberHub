在进行应用架构设计时，需要遵循以下通用的基本原则。

# 1、数据库规范

不同于大集中的主机时代，在云环境下，随着微服务的拆分，数据库的数量也会迅速增长，如果缺乏全局统一的规划设计，实际生产交易路径容易出现链路过长、涉及系统过多的问题，最终导致故障率上升且呈现常态化的态势。开发人员需要从全局视角，依据“上下游业务高内聚、不同业务间低耦合”的原则合理规划数据库部署策略（包括拆分及部署架构），以提高业务的整体可用性。

# 2、前后端分离

应用的前端和后端需要分离，前后端之间通过RESTful API交互。

# 3、 重视前期分析与设计

采用云开发范式进行需求分析及业务建模识别业务知识，在此基础上进行技术建模，将业务知识落地到工程代码。无论是只有大量CRUD 或少量业务逻辑的简单应用还是业务较为复杂的应用，均需重视前期的业务分析及设计工作，将业务知识显性化沉淀到合适的技术载体。

# 4、 合理拆分微服务

需综合考虑迭代节奏、版本耦合、资源利用、交易量、分工、部署复杂度、管理成本等，整体遵循“高内聚、低耦合”，合理控制粒度。尤其要避免为了拆分而拆分。

同时，应保持微服务内模块间的松耦合，不同模块之间的业务数据和能力原则上禁止相互依赖。

# 5、 遵守分层职责的约束

在云开发范式中，三层架构和四层架构的各层职责及层间依赖关系均已清晰定义。如果没有遵守这些约束，会导致各层代码的职责、调用关系变得混乱，难以维护。 因此，在开发过程中应严格遵守分层职责及层间依赖的约束，保持代码的整洁性以及可维护性。
