# 前端 UI 整体重构设计

- 日期：2026-03-29
- 项目：数据直通车
- 主题：壳层先行的前端 UI 优化与信息重排
- 当前分支：`master`

## 1. 背景

当前前端已经具备基础路由、角色导航和多个工作台页面，但整体观感仍明显停留在“原型增强版”阶段，主要问题集中在：

- 全局壳层偏厚，顶部、侧栏、主内容区的比例失衡，第一眼显得笨重。
- 首页同时承担导航页、总览页、任务页三种职责，信息优先级混乱。
- 大量内容通过相似卡片平铺表达，层级不够稳定，页面之间缺乏统一系统感。
- 业务页面各自形成局部样式，导致同一产品内的页面切换体验不连续。
- 当前视觉细节依赖渐变、胶囊按钮、较重阴影和多块容器叠加，未形成成熟业务后台的秩序感。

本次优化不是给单页“换皮”，而是对全站 UI 进行一次以壳层、模板、信息结构和视觉规则为核心的系统化收敛。

## 2. 目标

### 2.1 业务目标

- 让“数据直通车”从原型感界面升级为专业、克制、可持续演进的业务后台。
- 让用户进入系统后更快判断“我现在该做什么”和“系统现在处于什么状态”。
- 降低页面间的学习成本，让多工作台产品表现出统一的产品秩序。

### 2.2 设计目标

- 采用“冷静运营台”视觉方向：浅色、克制、信息效率优先。
- 保留“顶部一级导航 + 左侧二级导航”的双层结构，但显著压薄和收紧壳层。
- 将首页重构为“上作业、下总览”的混合型首页。
- 将重点页面收敛到有限的页面模板，避免每个页面单独长一套。
- 通过 token、排版、状态色、按钮层级和布局节奏建立一致的视觉系统。

### 2.3 非目标

- 不改后端接口、数据结构、权限模型和业务流程定义。
- 不在本轮重做数据地图等重交互页面的核心操作模型。
- 不为了视觉统一删除真实需要的入口，仅允许重排、合并或降级展示。

## 3. 已评估方案与决策

本次讨论过三条路径：

1. 轻改皮肤：仅调整颜色、字体、间距和局部组件视觉。
2. 壳层先行：优先统一全局框架，再重排首页，最后跟进代表性页面。
3. 页面逐个翻新：首页、知识生产台、发布中心分别单独翻新。

最终选择方案 2。原因如下：

- 当前问题不是单页不精致，而是缺少统一壳层与产品秩序。
- 仅改皮肤无法根治粗糙感，页面仍会像“旧结构上的新涂层”。
- 逐页翻新容易造成局部漂亮、全局不统一，后续维护成本更高。

## 4. 总体设计结论

### 4.1 视觉命题

产品整体转向“冷静运营台”：

- 情绪：专业、克制、沉稳、连续。
- 材质：低饱和浅色底、细描边、弱阴影、稳定留白。
- 能量：强调判断效率，而非平台展示感或大屏科技感。

### 4.2 内容命题

围绕四类核心内容组织页面：

- 导航：我在系统中的哪个模块。
- 作业：我当前需要推进什么。
- 状态：当前系统运行与发布是否健康。
- 证据：进一步下钻时看哪些列表、明细、结构或关系。

### 4.3 交互命题

交互不追求炫技，只保留对产品气质有帮助的三类动效：

- 页面和区块进入时的轻量层级过渡。
- 导航切换与当前态高亮的稳定反馈。
- 列表、抽屉、详情区切换时的连续过渡。

## 5. 全局壳层设计

### 5.1 顶部一级导航

保留顶部一级导航，不改为纯左栏主导航。原因：

- 现有路由结构已经按顶级模块分组，保留顶部一级导航有利于用户保留工作台心智。
- 顶部导航适合承载“首页总览 / 数据地图 / 知识生产台 / 发布中心 / 运行决策台 / 审批与导出 / 监控与审计”这类平级域。

优化原则：

- 显著降低头部高度，压缩品牌区宽度。
- 品牌信息只保留必要识别，不再占用过多横向空间。
- 一级导航从当前较厚的胶囊样式改为更薄、更紧、更稳定的导航标签样式。
- 角色切换和工具入口保留在头部右侧，但缩小权重，避免与模块导航竞争。

### 5.2 左侧二级导航

保留左侧二级导航，但由“厚重操作面板”改为“细侧栏目录”：

- 宽度收窄，降低背景装饰和容器感。
- 当前页高亮改为更明确但更克制的状态标识。
- 标签、状态标记和收起态统一规则，减少视觉跳动。
- 左栏只承担结构定位，不承担多余说明性内容。

### 5.3 主内容骨架

全站主内容区统一成稳定结构：

1. 页面标题区：标题、摘要、状态提示、关键跳转。
2. 页面动作区：筛选、搜索、主操作、上下文切换。
3. 页面主体区：主列表、主流程、主画布或主表单。
4. 辅助信息区：说明、最近变更、明细、详情或上下文证据。

主内容区不再依赖“每页自定义英雄区”，而是统一以产品化页面标题区承接内容。

## 6. 页面模板体系

为避免页面各自生长，本次建立 4 种页面模板。

### 6.1 模板一：指挥首页

适用页面：`/overview`

职责：

- 给出最重要的今日任务。
- 显示系统当前的关键状态。
- 提供继续工作和下钻入口。

首页不承担完整运营明细展开，只承担分发和判断。

### 6.2 模板二：流程工作台

适用页面：知识生产台及相近流程页。

结构：

- 标题区
- 当前步骤/状态区
- 主操作区
- 辅助说明或次级结果区

目标：

- 让用户始终清楚“现在做到哪一步”“下一步要做什么”“当前结果放在哪里”。

### 6.3 模板三：运营列表页

适用页面：发布中心、审批与导出、监控与审计、个人协作、系统设置。

结构：

- 页面摘要
- 关键指标条
- 筛选/搜索/批量动作
- 主列表
- 详情区或补充信息区

目标：

- 先扫状态，再看列表，再看详情。
- 尽量避免多个地位接近的卡片块并列竞争注意力。

### 6.4 模板四：资产浏览页

适用页面：数据地图、业务场景、资产图谱及相关浏览页。

结构：

- 顶部上下文条
- 左侧结构导航或筛选
- 中心主浏览区
- 右侧详情或证据区

目标：

- 保持浏览路径稳定。
- 在信息复杂时仍能保留上下文，不让用户迷失。

## 7. 首页重排方案

首页采用“混合型”布局，并明确“上作业、下总览”的优先级。

### 7.1 第一屏

第一屏只保留：

- 页面标题与一句摘要。
- 左侧：待处理事项区。
- 右侧：关键状态摘要区。

不再保留展示型 hero。首页的第一视图必须直接回答两个问题：

- 我现在要处理什么。
- 当前系统是否健康。

### 7.2 待处理事项区

合并当前“我的待办”和“重点风险”，形成统一的待处理事项区。

分组维度：

- 阻断发布
- 高风险
- 待补全
- 待审批

这样可避免用户扫描两个结构相似但语义重叠的板块。

### 7.3 关键状态摘要区

聚合以下信息：

- 候选发布状态
- 审批积压
- 运行健康
- 最近更新时间

这里保留摘要，不展开技术细节。首页只负责发出信号，并提供下钻入口。

### 7.4 下半区

下半区以“近期变更”为核心列表，承接“继续上次工作”的需求。

同时：

- 将搜索和热门领域降级为列表辅助工具。
- 将快速入口与快速定位合并，保留 3 到 4 个最高频入口。
- 所有入口都必须服务于任务推进，而不是作为导航陈列。

## 8. 代表性页面改造范围

### 8.1 知识生产台

知识生产台按“流程工作台”重构：

- 明确区分步骤状态、输入区、过程反馈区和结果区。
- 减少说明文字与操作区域的混杂。
- 把当前任务主线置于页面中心，把补充说明降到次级区域。

本轮不改其核心业务步骤，仅改信息排列和视觉组织。

### 8.2 发布中心

发布中心按“运营列表页”重构：

- 用一条稳定的状态摘要带替代部分分散指标块。
- 把筛选、刷新、发布动作统一到页面动作区。
- 主区域围绕候选场景、发布检查和版本上下文组织，减少容器碎片化。

### 8.3 个人协作 / 系统设置

这类轻量工具页不再使用与主业务页完全不同的表达方式，而是复用“运营列表页轻量版”：

- 简化标题区。
- 用统一列表与空态表达通知、收藏、最近浏览和设置项。

### 8.4 数据地图及相关浏览页

本轮主要统一壳层和外围信息区：

- 顶部上下文条统一。
- 左侧结构导航与右侧详情区的视觉规则统一。
- 中心主浏览区保留现有核心交互能力，不大改其关系浏览逻辑。

## 9. 视觉系统规则

### 9.1 色彩

- 以浅色、中性色为主要背景。
- 主色仅承担导航高亮、主操作和关键强调。
- 辅助色用于状态，不再让蓝绿两种品牌色同时大面积竞争。
- 风险、阻断、样例、原型等状态采用固定语义色，不再按页面单独定义。

### 9.2 排版

- 页面标题、区块标题、正文、说明、数据数字建立固定档位。
- 标题层级减少，避免一个页面同时存在多种接近但不一致的字号。
- 说明文案全部向“产品说明”风格收敛，不使用展示型口号化语言。

### 9.3 容器

- 以布局分区优先，卡片只在承载动作、数据组或详情块时使用。
- 全站边框、圆角、阴影统一收敛，减少“多个盒子堆叠”的观感。
- 相邻区域主要通过留白、排版和轻背景差区分，而不是厚重容器。

### 9.4 按钮与控件

仅保留清晰的层级：

- 主按钮
- 次按钮
- 文字按钮
- 危险按钮

所有链接型按钮、按钮型链接、导航型按钮统一语义和外观。

### 9.5 状态标签

统一状态体系：

- 正常
- 处理中
- 警示
- 阻断
- 样例/原型

同一状态在所有页面使用同一套颜色、边框和文字权重。

## 10. 交互与动效规则

- 页面进入动效轻量且统一，用于建立层级，不用于装饰。
- 导航激活态采用稳定过渡，不做跳跃式高亮。
- 列表和详情区的切换要保留空间连续性，避免“瞬间换屏”的断裂感。
- 动效必须在桌面和窄屏下都保持流畅，且可被简化或关闭。

## 11. 技术实现边界

### 11.1 允许修改

- `frontend/src/styles.css` 中的全局 token、壳层、页面骨架和关键页面样式。
- `frontend/src/App.jsx` 中的顶栏、侧栏、全局布局结构。
- 首页及 2 到 3 个代表性页面的 JSX 结构重排。
- 通用 UI 组件的样式层级统一。

### 11.2 不允许修改

- 后端 API 契约与路由语义。
- 权限控制逻辑。
- 复杂交互页面的核心操作模型。
- 样例页、原型页、正式页的业务标识语义。

## 12. 错误处理与状态表达

UI 重构后，以下反馈必须保持稳定：

- API 错误 toast 仍由全局壳层承接。
- 原型态、样例态提示继续保留，但改为统一且更收敛的状态条样式。
- 空态、加载态、错误态在首页、运营页、工具页之间采用一致语言和视觉。

## 13. 响应式与可访问性

- 桌面端优先，但必须保证常见窄屏宽度下导航、标题区和主列表不失真。
- 固定头部高度变薄后，主体区滚动关系要保持清晰。
- 继续保留跳转主内容能力和可见焦点态。
- 新的导航、按钮和标签样式不能依赖颜色作为唯一状态表达。

## 14. 验证方案

### 14.1 渲染验证

- 关键页面 render test 保持通过。
- 首页、知识生产台、发布中心和工具页能稳定渲染。

### 14.2 结构验证

- 顶栏、侧栏、主内容区在桌面和窄屏下比例稳定。
- 页面标题区、动作区、主体区的布局节奏一致。

### 14.3 交互验证

- 常用导航、入口、筛选、跳转保持可用。
- 首页重排后，不丢失高频入口和任务推进路径。

### 14.4 视觉验证

- 用户能明显感知系统从“原型增强版”升级为统一业务后台。
- 首页、知识生产台、发布中心在视觉上属于同一产品。

## 15. 实施顺序

建议按以下顺序执行：

1. 全局壳层与视觉 token 收敛。
2. 首页信息重排与入口合并。
3. 知识生产台重排。
4. 发布中心重排。
5. 个人协作/系统设置轻量统一。
6. 数据地图类页面壳层收口。

该顺序可以保证先建立系统规则，再将规则落到代表性页面，减少返工。

## 16. 最终决策摘要

- 采用“壳层先行”的整体重构路径。
- 视觉方向选择“冷静运营台”。
- 壳层结构保留“顶部一级导航 + 细侧栏”。
- 首页采用“混合型”，并执行“上作业、下总览”。
- 允许顺手重排首页信息、合并重复入口、调整部分页面信息结构。
- 本轮以视觉系统、页面模板和结构秩序为核心，不做后端和业务流程变更。
