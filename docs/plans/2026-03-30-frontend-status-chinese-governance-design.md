# 前端状态机中文展示治理设计稿

> 日期：2026-03-30
> 适用范围：发布中心、运行决策与知识包、数据地图
> 状态：设计已确认，待用户审阅后转实施计划

## 1. 背景

项目已经正式规定“前端状态机展示统一使用中文，代码与接口内部可保留英文状态编码”。当前已上线页面仍存在三类存量问题：

1. 部分徽标、概览卡和详情字段直接渲染接口返回的英文状态值。
2. 数据地图的状态筛选项直接把英文枚举暴露给业务用户。
3. 同一批页面各自维护状态 `tone` 判断，却没有统一的“英文编码 -> 中文展示名”收口层，后续极易再次漂移。

## 2. 治理范围

本轮只治理已经确认的同一批页面，不扩展到后端契约改造或全站扫尾：

1. `frontend/src/pages/PublishCenterPage.jsx`
2. `frontend/src/pages/KnowledgePackageWorkbenchPage.jsx`
3. `frontend/src/components/datamap/DataMapContainer.jsx`

对应正式设计真源同步到：

1. `docs/architecture/features/iteration-02-runtime-and-governance/07-发布检查、灰度发布与回滚.md`
2. `docs/architecture/features/iteration-02-runtime-and-governance/08-运行决策与知识包生成.md`
3. `docs/architecture/features/iteration-02-runtime-and-governance/10-数据地图浏览与覆盖追踪.md`

## 3. 方案对比

### 3.1 方案 A：页面内逐个修补

- 做法：在每个页面本地新增中文映射对象，逐处替换原始状态值。
- 优点：改动最少，推进最快。
- 缺点：重复定义严重，和最新开发规范中的“集中映射”要求冲突，后续极易再出现多套中文口径。

### 3.2 方案 B：共享前端状态展示层

- 做法：新增一个共享状态展示模块，统一维护不同领域状态的 `code -> { label, tone }`，页面只调用 helper。
- 优点：规则集中、便于测试、后续扩展时有唯一入口。
- 缺点：首轮需要多改几处引用和测试，但仍局限于前端展示层，可控。

### 3.3 方案 C：后端统一返回中文 `statusLabel`

- 做法：后端接口直接返回中文展示名，前端只负责渲染。
- 优点：前端最轻。
- 缺点：会把治理范围扩大到接口契约、后端测试和前后端联调，不适合作为首批已上线页面治理。

## 4. 选定方案

本轮采用方案 B：共享前端状态展示层。

### 4.1 共享模块设计

新增共享状态展示模块，建议路径为：

- `frontend/src/components/ui/statusPresentation.js`

模块职责：

1. 接收英文状态编码并归一化。
2. 返回统一中文展示名与 `tone`。
3. 为筛选项提供 `{ value, label }` 结构，而不是裸字符串数组。
4. 为未知状态提供统一兜底，不把原始英文状态值直接展示给业务用户。

首轮收口以下状态域：

1. `scene status`
2. `projection status`
3. `publish status`
4. `coverage status`
5. `decision / review status`
6. `unknown fallback`

建议返回结构：

```js
{
  code: "PUBLISHED",
  label: "已发布",
  tone: "good",
}
```

未知状态统一回退为：

```js
{
  code: "UNKNOWN",
  label: "未知状态",
  tone: "neutral",
}
```

## 5. 页面改造设计

### 5.1 发布中心

需要替换为中文映射的展示位：

1. 场景列表状态徽标
2. 当前场景头部状态徽标
3. 图谱投影状态详情字段
4. 图谱投影状态徽标
5. 版本发布状态徽标
6. 同批次概览卡中涉及状态名称的展示

约束：

1. 逻辑判断继续使用英文状态编码。
2. 所有展示文本统一从共享模块取值。

### 5.2 运行决策与知识包

需要替换为中文映射的展示位：

1. 图谱投影概览卡
2. 图谱投影状态徽标
3. 覆盖状态徽标
4. 决策结果徽标

约束：

1. 不改请求链路和运行上下文协议。
2. 只替换展示层，不改变业务流程。

### 5.3 数据地图

需要替换为中文映射的展示位：

1. 顶部“资产状态”筛选项
2. 同页所有直接出现的状态徽标或状态摘要

约束：

1. 内部过滤值仍保留英文编码，保证接口和现有逻辑兼容。
2. 用户看到的是中文标签，不再看到英文枚举值。

## 6. 测试与验收

首轮验收以现有前端测试为主，补充以下断言：

1. 发布中心渲染出的状态徽标与详情字段为中文。
2. 运行决策与知识包页的投影状态、覆盖状态、决策状态为中文。
3. 数据地图“资产状态”筛选项显示中文，但交互时提交的内部值仍为英文编码。
4. 未知状态回退为统一中文兜底，而不是原样渲染英文码。

## 7. 非目标

以下内容不在本轮范围内：

1. 后端接口新增 `statusLabel`
2. 数据库枚举改名
3. 全站所有页面一次性扫尾
4. 业务逻辑重构或状态机本身改造

## 8. 风险与控制

1. 风险：不同页面已经存在局部中文口径，接入共享层时可能出现文案回退。
   控制：以最新正式文档为准，测试断言固定中文展示名。
2. 风险：未知状态被新 helper 统一为“未知状态”后，调试信息不够直接。
   控制：业务展示只给中文；如需技术码，只保留在调试区或日志里。
3. 风险：共享模块过早做成大而全字典。
   控制：首轮只收口这三页确实用到的状态域，不做全站预设。
