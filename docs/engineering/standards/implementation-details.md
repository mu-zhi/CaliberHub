# 1、统一术语及命名规范

## 统一术语

业务和软件实现间统一术语，即从业务需求中提炼出统一术语（统一的词汇表），在业务方（或 领域专家）和开发团队间达成一致的理解。需给出领域术语的英文单词描述，之后设计与编码需使用该统一词汇。如此业务方、开发团队间通过统一术语进行沟通，降低彼此的沟通成本，加深团队对领域的理解，看到代码中的术语也可以做到见名识义。

- 【强制】每个项目都要有一份自己的核心领域词汇表。这个词汇表至少要包含中文、英文、缩写、备注四列。
- 【强制】《云开发范式：应用架构设计方法》强调相关术语保持全局的唯一性和一致性，不依赖于通过限界上下文来区分同一个词汇的二义性。
- 【强制】产品团队内的交流、文档、设计和代码都应该以词汇表为准。

## 命名规范

- **【强制】** CQRS，请求参数命名为xxCmd以及xxQuery。
- **【强制】**app层的应用服务类命名xxAppService。
- **【强制】**数据传输对象统一命名为xxDTO。
- **【强制】**app层数据转换使用mapstruct统一命名xxAssembler。
- **【强制】**领域层实体根据业务含义直接命名，领域服务命名为xxService。
- **【强制】**基础设施层数据转换统一命名为xxConverter。
- **【强制】**基础设施层数据持久化对象统一命名为xxPO。
- **【强制】**防腐层命名xxRepository或者xxSupport。
- **【强制】**对方法、类的命名需要概括其具备的能力。做到职责分配合理，单一职责。
- **【建议】**前端请求命名xxController，定时任务xxJob，微服务互调命名xxPort。
- **【建议】**应用层公共组件命名xxExecutor。
- **【建议】**用单词缩写命名时需要考虑下该缩写方式是否常见，难以缩写时，可以直接使用完整单词短语命名变量。遵循的原则是变量命名应该以含义清晰为首要前提，避免望文不知义。

## 前端状态机展示规范

- **【强制】**状态机在代码、接口字段、数据库枚举、领域枚举和内部常量中可以继续使用英文状态编码，但语义必须稳定，且与术语表和正式设计保持一致。
- **【强制】**一切面向业务用户的前端状态机展示统一使用中文，包括状态标签、状态徽标、筛选项、步骤条、时间轴、详情页状态字段，以及提示框、通知和空态中的状态名称。
- **【强制】**前端不得直接向业务用户暴露 `DRAFT`、`REVIEWED`、`PUBLISHED`、`FAILED` 一类英文状态值；若接口返回英文状态编码，必须先经过集中式中文映射再展示。
- **【强制】**同一状态机只维护一套正式中文展示口径，不允许在不同页面、不同组件或测试样例中散写多套近义中文。
- **【建议】**接口契约、前端主文档和测试断言同时维护“英文状态编码 -> 中文展示名”对照，避免联调、评审和验收阶段再次出现口径漂移。

# 2、分层架构

## 整体说明

《云开发范式：应用架构设计方法》制定了两种应用架构设计方法，分别是三层架构（Controller、Service、Entity、Infrastructure）、四层架构（Adapter、Application、Domain、Infrastructure）。

可根据应用类型选择简单应用架构设计方法（三层）或复杂应用架构设计方法（四层），具体选择方法参考方法对比与选型。

考虑到当前大数据应用支撑开发团队内已经试点实践的项目均选型为四层架构，我们重点先制定四层架构规范细则，具体内容如下。

### 四层架构规约

**分层职责：**

- **Adapter 层：**适配器层（Adapter）包含应用与外部交互的适配器，如 Web 控制器、定时任务适配器、消息适配器等，主要处理输入解析、验证、转换、序列化/反序列化、用户登录权限校验等。
- **Application 层：**应用服务层，面向用例设计，主要负责处理跨域业务流程的组合编排、事务处理等，可以调用领域层。简单查询场景(如：CQRS 架构中的 Query 部分，查询逻辑比较简单没有沉淀出领域模型等 )，应用层和基础设施层可以直接交互，但建议在应用层定义 support 接口实现依赖反转，让应用层不依赖基础设施层的具体实现。
- **Domain 层：**领域层，定义业务模型，通过领域服务和领域对象行为封装核心业务逻辑。在领域层定义 Support 接口，在 Infrastructure 层进行实现，利用依赖倒置实现业务领域和外部依赖的解耦。
- **Infrastructure 层：**包含工具类（如访问数据库、消息队列等中间件的客户端）、全局通用类（如通用配置、异常处理等）、对领域层及应用层所定义接口的实现（位于“supportimpl”分包中）。

**层间依赖规约：**

- 【**强制**】层间依赖约束细则详见：本文**5.2架构守护规约**
- 【**强制**】禁止数据对象透传。
    - 领域层不能直接使用基础设施层的 PO，应定义领域对象（实体/值对象），并在 support 接口中转换为领域对象（实体/值对象）返回。
    - 应用层不能直接使用基础设施层的 PO，应在应用层定义对应的 DTO，并在 support 接口中转换为该 DTO 返回。
    - 即使不同层的对象字段完全一致，也禁止直接使用。
- 【**强制**】通用技术校验在适配器层， 业务满足性校验在应用层，强业务属性检查沉淀到 entity。
- 【**建议**】不推荐使用领域服务，优先考虑实体行为和应用服务，领域服务作为最后折衷考虑。折衷时可考虑：涉及多实体的事务一致性逻辑，且属于重要业务概念。

**数据对象说明：**

在整个分层结构中主要包含三种数据对象，不同层之间需要进行转换：

- **DTO：**数据传输对象，在 Application 层和 Infrastructure 层使用，命名时以DTO后缀结尾。
- **Entity：**领域实体，封装业务实体的字段，字段要具有业务含义，承载业务属性和行为，命名时无特殊后缀。
- **Value Object**：承载业务属性，没有唯一标识和生命周期，不可变，当值对象的属性改变时，使用新的值对象进行替换，命名时无特殊后缀。
- **PO：**数据持久化对象，通常与数据库表一一对应，命名时以PO后缀结尾。

## 适配层设计

- 【**强制**】通用技术校验在适配器层实现，业务满足性校验在应用层，强业务属性检查沉淀到entity
- 【**强制**】禁止跨层调用，适配器层不能直接调用领域层。
- 【**建议**】异常处理统一在适配器层拦截处理。
    - 基础设施层不处理异常，直接抛出；对于异常自动重试这种异常可以进行捕捉。
    - 领域层抛出自定义业务异常。
    - 应用层不处理异常；对于系统异常不影响业务流程的情况下，应用层可以进行捕捉后打印日志。
    - 适配器层统一拦截异常，返回错误码以及友好错误提示。
- 【**建议**】适配器层按照功能分包，不同适配器代用不同的功能分包，比如Web控制器的功能分包为"web"，定时任务适配器为"schedule"，消息适配器为"consumer"。

## 应用层设计

- **【建议】**应该避免两个应用服务之间相互调用；
- **【建议】**应用层（Application）的主要职责是调用领域层（Domain）的业务知识进行应用逻辑编排。
    - 数据验证：业务满足性校验。
    - 数据转换：转换用户数据到领域模型格式，转换领域对象。
    - 业务编排：查询领域对象，完成领域对象业务逻辑。
    - 数据呈现：转换领域对象到DTO呈现给用户。
    - 功能支持：安全认证、事务控制、消息发送、权限校验。

## 领域层设计

领域层设计主要考虑业务规则的实现，其主要分为领域对象和领域服务。

(1) 定义业务模型，用领域对象行为封装核心业务逻辑；

(2) 用领域服务组织多个对象的行为；

对于无法决定放在应用层还是领域层的逻辑，可以考虑将其先放到应用层，采用先做厚应用层，然后逐步分析再做薄的思路，将跟领域对象有关的逻辑沉淀到领域层的对象中。

为了加深理解，可以考虑一个需求场景：

```sql
用户可以通过银行网页转账给另一个账号，支持跨币种转账。
同时因为监管和对账需求，需要记录本次转账活动。
```

从此场景中，可以识别出账户领域对象，本对象具备转入（金额）和转出（金额）的行为：

```sql
public class Account {

    private AccountId id;
    private AccountNumber accountNumber;
    private UserId userId;
    private Money available;
    private Money dailyLimit;

    public Currency getCurrency() {
        return this.available.getCurrency();
    }

    // 转入
    public void deposit(Money money) {
        if (!this.getCurrency().equals(money.getCurrency())) {
            throw new InvalidCurrencyException();
        }
        this.available = this.available.add(money);
    }

    // 转出
    public void withdraw(Money money) {
        if (this.available.compareTo(money) < 0) {
            throw new InsufficientFundsException();
        }
        if (this.dailyLimit.compareTo(money) < 0) {
            throw new DailyLimitExceededException();
        }
        this.available = this.available.subtract(money);
    }
```

则针对转账这一行为，有一个对应的领域服务组织收款和付款账户的行为：

```sql
//收款账户转入金额
sourceAccount.deposit(sourceMoney);
//付款账户转出金额
targetAccount.withdraw(targetMoney);
```

### 领域层设计规范如下

**实体设计规范：**

- **【强制】**领域实体使用充血模型，同时包含属性和行为。
- **【强制】**两个实体之间有共性时使用基类和子类的方式，避免在一个类里设计所有的属性。
- **【强制】**对于固定几种的选项、类型、状态等， 使用枚举类或者Boolean类型，而不是常量或者基本数据类型，枚举类可考虑放在valueobject下。
    正例：StatusEnum {ENABLE, DISABLE}，Boolean enable；反例：String status；
- **【强制】**实体对象禁止使用spring bean管理，禁止通过注入的方式引入其他服务。
- **【强制】**领域实体禁止使用public setter，通过业务语义化的表述修改方法签名，例如修改记录的状态，不要直接通过setStatus修改，定义业务行为，如enable，disable等方法修改状态。
- **【强制】**禁止使用lombok的@Data和@Setter注解（对应第5条）。
    说明：@Data默认包含了@Setter，会生成public setter方法
- **【强制】**禁止使用构造函数生成领域对象，禁止lombok的@AllArgsConstructor注解。
    说明：领域对象的生成是业务行为，可以在对象中定义create静态方法进行生成或者通过factory生成，更具有业务语义。
- **【强制】**领域对象使用builder模式，用于PO-Entity，以及DTO-Entity之间的对象转换，禁止使用set方法。可使用lombok的@builder和@superbuilder注解
- **【强制】**实体的创建应该在工厂方法、build方法、create静态方法、assember等中进行。
- **【强制】**实体不应该有默认值，默认值会污染实体，通过工厂或者build方法初始化
- **【强制】**实体需要满足业务完整性校验后才能被创建出来，即创建出来的实体必须是可信的。
    说明：实体属性的业务完整性校验可以放在factory或者自定义实体的build方法中
    例如，创建User对象时必须要求age大于18。
    正例1：factory创建，适用场景：创建规则比较复杂，或者依赖了repository获取相关配置或者规则等。
	正例2：create方法，适用场景：属性较少，业务规则简单。
	正例3：自定义builder，重写赋值方法或者build方法，适用场景：属性较多。
- **【强制】**任何实体的行为只能直接影响到本实体（和其子实体），任何实体的行为不能直接修改其他的实体类，确保代码可读性、可理解；另外一个实体对象的所有变更操作应该都是预期内的，如果一个实体能随意被外部直接修改的话，会增加代码bug的风险。
- **【强制】**实体属性的修改必须通过实体中的行为进行修改，且修改行为只能在application层和domain service中调用，禁止在其他模块的实体中修改。
- **【强制】**两个模块间的实体间禁止直接依赖，同模块中的实体和子实体是可以依赖的。如果跨模块需要依赖另外一个实体的属性时，只能通过依赖实体中的只读属性，定义实体的只读基类用于跨模块引用，减少重复属性的设置。
    说明：例如A实体需要依赖B实体中的字段值，技术实现如下：
    定义B实体的基类，BEntityProperty，其中只包含属性和get方法，在A实体中引用BEntityProperty，通过get方法获取B的属性，保证B实体中的属性不会再A实体中被改变。
    定义B实体的业务类BEntity extends BEntityProperty，在其中实现属性的变更行为。
- **【强制】**domain层定义的support接口必须传递EntityProperty对象，避免infrastructure调用entity中的行为。
- **【强制】**实体不可以强依赖其他模块实体、support接口或领域服务。一个实体的原则是高内聚、低耦合，即一个实体类不能直接在内部直接依赖一个外部的实体或服务，对外部对象的依赖性会直接导致实体无法被单测；以及一个实体无法保证外部实体变更后不会影响本实体的一致性和正确性。需要依赖外部参数时通过application层获取传递或者定义领域服务实现业务规则。
- **【强制】**不要直接把DTO传递到domain层，包括repository接口中, 应该转换为领域对象或者展开直接传递参数等方式。
- **【强制】**在领域层定义的防腐层接口应该采用业务处理需要的含义，而不是实现相关。
    说明：例如在假期领域中查询假期档案（员工的一些信息，在假期领域中作为假期档案），但底层是调用人事的api接口。
    正例：在domain层定义api接口，使用假期档案作为接口命名的一部分
- **【强制】**领域对象的成员变量类型应该按照业务概念进行定义，不能为了数据库存储而改变实体的成员变量类型。
    说明：例如数据库存储的是json类型，不能将成员变量修改为String，而是设置为对象，在PO中进行转换。
- **【建议】**对于实体所需要的数据设置，可以考虑在创建的时候的一同设置。
- **【建议】**在领域层，除了各种领域对象以外，实现业务代码逻辑的过程中，通常也会有使用普通对象或者接口的需求，这种是纯技术层面的需求。例如参数过多封装的入参对象，结果对象等，对于这种情况，遵循：谁使用，就由谁定义，就在那里定义。
- **【建议】**domain层单元测试尽可能做到全部覆盖，自动生成的get、builder方法无需测试。

**领域服务设计规范：**

- **【强制】**领域服务中不使用support的更新类操作，support的保存是领域实体的生命周期管理，领域实体不会将自身保存到数据库，因此通过关注点分离：业务逻辑放入领域模型，领域层使用更加OO的方式专注于业务；与外部世界的交互应该放入应用服务，application层去决定什么时候执行更新操作。
- **【建议】**不推荐使用领域服务，优先考虑实体行为和应用服务，领域服务作为最后折衷考虑。

## 基础设施层设计

- **【强制】**common包存放全局的工具类、配置类、枚举类、异常类，其他层可以直接引用
- **【强制】**client包存放中间件的通用客户端，不包含任何业务调用和转化逻辑；
- **【强制】**param包存放基础设施层内部流转的参数类，命名时无特殊后缀；
- **【强制】**converter包存放PO、DTO、实体、值对象、基础设施层内参数对象之间的转换逻辑；
- **【强制】**supportimpl：存放调用外部服务或数据库具体实现类，supportimpl包内部可以按应用层实现和领域层实现分包。
- **【建议】**如果微服务内涉及多个模块，可以按模块进行第一级的分包，在模块内存再放dao、supportimpl、converter等子包；
- **【建议】**对象之间的转换推荐使用 MapStruct，禁止使用 bean copy 的方式，反射会影响系统性能。
- **【建议】**基础设施层不处理异常，直接抛出；对于异常自动重试这种异常可以进行捕捉，异常处理统一在适配器层拦截处理，返回错误码和友好提示信息。

# 3、防腐设计

**防腐层的意义**：很多时候，我们的系统都会依赖其他的系统来完成业务流程，而被依赖的系统可能存在着不合理的数据结构、API、协议或技术实现。如果在业务逻辑中直接依赖外部系统的实现，会导致我们的系统被“腐蚀”。因此可以在系统间加入一个防腐层，将外部依赖和内部逻辑隔离开，无论外部怎么变化，内部的逻辑可以尽可能地保持不变。防腐层存在目的就是为了进行上下文映射。也就是一个领域要使用其他的领域的信息，需要一个“翻译器”来将外域的信息翻译成本领域的概念。防止其他领域模型侵害本领域模型。

- 【**强制**】调用外界接口服务，应严格防腐，应定义 support 接口，由基础设施层实现。
- 【**强制**】应用层简单查询也不直接通过 mapper 与数据库交互，应定义 support 接口解耦。

# 4、实体充血细则

充血模型（Rich Domain Model）是指在设计实体时，包含实体的所有属性和该实体相 关的业务逻辑，其优点是面向对象。使用充血模型方式时，应用层是很 薄的一层，仅仅封装事务和跨模块的业务逻辑。

建议实践中将应用层中属于某个实体的业务逻辑，逐步沉淀到相应实体中，让实体 适当充血。

- **【强制】**数据持久化逻辑不适合充血到实体中，会引起涨血；
- **【强制】**实体的属性与该实体的相关业务逻辑（原子化行为或组合逻辑行为）沉淀到实体内，如果一开始不能识别出充血行为，也没关系；
- **【强制】**领域层实体应定义业务语义化的方法修改属性。
    - 例如修改记录的状态，建议定义业务行为如 enable，disable 等方法修改状态，不建议直接通过 setStatus修改。
- **【强制**】实体相关的强业务属性检查沉淀到实体内；
- **【强制】**实体对象不能使用Spring容器托管，spring bean管理的通常是单例，而实体对象是多例的；
- **【强制】**领域对象（实体/值对象）不应使用 lombok 的 @Data 和 @Setter 注解：
    - @Data 和 @Setter 注解会引入set方法，而set是技术上的执行和手段，并不是业务意图。
    - DTO可以使用@Data注解。
- **【建议】**不应使用构造函数生成实体，禁止 lombok 的 @AllArgsConstructor 注解，可在对象中定义 create 静态方法进行生成或者通过 factory 统一管理实体的创建，防止实体的构建在代码中散落 ；
- **【建议】**领域对象（实体/值对象）使用builder模式，用于PO-Entity，以及DTO-Entity之间的对象转换，禁止使用set方法。可使用lombok的@builder和@superbuilder注解。
    - 实体创建出来就是业务上合法的，如果有一些业务规则整个生命周期都需要满足（比如订单总价与订单项单价、数量保持一致），可重写build()方法， 可参考
- **【建议】**实体本身的定义里，属性不应有默认值，可在工厂初始化时设置属性默认值。

# 5、通用细则

## 框架使用技巧

- 【**强制**】使用org.springframework.scheduling.annotation.Async注解异步处理的时候 必须在Spring Boot启动类或Configuration类添加org.springframework.scheduling.annotation.EnableAsync类，否则异步处理不生效。
- 【**强制**】避免使用new Thread().start的方式进行线程的创建，应该使用ExecutorService等对线程进行管控。
- 【**强制**】不应该在代码中散落魔法值的使用，应该定义常量或枚举。
- 【**强制**】成员变量都有其明确的语义，不能作为容器存储其他语义的数据。
- 【**强制**】方法的入参不应该被修改，如果有返回数据的必要性，可以使用command对象或者值对象。
- 【**强制**】通用技术校验在适配器层(使用spring的校验注解)， 业务满足性校验在应用层，强业务属性检查沉淀到实体内。
- 【**建议**】方法的可见性上，如暂无外部使用需要，可以先声明为private，后续如有需要再调整可见性
- 【**建议**】如果方法入参为列表，可使用Collections.unmodifiableList()保证入参列表不可变。
- 【**建议**】代码文件推荐用UTF-8编码
- 【**建议**】 synchornized 应该单独放置明确语义,不能到处散落，便于理解其使用方案，避免并发相关问题。
- 【**建议**】禁止使用两种及以上的 ORM 框架，避免增加维护成本。
- 【**建议**】同一项目中，API 调用的方式尽可能统一，避免额外的技术负担。
- 【**建议**】叶子节点的实体类要加final，避免直接通过继承方式修改。
- 【**建议**】 缓存等与业务逻辑不直接相关的辅助能力，尽可能不要入侵业务代码，可以使用 AOP 等方式处理。
- 【**推荐**】使用云效进行编码的架构守护。
- 【**建议**】方法调用尽量保持入参的清晰明确，遵循最小依赖原则。
- 【**建议**】避免使用不具备业务含义的泛容器对象，例如Map、JSONObject等
- 【**建议**】依赖注入方式主要有基于构造函数和基于字段两种，推荐优先考虑基于构造函数的依赖注入方式，其次是基于字段的依赖注入方式（@Autowired）
    - 采用基于构造函数的依赖注入方式，有助于践行类的单一职责。当你注入的太多的时候，这个构造方法的参数就会很庞大，自然而然的会想一下“这个类是不是违反了单一职责思想?”。 但是使用基于字段的依赖注入不会让你察觉，你会很沉浸在@Autowire当中。
    - 基于字段的依赖注入方式与Spring的IOC机制紧密耦合，Spring不推荐。
