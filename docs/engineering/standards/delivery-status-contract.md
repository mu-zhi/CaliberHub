# 交付状态契约

本文定义项目级 `Delivery Status（交付状态）` 契约，用于约束“当前开发进度 / 下个阶段工作”的唯一真源、字段口径、状态流转、更新时间点和禁止事项。本文解决“怎么写、什么时候必须写、哪些内容不能写”；最高原则以 [overall-principles.md](overall-principles.md) 为准，流程门禁以 [../collaboration-workflow.md](../collaboration-workflow.md) 为准。

---

## 1. 适用范围

1. 本契约只约束团队内部日常推进、任务接力、阶段验收前的滚动交付状态。
2. 当前开发进度与下个阶段工作统一维护在 [../current-delivery-status.md](../current-delivery-status.md)。
3. 架构方案、正式特性设计、实施计划、用户手册和历史复盘不得充当并行状态入口。
4. 长期蓝图、需求池、临时想法和历史周报不进入当前交付状态文档。

## 2. 唯一真源约束

1. [../current-delivery-status.md](../current-delivery-status.md) 是仓库内唯一 `Single Source of Truth（唯一真源）`。
2. `README.md`、`docs/README.md`、`docs/plans/README.md`、正式设计正文、评审结论和聊天记录只允许放入口链接或引用，不允许并行维护滚动摘要。
3. 同一工作项只能在状态文档中保留一份当前状态；需要补充细节时，回链到设计、计划或评审文档，不新增第二份“当前进度”。

## 3. 字段契约

### 3.1 仓库总览

仓库总览必须至少包含以下字段：

| 字段 | 是否必填 | 约束 |
| --- | --- | --- |
| 当前阶段 | 是 | 只写当前团队真正处于的阶段，不写远期蓝图 |
| 本阶段目标 | 是 | 用一句到三句话写清本阶段要完成的交付结果 |
| 本阶段退出条件 | 是 | 必须可验证，可使用勾选项 |
| 最近更新时间 | 是 | 使用 `YYYY-MM-DD` |

### 3.2 进行中工作项

进行中工作项必须至少包含以下字段：

| 字段 | 是否必填 | 约束 |
| --- | --- | --- |
| 工作项 | 是 | 使用可辨识的交付对象名称，不写口号 |
| 归属阶段 | 是 | 指向当前阶段中的具体子阶段 |
| 来源设计 | 是 | 指向正式设计真源；不得只指向聊天记录 |
| 来源计划 | 条件必填 | 进入 `implementing（实施中）` 前必须指向实施计划；纯只读评审或纯文档同步如经明确认定，可写明豁免理由 |
| 测试文档 | 条件必填 | 进入 `reviewing（评审验证中）` 前必须指向 `docs/testing/features/` 下的对应测试文档；纯只读评审或纯文档同步如经明确认定，可写明豁免理由 |
| 当前状态 | 是 | 只能使用本文定义的状态枚举 |
| 最新完成 | 是 | 只写最近一次已落地、可验证的完成事实 |
| 下一动作 | 是 | 必须是单步可执行动作，粒度不能大于一个实施计划任务 |
| 退出条件 | 是 | 必须可验证，不能写“继续完善” |
| 阻塞项 | 是 | 没有阻塞时写“无” |
| 责任人 | 是 | 写明确角色、人名或“待指定”；禁止留空 |
| 最后更新时间 | 是 | 使用 `YYYY-MM-DD` |

### 3.3 下一阶段工作

下一阶段工作用于记录已经具备进入条件、但尚未启动的下一棒工作，必须至少包含以下字段：

| 字段 | 是否必填 | 约束 |
| --- | --- | --- |
| 工作项 | 是 | 必须是可单独启动的工作，不写需求池 |
| 启动前置条件 | 是 | 写清哪些条件满足后才能启动 |
| 对应设计/计划 | 是 | 至少指向一份设计或计划文档 |
| 启动后第一步 | 是 | 必须是启动后立即可执行的第一步 |
| 责任人 | 是 | 写明确角色、人名或“待指定” |

### 3.4 近期待验收与风险

1. 近期待验收必须至少包含 `工作项`、`验证命令`、`验证结果`、`状态`、`责任人`。
2. 风险与外部依赖必须至少包含 `风险/依赖`、`影响范围`、`当前状态`、`应对动作`、`责任人`。

## 4. 状态枚举与流转

### 4.1 固定状态枚举

当前状态只允许使用以下枚举：

| 枚举值 | 中文口径 | 使用说明 |
| --- | --- | --- |
| `proposed` | 已识别，未进入方案 | 工作项已被识别，但尚未开始正式方案收敛 |
| `brainstorming` | 方案中 | 正在做需求收敛、边界澄清和方案比较 |
| `planning` | 计划中 | 设计已确认，正在拆实施计划 |
| `implementing` | 实施中 | 已进入执行，正在按计划推进 |
| `reviewing` | 评审验证中 | 实现已落地，正在做验证、审查或验收准备 |
| `fixing` | 缺陷修复中 | 只能由 `reviewing` 进入；代码检视或测试验收发现 `P0/P1` 缺陷后，进入主动修复 |
| `blocked` | 阻塞中 | 由于依赖、决策、资源或缺陷暂时无法推进 |
| `done` | 完成 | 已通过约定验证并完成交接 |
| `dropped` | 终止 | 明确不再继续推进 |

### 4.2 状态流转约束

1. 正常主路径为：`proposed -> brainstorming -> planning -> implementing -> reviewing -> done`；在 `reviewing` 阶段如发现 `P0/P1` 缺陷，可进入 `fixing` 后再回到 `reviewing`。
2. 任一进行态可以临时进入 `blocked`；解阻后必须回到原状态或进入下一合法状态，不新开并行条目。
3. 不允许从 `brainstorming` 直接进入 `implementing`。
4. 不允许从 `implementing` 直接进入 `done`。
5. `reviewing` 期间不允许直接修改实现代码；若需改代码，必须先切换为 `fixing`，完成后回到 `reviewing`。
6. `fixing` 只能由 `reviewing` 进入；修复完成后必须重新进入 `reviewing`，不允许从 `fixing` 直接进入 `done`。
7. 没有验证证据，不允许标记为 `done`。
8. 存在未关闭 `P0` 或 `P1` 缺陷时，不允许标记为 `done`；存在 `P2` 遗留缺陷时，测试文档结论必须显式说明。
9. 终止项统一标记为 `dropped`，并在 `最新完成` 或 `阻塞项` 中写清终止原因。

## 5. 更新时间点

以下时点必须同步 [../current-delivery-status.md](../current-delivery-status.md)：

1. 设计确认后：当天补齐 `来源设计`、`当前状态`、`下一动作`。
2. 实施计划落地后：当天补齐 `来源计划` 和计划启动顺序。
3. 每完成一批任务或每日收工前：更新 `最新完成` 和 `下一动作`。
4. 任务交接前：必须同步 `当前状态`、`阻塞项`、`责任人`、`最后更新时间`。
5. 出现新阻塞时：在当次工作时段内改为 `blocked` 并写清阻塞原因。
6. 实现完成后、进入 `reviewing` 前，必须先同步 `reviewing` 所需测试文档路径及相关验证入口，并确认 `code-reviewing（代码检视）` 已通过；纯只读评审或纯文档同步如经明确认定，可写明豁免理由。
7. 验证完成并具备交接条件后：改为 `reviewing`、`fixing` 或 `done`，并补充验证信息。

## 6. 禁止事项

1. 禁止在多个文档并行维护“当前开发进度”。
2. 禁止把长期蓝图、需求池、临时想法混入当前交付状态。
3. 禁止没有 `来源设计` 的工作项进入 `planning` 及之后的状态。
4. 禁止没有 `来源计划` 或没有豁免说明的工作项进入 `implementing`。
5. 禁止进入 `reviewing` 后没有 `测试文档` 或没有豁免说明。
6. 禁止把“继续优化”“联调一下”“后续完善”这类不可执行描述写成 `下一动作`。
7. 禁止没有验证命令或验证结果的工作项标记为 `done`。
8. 禁止超过约定周期未更新而仍长期显示为“实施中”“评审验证中”或“缺陷修复中”。

## 7. 模板约束

[../current-delivery-status.md](../current-delivery-status.md) 的结构固定为：

1. `仓库总览`
2. `进行中工作项`
3. `下一阶段工作`
4. `近期待验收`
5. `主要风险与外部依赖`

如需新增小节，应先确认不会形成第二份并行状态口径，再更新本契约。
