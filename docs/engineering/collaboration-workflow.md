# 协作工作流

本文用于固化 `Codex（代码智能体）` 在本项目中的共享协作协议，统一“先做设计、再写计划、后执行计划、最后完成分支”的默认工作方式。该文档只写团队共享规则，不沉淀个人提示词习惯、模型偏好或临时实验配置。

---

## 1. 共享协作目标

1. 把高频协作入口固定为少数可复用意图，减少每次重新描述流程的成本。
2. 把方案脑暴、计划拆解、执行实施、任务间审查、变更评审、接口文档同步、项目文档同步拆成稳定链路，避免任务中途口径漂移。
3. 让共享规则保持轻量，只保留项目边界、技能触发、强制验证、文档同步和交付格式。
4. 把当前开发进度与下个阶段工作收口到仓库级唯一状态入口，降低任务接力成本并暴露真实阻塞。
5. 在状态真源和技能门禁稳定后，支持“规则触发的半自动多智能体协作”：根据工作项状态优先切换到固定角色智能体，降低人工重新分派成本。

---

## 2. 基础工作流

### 2.1 总流程

默认基础工作流固定为四段：

1. `Brainstorming（方案脑暴技能）`
2. `Writing Plans（实施计划编写技能）`
3. `执行计划`
4. `完成分支`

对应约束如下：

1. `Brainstorming（方案脑暴技能）` 阶段，`AI（人工智能）` 先通过一问一答理解需求，探索多种方案，分段呈现设计并等待确认。
2. 确认后的正式设计结论优先回写项目主文档；其中前端页面结构、导航、交互和中文界面口径写入 `docs/architecture/frontend-workbench-design.md`，方案对象、运行主线、接口契约和治理边界写入 `docs/architecture/system-design.md`，已经从主方案拆出的场景级特性文档写入 `docs/architecture/features/`。特性文档是主方案的场景化展开，不替代主文档。`docs/plans/` 仅承接过程稿或尚未并入主文档的中间设计稿。
3. 涉及功能新增、行为变更、场景实现或能力增强的开发任务，在进入 `Writing Plans（实施计划编写技能）` 前，必须先把当前能力沉到对应特性文档；并且自本轮原则调整起，新增或重写特性文档时必须在同一轮协作中同步产出对应实施计划。没有对应特性文档或缺少同步实施计划时，不得写实施计划以外的后续内容或启动实现；若无法判断当前任务是否触发该门禁，默认按触发处理。
4. `Writing Plans（实施计划编写技能）` 阶段，把设计拆成每步 2 至 5 分钟的极小任务；每个任务必须包含精确文件路径、完整代码、测试命令与预期输出，并且计划文件必须明确引用所依赖的特性文档路径。对于新建特性文档，该计划文件视为同轮必交付物，不允许延后补票。
5. 确认后的实施计划统一保存到 `docs/plans/`，文件命名建议为 `YYYY-MM-DD-<topic>-implementation-plan.md`；特性文档与实施计划默认成对出现，除非在状态文档、评审记录或计划文件中写明经批准的临时豁免。
6. `执行计划` 阶段默认优先 `subagent-driven-development（子代理驱动开发技能）`；如果用户明确要求批量执行或独立会话执行，则改用 `executing-plans（批量执行计划技能）`。执行前必须同时引用对应特性文档与实施计划；实现完成后，不直接进入测试，而是先进入“`Codex（代码智能体）` 开发 -> `Claude Code` 代码检视 -> `Codex（代码智能体）` 修复 -> `Claude Code` 复检 -> 测试验收”的执行内子流程。
7. `完成分支` 阶段只在全部验证通过后进入，并统一走 `finishing-a-development-branch（开发分支收尾技能）`。
8. 当前开发进度与下个阶段工作统一维护在 `docs/engineering/current-delivery-status.md`；该文件是团队内部日常推进与任务接力的唯一真源。

### 2.2 强制门禁

1. 所有实现任务强制执行 `TDD（测试驱动开发，Test-Driven Development）`。
2. `TDD（测试驱动开发，Test-Driven Development）` 必须完整经历 `Red（先写失败测试）` -> `Green（最小实现通过）` -> `Refactor（重构保持绿色）`，且必须明确看到测试先失败。
3. 任务之间默认自动请求审查；每个任务完成后先做规格符合性检查，再做代码质量检查，未关闭问题前不得进入下一任务。
4. `AI（人工智能）` 在执行前默认先检查可用技能，并在合适时机触发对应技能，不跳过技能门禁直接实施。
5. 实现完成后，必须先执行 `code-reviewing（代码检视技能）`；代码检视未通过时，由 `Codex（代码智能体）` 修复并再次送检，复检通过后才进入 `feature-test-report（特性测试与验收技能）` 所承载的测试阶段。
6. 设计确认后、实施计划落地后、实现完成后、任务交接前，必须同步 `docs/engineering/current-delivery-status.md`；未同步视为未完成交接。
7. 仓库是 `AI（人工智能）` 与团队共享执行上下文的唯一真源；凡影响实现、评审、验收或交接的正式约束，必须回写仓库文档、脚本、配置或测试。仅存在于聊天记录、口头说明或临时备注中的约束，不视为已生效约束。
8. 实施计划中的测试入口、`Mock（模拟对象）`、纯逻辑抽离、测试文档骨架、脚本与断言工具等“脚手架任务”必须先于对应业务任务出现；默认不新增额外任务标签，但要求执行顺序先固化边界、契约和反馈回路，再展开业务实现。
9. 能通过 `lint（静态检查）`、结构校验脚本、测试、`CI（持续集成，Continuous Integration）` 或目录约束表达的规则，应在对应基础设施就绪后升级为可执行门禁；基础设施未就绪前，暂按“文档约束 + 人工检查点”执行，并在状态文档或评审记录中显式写明仍处于人工门禁阶段。
10. 新增目录、共享组件、公共接口、页面主路径、数据契约、导航结构与协作规则时，必须同步补齐可复用的 `Golden Path（黄金路径样例）` 样例或模板；存量约束按优先级逐步补齐，当前默认从“数据地图高可见层首个真实执行样本”开始试点。
11. 任何例外处理都必须在对应计划、评审或状态文档中写明豁免原因、临时边界、回收时机与责任人；未写明的例外，不允许长期存在。

### 2.3 交付状态真源

1. `docs/engineering/current-delivery-status.md` 是团队内部“当前开发进度 / 下个阶段工作”的唯一入口。
2. 该文件只记录当前阶段、进行中工作项、下一阶段工作、近期待验收和主要风险，不替代正式设计、实施计划和历史复盘。
3. 任何状态项都必须能回链到正式设计真源与实施计划；若属于纯只读评审或纯文档同步，应显式写明豁免理由或对应协作文档。

## 3. 六个稳定入口

| 入口 | 目标 | 默认动作 | 主要输出 |
| --- | --- | --- | --- |
| `做方案` | 收敛需求、边界与实现路径 | 执行 `Brainstorming（方案脑暴技能）`，不直接编码 | 设计确认结果、正式主文档路径或设计稿路径 |
| `写计划` | 把设计变成可执行任务清单 | 执行 `Writing Plans（实施计划编写技能）` | 计划文件路径、任务拆分结果 |
| `按方案实现` | 按已确认设计和计划落地实现 | 默认优先 `subagent-driven-development（子代理驱动开发技能）`，批量场景用 `executing-plans（批量执行计划技能）` | `改动摘要`、`执行命令`、`服务状态`、`剩余风险` |
| `评审这次改动` | 对现有改动做只读评审 | 重点找回归、契约漂移、缺失验证和缺失文档 | 问题发现项优先、结论次之 |
| `同步接口文档` | 保持接口说明与实现一致 | 以代码和注解为真源，优先同步 `SpringDoc（接口文档生成框架，SpringDoc OpenAPI）` / `OpenAPI（开放接口描述规范，OpenAPI Specification）` 输出 | 已同步接口范围、可访问地址、仍需人工维护的补充说明 |
| `更新项目文档` | 让主方案、开发文档和 README 导航保持一致 | 先按路由映射分类，再做全量一致性检查 | 文档同步结果 |

## 3.1 半自动多智能体协作

当用户明确要求多智能体协作，或团队进入规则触发的半自动编排模式时，固定采用以下四类角色，不按专题无限拆分：

1. `Design Agent（设计智能体）`
   - 对应技能：`brainstorming`、`feature-doc-coverage-mapping`、`feature-doc-authoring`
   - 负责状态：`proposed`、`brainstorming`、`planning（特性文档缺口补齐）`
2. `Plan Agent（计划智能体）`
   - 对应技能：`writing-plans`
   - 负责状态：`planning（实施计划编写）`
3. `Build Agent（实现智能体）`
   - 对应技能：`subagent-driven-development` / `executing-plans`、`test-driven-development`
   - 负责状态：`implementing`、`fixing`
4. `Review Agent（评审智能体）`
   - 对应技能：`requesting-code-review`、`code-reviewing`、`feature-test-report`、`change-review`
   - 负责状态：`reviewing`

补充说明：

1. 当需求还停留在“主方案已经有能力，但尚未拆到特性文档”的阶段时，不直接进入 `Plan Agent（计划智能体）`；先执行 `feature-doc-coverage-mapping` 形成映射清单。
2. `feature-doc-authoring` 在 `gate mode` 下负责特性文档最低完备性门禁，在 `authoring mode` 下允许直接补写或新建目标特性文档；它可以承接 `feature-doc-coverage-mapping` 的结果做批量补文档，但不替代能力映射本身。

触发约束如下：

1. 状态流转优先触发角色推荐或唤起；在 automation（自动化）模式下，允许角色在门禁下直接写入对应正式产物。
2. 半自动编排可被用户即时指令覆盖。
3. 若任务规模很小、上下文高度集中，允许退回单智能体执行。

### 3.2 Automation 接入原则

当团队启用 automation（自动化巡检）承接半自动编排时，固定遵循以下原则：

1. automation 可以读取 `docs/engineering/current-delivery-status.md`、对应特性文档、对应实施计划与必要的测试文档，并允许在门禁下直接写入：
   - `docs/architecture/features/` 下目标特性文档
   - `docs/plans/` 下目标实施计划
   - `docs/engineering/current-delivery-status.md` 中当前工作项的 `当前状态`、`最新完成`、`下一动作`、`最后更新时间`
2. automation 不允许自动新建工作项，也不允许自动重写 `来源设计`、`来源计划` 与 `退出条件`。
3. `planning（计划中）` 状态若发现特性文档缺失、映射不清或门禁未通过，优先进入 `Design Agent（设计智能体）` 的补文档链路；特性文档通过门禁后，再进入 `Plan Agent（计划智能体）`。`implementing（开发中）` 状态优先进入 `Build Agent（实现智能体）`；`reviewing（评审验证中）` 状态优先进入 `Review Agent（评审智能体）`；`fixing（修复中）` 状态回退到 `Build Agent（实现智能体）`。
4. automation 若发现状态、特性文档、实施计划或测试文档之间存在冲突，只输出问题项和建议动作，不自动回填文档。
5. 第一轮 automation 试运行必须绑定真实样本工作项；当前默认样本为“数据地图高可见层首个真实执行样本”。
6. 面向“主方案能力是否已拆到特性文档”的自动化巡检，默认按 `feature-doc-coverage-mapping -> feature-doc-authoring` 串联执行：先产出“能力项 -> 特性文档 + 实施计划”映射清单，再只对命中的目标文档做最低完备性检查。巡检结果必须显式指出两类缺口：一类是主方案能力尚未拆到特性文档，另一类是特性文档已存在但缺对应实施计划。
7. 子代理模型与推理强度的复杂度路由配置，统一放在 `ai/project/agents/model-routing.json`；本文件只保留入口与职责，不维护阈值细节。

## 4. 分阶段协议

### 4.1 方案会话

1. 默认用于新需求、模糊需求、跨模块需求、行为变更需求或用户明确说“做方案”的场景。
2. 方案阶段执行 `Brainstorming（方案脑暴技能）`，通过一问一答精炼设计，并至少给出 2 至 3 种方案与权衡。
3. 方案阶段按节分段呈现设计供用户确认；未经确认，不进入计划拆解或实现阶段。
4. 设计确认后，优先按项目文档路由原则回写正式主文档；只有过程稿或暂不适合直接并入主文档的中间设计，才写入 `docs/plans/`。
5. 若需求属于功能新增、行为变更、场景实现或能力增强，且后续要进入开发，必须先补对应特性文档，并在同一轮协作中同步产出对应实施计划，再进入计划 / 实现阶段。
6. 特性文档是主文档的场景化展开，不是并行真源；若当前场景暴露出主文档总纲缺口，应先回写主文档，再同步特性文档；当特性文档与主文档发生冲突时，以主文档为准，且当次任务必须同步修正特性文档。
7. 设计确认当天，应同步 `docs/engineering/current-delivery-status.md`，补齐 `来源设计`、`当前状态` 与 `下一动作`。

### 4.2 计划会话

1. 默认用于设计已确认、对应特性文档已补齐，且用户要求进入实施准备阶段的场景。
2. 计划阶段执行 `Writing Plans（实施计划编写技能）`，把设计拆成 2 至 5 分钟级别的小任务；自动化默认主模型为 `gpt-5.4 + xhigh`，备选为 `claude-sonnet-4-6-20260218`。
3. 只有满足最低完备性的特性文档，才可视为“已补齐”。最低标准固定为：遵循 `docs/engineering/standards/scenario-feature-doc-standard.md`，并至少写明场景背景、目标、功能说明、涉及治理对象、接口或数据边界、前端承载页面或工作台，以及本场景不负责什么；一句话占位文档不得用于过门禁。
4. 每个计划文件必须在开头明确引用对应特性文档路径；若当前需求属于功能新增、行为变更、场景实现或能力增强，但还没有对应特性文档，或特性文档已存在但尚未同步出对应计划，必须退回方案文档同步阶段，不得直接进入后续实现。
5. 每个任务必须写明精确文件路径、完整代码、测试命令、预期输出，避免“后续补充”“按类似方式实现”等占位描述。
6. 实施计划写入 `docs/plans/` 后，才进入执行阶段。
7. 实施计划落地当天，应同步 `docs/engineering/current-delivery-status.md`，补齐 `来源计划`、计划启动顺序和首个执行动作。

### 4.3 执行会话

1. 默认用于用户明确说“按方案实现”或设计与计划都已确认后的实施阶段。
2. 执行时必须先引用已确认的特性文档与实施计划，再开始编码、改文档或跑命令；缺少任一文档时，应退回补文档而不是直接实现。
3. 执行方式默认优先 `subagent-driven-development（子代理驱动开发技能）`：主 `AI（人工智能）` 派发子代理逐任务实施，并在每个任务后做双阶段审查；自动化默认主模型为 `gpt-5.2-codex + high`，满足高风险条件时升级到 `gpt-5.2-codex + xhigh`，备选为 `qwen3-coder-plus`。
4. 若用户明确要求批量执行、独立会话执行或一次跑完整个计划，则改用 `executing-plans（批量执行计划技能）`，并按批次设置检查点。
5. 执行过程中若发现目标需要变化，应退回方案阶段重新确认，不在实现中途直接改口径。
6. 代码变更完成后，先执行 `code-reviewing（代码检视技能）`：由 `Claude Code` 进行只读代码检视；默认主模型为 `claude-sonnet-4-6-20260218`，优先使用 `review` skill；未通过时由 `Codex（代码智能体）` 修复，并再次进入 `code-reviewing（代码检视技能）` 复检。
7. 代码检视通过后，执行 `feature-test-report（特性测试与验收技能）`，在 `docs/testing/features/` 下生成或更新测试文档，并按验收结果决定是否进入测试后修复。
8. 测试与代码检视全部通过后，按项目运行规范重启受影响服务并做轻量验活。
9. 每完成一批任务或每日收工前，应更新 `docs/engineering/current-delivery-status.md` 中对应工作项的 `最新完成` 与 `下一动作`。
10. 任务交接前，必须更新 `当前状态`、`阻塞项`、`责任人` 与 `最后更新时间`；若出现新阻塞，应在当次工作时段内改为 `blocked（阻塞中）` 并写清原因。

## 5. 实施门禁与审查通道

1. 所有实现默认执行 `TDD（测试驱动开发，Test-Driven Development）`，没有先失败的测试，不写生产代码。
2. `TDD（测试驱动开发，Test-Driven Development）` 必须保留可核验的 `Red（先写失败测试）` -> `Green（最小实现通过）` -> `Refactor（重构保持绿色）` 证据。
3. 任务间自动审查是基础门禁：
   - 先做规格符合性检查
   - 再做代码质量检查
4. 如执行方式是 `subagent-driven-development（子代理驱动开发技能）`，默认在每个任务之间触发 `requesting-code-review（请求代码审查技能）` 或同等审查动作。
5. `评审这次改动` 只做只读评审，不直接实现修复。
6. 评审固定关注五类问题：
   - 回归风险
   - 契约漂移
   - 缺失验证
   - 缺失文档
   - 会被 `AI（人工智能）` 继续复制的坏模式
7. 前端若展示状态机相关状态，界面口径统一使用中文；代码、接口字段、内部枚举和值对象可保留英文状态编码，但必须经过集中映射后再展示，不得把 `DRAFT`、`PUBLISHED`、`FAILED` 一类英文状态值直接暴露给业务用户。
8. 输出顺序固定为问题发现项优先、总结次之；若无明确问题，也要写出残余风险或测试空白。
9. 以下任务默认视为中高风险，执行阶段收尾前应补一次评审：
   - 同时改动多个模块或前后端联动
   - 涉及公共接口、数据契约、导出链路或鉴权边界
   - 涉及导航、路由、主流程布局或关键业务文案
   - 涉及文档拓扑、共享触发语或协作规则调整
10. 若评审结论改变了当前可交付状态、阻塞项或下个阶段工作，应在评审结束前同步 `docs/engineering/current-delivery-status.md`。
11. 当工作项从 `implementing（实施中）` 进入 `reviewing（评审验证中）` 时，`Review Agent（评审智能体）` 默认补一次轻量“坏模式扫描”；若发现重复实现、脆弱测试、命名漂移、失效计划或未回写主文档等问题，优先回填到对应测试文档、特性文档或评审记录，而不是另起并行台账。
12. 当工程协作需要复用“已打开 `ChatGPT（对话式人工智能产品）` 页面”的浏览器上下文时，优先通过 `ai/skills/chatgpt-browser-bridge/SKILL.md` 调用 `scripts/chatgpt_browser_bridge.py`；它是内部协作桥接能力，不替代通用浏览器自动化。

## 6. 完成证据与分支收尾

所有“按方案实现”类任务收尾时，统一输出以下四项：

```text
改动摘要：
执行命令：
服务状态：
剩余风险：
```

其中：

1. `改动摘要` 只写本次真正落地的核心变更。
2. `执行命令` 记录本次跑过的关键构建、测试、校验或同步命令。
3. `服务状态` 默认检查 `8082` 和 `5174`；若涉及接口契约，追加 `/v3/api-docs` 或相关接口校验结果。
4. `剩余风险` 只保留仍未消除的问题、限制或待补验证。
5. 输出四项证据前，应先确认 `docs/engineering/current-delivery-status.md` 已反映最终状态、下一动作或交接结果。

在四项证据之后，如果全部测试通过，统一进入 `finishing-a-development-branch（开发分支收尾技能）`，并只提供以下四个选项：

1. 本地合并回基线分支
2. 推送并创建 `PR（合并请求，Pull Request）`
3. 保留当前分支，稍后处理
4. 丢弃本次工作

未通过测试前，不进入分支收尾选项。
