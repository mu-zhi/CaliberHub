- **对应特性文档：** `docs/architecture/features/iteration-01-knowledge-production/03-资产建模与治理对象编辑.md`

## 二、验收范围与边界

### 2.1 覆盖项
- [x] **候选对象转换能力：** 提供 `ImportCandidate`（候选草案数据）向 `Scene` / `Plan` 等受控图谱资产的数据平滑搬运与映射。
- [x] **主数据真实落位：** 新建资产主数据由 H2 库正式切换为真实关系型 MySQL 控制库。
- [x] **状态机闭环限制：** 证明草稿态 (DRAFT) 锁定后在未达成门禁时无法流转为不可变的已发布态 (PUBLISHED)。
- [x] **强制入参校验网关 (Publish Gate)：** 场景发布前拦截并校验必备规则约束的闭环验证（例如校验 InputSlotSchema）。

### 2.2 不覆盖项（本次延后或不涉及验证）
- 前端页面的结构化表单级交互与展示对齐逻辑。
- 对未决 `Policy` 和复杂 `OutputContract` 自定义子字段的深层逻辑渲染。
- 图数据库 (Neo4j等) 与关系库同步投影的双写性能和一致性监控指标，目前聚焦主库真实行为保障。

## 三、测试案例与执行记录

### 3.1 TC-01：基于 API 验证候选场景转换为正式资产
- **输入：** 使用合规候选 JSON 载荷调用 `POST /api/import/candidates/{candidateCode}/confirm` 接口，包含新定义的代发查询口径。
- **预期输出：** `200 OK`，HTTP 响应实体中返回全新生成的、基于真实 DB Id 映射的正式 `Scene` 资产信息以及状态标识变更为锁定。
- **实际结果：** 顺利响应，`sceneId` 及资产落库正常，旧表中标志位正确更新为 `SCN-xxx` 的关联标识。**（通过）**

### 3.2 TC-02：资产发布强校验门禁阻断验证 (Publish Gate)
- **输入：** 获取 TC-01 生成的 `Scene Id`，未补齐 `InputSlotSchema` 的情况下直接调用 `POST /api/scenes/{id}/publish`。
- **预期输出：** 触发异常和拦截，返回标准提示：`发布失败，控制资产不完整：输入槽位：至少需要 1 条输入槽位定义`。
- **实际结果：** 精确命中了校验控制异常拦截器，响应给前端 `400 Bad Request` 并提供错误枚举反馈。**（通过）**

### 3.3 TC-03：发布通过与 Knowledge Package (运行时决策) 联通验证
- **输入：** 手动对 TC-01 资产追加一条 `InputSlotSchema` 记录，随后重调 `POST /api/scenes/{id}/publish`；在快照定版（生成 snapshotId 1）后，调用知识联机查询 `POST /api/graphrag/query` 请求对应场景但故意缺漏必填特征。
- **预期输出：** 
  1. Scene 成功转换为 `PUBLISHED`。
  2. 运行时知识引擎给出知识包，因特征不足，决策标识返回面向画布交互的补充类判定 `IDENTIFIER_REQUIRED` (或者明确提示需要必填标识)。
- **实际结果：** 完全满足以上预期；API 后端已按照口径产出了面向决策澄清的产品包。**（通过）**
