## 三、测试案例与执行记录

### 3.1 TC-01：候选实体图谱构建
- **输入：** 导入代发/薪资域样板材料。
- **预期输出：** 返回任务级候选图谱，节点类型覆盖 `CANDIDATE_SCENE / CANDIDATE_PLAN / IDENTIFIER / FIELD_CONCEPT / TIME_SEMANTIC / SOURCE_TABLE / SOURCE_COLUMN / JOIN_RELATION / CANDIDATE_EVIDENCE_FRAGMENT`。
- **实际结果：** 已执行并通过。`ImportTaskApiIntegrationTest` 已验证 `GET /api/import/tasks/{taskId}/candidate-graph` 返回 `200`，样板材料会生成 `CANDIDATE_SCENE / CANDIDATE_PLAN / IDENTIFIER / FIELD_CONCEPT / TIME_SEMANTIC / SOURCE_TABLE / CANDIDATE_EVIDENCE_FRAGMENT` 七类节点，以及 `SCENE_HAS_PLAN / PLAN_USES_SOURCE_TABLE / PLAN_USES_TIME_SEMANTIC / SCENE_USES_IDENTIFIER / NODE_SUPPORTED_BY_EVIDENCE` 五类边。`ImportCandidateGraphBuildServiceTest` 已补充 `SOURCE_COLUMN`、`SOURCE_COLUMN` 与 `PLAN_USES_SOURCE_COLUMN`、`JOIN_RELATION` 与 `PLAN_USES_JOIN_RELATION` 的场景级构建与高层边缘连接验证，但当前规则样板接口样例仍未稳定覆盖 `source_columns / join_relations` 的 API 级联调。

### 3.2 TC-02：高风险节点人工确认回写
- **输入：** 接受一个 `TIME_SEMANTIC` 节点和一个 `SOURCE_TABLE` 节点。
- **预期输出：** 任务预处理结果与候选场景载荷被回写，进入建模页面时可见已接受提示。
- **实际结果：** 已执行并通过。`POST /api/import/tasks/{taskId}/candidate-graph/review` 接受 `TIME_SEMANTIC` 与 `SOURCE_TABLE` 节点后，`POST /api/import/candidates/{candidateCode}/confirm` 返回的正式 `Scene（业务场景）` 已包含 `sqlVariantsJson.default_time_semantic = TRX_DT`，且 `sqlVariantsJson.source_tables` 保留 `PDM_VHIS.T05_AGN_DTL`；前端 `knowledge-import-utils.test.js` 与 `KnowledgePage.render.test.jsx` 也已验证候选图归一化与页面渲染接线。

### 3.3 TC-03：导入 -> 复核 -> 确认 -> 发布闭环
- **输入：** 无草稿直建模式导入样板材料，先复核候选图，再确认候选场景并发布。
- **预期输出：** 形成正式 `Scene（业务场景）`，且确认后的时间语义和来源线索已进入正式载荷。
- **实际结果：** 已执行并通过。`MvpKnowledgeGraphFlowIntegrationTest` 已覆盖“导入 -> 候选图查询 -> 接受高风险节点 -> `confirmCandidate` -> 质检确认 -> 发布 -> 数据地图 / 影响分析 / 运行查询”闭环，且断言确认后的时间语义与来源表线索进入正式 `sqlVariantsJson`，发布响应、投影状态与数据地图保持同一 `snapshotId`。

### 3.4 TC-04：同一业务场景下多取数方案归一到共同场景
- **输入：** 两段标题分别为“方法1：按协议号查询代发明细”“方法2：按协议号查询代发明细”的材料，它们输出字段一致，但来源表不同。
- **预期输出：** 预处理结果被归一为 1 个 `Candidate Scene（候选场景）` 和多个 `Candidate Plan（候选方案资产）`；候选图中只保留一个场景节点，并通过多条 `SCENE_HAS_PLAN` 挂接多个方案节点。
- **实际结果：** 已执行并通过。`ImportSceneNormalizationSupportTest` 和 `ImportCandidateGraphBuildServiceTest` 已覆盖“同场景多方案 -> 单场景多方案节点”规则；本地真实接口烟测也已验证 `POST /api/import/preprocess` 返回 `sceneCount = 1`、`firstSceneVariantCount = 2`，随后 `GET /api/import/tasks/{taskId}/candidate-graph` 返回 `sceneNodes = 1`、`planNodes = 2`、`sceneHasPlanEdges = 2`。

## 五、遗留风险与测试缺口

- **风险 1：`SOURCE_COLUMN / JOIN_RELATION` API 联调样例仍依赖样板模板覆盖**
  构建单测已覆盖这两类节点及 `PLAN_USES_SOURCE_COLUMN` / `PLAN_USES_JOIN_RELATION` 边连接，但当前稳定样板仍未固定产出这两类字段，`/api/import/tasks/{taskId}/candidate-graph` 的高层回归暂未形成稳定断言。
  **状态：** identified

- **风险 2：前端仍缺浏览器级交互回归**
  当前前端验证覆盖了候选图归一化工具、`KnowledgePage` SSR 渲染和生产构建，但还没有 Playwright 级别的点击复核、接受/驳回、焦点切换与表单刷新链路回归。
  **状态：** identified

- **风险 3：多方案命名质量依赖上游上下文密度**
  已通过“方法名归一 + 来源表补充 + 适用时段”三元规则增强同场景方案命名，能区分常见同名多方案；剩余风险是输入不含 `applicable_period`、`source_tables` 或方法前缀信息时仍需人工侧重确认。
  **状态：** identified

- **已关闭项**
  1. 高风险候选接受结果未稳定回写到正式载荷：已通过 `ImportTaskApiIntegrationTest` 与 `MvpKnowledgeGraphFlowIntegrationTest` 关闭。
  2. 候选图编码映射存在不一致风险：已通过 `ImportCandidateCodeSupport` 统一编码规则，并由后端单测覆盖节点 / 边与候选场景的 `sceneCandidateCode` 映射关闭。

## 六、最终结论

- 当前测试报告已按最新实现与复验结果收口：候选图构建、查询、人工复核、候选确认回写、前端工作台渲染接线，以及“导入 -> 复核 -> 确认 -> 发布”闭环均已有自动化验证。
- “同一业务场景下多种取数方案必须汇合到共同场景节点”已被补成正式约束，并有单测、回归测试和真实接口烟测三层验证。候选方案命名增强已覆盖“方法名/来源表/适用时段”规则。
- **结论：本阶段核心主链路高层回归与命名增强已覆盖；剩余事项收敛为 `source_columns / join_relations` API 规则样板覆盖与 Playwright 级交互回归。**
