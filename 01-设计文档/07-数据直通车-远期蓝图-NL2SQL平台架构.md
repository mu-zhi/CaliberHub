## 一、蓝图定位

数据直通车 2.0 的终态目标是构建 NL2SQL（自然语言转SQL）一站式平台：

- 业务用户用自然语言发起取数，系统基于结构化知识资产自动生成 SQL
- 口径治理阶段产出的数据地图，成为 NL2SQL 的知识底座
- 成功的 SQL 自动沉淀为可复用模板，逐步减少对 AI 的依赖

### 从口径治理到 NL2SQL 的演进路径

```
第一阶段 口径治理MVP     → 结构化场景资产
第二阶段 引用与版本      → 语义视图 + 字典 + 引用关系
第三阶段 对齐与导出      → 元数据对齐 + 服务说明
第四阶段 语义检索        → ES + 向量检索 + 同义词
─────────────────────────────────────────────
远期 NL2SQL        → 自然语言查询 + SQL生成 + 安全执行
```

> 只有当 第一阶段至第四阶段 产出了足够的结构化资产（≥ 200 个已发布场景、覆盖 TOP10 业务域），NL2SQL 才具备启动条件。

## 三、微服务划分（远期）

> 当前阶段为单体应用。以下微服务划分在系统演进到 NL2SQL 阶段时适用。

### 3.1 svc-nl2sql（查询服务）

- **职责**：解析用户自然语言问句，组装上下文（领域词/映射、表 Schema 摘要、历史交互），调用推理接口生成候选 SQL，打分选优，生成自然语言解释
- **四层结构**：
  - Adapter：`QueryController`
  - Application：`QueryAppService`（权限钩子、知识检索、候选评估）
  - Domain：`Query`、`NLIntent`、`CandidateRankService`
  - Infrastructure：`ModelPort`（LLM）、`KnowledgePort`、`HistoryRepo`

### 3.2 svc-query-gateway（查询执行网关）

- **职责**：只读执行、DML/DCL 拦截、超时/行数限制、敏感字段脱敏、审计落库
- **安全防线**：只读 → 拦截 → 限流 → 脱敏 → 审计

### 3.3 svc-knowledge（知识管理）

- **职责**：口径治理系统的演进体，承载术语/映射、场景、标注评审、元数据同步、主题节点树（原业务树）浏览
- 由当前的口径治理单体应用拆分而来

### 3.4 svc-config（配置与策略）

- **职责**：全局参数、数据源连接、模型端点、灰度开关

## 四、NL2SQL 关键流程

```
用户输入自然语言
    │
    ▼
意图理解 + 上下文组装（术语/映射/Schema/历史）
    │
    ▼
调用 LLM 推理 → 候选 SQL 列表
    │
    ▼
规则 + 知识打分 → 选优 → 生成解释
    │
    ▼
SQL 安全网关（只读/拦截/限流/脱敏/审计）
    │
    ▼
执行查询 → 结果 + SQL + 解释返回用户
    │
    ▼
成功 SQL 沉淀为可复用模板（自强化循环）
```

## 五、安全与合规设计（远期）

| 安全层 | 措施 |
|--------|------|
| 只读 | 禁止 DML/DCL；黑名单（关键字/函数）；白名单（库/表/视图） |
| 限流 | 连接池隔离、并发阈值、查询队列；成本预估拦截大扫 |
| 脱敏 | 列级策略（正则/标签匹配），策略模板化（掩码/置空/哈希） |
| 审计 | 不可篡改落库；记录用户/SQL/表名/行数/耗时/策略命中 |

## 九、M3 实施下钻（原 09 文档合并）

> 本章是“远期蓝图”中的当前可实施部分，用于承接 M3（对齐与导出 + 可控取数）。  
> 该部分替代原独立文档《09-数据直通车-M3可控取数蓝图》。

### 9.1 M3 核心链路

```text
自然语言查询 -> 图谱增强检索 -> 计划路由 -> Plan(IR) -> 风险门禁 -> Service Spec/SQL模板 -> 执行侧
```

核心目标：输出可解释、可审计、可回放的 Plan(IR)，而不是黑盒 SQL。

### 9.2 Service Spec（发布态契约）

建议最小结构：

```json
{
  "spec_code": "PAYROLL_DETAIL_QUERY",
  "spec_version": "1.0.0",
  "scene_ref": {"scene_id": 1001, "scene_code": "PAYROLL_DETAIL_QUERY"},
  "input_schema": [],
  "output_schema": [],
  "routing_rules": [],
  "risk_policy": {},
  "examples": [],
  "error_model": []
}
```

字段说明：

1. `input_schema`：参数名、类型、必填、校验规则、派生规则引用。
2. `output_schema`：字段说明、敏感等级、脱敏策略（M3 强制）。
3. `routing_rules`：按时段/条件选取 Plan。
4. `risk_policy`：审批级别、行数与成本约束、拒绝条件。

### 9.3 Plan(IR)（运行时白盒中间态）

建议最小结构：

```json
{
  "query_id": "q_20260227_001",
  "spec_ref": {"spec_code": "PAYROLL_DETAIL_QUERY", "spec_version": "1.0.0"},
  "scene_ref": {"scene_id": 1001, "scene_code": "PAYROLL_DETAIL_QUERY"},
  "plan_ref": {"variant_name": "2014至今主表", "plan_id": "plan_01"},
  "resolved_params": [],
  "missing_params": [],
  "evidence": [],
  "risk_decision": {"decision": "allow|need_approval|deny", "reasons": []},
  "audit_explain": []
}
```

审计最小集：

1. 命中 Scene/Spec/Plan。
2. 参数来源（输入/默认/派生）。
3. 路由依据（命中规则表达式）。
4. 风险判定与原因。

### 9.4 路由 DSL 与风险策略

路由 DSL 示例：

```text
rule "payroll_recent"
priority 100
when date_range.overlaps("2014-01-01", "NOW") and channel in ["retail","branch"]
then plan "PAYROLL_DETAIL_QUERY#plan_2014_now"
```

风险决策输出：

1. `allow`：允许输出可执行模板参数。
2. `need_approval`：需 OA 审批后放行。
3. `deny`：不允许执行。

### 9.5 Graph-RAG 检索与排序

召回层：

1. 关键词召回（Scene/Spec/Term）。
2. 向量召回（Scene/Plan/Term/Column）。
3. 图扩展（1-2 跳引用关系）。

排序函数示意：

```text
score = 0.45 semantic + 0.25 graph + 0.15 trust + 0.10 fit - 0.05 risk_cost
```

### 9.6 M3 前置依赖

1. M1 必须保证 `applicable_period/source_tables` 完整。
2. M1 建议补齐 `id_mapping_notes`。
3. M2 必须完成 `Dictionary/SemanticView/Version/Reference`。

### 9.7 M3 落地里程碑

1. M3-A（对齐期）：完成 `alignment_report` 与发布门禁接入。
2. M3-B（契约期）：自动生成 Service Spec，提供 `GET /api/service-specs/{specCode}`。
3. M3-C（计划期）：提供 `POST /api/nl/query` 输出 Plan(IR)，执行侧人工确认后执行。

### 9.8 M3 验收指标

1. Plan(IR) 生成 P95 <= 5s（不含执行侧）。
2. 路由可解释率 100%。
3. 风险策略命中可审计（抽检通过率 100%）。
4. 高敏字段必须触发审批或脱敏策略。
