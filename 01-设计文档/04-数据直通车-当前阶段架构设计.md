# 数据直通车 · 当前阶段架构设计

> **目标**：定义口径治理 MVP 及后续 第二阶段/第三阶段 迭代的系统架构，只包含当前阶段需要落地的技术方案。
>
> **范围**：覆盖口径治理 MVP 到第二阶段/第三阶段的模块边界、系统分层与部署关系；不包含远期 NL2SQL（自然语言转 SQL）平台的完整实现设计。
>
> **读者**：开发团队。
>
> **当前阶段定位**：口径治理 M1/M2/M3 三阶段建设，是数据直通车 2.0 远期蓝图的基础。
>
> **术语说明**：文档中的技术术语请参考 [统一术语表](08-数据直通车-统一术语表.md)。本系统采用"六资产模型"作为核心治理资产。

## 一、系统定位

**数据直通车·口径治理系统** 是银行大数据部门的内部知识管理工具，将散落的口径文档结构化为可浏览、可搜索、可复用的业务场景资产。

### 边界声明

| ✅ 做 | ❌ 不做 |
|------|--------|
| 口径文档结构化导入 | SQL 执行（由数据直通车 1.0 承担） |
| 业务场景管理（CRUD + 发布） | 审批流程（由行内 OA 承担） |
| 场景浏览与搜索 | 权限裁决（由外部平台控制） |
| LLM 辅助抽取 | NL2SQL 自然语言查询（远期蓝图） |
| 结构化资产版本管理（第二阶段） | 实时数据处理 |

## 三、角色模型

| 角色 | 定义 | MVP 阶段职责 |
|------|------|-------------|
| **数据支持组成员** | 总行开发同事 | 导入口径文档、编辑场景、发布场景 |
| **一线业务人员** | 分行数据分析师 | 浏览已发布场景、搜索口径 |

> **M1 阶段**：默认不启用细粒度权限（与接口配置 `caliber.security.enabled=false`、`require-write-auth=false` 一致），业务上可视为操作员共享能力；如开启写鉴权，则按接口文档角色矩阵收敛写权限。
>
> **M2 阶段**：引入角色区分和简化审核流程
>
> **M3 阶段**：引入多人审核（语义/技术/合规）与影响分析

## 四、核心流程

### 4.1 导入流程

```
用户粘贴口径文档
    │
    ▼
后端拼装提示词 + 调用 LLM
    │
    ▼
LLM 返回 CALIBER_IMPORT_V2 JSON
    │
    ▼
前端展示对照视图（左原文 / 右结构化）
    │
    ▼
用户确认/修正 → 保存为草稿
```

### 4.2 编辑与发布流程

```
                ┌──────────┐
                │ 手动创建  │
                └────┬─────┘
                     │
                     ▼
┌────────┐    ┌────────────┐    ┌──────────────┐
│ LLM导入 │───▶│   DRAFT    │───▶│  PUBLISHED   │
└────────┘    │  （草稿）   │    │ （已发布）    │
              └────────────┘    └──────────────┘
                   ▲  │              │
                   │  ▼              ▼
                  编辑/修改      可浏览/可搜索
```

> **状态流转**：M1 只有 DRAFT 和 PUBLISHED 两个状态。发布需填写验证日期和变更摘要（建立可信追溯链）。

### 4.4 一致性与并发控制（补充）

1. DRAFT→PUBLISHED 必须在同一事务内完成状态变更与审计字段写入。  
2. Scene 编辑采用乐观锁（`row_version`/ETag）避免并发覆盖。  
3. 版本冲突返回 409（`CAL-SC-409`），由前端提示用户合并后重试。  
>
> **质量门禁**：M1 引入质量评分卡机制，对抽取结果进行硬缺失项检查（如 source_tables 为空时禁止发布）。

### 4.3 浏览与搜索流程

```
用户进入场景列表
    │
    ├── 按业务域筛选
    ├── 按状态筛选（DRAFT / PUBLISHED）
    └── 输入关键词搜索（标题/描述/SQL 全文）
    │
    ▼
点击场景 → 查看详情页（完整结构化信息 + SQL 高亮）
```

## 五、数据模型

### 5.1 MVP 阶段（第一阶段）

两张核心表。详细字段定义见 [MVP定义与交付计划](03-数据直通车-MVP定义与交付计划.md) 第四节。

```
caliber_domain  1 ──── * caliber_scene
  (业务域)                (业务场景)
    │                        │
    ├── domain_overview      ├── sql_variants_json  (取数方案数组)
    ├── common_tables        ├── code_mappings_json (码值说明)
    └── contacts             └── contributors       (口径提供人)
```

**核心设计决策**（基于真实口径文档验证）：
1. **取数方案（sql_variants）**：替代原 sql_blocks，新增 `applicable_period` 和 `source_tables`，支持"按时间选表"的路由逻辑（如代发明细横跨 8+ 张表）
2. **域级业务概述（domain_overview）**：承载口径文档开头的大段业务背景知识（如 5000 字的借记卡/存折定义）
3. **码值说明（code_mappings）**：嵌入场景 JSON，不独立为字典实体（M1 阶段码值嵌在 SQL 注释里，量级不大）
4. **口径提供人（contributors）**：场景级别标注权威来源
5. **长文本与附件存储策略**：口径文本类大字段采用分表存储（主表存高频检索字段，扩展表存长文本）以控制主表膨胀；仅在包含图片、PDF（便携式文档格式）附件时使用对象存储，数据库只保存对象引用与元数据

### 5.2 第二阶段（引用与版本）

拆出独立实体，建立引用关系：

```
caliber_domain  1 ──── * caliber_scene
                              │
                              │ 引用
                              ▼
                    caliber_semantic_view  (语义视图)
                    caliber_dictionary     (字典，从 code_mappings 升级)
                    caliber_scene_version  (版本记录)
```

### 5.3 第三阶段（对齐与导出）

增加对齐和导出能力：

```
caliber_scene ──── caliber_alignment_report  (对齐报告)
                       │
                       ▼
                  元数据平台 (外部)

caliber_scene ──── caliber_service_spec  (服务说明/导出)
caliber_scene ──── caliber_derivation_rule  (派生规则)
```

## 六、技术选型

### 6.1 MVP 技术栈

| 层级 | 选型 | 理由 |
|------|------|------|
| 后端框架 | Spring Boot 3.x | 团队熟悉，银行标准 |
| 构建工具 | Maven | 银行标准 |
| 数据库 | MySQL 8.0 | 银行标配，MVP 数据量极小 |
| ORM | MyBatis / MyBatis-Plus | 团队熟悉，JSON 列支持良好 |
| LLM 调用 | HTTP Client（RestTemplate / OkHttp） | 直接调用行内 LLM API |
| 前端框架 | Vue 3 或 React（按团队决定） | 前后端分离 |
| SQL 编辑器 | CodeMirror / Monaco Editor | 前端 SQL 语法高亮 |
| 搜索 | MySQL FULLTEXT | MVP 数据量 < 1000，够用 |

### 6.2 第二阶段 新增

| 组件 | 用途 | 引入条件 |
|------|------|---------|
| Elasticsearch | 语义检索、同义词匹配 | 场景数 > 500 且搜索体验不足时 |
| Redis | 缓存热点场景 | 并发用户 > 50 时 |

### 6.3 明确不引入的组件

以下组件属于远期蓝图范畴，当前阶段**不引入**：

| 组件 | 属于远期蓝图的原因 |
|------|------------------|
| 向量数据库（Milvus/Pinecone） | NL2SQL 语义检索需求，当前阶段无此需求 |
| 图数据库（Neo4j） | 知识图谱查询需求，当前用关系表的引用关系足够 |
| 消息队列（Kafka/RabbitMQ） | 异步流处理需求，MVP 全同步 |
| 模型服务（PyTorch/TF Serving） | 模型训练和推理，MVP 只调用现成 LLM API |
| K8s/容器编排 | MVP 直接部署到 VM 即可 |
| 微服务拆分 | MVP 单体足够，第二阶段 仍为单体 |
| GraphQL | 远期数据聚合需求 |

## 七、代码分层

### 7.1 MVP 阶段：经典三层

```
src/main/java/com/bank/caliber
├── controller/          # REST 接口
├── service/             # 业务逻辑
├── repository/          # 数据访问
├── model/
│   ├── entity/          # 数据库实体
│   └── dto/             # 请求/响应对象
└── config/              # 配置
```

### 7.2 第二阶段 阶段：演进为 DDD 四层

当引入独立实体（语义视图、字典）和引用关系后，重构为 DDD 分层：

```
src/main/java/com/bank/caliber
├── adapter/
│   └── web/             # Controller + 异常处理
├── application/
│   ├── dto/             # 请求/响应对象
│   ├── assembler/       # DTO ↔ Entity 转换
│   └── service/         # 应用服务（流程编排）
├── domain/
│   ├── model/           # 实体 + 值对象
│   ├── repository/      # 仓储接口
│   └── service/         # 领域服务
└── infrastructure/
    ├── persistence/     # 仓储实现 + PO + Converter
    └── external/        # LLM 调用、元数据平台对接
```

详细接口定义见 [MVP定义与交付计划](03-数据直通车-MVP定义与交付计划.md) 第六节。

|------|------|------|

┌─────────────────────────┐    ┌─────────────────────────┐

## 十一、安全与合规（MVP 最小集）

| 安全项 | MVP 方案 | 第二阶段 增强 |
|--------|---------|--------|
| 身份认证 | 接入行内统一登录（SSO） | 不变 |
| 操作审计 | 记录创建/编辑/发布操作到日志表 | 审计不可篡改 |
| 数据分类 | 场景标记敏感字段（文本标注） | 敏感字段关联脱敏规则 |
| SQL 安全 | SQL 原文只读展示，不执行 | 不变 |
