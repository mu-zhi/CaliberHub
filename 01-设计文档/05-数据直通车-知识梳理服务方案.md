# **知识梳理服务方案**

> **目标**：把散落在 SQL（结构化查询语言）文件、文档、聊天记录中的"业务口径"结构化沉淀为可浏览、可搜索、可复用的业务场景资产。

> **范围**：覆盖知识梳理服务在口径治理 M1/M2/M3 阶段的能力边界、概念模型、Plan(IR) 与阶段演进路径；不包含执行引擎实现。

> **读者**：数据支持组、产品经理、架构师、开发与测试团队。

> **当前阶段定位**：数据直通车 2.0 远期蓝图中的基础能力设计，聚焦口径治理 M1/M2/M3 阶段的大模型场景。

> **边界**：本方案不做执行引擎；执行、取数、计算由外部平台负责。

> **版本说明**：本方案已基于 3 份真实口径文档现状（零售客户信息查询、零售客户信息变更、代发明细查询）进行验证和调整，确保设计贴合实际。

---

## **0. 背景与问题陈述**

### **0.1 真实现状（从口径文档中提炼）**

通过分析 3 份真实口径文档，我们发现以下共性特征：

| 特征 | 实际情况 | 举例 |
|------|---------|------|
| **基本单位** | 以"业务场景"为核心（查询户口信息、客户密码修改…） | 单份文档含 5-20 个场景不等 |
| **SQL 是主体** | SQL 不是"附件"，是口径的核心载体 | 文档中 80% 内容是 SQL + 注释 |
| **结构极度不统一** | 有的只有 SQL，有的有描述+SQL，有的只有文字指引 | "查询线上申请卡片未领卡"仅有注释无可执行 SQL |
| **历史表分裂** | 同一业务横跨不同年代的不同表 | 代发明细横跨 2003-至今的 8+ 张表 |
| **码值嵌在注释里** | 不是独立字典，而是散在 SQL 注释中 | `'P1' 支付密码修改` 直接写在注释里 |
| **业务背景是大段文本** | 文档开头常有数千字的业务定义 | 借记卡/财富账户/存单定义约 5000 字 |
| **口径提供人是权威** | 部分标注提供人，部分缺失 | "谢华勇/80274711" |
| **跨文档引用不存在** | 每个文档自包含，同一码值可能重复解释 | CARD_GRD_CD 在同一文件中出现 2 次 |

### **0.2 我们要解决的根问题**

- 不是"把数据查出来"，而是把**散落在 SQL 文件和人脑中的口径经验**结构化保存，让"怎么查、查哪张表、要注意什么"可搜索、可浏览、可传承。
- 当前最紧迫的需求是：**新人接手时找得到、看得懂、不出错**。

## **1. 一句话定位与核心价值**

**知识梳理服务是数据团队的"口径经验结构化工具"**：

将散落的 SQL 文件、注释、个人经验整理为可浏览、可搜索的业务场景资产。

价值：
- 对数据团队：从"问人/翻文档/试错" → "搜得到、看得懂、能复用"
- 对新人：从"不知道查哪张表" → "按场景找到 SQL + 注意事项"
- 对管理：从"口径在个人电脑里" → "口径在系统里，人走了知识还在"

## **2. 核心概念模型**

### **2.1 资产类型与阶段映射**

基于真实文档验证，资产类型按阶段渐进引入：

| 资产类型 | 定义 | 引入阶段 | 理由 |
|---------|------|---------|------|
| **业务域（Domain）** | 一个业务领域的分组容器 | M1（MVP） | 对应实际文档的文件级别划分 |
| **业务场景（Scene）** | 解决一个具体取数问题的完整口径 | M1（MVP） | 真实文档的基本组织单位 |
| **取数方案（Data Retrieval Plan）** | 同一场景在不同时期/条件下的 SQL | M1（MVP） | 代发明细横跨 8+ 张表，必须支持 |
| 语义视图（Semantic View） | 怎么读懂一类数据记录 | M2 | 当前场景量不足以支撑独立实体 |
| 字典（Dictionary） | 码值与术语解释 | M2 | 当前码值嵌在注释里，量级不大 |
| 派生规则（Derivation Rule） | 参数转换规则 | M3 | 依赖引用关系图谱 |
| 主题节点（Topic Node） | 业务概念导航 | M3 | 依赖足够的场景积累 |
| 服务说明（Service Spec） | Scene 的发布形态 | M3 | 依赖成熟的场景资产 |

### **2.2 M1 阶段概念模型（MVP）**

```
业务域（Domain）
├── 域名称：零售客户信息查询
├── 业务概述（富文本）：借记卡、财富账户、存单、存折、户口系统…
├── 常见数据表清单（文本）：F3EACDTAP、CSEASDTAP、DEE_EAC_INF_T…
├── 负责人信息：谢鑫、谢华勇
│
└── 业务场景（Scene）[]
    ├── 场景标题：查询借记卡等级
    ├── 场景描述
    ├── 口径提供人
    ├── 注意事项（富文本，极其关键）
    ├── 结果字段描述
    ├── 限制字段描述
    ├── 取数方案（Data Retrieval Plan）[]  ← 核心设计
    │   ├── variant_name：当前主表查询
    │   ├── applicable_period：2014至今
    │   ├── description：使用 PDM 表判断卡类别
    │   ├── sql_text：SELECT ... FROM PDM_VHIS.T03_CORE_CUST_EAC_ATCH_INF_S ...
    │   ├── source_tables[]：PDM_VHIS.T03_CORE_CUST_EAC_ATCH_INF_S
    │   └── notes：受上云影响，主机表已不适用
    ├── 码值说明（嵌入 JSON，不独立为字典实体）
    │   └── [{ code: "CARD_GRD_CD", mappings: {"020":"金卡","040":"金葵花卡",...} }]
    └── 状态（草稿/已发布）
```

**为什么需要"取数方案"？**

以代发明细查询为例，同一个业务场景需要查不同年代的不同表：

| 方案名 | 适用时段 | 表名 | 备注 |
|--------|---------|------|------|
| 早期柜面代发 | 2003-2004 | CORINFO_PKTRSLOGP_YEAR | 数据不连续 |
| 合作方交易历史1 | 2004-2007 | EPHISTRXP1 | 日数据最高 108 万 |
| 合作方交易历史2 | 2004-2007 | EPHISTRXP2 | 与 EPHISTRXP1 数据有不同 |
| 统一代发表 | 2004-2013 | UNICORE_EPHISTRXP_YEAR_20220330 | 缺 12 月数据 |
| A2 代发明细 | 2013 | ODS_C3A3_A2DPADTLP_YEAR | 仅 2013 年 |
| 当前主表 | 2014至今 | T05_AGN_DTL | 当前主要使用 |

如果场景只能挂一段 SQL，则无法表达这种"按时间选表"的路由逻辑。

### **2.3 M2/M3 阶段概念演进**

**M2（引用与版本）**：在 M1 基础上拆出语义视图、字典为独立实体，建立引用关系，引入版本管理。

**M3（对齐与导出）**：在 M2 基础上增加派生规则、主题节点、服务说明、元数据对齐、影响分析。

### **2.4 长文本与附件存储策略（M1 生效）**

- 口径文本类长内容（如域级业务概述、场景注意事项）采用分表存储：主表保留高频检索字段，扩展表承载长文本，避免主表体量持续膨胀导致查询性能下降。
- 对象存储只用于包含图片、PDF（便携式文档格式）附件的场景，数据库仅保存对象键、访问路径和文件元数据。
- 纯文本与 SQL（结构化查询语言）正文不进入对象存储，继续在关系库中按结构化字段管理。

## **4. 端到端流程**

### **4.1 M1 阶段：3 步流程**

```
           ┌──────────────┐
           │ 手动录入      │  ← 主路径（口径提供人直接填写）
           │ （表单+SQL）  │
           └──────┬───────┘
                  │
                  ▼
┌─────────┐   ┌────────────┐   ┌──────────────┐
│ LLM 导入 │──▶│   DRAFT    │──▶│  PUBLISHED   │
│（辅助）  │   │  （草稿）   │   │ （已发布）    │
└─────────┘   └────────────┘   └──────────────┘
                  ▲  │              │
                  │  ▼              ▼
                 编辑/修改      可浏览/可搜索
```

**关键设计决策**：
- **手动录入是主路径**，因为口径提供人自己就是权威来源，不需要 LLM 代劳
- **LLM 导入是辅助加速器**，适用于从已有的大段 SQL 文件批量导入
- **不强制走流水线**，任何场景都可以从空白表单开始手动填写

### **4.2 M1 导入流程（LLM 辅助路径）**

```
用户粘贴口径文档（SQL 文件全文）
    │
    ▼
后端拼装提示词 + 调用 LLM
    │
    ▼
LLM 返回 CALIBER_IMPORT_V2 JSON（含取数方案）
    │
    ▼
前端展示对照视图（左原文 / 右结构化）
    │
    ▼
用户确认/修正 → 保存为草稿
```

### **4.3 阶段契约（用于研发拆解与验收）**

| 阶段 | M1 做什么 | M2 做什么 | M3 做什么 |
|------|----------|----------|----------|
| 采集解析 | 用户粘贴文本 / 上传文件 | 同 M1 + 支持图片 OCR | 同 M2 + 批量导入 |
| 抽取 | LLM 辅助抽取场景+取数方案 | 同 M1 + 抽取字段级语义 | 同 M2 + 抽取派生链 |
| 对照展示 | 左原文右结构化，支持编辑 | 同 M1 | 同 M1 |
| 对齐 | 不做 | 不做 | 与元数据平台表/字段对齐 |
| 规范化 | 不做 | 命名归一（建议，非强制） | 模板化表达 + 双轨对照 |
| 融合与冲突 | 不做 | 不做 | 冲突分级 + 可信来源优先 |
| 审核 | 不做（一人操作） | 单人确认 | 多人审核（语义/技术/合规） |
| 发布 | 草稿→已发布（状态切换） | 同 M1 + 版本号 | 同 M2 + 影响分析 + 一致性校验 |

## **5. 导入后对照视图**

- 展示目标：导入完成后，让用户同时看到"原始信息片段"和"系统整理后的结构化信息"。
- 展示方式：左侧原始信息（SQL 文件原文），右侧结构化信息（场景、取数方案、注意事项）。
- 映射能力：支持高亮一对一/一对多映射，点击结构化字段可定位到对应原始信息片段。
- 交互动作：支持"接受""编辑""标记待确认"，并将动作写入草稿变更记录。

## **6. 域级业务概述（M1 新增概念）**

### **6.1 为什么需要域级业务概述**

真实口径文档中，"零售客户信息查询"文件开头有约 5000 字的业务背景知识：借记户口、财富账户、存单、存折、户口系统、卡片等级、账户分类、制卡模式等。

这些知识：
- **不属于任何单一场景**——是理解所有场景的前提
- **不是字典**——不是码值映射
- **是领域背景知识**——帮助新人理解整个业务域

### **6.2 在业务域实体中承载**

```
caliber_domain
├── domain_name: "零售客户信息查询"
├── domain_overview: "（富文本）借记户口定义、财富账户定义……"   ← 新增
├── common_tables: "F3EACDTAP、CSEASDTAP……"              ← 新增
├── contacts: "谢鑫、谢华勇"                               ← 新增
└── sort_order: 1
```

## **7. 取数方案设计（M1 核心新增）**

### **7.1 取数方案结构**

```json
{
  "sql_variants": [
    {
      "variant_name": "当前主表查询",
      "applicable_period": "2014至今",
      "description": "使用 PDM 表 T05_AGN_DTL",
      "sql_text": "SELECT EVT_ID AS \"事件编号\" ... FROM PDM_VHIS.T05_AGN_DTL ...",
      "source_tables": ["PDM_VHIS.T05_AGN_DTL"],
      "notes": "限制字段：AGN_BCH_SEQ 代发批次号, EAC_NBR 收款户口号"
    },
    {
      "variant_name": "统一历史表",
      "applicable_period": "2004-2013",
      "description": "使用统一核心历史表",
      "sql_text": "SELECT ... FROM LGC_EAM.UNICORE_EPHISTRXP_YEAR_20220330 ...",
      "source_tables": ["LGC_EAM.UNICORE_EPHISTRXP_YEAR_20220330"],
      "notes": "注意：缺12月数据"
    }
  ]
}
```

### **7.2 与原 sql_blocks 的关系**

sql_variants 是 sql_blocks 的超集。原来只有一段 SQL 的场景，就只有一个方案（applicable_period 为空或"全时段"）。新增字段：`applicable_period`、`source_tables`。

## **8. 大模型抽取策略**

### **8.1 M1 定位：辅助加速器**

| 场景 | LLM 的作用 |
|------|----------|
| 用户粘贴一份 1000 行的 SQL 文件 | LLM 自动拆分为多个场景 + 识别取数方案 + 提取注意事项 |
| 用户手动创建场景 | 不需要 LLM |
| LLM 不可用 | 用户照常手动录入，系统 100% 可用 |

### **8.2 抽取规则**

| 规则 | 说明 |
|------|------|
| SQL 原文保真 | SQL 原文绝对不可改写 |
| 保守抽取 | 宁缺毋造，低置信度明确标注 |
| 码值内嵌 | 从 SQL 注释中提取的码值说明放入场景的 code_mappings JSON |
| 取数方案识别 | 识别同一场景中按时间/条件区分的多段 SQL，拆为 sql_variants |

## **9. 审核机制（按阶段渐进）**

| 阶段 | 审核方式 | 理由 |
|------|---------|------|
| M1 | 不做审核，一人操作 | 团队 < 5 人，信任口径提供人 |
| M2 | 单人确认 | 编辑者以外的一人确认后发布 |
| M3 | 多人审核（语义/技术/合规） | 场景量大、影响范围广 |

---

## **10. 共享语义机制（M2/M3）**

> M1 阶段不引入。当场景数 < 100 时，允许码值说明在不同场景中冗余存在。

M2 引入引用关系：场景引用字典/语义视图，UI 按引用动态渲染。
M3 引入版本策略与影响分析。

---

## **11. 对齐机制（M3）**

> M1/M2 阶段不做对齐。用户直接填写表名+字段，系统原样保存。

---

## **12. 冲突与融合机制（M3）**

> M1/M2 阶段不做冲突融合。M3 阶段引入可信来源优先级 + 冲突分级。

## **13. 真实案例验证**

### **13.1 零售客户信息查询**

- **验证点**：~20 个场景 + 5000 字业务概述 + 部分场景缺 SQL
- **适配**：业务概述 → domain_overview；缺 SQL 场景 → sql_variants 为空数组

### **13.2 代发明细查询**

- **验证点**：同一业务横跨 2003-至今的 8+ 张表
- **适配**：8 张表 → 8 个取数方案，各标注 applicable_period

### **13.3 零售客户信息变更**

- **验证点**：码值散在注释中 + 按日期分段查询 + 多步骤链路
- **适配**：码值 → code_mappings JSON；分段查询 → 取数方案；多步骤 → 场景描述说明

## **14. 风险与应对**

| 风险 | 应对 | 阶段 |
|------|------|------|
| LLM 抽取不稳定 | 手动录入是主路径，LLM 只是加速器 | M1 |
| 文档结构极度不统一 | 场景表单灵活（除标题外均可选填） | M1 |
| 历史表分裂 | 取数方案机制 | M1 |
| 业务背景知识无处放 | 域级 domain_overview 字段 | M1 |
| 码值重复维护 | M1 允许冗余，M2 抽为独立字典 | M1→M2 |

## **15. 成功度量**

| 指标 | M1 目标 | M2 目标 |
|------|--------|--------|
| 已录入场景数 | ≥ 50 | ≥ 200 |
| 覆盖业务域数 | ≥ 3（TOP3 高频域） | ≥ 10 |
| 新人找到口径的平均时间 | 从"问人 30 分钟"降到"搜索 3 分钟" | < 1 分钟 |
| 口径提供人录入一个场景耗时 | 手动 < 15 分钟，LLM 辅助 < 5 分钟 | 同 M1 |
