# 数据直通车 · 项目背景与定位

> **目标**：明确项目起点、核心痛点与阶段目标，为“口径治理”与“知识梳理服务”提供统一业务背景。  
>
> **范围**：覆盖数据直通车 1.0 现状、2.0 远期蓝图、当前核心痛点与本阶段切入点；不展开接口与实现细节。  
>
> **读者**：业务负责人、数据支持组、产品经理、架构与开发团队。  
>
> **当前阶段定位**：口径治理建设期的定位文档，用于统一项目北极星与边界。  
>
> **术语说明**：术语口径以 [统一术语表](08-数据直通车-统一术语表.md) 为准。

## 一、我们是谁

我们是银行大数据部门的公共支持组，主要职责是支持分行业务人员的临时取数需求。分行用户通过内部工单发起取数申请，经过层层审批后到达我们这里，我们写 SQL（结构化查询语言）从数据湖（高斯数据库，GaussDB）中查询并交付数据。

过去十年，我们处理了数万个取数需求，积累了大量可复用经验，包括：

- 大量高频重复的取数场景（例如交易流水查询、客户信息查询等）
- 大量 SQL 示例和处理规则
- **数十万字的数据口径文档**（说明业务定义、字段含义、使用哪些表、SQL 示例、注意事项等）

我们已有一个系统**"数据直通车"**上线，具备模板取数、模板开发、日志记录、系统管理，并已集成行内 OA（办公自动化）审批流程。

## 二、数据直通车的演进

### 2.1 数据直通车 1.0 · 模板取数

我们在处理了几千次数据查询需求后发现一个关键事实：**70% 的需求是重复的**。比如交易流水查询，很多业务同事都要查，用 SQL 实现也很固定，每次查询的区别只是查询时间和客户信息不同。

于是我们将高频需求抽象为**查询模板**：

1. 一线业务同事点击查询模板
2. 选择自己需要的字段，填写查询日期和客户信息
3. 补充申请信息（申请事由、说明附件、选择审批人），提交申请
4. **系统后台**：接收申请 → 将查询参数替换模板中的变量 → 生成可执行 SQL → 发送到数据湖查询 → 查询结果上传到工单
5. 业务人员下载文件获取数据

**1.0 的核心瓶颈：**

| 编号 | 瓶颈 | 说明 |
|------|------|------|
| 1 | **供给依赖人力** | 查询模板由总行开发同事手工上线，需要配置源表和数据口径，这些信息会变化，需不断维护 |
| 2 | **场景无法穷尽** | 总行开发同事不在一线，无法穷尽所有查询场景 |
| 3 | **模板太僵硬** | 模板不能灵活变化，复杂需求仍需人工写 SQL |

### 2.2 数据直通车 2.0 · 智能化数据服务（远期蓝图）

**核心思路**：不再依赖人力上线模板，而是通过系统自动生成 SQL，让系统越用越智能。

我们在过往处理的需求中，发现可以按照不同类型划分——零售、对公、运营、核心系统、公共信息、财富管理等，每个类型下还可以继续细分。这些有体系的类型共同组成了我们的**数据地图**（内部曾称"业务树"），每个节点包含详细的业务说明。

两个关键能力：

- **数据地图**：将积累的业务知识结构化为可浏览、可搜索的主题节点树（Topic Node），并在节点下组织业务场景与语义资产
- **自然语言查询**：首页搜索框支持自然语言输入，基于业务知识自动生成 SQL，是数据地图的补充

> **2.0 是一个远期蓝图，实现它的前提是解决最根本的问题——口径治理。**

## 三、核心痛点分析

我们发现系统的发展瓶颈不在"取数执行"或"审批"，而在 **知识与模板的供给侧**。

### 3.1 口径文档：信息丰富但结构不统一

目前口径文档按近百个业务分类组织，每个分类下有多个细分场景文档，但文档格式偏自由式——SQL 代码块 + 中文解释 + 列表 + 超链接/历史备注混杂。

**存量口径文档形态非常多**：

| 类型 | 示例 |
|------|------|
| 纯 SQL 脚本 | 多段 SQL、CTE、多个 SELECT，甚至一份脚本包含多个"场景" |
| 规则说明/流程说明 | 没有 SQL，但有筛选条件、口径定义、注意事项 |
| Excel/截图式文档 | 表格 + 文字说明 + 局部 SQL |
| 混合型 | 背景描述 + SQL + 口径校验要求 + 合规提醒 |

**口径文档存在的六大问题**：

| 编号 | 问题 | 说明 |
|------|------|------|
| 1 | **找不到** | 散落在邮件、网盘、聊天群里 |
| 2 | **不可信** | 没人知道是否验证过、何时验证 |
| 3 | **不可复用** | 关键条件/输入参数不清晰 |
| 4 | **不可治理** | 表血缘/敏感字段/脱敏规则缺失 |
| 5 | **不可追溯** | 版本、变更摘要、审计链路不完整 |
| 6 | **无法自动化** | 不能稳定导入成系统资产，无法导出为 RAG/知识库素材 |

### 3.2 业务理解门槛高

口径文档不只是"给机器看的"，人也看不懂：

- **人看困难**：新员工需要逐行读 SQL 才能理解，定位"口径、输入、输出、关键表、坑点"效率低
- **业务数据难理解**：产出的 Excel 数据并不直观——比如埋点日志，一线业务同事根本看不懂，每次都需要业务专家解释。我们需要把这些解释积累下来，后期通过 AI 来学习业务说明，产出人人都能读懂的报告

### 3.3 五大核心痛点（从使用者视角）

| 编号 | 痛点 | 典型场景 |
|------|------|---------|
| 1 | **语义断层** | 看不懂交易码、状态码、JSON 字段含义（如手机银行日志里的交易码、验证方式码） |
| 2 | **路径迷失** | 复杂查询需多表多条件组合，新手不知从何入手（如收单查询路径依赖专家经验） |
| 3 | **身份迷宫** | 跨系统唯一键不一致，需链式映射（证件号→客户号→一网通编号→日志） |
| 4 | **共享语义维护成本高** | 一处修改需同步改多份文档，导致新旧不一致 |
| 5 | **需求边界不清** | 用户不知道系统能/不能解决什么问题，导致反复试错、沟通成本高 |

### 3.4 模板供给瓶颈

我们希望数据直通车未来能开放给业务同事参与模板开发，但当前模板开发流程由支持组主导，无法覆盖银行内部复杂业务口径。这限制了系统扩展能力。

**结论：系统瓶颈在供给侧，口径治理是一切的基础。不解决口径文档的结构化问题，数据地图无法建立，自然语言查询也缺少可靠的知识来源。**

## 四、从口径治理到数据地图

口径治理与数据地图不是两个独立系统，而是同一件事的两个阶段：

```
┌──────────────────────────────────────────────────────┐
│  第一阶段：口径治理（数据结构化）                      │
│  将散落的口径文档 → 结构化语义资产                     │
│  （可编辑、可校验、可发布、可追溯）                     │
├──────────────────────────────────────────────────────┤
│  第二阶段：数据地图（知识服务化）                      │
│  结构化资产 → 可浏览、可搜索、可导航的知识图谱          │
│  （供查询引擎、RAG、模板生成消费）                     │
└──────────────────────────────────────────────────────┘
```

**知识梳理服务** 是口径治理阶段中的一个大模型应用场景——利用 AI 能力自动解析、抽取、结构化口径文档，减少人力成本，提高系统自主能力。

### 4.4 NL → Plan(IR) → SQL 路径

基于总体设计方案（2026-02-26），我们确立了金融场景下的可控取数路径：

```
自然语言问题
    │
    ▼
Plan(IR)  ← 可控取数计划（白盒、可审计）
    │
    ▼
SQL（模板填参/受控生成）
    │
    ▼
执行侧
```

**核心观点**：金融场景下直接让大模型端到端生成 SQL 难以满足可控与审计要求。Plan(IR) 作为中间态，输出选中的 Scene/Plan、参数填充、风险门禁与证据解释，而不是直接 SQL。

### 4.1 口径知识天然是"图谱型"的

我们的口径知识不是孤立页面，本质上是**大量关系**：

- 场景 → 使用 → 表
- 表 → 包含 → 字段
- 字段 → 定义 → 业务语义/口径
- 场景 → 输出 → 字段白名单
- 场景 → 依赖 → SQL 段
- SQL 段 → 引用 → 表/字段（血缘/依赖）
- 版本 → 替换/废弃 → 旧版本
- 字段/口径 → 具有 → 敏感等级/脱敏规则
- 文档版本 → 由谁在何时验证 → 审计记录

这些关系一旦结构化存下来，就已经拥有了"图谱的节点与边"。

### 4.2 分阶段落地路线

| 阶段 | 定位 | 核心目标 |
|------|------|---------|
| **第一阶段(M1)** | 可生成图谱 | 不强调图查询能力，但数据模型按图思路设计；解决结构化录入 + 自动抽取 + 审计版本 + JSON 输出 |
| **第二阶段(M2)** | 用图谱能力反哺系统 | 影响分析（表变更影响哪些场景）、一致性治理、高频场景运营、RAG 增强 |
| **第三阶段(M3)** | 智能化/自动化 | 元数据对齐 + Service Spec导出 + Plan(IR) + Graph-RAG |
| **远期** | NL2SQL平台 | 自然语言查询 + SQL生成 + 安全执行 |

### 4.3 六资产模型（核心治理资产）

基于总体设计方案，统一采用六资产模型：

| 资产类型 | 英文术语 | 说明 |
|---------|---------|------|
| 业务场景 | Scene | 最小治理单元，承载口径定义、输入输出、Plan、风险提示与审计信息 |
| 语义视图 | Semantic View | 解释一类数据记录如何理解（粒度、关键字段、常用 join/过滤） |
| 字典 | Dictionary | 码值/枚举/术语解释映射，支撑共享语义与消歧 |
| 派生规则 | Derivation Rule | 身份/参数映射链（如证件号→客户号→协议号） |
| 主题节点 | Topic Node | 导航与聚合，组织场景与语义资产 |
| 服务说明 | Service Spec | Scene 发布态对外契约（输入/输出/路由/风险/错误模型） |

> **重要说明**：本项目采用"六资产模型"作为核心治理资产体系，详见[统一术语表](08-数据直通车-统一术语表.md)。

### 4.5 概念模型对齐（以知识梳理服务方案为准）

本项目统一采用六资产概念模型：

- 数据语义视图（Semantic View）
- 业务场景（Scene）
- 字典（Dictionary）
- 派生规则（Derivation Rule）
- 主题节点（Fact / Topic Node）
- 服务说明（Service Spec）

旧表达与新模型的对应关系如下：

| 旧表达 | 当前统一口径 |
|------|-------------|
| 数据主题 / 业务树节点 | 主题节点（Topic Node），用于导航与聚合 |
| 模板定义 | 业务场景（Scene）在执行侧的消费形态 |
| 对外接口说明 | 服务说明（Service Spec），即场景发布态 |
| 字段解释散落在场景中 | 语义视图 + 字典（可被场景引用） |

### 4.6 Plan 与 Plan(IR)

| 概念 | 说明 |
|------|------|
| **Plan（取数方案）** | Scene 内的 SQL 变体集合，用于表达历史表分裂与路由（applicable_period + source_tables） |
| **Plan(IR)（中间态计划）** | 运行时中间态（白盒），输出选中的 Scene/Plan、参数填充、风险门禁与证据解释；不是直接 SQL |

> **重要说明**：Plan(IR) 是 M3 阶段的核心产出，作为自然语言到 SQL 的可控中间态。

## 五、整体架构定位

```
  ┌─────────────────────────────────────┐
  │   业务用户层（数据分析师）            │
  └─────────────────────────────────────┘
                ↓ 自然语言查询 / 模板取数
  ┌─────────────────────────────────────┐
  │   数据直通车（查询系统）              │
  │   - 自然语言转SQL引擎                │
  │   - 智能审批                         │
  │   - 执行引擎                         │
  └─────────────────────────────────────┘
                ↓ 读取知识
  ┌─────────────────────────────────────┐
  │   数据地图（知识层）                  │
  │   - 主题节点（导航）                 │
  │   - 业务场景（最小治理单元）          │
  │   - 语义视图 / 字典 / 派生规则        │
  │   - 服务说明（发布态）               │
  │   ← 口径治理完成后，自然成为数据地图   │
  └─────────────────────────────────────┘
                ↑ 生产知识（含AI辅助）
  ┌─────────────────────────────────────┐
  │   口径治理（知识治理）                │
  │   - 知识梳理服务（大模型场景）        │
  │   - 场景编辑 / 版本管理 / 质量校验    │
  └─────────────────────────────────────┘
                ↑ 维护知识
  ┌─────────────────────────────────────┐
  │   业务专家层（总行业务人员）          │
  └─────────────────────────────────────┘
```
### 系统边界

| ✅ 做 | ❌ 不做 |
|------|--------|
| 沉淀语义资产（口径、解释、规则） | 执行引擎（不跑 SQL） |
| 版本管理与审计追溯 | 权限裁决（由外部平台控制） |
| 影响分析与发布管理 | 实时数据处理 |
| 输出服务说明与配置 | 手工 YAML 维护（使用表单代替） |

### 关键约束与设计原则

| 编号 | 原则 | 说明 |
|------|------|------|
| 1 | **最小治理单元：业务场景** | 版本、审计、发布、导出的原子对象是场景（不是业务领域） |
| 2 | **三层结构** | 业务领域 → 业务场景 → SQL 逻辑块 |
| 3 | **强门禁** | SQL 抽取的表名必须可见，且必须能去元数据平台匹配 |
| 4 | **合规最小集** | 敏感字段必须可标注，且必须填脱敏规则才能发布 |
| 5 | **可信度机制** | 每次发布必须填写验证日期 + 变更摘要，形成可追溯链路 |
| 6 | **导入不完美但不丢信息** | 无法识别的信息保留原始文本，以便人工补齐 |
| 7 | **人机双输出** | 同时满足人类可读（页面渲染）和机器可用（标准 JSON） |
| 8 | **Plan(IR) 白盒机制** | 可控取数计划作为自然语言到 SQL 的中间态，保证可审计可回滚 |

### 为什么最小治理单元选"业务场景"

- **验证可操作**：场景级验证可落地——拿一组样例参数跑 SQL、对输出做复核；领域级验证成本太高，容易形式化
- **版本粒度合适**：改一个场景只影响这一个场景的版本，审计清晰；如果用领域做单元，每次改一个场景都要发整个领域新版本
- **RAG 检索稳定**：场景级文档边界明确，不会出现"召回到领域的某个片段，但混着别的场景的 SQL"
- **模板天然母体**：场景天然有输入/输出/SQL/注意事项，可直接演进成模板定义

> 业务领域（通过主题节点组织）是场景的**组织与导航层**——承载共性背景、术语、常用表，不承担最小门禁与复核压力。
