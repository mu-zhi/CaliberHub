# 数据直通车 · MVP 定义与交付计划

> **目标**：定义 6-8 周可交付的最小可用系统，验证"口径文档 → 结构化资产 → 可浏览数据地图"的端到端可行性。
>
> **范围**：覆盖 M1（口径治理 MVP）目标、边界、用户故事、数据模型、技术架构与交付节奏；不包含执行侧 SQL 运行、审批流实现与 NL2SQL 引擎。
>
> **读者**：开发团队、产品负责人、业务专家。
>
> **当前阶段定位**：口径治理 MVP，是数据直通车 2.0 远期蓝图的第一块基石。

---

## 一、MVP 的一句话定义

> **能导入一份口径文档，看到结构化结果，手动修正后保存为可浏览的业务场景。**

不做 NL2SQL，不做查询执行，不做审批流程；权限能力仅保留配置开关，M1 默认不启用细粒度鉴权。只做"知识从散落到结构化"这一件事。

---

## 二、MVP 边界（做什么 / 不做什么）

### ✅ MVP 范围内

| 能力 | 说明 | 验证目标 |
|------|------|---------|
| **手动创建场景** | 空白表单逐字段填写，主路径 | 验证不依赖 LLM 也能结构化录入 |
| **LLM 辅助导入** | 粘贴口径文档 → LLM 抽取 → 输出 CALIBER_IMPORT_V2 JSON | 验证行内 LLM 的抽取质量 |
| **导入对照视图** | 左侧原文、右侧结构化结果，支持高亮映射 | 验证用户能否快速确认/修正抽取结果 |
| **场景编辑器** | 编辑标题、描述、取数方案、码值说明、注意事项 | 验证编辑体验是否可用 |
| **取数方案管理** | 同一场景支持多段 SQL（按时间/条件区分） | 验证"历史表分裂"场景可表达 |
| **业务域管理** | 域名称 + 业务概述（富文本）+ 常见数据表 | 验证大段业务背景知识有处安放 |
| **场景列表与详情** | 按业务域分组浏览已保存的场景 | 验证结构化资产的可浏览性 |
| **场景状态管理** | DRAFT → PUBLISHED 两个状态，一人操作 | 最小可用的发布流程 |
| **全文搜索** | 按场景标题、描述、SQL 内容搜索 | 验证"找得到"这个基本需求 |

### ❌ MVP 范围外（后续迭代）

| 能力 | 原因 | 归属阶段 |
|------|------|---------|
| NL2SQL / 自然语言查询 | 远期蓝图，依赖大量结构化知识积累 | 远期 |
| Plan(IR) 引擎 | 需要 M2 的引用关系图谱基础 | M3 |
| Graph-RAG | 需要 M2 的语义资产积累 | M3 |
| 查询执行引擎 | 1.0 已有，MVP 不重建 | 第三阶段及以后 |
| 审批流程 | 小团队初期不需要多人审批 | 第二阶段 |
| 语义视图（独立实体） | MVP 中场景自身包含字段解释即可 | 第二阶段 |
| 字典（独立实体） | MVP 中码值说明嵌入场景 JSON（code_mappings） | 第二阶段 |
| 派生规则（独立实体） | MVP 中仅文字描述派生关系 | 第三阶段 |
| 版本管理 / 版本对比 | MVP 只需"当前版本"，不做版本链 | 第二阶段 |
| 引用策略（锁定/兼容/最新） | MVP 中无独立资产，不需要引用策略 | 第二阶段 |
| 影响分析 | 依赖引用关系图谱，MVP 无此基础 | 第三阶段 |
| 多人审核（语义/技术/合规） | 小团队一人操作即可 | 第二阶段 |
| 对齐机制（元数据平台对接） | 依赖行内元数据平台 API，MVP 先跳过 | 第二阶段 |
| 服务说明导出 | 依赖成熟的场景资产 | 第二阶段 |
| 前端主题树 / 数据地图导航 | MVP 用扁平列表 + 搜索替代 | 第二阶段 |

## 三、核心用户故事

### US1：手动创建一个业务场景

> 作为**数据支持组成员（口径提供人）**，我想手动创建一个业务场景并逐字段填写，把我脑中的口径经验录入系统。

**验收标准**：
1. 提供"手动创建"入口，生成空白场景表单
2. 可以填写场景标题、描述、注意事项
3. 可以添加多个取数方案（方案名、适用时段、SQL 原文、来源表、备注）
4. 可以添加码值说明（code_mappings JSON）
5. 可以选择业务域归属（下拉选择）
6. 保存后状态为 DRAFT

### US2：通过 LLM 辅助导入口径文档

> 作为**数据支持组成员**，我想把一份口径文档粘贴到系统中，让系统自动抽取出结构化信息，加速录入过程。

**验收标准**：
1. 粘贴 Markdown/纯文本/SQL 文件格式的口径文档到输入框
2. 系统调用 LLM 返回 CALIBER_IMPORT_V2 JSON（含取数方案）
3. 页面展示对照视图：左侧原文，右侧结构化字段
4. 多场景文档能正确拆分为多个 scene
5. 同一场景的多段 SQL 能识别为不同方案（按时间/条件区分）
6. SQL 原文保留，不被改写
7. 用户可以在对照视图中直接编辑修正后保存为草稿

### US3：发布场景

> 作为**数据支持组成员**，我想把确认无误的草稿发布，让其他同事可以浏览和搜索到。

**验收标准**：
1. 点击"发布"按钮，状态从 DRAFT 变为 PUBLISHED
2. 发布时填写验证日期和变更摘要（表单必填）
3. 发布后场景出现在场景列表中

### US4：浏览和搜索场景

> 作为**数据支持组成员或一线业务人员**，我想浏览所有已发布的场景，或按关键词搜索，快速找到我需要的口径。

**验收标准**：
1. 场景列表按业务域分组展示
2. 支持按标题、描述、SQL 内容全文搜索
3. 点击场景进入详情页，展示完整的结构化信息
4. 详情页中 SQL 代码高亮显示

### US5：管理业务域信息

> 作为**数据支持组成员**，我想维护业务域的基础信息（名称、业务概述、常见数据表、负责人），为场景提供分组和背景知识。

**验收标准**：
1. 可以创建新的业务域（名称、排序）
2. 可以编辑域级业务概述（富文本，承载大段业务背景知识）
3. 可以编辑常见数据表清单（文本）
4. 可以编辑负责人信息
5. 场景列表按业务域分组展示时，可展开查看域级业务概述

## 四、数据模型（MVP 最小集）

MVP 只需要两张核心表：

### 4.1 场景表（caliber_scene）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | BIGINT | 是 | 主键 |
| scene_code | VARCHAR(64) | 是 | 场景唯一编码（业务标识） |
| scene_title | VARCHAR(200) | 是 | 场景标题 |
| domain_id | BIGINT | 否 | 关联业务域 ID |
| status | ENUM | 是 | DRAFT / PUBLISHED |
| scene_description | TEXT | 否 | 场景描述 |
| caliber_definition | TEXT | 否 | 口径定义 |
| applicability | TEXT | 否 | 适用范围 |
| boundaries | TEXT | 否 | 边界说明 |
| inputs_json | JSON | 否 | 输入参数（限制字段描述） |
| outputs_json | JSON | 否 | 输出字段（结果字段描述） |
| sql_variants_json | JSON | 否 | 取数方案数组（替代原 sql_blocks） |
| code_mappings_json | JSON | 否 | 码值说明（从 SQL 注释中提取的码值映射） |
| caveats_json | JSON | 否 | 注意事项 |
| contributors | VARCHAR(500) | 否 | 口径提供人（权威来源标注） |
| unmapped_text | TEXT | 否 | 未映射原文 |
| quality_json | JSON | 否 | 质量评估（置信度、警告、错误） |
| raw_input | MEDIUMTEXT | 否 | 导入时的原始文档（用于对照视图） |
| verified_at | DATETIME | 否 | 验证日期 |
| change_summary | TEXT | 否 | 变更摘要 |
| created_by | VARCHAR(64) | 是 | 创建人 |
| created_at | DATETIME | 是 | 创建时间 |
| updated_at | DATETIME | 是 | 更新时间 |

**sql_variants_json 结构**（替代原 sql_blocks_json）：

```json
[
  {
    "variant_name": "当前主表查询",
    "applicable_period": "2014至今",
    "description": "使用 PDM 表",
    "sql_text": "SELECT ... FROM PDM_VHIS.T05_AGN_DTL ...",
    "source_tables": ["PDM_VHIS.T05_AGN_DTL"],
    "notes": "限制字段：AGN_BCH_SEQ, EAC_NBR"
  }
]
```

**code_mappings_json 结构**（码值说明，从 SQL 注释中提取）：

```json
[
  {
    "code": "CARD_GRD_CD",
    "description": "卡等级代码",
    "mappings": { "020": "金卡", "040": "金葵花卡", "060": "钻石卡" }
  }
]
```

### 4.2 业务域表（caliber_domain）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | BIGINT | 是 | 主键 |
| domain_code | VARCHAR(64) | 是 | 域编码 |
| domain_name | VARCHAR(100) | 是 | 域名称（如"零售客户信息查询"） |
| domain_overview | MEDIUMTEXT | 否 | 业务概述（富文本，承载大段业务背景知识） |
| common_tables | TEXT | 否 | 常见数据表清单 |
| contacts | VARCHAR(500) | 否 | 域负责人/联系人 |
| sort_order | INT | 否 | 排序 |
| created_at | DATETIME | 是 | 创建时间 |
| updated_at | DATETIME | 是 | 更新时间 |

> **设计决策**：
> 1. **取数方案替代 SQL 块**：同一场景可能横跨多张历史表（如代发明细 8+ 张表），`sql_variants` 通过 `applicable_period` 和 `source_tables` 字段支持"按时间选表"。只有一段 SQL 的场景就只有一个方案。
> 2. **码值说明嵌入场景 JSON**：当前码值散在 SQL 注释中，量级不大（< 50 个），不值得独立为字典实体。第二阶段场景量大时再升级为独立字典表。
> 3. **域级业务概述**：真实口径文档开头常有 3000-5000 字的业务背景知识（如借记卡/存折/财富账户定义），不属于任何单一场景，由域表 `domain_overview` 承载。
> 4. **JSON 列简化**：MVP 阶段将取数方案、码值等存为 JSON 列，避免过早拆分十几张关联表。当资产量增长到需要跨场景查询字段级信息时，再拆分为独立表。
> 5. **长文本与附件存储策略**：口径文本类大字段采用分表存储（主表保留高频检索字段，扩展表承载长文本）以控制主表规模并保障查询性能；仅当内容包含图片、PDF（便携式文档格式）附件时，才使用对象存储，数据库仅保存对象引用与元数据。

## 五、技术架构（MVP 最小技术栈）

```
┌──────────────────────────────────────────┐
│  前端（单页应用）                          │
│  场景列表 · 对照视图 · 场景编辑器 · 搜索    │
└──────────────────────────────────────────┘
              ↕ RESTful API
┌──────────────────────────────────────────┐
│  后端（Spring Boot 单体应用）              │
│  ┌────────────┐  ┌────────────────────┐  │
│  │ 场景管理    │  │ LLM 预处理服务     │  │
│  │ CRUD+发布   │  │ 调用行内LLM API   │  │
│  └────────────┘  └────────────────────┘  │
└──────────────────────────────────────────┘
              ↕
┌──────────────────────────────────────────┐
│  MySQL（元数据存储）                       │
│  caliber_scene · caliber_domain           │
└──────────────────────────────────────────┘
```

### 5.1 技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| 后端 | Java Spring Boot（单体） | 团队熟悉，银行标准技术栈 |
| 前端 | 待定（Vue 或 React） | 按团队技术栈选择 |
| 数据库 | MySQL | 银行标配，MVP 数据量极小 |
| LLM 调用 | 行内 LLM API（HTTP 调用） | 需先确认可用性 |
| 搜索 | MySQL LIKE / FULLTEXT | MVP 阶段够用，数据量 < 1000 条 |

### 5.2 不引入的技术组件

| 组件 | 为什么不引入 |
|------|------------|
| Elasticsearch | 数据量 < 1000，MySQL 全文搜索够用 |
| Redis | 无高并发场景 |
| Kafka / RabbitMQ | 无异步任务 |
| 向量数据库 | 无语义检索需求 |
| Neo4j | 无图查询需求 |
| K8s | MVP 可直接部署到 VM |
| 微服务拆分 | 单体应用完全满足 |

### 5.3 代码分层（简化 DDD）

```
src/main/java/com/bank/caliber
├── controller/          # REST 接口
│   ├── SceneController  # 场景 CRUD + 发布
│   └── ImportController # LLM 预处理 + 导入
├── service/             # 业务逻辑
│   ├── SceneService     # 场景管理
│   └── LlmService       # LLM 调用封装
├── repository/          # 数据访问
│   ├── SceneRepository
│   └── DomainRepository
├── model/               # 数据模型
│   ├── entity/          # 数据库实体
│   └── dto/             # 请求/响应对象
└── config/              # 配置
```

> **设计决策**：MVP 不采用完整的 DDD 四层架构（Adapter/Application/Domain/Infrastructure），而是用经典三层（Controller/Service/Repository）。原因是 MVP 业务逻辑简单（CRUD + LLM 调用），四层架构在此阶段引入不必要的复杂性。当系统演进到 第二阶段（引用关系、版本管理）时，再重构为 DDD 分层。

## 六、API 接口设计（MVP 最小集）

### 6.1 LLM 辅助导入

```
POST /api/import/preprocess
Request:  { "rawText": "原始口径文档全文", "sourceType": "PASTE_MD" }
Response: { "success": true, "data": { CALIBER_IMPORT_V2 JSON（含 sql_variants） } }
```

### 6.2 场景 CRUD

```
POST   /api/scenes                    # 创建场景（从 JSON 或手动空白）
GET    /api/scenes                    # 场景列表（支持 ?domainId=&status=&keyword= 过滤）
GET    /api/scenes/{id}               # 场景详情（含 sql_variants、code_mappings）
PUT    /api/scenes/{id}               # 更新场景
POST   /api/scenes/{id}/publish       # 发布场景
DELETE /api/scenes/{id}               # 删除草稿
```

### 6.3 业务域

```
GET    /api/domains                   # 业务域列表
POST   /api/domains                   # 创建业务域
GET    /api/domains/{id}              # 域详情（含 domain_overview）
PUT    /api/domains/{id}              # 更新业务域
```

## 九、LLM 定位与降级策略

MVP 的核心价值不依赖 LLM。手动录入是主路径，LLM 是加速器。

| 路径 | 使用场景 | 说明 |
|------|---------|------|
| **主路径：手动录入** | 口径提供人直接填写 | 不需要 LLM，系统 100% 可用 |
| **辅助路径：LLM 导入** | 从已有大段 SQL 文件批量导入 | LLM 抽取 → 人工修正 → 保存 |
| **LLM 降级** | LLM 不可用或质量差 | 回退到手动录入，无任何功能损失 |

> **关键决策**：即使 LLM 完全不可用，MVP 仍然有价值——"结构化表单 + 取数方案编辑 + 保存 + 浏览 + 搜索"本身就解决了"口径文档找不到"的问题。LLM 只是加速器，不是前提条件。交付计划中，LLM 集成放在 Week 5（而非 Week 1-2），确保核心流程不阻塞于 LLM 可用性。
