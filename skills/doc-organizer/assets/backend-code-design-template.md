# 代码设计文档模板

**适用场景**：模块后端实现

## 编写指南

- **核心聚焦**：业务规则、状态流转、事务边界、并发控制
- 明确数据校验、事务边界
- 不得包含数据库表结构/字段类型/索引清单（引用系统架构的数据库设计）
- 不得包含完整测试用例矩阵与测试报告，归档到"测试与质量"
- 不得包含运维排障手册（初期不强制）
- 错误处理通常是全局统一的，写在每个模块文档中是重复。模块特有的错误处理极少，一般无需编写
- 不要重复叙述 API 列表，而是引用 API 规范文档
- 文档不需要撰写目录结构，其是实现细节，随代码演进会过时，代码本身就是最好的目录结构文档
- 日志设计不单独设立章节，如有模块特有的日志需求，融入"安全设计"章节
- 后端实现文档应聚焦架构设计，不写技术栈特定细节，各语言实现差异应体现在代码中

**章节选择指引**：
| 章节 | 简单 CRUD | 复杂模块 |
|------|----------|---------|
| 1. 文档概述 | ✅ 必写 | ✅ 必写 |
| 2. 架构设计 | ⚪ 可省 | ✅ 展开 |
| 3. 数据传输对象 | ⚪ 可省 | ⚪ 按需 |
| 4. 核心业务逻辑 | ✅ 必写 | ✅ 必写 |
| 5. 数据访问与依赖 | ✅ 必写 | ✅ 必写 |
| 6. 外部服务集成 | ⚪ 不写 | ✅ 按需 |
| 7. 缓存设计 | ⚪ 按需 | ✅ 展开 |
| 8. 安全设计 | ✅ 必写 | ✅ 必写 |
| 9. 异步任务设计 | ⚪ 不写 | ✅ 按需 |
| 10. 性能目标 | ⚪ 不写 | ✅ 按需 |
| 11. 配置项 | ⚪ 按需 | ✅ 必写 |
