# 02 核心概念与设计理念

## 1. 文档说明

- 目标：解释系统中的关键概念和设计取舍。
- 范围：核心资产、工作模式、设计原则。
- 读者：全体角色。
- 当前阶段定位：指导日常使用和评审讨论。

---

## 2. 核心概念

1. 业务场景：正式语义组织单元，描述“解决什么问题、主对象是什么、默认时间怎么解释、标准输出有哪些”。
2. 方案资产：同一业务场景下可并行维护多个 `Plan（方案资产）`，用于表达不同时间段、不同入口条件和不同历史路径。
3. 业务领域：场景的管理边界，承载域级概述、常见表、联系人。
4. 码值说明：字段码值及其业务含义，当前嵌入场景内维护。
5. 数据地图：对治理成果的业务化展示，包含主题节点、场景及其引用关系。

---

## 3. 设计理念

1. 手动优先，`LLM（大语言模型，Large Language Model）` 加速：人工录入和修订是主路径，`LLM（大语言模型，Large Language Model）` 用于提效。
2. 导入不完美，但不丢信息：无法识别的内容保留到“LLM 未识别内容”。
3. 引用优先，不做复制：同一知识尽量一处维护、多处引用。
4. 人机双输出：页面可读，接口可机读。
5. 中文语境优先：界面文案、流程说明优先使用中文业务表达。

---

## 4. 为什么要有方案资产

在实际场景中，经常出现历史表与现行表并存、不同时段口径差异等情况。
如果只有单段 `SQL（结构化查询语言，Structured Query Language）`，系统无法准确表达这些差异。

`Plan（方案资产）` 用于解决三类问题：

1. 时段差异：不同年份使用不同表结构。
2. 口径差异：同一场景对不同客群有不同筛选条件。
3. 迁移过渡：新旧模型并行期间需要双轨表达。
