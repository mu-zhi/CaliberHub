## 2. 这些资料在项目里的角色

这批文档在项目里承担 3 个角色：

1. 顶层方法参考：帮助判断当前方案是否符合企业级 `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` / `NL2SQL（自然语言转 SQL，Natural Language to SQL）` 的共性做法。
2. 设计校准参考：帮助我们判断哪些能力应该在知识图谱上游完成，哪些能力应留给下游 `NL2SQL`。
3. 评审对照参考：后续组内评审或方案质询时，可以直接回到官方资料，而不是只回到项目内部文档。

## 3. 当前最值得直接吸收的共性结论

这 5 份资料合起来，当前最稳定的共识有 6 条：

1. 企业级 `NL2SQL（自然语言转 SQL，Natural Language to SQL）` 不能直接面向全库生成，必须先缩到正确领域或正确场景。
2. `GraphRAG（图检索增强生成，Graph Retrieval-Augmented Generation）` 不是单一检索器，而是图、文本、规则、示例共同参与检索。
3. 企业问题不是只有一种检索模式，至少要区分局部问题、全局问题和逐步扩展问题。
4. 知识图谱不只来自文档，还来自结构化、半结构化、非结构化多类数据源。
5. 图的价值不只是“召回更多信息”，还在于路径解释、证据追溯和结果可审计。
6. 历史 `SQL（结构化查询语言，Structured Query Language）` 适合作为示例和证据，不适合作为真相本体。

## 4. 对数据直通车的直接启发

落到数据直通车，最关键的不是“照搬官方产品形态”，而是吸收这几条做法：

1. 先做 `Domain（业务领域）` / `Scene（业务场景）` 收敛，再做下游生成。
2. 运行时优先返回问题相关业务子图，而不是直接把整批表结构扔给模型。
3. 把证据、版本、表间关联关系、派生规则放到知识底座，而不是只放表字段元数据。
4. 把外部最佳实践当作方法依据，把口径文档现状当作本地知识来源和验证样例来源。

## 5. 使用方式

1. 如果要解释“为什么我们的方案不是大模型直接写 `SQL（结构化查询语言，Structured Query Language）`”，先看 `09`。
2. 如果要解释“为什么业务子图不是单一检索结果”，先看 `10` 和 `11`。
3. 如果要解释“知识加工层为什么必须接多源数据”，先看 `12`。
4. 如果要解释“官方方法为什么还强调调优和成本”，再看 `13`。
